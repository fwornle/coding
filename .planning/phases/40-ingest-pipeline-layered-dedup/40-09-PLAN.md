<objective>
Close **CR-02** (REVIEW.md offset 83-109, VERIFICATION.md gap #2): the `if (candidate.id === entity.id) continue;` self-identity guard in all 3 matchers is **dead code on the happy path** (the pipeline calls dedup BEFORE putEntity, so input entity.id is freshly-minted; collision is astronomical) AND **actively wrong on the legacy-id re-extraction path** (when an extractor re-emits an existing stored entity at its same id, the guard skips the perfect match and the pipeline silently writes a duplicate).

The fix is a one-line deletion in each of three files:

- `~/Agentic/km-core/src/dedup/JaccardNameMatcher.ts:53` — `if (candidate.id === entity.id) continue;`
- `~/Agentic/km-core/src/dedup/CosineEmbeddingMatcher.ts:123` — `if (candidates[i].id === entity.id) continue;`
- `~/Agentic/km-core/src/dedup/LLMSemanticMatcher.ts:132` — `.filter((c) => c.id !== entity.id)` clause inside the `existingNames` construction

Each deletion is accompanied by a header-comment paragraph explaining the rationale (CR-02 from 40-REVIEW.md). Per the REVIEW.md "Fix" recipe at offset 96-108, the correct treatment is: "Self-write protection is the store's job, not the matcher's" — by construction of D-46 (active-only candidate pool), an exact id collision IS the same logical entity, which IS what dedup is meant to catch.

This plan ships 3 new tests (one per matcher) asserting the legacy-id re-extraction case is now matched correctly. The 23 pre-existing matcher unit tests must keep passing — none of them rely on the guard's behavior (the existing test fixtures all use fresh entities with distinct ids, so the guard never fired).

Cross-repo workflow: code edits land in `~/Agentic/km-core/` with commits prefixed `fix(40-09): ...` / `test(40-09): ...`. SUMMARY.md lands in coding/ worktree with `docs(40-09): ...`.
