## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user click → store mutation | Row click triggers atomic store write with id originating from server-fetched entity list |
| selectedNodeId → DOM selector | scrollIntoView query uses CSS.escape (already in place) to defuse selector-injection if id contains a `]` |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-56-02-01 | Tampering | HistorySidebar row id flows into setState | mitigate | id originates from km-core API list (already trusted); set wrapped in `useViewerStore.setState({...})` so the surface is identical to existing handlers |
| T-56-02-02 | Information Disclosure | data-history-id renders entity id into DOM | accept | Entity ids are already rendered into aria-labels, click handlers, and the existing scroll selector logic — no new disclosure |
| T-56-02-03 | Injection | querySelector(`[data-history-id="${CSS.escape(id)}"]`) | mitigate | CSS.escape is ALREADY in place at HistorySidebar.tsx:107; this plan preserves it |
| T-56-02-04 | DoS | scrollIntoView fires on every selectedNodeId change | accept | Constant-time DOM op; throttling unnecessary for typical entity counts (≤200) |
| T-56-02-05 | XSS via highlight class injection | className built from store data | accept | Class strings are literals from this file; not user-supplied; no template-injection vector |

All threats LOW. The data-history-id fix narrows a latent dead-code window but does not open any new surface.
</threat_model>
