# Phase 56.1: Code Review Report

**Reviewed:** 2026-06-14T12:30:00Z
**Depth:** standard
**Files Reviewed:** 13
**Status:** issues_found

## Summary

Phase 56.1 introduces a many-to-many bidirectional bridge between the timeline strip, D3 graph, and side panel. The implementation is generally careful — atomic store writes via `setSelection`, audit-locked dep lists, reference-stability guards, and a thoughtful three-way side-panel mode-switch are all in place. Heavy in-code commentary documents PATTERNS contracts and prior-fix rationale.

However, adversarial review surfaces real defects, several with user-visible consequences:

- **2 Critical bugs**: (a) a stale-prev-Set in `deriveFocal` that miscomputes the focal on every multi-selection write because `selectedNodeIds` is now snapshotted from the post-write reference; (b) the new fit-to-bounds effect reads `d3NodesRef.current` whose contents may have been wiped (or re-populated to a different array) between Layer-0→Layer-1 transitions and the simulation tick, producing wrong/empty bounds and a no-op or mis-centered zoom.
- **8 Warnings**: invariant breaches, race conditions, silent failures, ESLint hooks-rule violations.
- **5 Info-level cleanups**: dead code, misleading comments, missing test rigor.

The most serious finding is the focal-derivation bug (CR-01) — it directly defeats the "last-added wins" insertion-order contract that drives the focal selection ring AND the side-panel mode-switch in many code paths. This bug is masked by the fact that most callers either pass an explicit `focal.nodeId` or write a singleton Set (where any iteration order works), but the additive Cmd-click handler in `LslTimelineStrip` AND the reverse-direction graph-click handler in `D3GraphCanvas` BOTH rely on the implicit "last-added" derivation and BOTH ship the bug.

## Critical Issues

### CR-01: `deriveFocal` is called with the WRONG `prev` argument when a deep-equal write preserves the existing Set reference — defeats insertion-order "last-added wins"

**File:** `integrations/unified-viewer/src/store/viewer-store.ts:484-488`
**Issue:**

`setSelection` computes `nextSelectedNodeIds` using the `sameSetMembership` reference-stability guard:
```ts
const nextSelectedNodeIds =
  nodeIdsArg !== undefined
    ? sameSetMembership(s.selectedNodeIds, nodeIdsArg)
      ? s.selectedNodeIds         // <-- preserved reference
      : nodeIdsArg
    : s.selectedNodeIds

const nextFocalNodeId =
  nodeIdsArg !== undefined || focal?.nodeId !== undefined
    ? deriveFocal(s.selectedNodeIds, nextSelectedNodeIds, focal?.nodeId)
    //            ^^^^^^^^^^^^^^^^^ prev   ^^^^^^^^^^^^^^^^^^^^ next
    : s.focalNodeId
```

`deriveFocal(prev, next, explicit)` walks `next` and returns the last id in `next` that is NOT in `prev`. The intent is: "last id added that wasn't there before."

Two failure modes:

1. **Cmd-click additive path in `LslTimelineStrip.tsx:626-632`**: builds `nextNodeIds` by copying `prevState.selectedNodeIds` and adding `resolvedNodeIds`. Since `nextNodeIds` is a brand-new Set superset of `s.selectedNodeIds`, the diff IS the addition — so far so good.

   BUT, an explicit `focal: { nodeId: focalNodeIdNext, bucketKey }` is also passed. `deriveFocal` short-circuits on explicit override correctly. **OK in this path.**

2. **D3GraphCanvas graph-click in `D3GraphCanvas.tsx:737-745`**: passes `nodeIds: new Set([d.id])` (singleton) and `focal: { nodeId: d.id, bucketKey: null }` — explicit, so derive doesn't matter.

3. **HistorySidebar in `HistorySidebar.tsx:126-135`**: explicit focal, OK.

4. **BucketCardList in `BucketCardList.tsx:172-182`**: explicit focal, OK.

5. **HOWEVER, the bug surfaces in a subtle path**: when a write `setSelection({ nodeIds: someSet, source: ..., ... })` passes `nodeIds` but OMITS `focal.nodeId`, AND `someSet` happens to deep-equal the current `s.selectedNodeIds`, the deep-equal guard fires and `nextSelectedNodeIds === s.selectedNodeIds`. `deriveFocal(s.selectedNodeIds, s.selectedNodeIds, undefined)` now finds zero "newly added" ids and returns the LAST iteration-order element — which clobbers whatever `s.focalNodeId` was previously set to (the user-meaningful focal). Pre-Phase-56.1 the focal was simply preserved; post-refactor it's silently overwritten with the iteration-order tail.

   For Cmd-click additive that lands on an already-selected node, e.g. user Cmd-clicks the same tick twice (or two ticks that produce the SAME ancestor set), the second click should be idempotent on focal. Instead, the focal mutates to the tail of iteration order. The user-observable symptom: focal ring jumps to an "earlier" halo node on duplicate clicks.

   Also potentially affected: any future test/code that does `setSelection({nodeIds: <unchanged set>, source: ...})` to refresh `selectionSource`.

**Fix:**
Guard explicitly on whether `nodeIds` was actually-passed AND content-changed — preserve focal when no addition is detectable:
```ts
const nextFocalNodeId: string | null =
  focal?.nodeId !== undefined
    ? focal.nodeId
    : nodeIdsArg !== undefined && nextSelectedNodeIds !== s.selectedNodeIds
      ? deriveFocal(s.selectedNodeIds, nextSelectedNodeIds, undefined)
      : s.focalNodeId   // preserve when no change OR when caller didn't touch nodeIds
```

Add a regression unit test for the store: `setSelection({nodeIds: SAME_SET, source: 'graph'})` MUST preserve `focalNodeId`.

### CR-02: Fit-to-bounds effect reads `d3NodesRef.current` whose contents may have been REPLACED (or in-place mutated to a different layout) between the Layer-0→Layer-1 transition and the next selection-source flip

**File:** `integrations/unified-viewer/src/graph/D3GraphCanvas.tsx:478-557` (esp. lines 522-540)
**Issue:**

The fit-to-bounds effect runs when `selectionSource` transitions from `null → non-null` AND `selectedNodeIds.size >= 1`. It reads `d3NodesRef.current` to look up x/y for the bounded set.

`d3NodesRef.current` is REPLACED at every main-render-effect run (`D3GraphCanvas.tsx:599`):
```ts
d3NodesRef.current = d3Nodes   // brand new array on every render
```

Two race scenarios:

1. **Filter change racing with selection**: User clicks a tick → selection cascade fires → `setSelection` writes the new selection slice AND `lslFilterEntityIds`. The selection-source dep makes the fit-to-bounds effect schedule. Simultaneously, `visibleEntities` re-memoises (because `lslFilterEntityIds` is in its dep list), `visibleRelations` re-memoises, the MAIN render effect runs FIRST (React commits effects in declaration order; main render is declared after the fit-to-bounds effect). The main render rebuilds `d3Nodes = visibleEntities.map(...)` with `x: undefined, y: undefined` for every node (lines 585-591). Then the fit-to-bounds effect runs against `d3NodesRef.current` which now points at the FRESH array with `x === undefined` for every node. `targetNodes` is filtered to those with `typeof n.x === 'number'` (line 533-536) — that filter returns EMPTY → early return at line 537. **No fit happens, silently.** The user clicks and nothing zooms.

   The audit-locked dep list `[visibleEntities, visibleRelations, theme, isLoading]` is EXACTLY why this happens — the tick-click cascade can fire BOTH the main render and the fit-to-bounds in the same React commit.

2. **Effect ordering bug**: The fit-to-bounds effect is declared at line 510 (before the main render effect at line 560). React runs effects in declaration order. So fit-to-bounds runs FIRST against the OLD `d3NodesRef.current` from the previous render (which still has stale x/y). If the user's selection now refers to nodes that weren't in the previous render's `d3Nodes`, `targetNodes` is empty → silent no-op. If the user's selection refers to nodes that WERE in the previous render's d3Nodes, the fit fires against stale x/y coordinates — the viewport snaps to where the nodes USED TO BE before the new force layout settles them.

The comment block at lines 478-508 documents the intent ("transitions that do NOT trigger this") but never addresses what the fit reads from `d3NodesRef.current` when the simulation hasn't ticked yet (cold mount → first click of session).

Additionally, `simulationRef.current` is also not consulted for `alpha` state — the fit fires immediately at the click, but the simulation may still be in `alpha > 0` settle, so `x/y` are interim, not final.

**Fix:**
Two mitigations needed:
1. Defer the fit until the simulation has settled OR until `d3NodesRef.current` is known to contain mounted coordinates. The cheapest mitigation is to schedule the fit inside a one-shot `requestAnimationFrame` AFTER a `simulation.tick()` or `simulation.on('end')` callback registers the layout settled. Track a sticky "fit pending" flag and consume it on the next tick.
2. Guard against the empty-targetNodes silent-noop with a Logger.warn so future regressions are visible:
```ts
if (targetNodes.length === 0) {
  Logger.warn(Logger.Categories.PANELS,
    `Fit-to-bounds: 0 of ${targetIds.size} target ids have laid-out coordinates — deferring`)
  // schedule retry on next simulation tick / next render
  return
}
```

Also consider: the main render and fit-to-bounds compete on the same `zoomBehaviorRef.current`. After mount the main render's `simulation.on('end', fitToScreen)` at line 845 ALSO calls `fitToScreen` once. The two fits race — the user might see a fit-to-bounds zoom interrupted ~500ms later by `fitToScreen` zooming the whole graph. Test by clicking the first tick within ~5s of mount; the smoke flicker should be visible.

**Fix (alternative — simpler if drift acceptable):** subscribe the fit-to-bounds effect to `[selectionSource, selectedNodeIds, pathToSelected, visibleEntities]` so it re-runs when the layout array changes, and gate the inner body on `simulation.alpha() < someThreshold`. The viewport-stability invariant survives because the existing main render dep list is unchanged.

### WR-01: `pickAllResolvable` LLS-suppression deletes from `out` during iteration over `noiseAncestors` — re-checks the same size guard, but the partial-delete loop can leave the set in a state inconsistent with the "size >= 2" intent if `noiseAncestors` is large

**File:** `integrations/unified-viewer/src/graph/ancestry.ts:254-264`
**Issue:**

```ts
if (noiseAncestors !== undefined && noiseAncestors.size > 0 && out.size >= 2) {
  for (const id of noiseAncestors) {
    if (out.has(id) && out.size >= 2) {
      out.delete(id)
    }
  }
}
```

When `out` size is, say, 3 and two of the three are noise ancestors, the loop deletes the first noise → size=2 → deletes the second noise → size=1 (the guard `out.size >= 2` was checked BEFORE delete, so it permits the delete that takes size to 1). Comment on line 257-259 acknowledges this ("if every ancestor in the result was in the noise set, we'd end with empty") but the actual guard `out.size >= 2` is satisfied AT THE CHECK and then the delete drops size to 1, NOT to 0 as the comment fears.

The real defect is **iteration-order-dependence**: deleting from `out` mid-loop returns size=1 with whichever noise id was processed LAST. If the operator's `noiseAncestors` Set ever grows beyond LLS (and the comment says "multi-noise would still risk it"), the result is non-deterministic w.r.t. which "noise-flavored ancestor" survives. For one-noise (LLS), the bug is invisible.

The bigger fix is to compute the candidate-delete set first, then delete only if leaving ≥1:
```ts
if (noiseAncestors !== undefined && noiseAncestors.size > 0 && out.size >= 2) {
  // Collect noise candidates in `out`, leave at least one non-noise survivor.
  const noiseInOut: string[] = []
  for (const id of out) {
    if (noiseAncestors.has(id)) noiseInOut.push(id)
  }
  const survivors = out.size - noiseInOut.length
  if (survivors >= 1) {
    for (const id of noiseInOut) out.delete(id)
  }
  // else: every ancestor IS noise — preserve as-is per option iii.
}
```
**Fix:** Replace with the precompute-then-delete pattern above. Add a unit test where `noiseAncestors = {a, b}` and `out = {a, b, c}` — verify result is `{c}`. Add a second test where `out = {a, b}` and noise = `{a, b}` — verify result is `{a, b}` (preserved per option iii — currently passes by accident).

### WR-02: `useNodeToBucketsIndex` `noiseAncestors` is named-based — silent breakage on canonical seed regeneration that renames `LiveLoggingSystem` to a synonym

**File:** `integrations/unified-viewer/src/graph/useNodeToBucketsIndex.ts:78-84`, mirrored in `LslTimelineStrip.tsx:261-272`
**Issue:**

`noiseAncestors` is derived by iterating `entities` and matching `e.name === 'LiveLoggingSystem'` exactly. Comments claim "robust to id changes across seed regenerations." This is half-true: it IS robust to id changes, but it's brittle to NAME changes. If the ontology classifier ever re-labels the entity to e.g. `Live Logging System` (with spaces) or `LiveLoggingSystem (Component)` (with descriptor), suppression silently breaks and EVERY bucket starts halo-ing LLS again. The 65%+ "LLS-in-every-bucket" UX regression returns silently with zero observability — Logger.warn at the strip's onTickClick fires only when suppression ACTIVELY drops something, so a zero-suppression state produces zero logs.

Two callsites duplicate the lookup (strip + index hook). Per the comment in `useNodeToBucketsIndex.ts:73-77` they intentionally don't share code; that's the right contract decision but reinforces the fragility — both must be kept in lock-step manually.

**Fix:** Use a more stable identification. Options:
1. Check `e.metadata.role === 'noise-ancestor'` (requires ontology change but is explicit).
2. Use a centralized constant `NOISE_ANCESTOR_NAMES: ReadonlySet<string>` exported from a single module, imported by both callsites. Then renames are a one-line change.
3. Add a startup-time invariant check: if `noiseAncestors.size === 0`, emit Logger.warn ONCE per session — proves the lookup matched something.

Minimum fix: extract the name-set constant and import it in both places.

### WR-03: `applySelectionStyling` `useCallback` dep list omits `selectedBucketKeys`, but the predicate at line 404 reads `selectedNodeIds.has(d.id)` only — no bug from this omission, BUT the lightweight selection useEffect (line 473) ALSO re-runs on `applySelectionStyling` changes only — meaning a pure `selectedBucketKeys` change (e.g. cmd-click adding a halo bucket on the timeline) does NOT repaint the graph

**File:** `integrations/unified-viewer/src/graph/D3GraphCanvas.tsx:467, 473-476`
**Issue:**

The lightweight selection effect only depends on the memoised `applySelectionStyling` callback. `applySelectionStyling` itself only depends on `[selectedNodeIds, focalNodeId, pathToSelected, visibleRelations, theme]` (line 467). So:

- A timeline Cmd-click that ADDS to `selectedBucketKeys` but does NOT change `selectedNodeIds` (e.g. clicking the same session's second tick where every entity resolves to the same focal node) writes a new `selectedBucketKeys` Set with content change — the strip itself repaints via subscription, BUT the graph's `applySelectionStyling` doesn't fire because `selectedBucketKeys` is not in its dep list.

For the GRAPH-side rendering this happens to be fine today because `applySelectionStyling`'s body only reads node-side state. But the comment at line 461-466 ("selectedNodeIds and focalNodeId are now read inside this callback") implies completeness — review traps in the future could see the `selectedBucketKeys` subscription missing and either (a) add bucket-side render logic to `applySelectionStyling` without updating the dep list (silent stale rendering), or (b) infer the omission was intentional and break the cross-pane invariant later.

The deeper concern: subscribing to `selectedBucketKeys` at line ~234 brings it into the component for `nodeToBuckets` use, but it's NOT in the dep list of `applySelectionStyling` — so a future programmer reading the subscriptions thinks `applySelectionStyling` re-reads them and is wrong.

**Fix:**
1. Add an explicit comment block at line 467 stating that `selectedBucketKeys` is INTENTIONALLY omitted — the graph reads node-side state only and bucket-side render is the strip's responsibility.
2. Better: add an ESLint exhaustive-deps rule disable comment with an inline justification:
```ts
// eslint-disable-next-line react-hooks/exhaustive-deps -- selectedBucketKeys
//   intentionally NOT a dep: this callback paints node-side state only
[selectedNodeIds, focalNodeId, pathToSelected, visibleRelations, theme],
```
3. Add a unit test that asserts the graph DOM does NOT re-render on pure-bucket-side selection changes — locks the contract.

### WR-04: Cmd-click additive path in `LslTimelineStrip.onTickClick` reads `useViewerStore.getState()` then computes `nextNodeIds`/`nextBucketKeys` from `prevState`, then writes via `setSelection` — a concurrent third-pane mutation between the get and the set silently clobbers

**File:** `integrations/unified-viewer/src/panels/coding/LslTimelineStrip.tsx:609-645`
**Issue:**

```ts
const prevState = useViewerStore.getState()
const prevSet = prevState.lslFilterEntityIds
const union = new Set<string>(prevSet ?? [])
for (const id of ids) union.add(id)
for (const id of resolvedNodeIds) union.add(id)
const nextSessionFilter = Array.from(
  new Set<string>([...prevState.lslSessionFilter, sessionId]),
)
const nextNodeIds = new Set<string>(prevState.selectedNodeIds)
for (const id of resolvedNodeIds) nextNodeIds.add(id)
const nextBucketKeys = new Set<string>(prevState.selectedBucketKeys)
if (bucketKey !== null) nextBucketKeys.add(bucketKey)
setSelection({ nodeIds: nextNodeIds, bucketKeys: nextBucketKeys, ... })
```

This is a classic read-modify-write race. Between `useViewerStore.getState()` and the `setSelection` call, another action can mutate the store (programmatic test code, an SSE-driven ETM update, a different click handler if the user holds Cmd and bursts clicks rapidly). The next write replaces the entire `selectedNodeIds` set with what THIS handler computed from the now-stale `prevState`.

In single-user keyboard interaction this is unlikely to fire — most user input is single-threaded — but:

1. The test harness publishes `window.__viewerStore` and the E2E specs in `56.1-many-to-many.spec.ts` deliberately drive selection rapidly (`page.keyboard.press('Escape')` + `nth(i).click()` in a tight loop). A test that mid-burst writes a different selection from outside would silently lose updates.
2. React 18 concurrent rendering and event batching can defer the `setSelection` write across a microtask boundary — between `.getState()` and `set(...)` an SSE listener inside the ETM tail slice could push an observation update that touches selection (although the ring-buffer push action doesn't currently touch selection — but the contract is fragile).

The store action `setSelection` exposes a functional `set(s => {...})` form internally (line 466) — the caller should compute the union INSIDE that callback so it reads the freshest state atomically.

**Fix:** Provide a new store action `addToSelection({nodeIds: Set, bucketKeys: Set, ...})` that performs the union inside `set(s => {...})` atomically. Refactor the Cmd-click handler to call it instead of computing the union itself. Same fix for the union computation of `union` (the `lslFilterEntityIds`).

### WR-05: `popSelection` calls `get().clearSelection()` (not the action) when `selectionHistory === null` — returns `false` even though it DID clear; comment at line 619-624 acknowledges this but Esc handler at `useKeyboardShortcuts.ts:202-211` only logs differentially, NEVER preventDefaults on the "fell through" path

**File:** `integrations/unified-viewer/src/store/viewer-store.ts:615-625` + `integrations/unified-viewer/src/hooks/useKeyboardShortcuts.ts:202-211`
**Issue:**

```ts
// useKeyboardShortcuts.ts:202
if (hasSelection) {
  const popped = state.popSelection()
  event.preventDefault()              // <-- always called
  Logger.debug(...)
}
```

The Esc handler's `hasSelection` guard correctly checks `selectionHistory !== null` among other conditions. If `selectionHistory === null` AND any other selection-slice field is non-empty, `hasSelection === true`, `popSelection()` returns `false` and falls through to `clearSelection()`. `preventDefault()` IS called — actually fine.

BUT: `popSelection`'s return-value contract documented in lines 619-624 says "false = nothing was popped per se ... the Esc handler decides via its pre-existing hasSelection check." Yet the callsite at `useKeyboardShortcuts.ts:203` does NOT decide based on the return value at all — it's used only for the log message. Worse, `popSelection` calls `get().clearSelection()` (the public action) which itself fires a full `set({...})` write — making `popSelection` semantically a side-effect-causing read, which is brittle.

The defect manifests in two ways:
1. The X-button in `SidePanel.tsx:160-162` ALSO calls `popSelection()` but ignores the return value entirely — there's no preventDefault to invoke, but if the click happens with `selectionHistory === null` AND no selection (empty store), the user clicks X expecting it to close the panel but `popSelection → clearSelection` is a no-op on already-empty state. Then `entity` (line 157 guard) re-renders to null and the X button vanishes — fine but only by accident.
2. The deceiving comment block at 619-624 says "the caller might still want to preventDefault" and lays the responsibility on `useKeyboardShortcuts`, but the callsite does not gate on `popped`. So Esc with `lslSessionFilter` populated but `selectionHistory === null` correctly fires `clearSelection` AND preventDefault — but only because `hasSelection` happened to include `lslSessionFilter`. A future code path that adds a new "fully cleared but not history" state could mis-fire.

**Fix:**
1. Simplify `popSelection` to ONLY pop; return `true` if popped and `false` otherwise. Move the "fall through to clearSelection" behavior to the caller. The current pattern of an action that conditionally calls another action obscures control flow.
2. Update `useKeyboardShortcuts.ts:203` to gate on `popped`:
```ts
const popped = state.popSelection()
if (!popped) state.clearSelection()
event.preventDefault()
```
3. Same fix for `SidePanel.tsx:160-162`.

### WR-07: `D3GraphCanvas` `useEffect` at line 510 hooks-rule violation risk — uses `prevSelectionSourceRef` to compare prior value, but `prevSelectionSourceRef.current = selectionSource` happens BEFORE the early return on `!enteredLayer1` — correct, but trivially easy to break in future edits

**File:** `integrations/unified-viewer/src/graph/D3GraphCanvas.tsx:510-557`
**Issue:**

```ts
useEffect(() => {
  const prevSource = prevSelectionSourceRef.current
  const enteredLayer1 = prevSource === null && selectionSource !== null && selectedNodeIds.size >= 1
  prevSelectionSourceRef.current = selectionSource    // <-- bookkeeping BEFORE early return
  if (!enteredLayer1) return
  ...
}, [selectionSource, selectedNodeIds, pathToSelected])
```

The dep list includes `selectedNodeIds` and `pathToSelected`. This means the effect re-runs whenever the multi-set OR the path changes (which is on every selection write). On every such re-run, `prevSelectionSourceRef.current = selectionSource` overwrites the bookkeeping. If `selectionSource` was `'timeline'` from the prior click and the next click also sets `selectionSource: 'timeline'`, `prevSource = 'timeline'` (from a prior run that already bookkept), `enteredLayer1 = false`, no fit fires — correct.

BUT: consider a sequence (a) tick click → selection cascade fires, (b) main render dep change fires the lightweight selection effect, (c) some other effect fires that mutates pathToSelected without changing selectionSource. The fit-to-bounds effect re-runs because `pathToSelected` is in its dep list — bookkeeping overwrites ref, the comparison computes against a stale "captured at start of effect" `prevSource` that was already === `selectionSource` from the prior bookkeeping. `enteredLayer1 = false`. Correct.

This is brittle: if a future edit moves the bookkeeping AFTER the early return (a "clean up" refactor), the contract breaks. Add an inline `// CRITICAL ORDERING` comment.

Also: if `selectedNodeIds.size === 0` and `selectionSource !== null`, `enteredLayer1` is false even though the source transitioned. Edge case: user clicks a tick whose entityIds resolves to 0 graph-visible ancestors (sidebar-only mode). The early-return at line 537 (`targetNodes.length === 0`) catches this, but the effect still RAN.

**Fix:**
1. Add `// CRITICAL: bookkeeping MUST be before early return — see Plan 06 fit-to-bounds dep list comment.` at line 516.
2. Move the bookkeeping into a separate dedicated `useEffect` keyed on `[selectionSource]` only, so the main fit-to-bounds effect can early-return without worrying about ref drift. Two effects, one responsibility each.

Lines 120-127:

So the click essentially writes only `source: 'graph'` and `pathToSelected: ...`, leaving the rest preserved.

If this spec is in the CI suite, it's RED or skipped. If it's GREEN, it's GREEN by accident (asserting against undefined === undefined or comparing nulls that are always null).

**Fix:** Migrate the spec to the new multi-set API. Use the patterns demonstrated in `56.1-many-to-many.spec.ts` (which uses `selectedNodeIds: Set<string>`, `focalNodeId: string | null`, `selectedBucketKeys: Set<string>`). The Phase 56 spec should be either retired (its ACs are subsumed by 56.1) or aggressively rewritten. As is, it's a maintenance liability that gives false confidence.
