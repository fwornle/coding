# Phase 56.1 Plan 03: D3GraphCanvas Two-Tier Halo + Drill-Collapse Summary

**Refactored `D3GraphCanvas.tsx` to consume the Plan 01 many-to-many selection store (`selectedNodeIds: ReadonlySet<string>` + `focalNodeId: string | null`) and render the Phase 56.1 D-4 two-tier ring contract (red focal + lighter-blue halo with data-focal/data-halo observability markers), with D-5 drill-collapse click semantics that always shrink any prior halo to single-focal mode on graph click. The audit-locked Phase 56-PATTERNS Locked Contract #3 viewport-stability invariant (main render-effect dep list = `[visibleEntities, visibleRelations, theme, isLoading]`) carries forward verbatim.**

## Accomplishments

### (a) Two-tier render mechanism

The Phase 56 single-ring `applySelectionStyling` block (`.filter((d) => d.id === selectedNodeId).select('.selection-ring').attr('opacity', 1)`) is replaced with a per-node `.each(function (d) { ... })` callback that paints THREE tiers using ONE `.selection-ring` circle per node (initialised once at mount with opacity 0):

| Predicate | Stroke | Opacity | Stroke-width | data-focal | data-halo |
|---|---|---|---|---|---|
| `d.id === focalNodeId` | `#ff0000` | `1` | `4` | `"true"` | `null` |
| `selectedNodeIds.has(d.id) && d.id !== focalNodeId` | `#60a5fa` | `0.6` | `2` | `null` | `"true"` |
| else | (preserved) | `0` | (preserved) | `null` | `null` |

`data-focal` / `data-halo` are written on the parent `.node <g>` so Plan 06's E2E spec can select by attribute without RGB sniffing. The two markers are mutually exclusive: every branch clears the sibling attr in lockstep, so a node never carries both at once.

The no-selection branch fires when BOTH `focalNodeId === null && selectedNodeIds.size === 0`. Per Plan 01's `deriveFocal` invariant these are equivalent (focal === null iff set is empty), but the explicit two-condition predicate documents the contract intent and survives any future regression of `deriveFocal`.

### (b) Drill-collapse click payload

The node `.on('click', ...)` handler writes through the canonical `setSelection` action (NOT inline `setState`) with the D-5 drill-collapse payload:

```typescript
useViewerStore.getState().setSelection({
  nodeIds:    new Set<string>([d.id]),     // collapse any prior halo to single
  bucketKeys: new Set<string>(),           // clear timeline halo
  focal:      { nodeId: d.id, bucketKey: null },
  pathToSelected: new Set<string>(path.nodeDepths.keys()),
  highlightedRowKey: d.id,
  source: 'graph',
})
```

`event.stopPropagation()` is preserved — without it the SVG background click handler fires next and immediately clears the selection we just wrote.

Background SVG click already routes through `useViewerStore.getState().clearSelection()` (Phase 56 CR-02 fix carried forward). Plan 01 extended `clearSelection()` to cover all four new selection fields plus the LSL filter slice plus `pathToSelected` — no per-pane handler logic needed here.

### (c) Audit-locked dep-list invariant preserved

The main render-effect dep list at the tail of the file is unchanged verbatim:

```typescript
}, [visibleEntities, visibleRelations, theme, isLoading])
```

This is Phase 56-PATTERNS Locked Contract #3 — listing ANY selection field there rebuilds the entire SVG + restarts the force simulation on every click, regressing the viewport-stability invariant the audit ended in Phase 56 R3. The comment block above the dep list is updated to enumerate every Phase 56.1 selection field name (`selectedNodeIds`, `focalNodeId`, `selectedBucketKeys`, `focalBucketKey`) plus the carried-forward `pathToSelected` as the canonical "MUST NOT appear here" list.

The renamed `applySelectionStyling` useCallback dep list is `[selectedNodeIds, focalNodeId, pathToSelected, visibleRelations, theme]` (Phase 56's `[selectedNodeId, pathToSelected, visibleRelations, theme]` with the rename + `focalNodeId` added).

## Decisions Made

See `key-decisions` in frontmatter. Highlights:

- **Centralised per-node tier decision in `applySelectionStyling`.** Mount-time `.selection-ring` opacity defaults to 0; the tail-call `applySelectionStyling(svg)` paints the correct tier from the current store snapshot. Phase 56's mount-time `selectedNodeId === d.id ? 1 : 0` predicate was a second source of truth for the visual decision; removing it eliminates drift risk between mount and steady-state.
- **Explicit two-condition no-selection guard.** `focalNodeId === null && selectedNodeIds.size === 0` is belt-and-braces given Plan 01's `deriveFocal` invariant, but it documents the contract intent for any future reader and survives a deriveFocal regression.
- **Parallel observability markers.** `data-focal` / `data-halo` attrs are written in lockstep with the inline stroke attrs inside the same `.each` callback. Plan 06's E2E selectors can match by data-attr without RGB sniffing; the two markers are mutually exclusive so a single node never carries both.
