# Phase 56.1: Unified Viewer — Many-to-Many Temporal-Knowledge Bridge

<domain>
Replace Phase 56's one-to-one selection sync with a many-to-many temporal-knowledge bridge across the unified viewer's three coding-tab panes (D3 graph, LSL timeline strip, history sidebar):

- A timeline tick lights up ALL graph nodes it touched (multi-ring halo) + ALL entities in the sidebar (History-style card list).
- A graph node lights up ALL ticks where that node was touched, across all sessions visible in the current window (multi-highlight on timeline).
- Drilling into a halo node OR a bucket card collapses the halo to single-focal mode (Phase 56 behavior).

This is an evolutionary refactor of the Phase 56 store contract: `selectedNodeId: string | null` is promoted to `selectedNodeIds: Set<string>`, and the focal entity is derived from the set ("most recently added"). Several Phase 56 PATTERNS contracts need re-statement (not breaking — they still hold; the predicate's input shape changes).
</domain>

<decisions>

### D-1 — Data model: PROMOTE single-selection to multi-selection (foundational)

`selectedNodeId: string | null` is **removed** from `useViewerStore` state. Replaced by:

```ts
selectedNodeIds: Set<string>             // empty Set = no selection
focalNodeId: string | null               // derived: "most recently added to selectedNodeIds"
                                          //   OR null when set is empty
selectedBucketKeys: Set<string>          // multi-selected timeline buckets (reverse of selectedNodeIds)
focalBucketKey: string | null            // derived: most recently added bucket
```

`focalNodeId` / `focalBucketKey` are derived inside the store (via a setter that tracks insertion order), NOT stored as separate fields users can write. This guarantees the focal is always consistent with the set membership.

**Touched-elsewhere implications:**
- `setSelection` action signature extended: accepts `nodeIds: Set<string>` (or array) instead of `nodeId: string | null`; the action computes focal as last-added; existing single-element callers pass `new Set([id])`.
- `clearSelection()` clears both selectedNodeIds and selectedBucketKeys to empty Sets.
- `pathToSelected` is now computed from `focalNodeId` only (not every member of selectedNodeIds — visual noise).
- Every reader that previously did `state.selectedNodeId === node.id` becomes `state.selectedNodeIds.has(node.id)` (for halo) OR `state.focalNodeId === node.id` (for focal/red-ring/sidebar-mount).

**This is a deliberate evolution of Phase 56 contract #5.** The audit acceptance grep (`useViewerStore.setState` count outside the store = 0) still applies; the field name in PATTERNS contract #2 and #6 must be re-stated to `selectedNodeIds` + `focalNodeId`.

### D-2 — Touched predicate: reuse pickFirstResolvable (consistent with Phase 56)

A bucket B "touches" node X iff at least one entity in `B.entityIds` resolves (via the existing `pickFirstResolvable(B.entityIds, visibleIds, relations)` from `graph/ancestry.ts`) to X. Same predicate runs in both directions.

**Forward direction (tick click → graph multi-ring):**
For the clicked bucket, the set of touched nodes is `{ pickFirstResolvable(B.entityIds, ...) }` — typically 1–3 nodes per bucket. Note: `pickFirstResolvable` currently returns the FIRST resolvable, but the new contract requires the FULL set. Extend `pickFirstResolvable` (or add a sibling `pickAllResolvable`) that returns `Set<string>` instead of `string | null`.

**Reverse direction (node click → timeline multi-highlight):**
For the clicked node X, look up `nodeToBuckets[X.id]` in the pre-index (see D-3) — returns the touching bucket-key set in O(1).

### D-3 — Reverse-lookup mechanism: client-side pre-indexed Map (no backend)

Build a `nodeToBuckets: Map<string, Set<string>>` reverse index once per data-load:

```ts
// pseudocode in a new src/graph/useNodeToBucketsIndex.ts hook
for (const bucket of allBuckets) {
  const bucketKey = `${bucket.id}|${bucket.startAt}`
  const touched = pickAllResolvable(bucket.entityIds, visibleIds, relations)
  for (const nodeId of touched) {
    const existing = nodeToBuckets.get(nodeId) ?? new Set()
    existing.add(bucketKey)
    nodeToBuckets.set(nodeId, existing)
  }
}
```

**Rebuild triggers:**
- Sessions load (initial fetch OR refetch)
- `visibleIds` changes (filters toggled — class, level, layer, learning source)
- `relations` changes (graph refetch)

**Memory budget:** ~1 entry per (visible node × average touching buckets). For typical session counts (~50 buckets/day, ~3 nodes touched per bucket, ~50 visible nodes), this is well under 1MB.

**No backend work needed.** No new server endpoint. No HTTP round-trip per click.

### D-4 — Visual treatment: two-tier (halo + focal), trace from focal only

**Graph side:**
- Halo nodes (in `selectedNodeIds` but not focal): `ring-2 ring-blue-300/60` (lighter blue, 60% opacity)
- Focal node (`focalNodeId === node.id`): `ring-2 ring-red-500` (matches Phase 56 single-selection red ring)
- Non-trace nodes (not in `selectedNodeIds` and not in `pathToSelected`): faded as today
- `pathToSelected` trace drawn from `focalNodeId` ONLY, not from every halo node (would be too noisy at N=5+)

**Timeline side:**
- Halo buckets (in `selectedBucketKeys` but not focal): light blue fill tint (`bg-blue-200/40`) — distinct from existing pink/rose bucket color
- Focal bucket (`focalBucketKey === bucketKey`): existing `ring-blue-500` from Phase 56
- 0-obs ticks: grey-out + pointer-events-none (Phase 56 contract #4, preserved)

**Sidebar:**
- Multi-selection mode (`selectedNodeIds.size > 1` OR `selectedBucketKeys.size > 0`): renders a **History-style card list** of entities from the active set (mode driven by `selectionSource` — `'timeline'` → cards for bucket's entities; `'graph'` → cards for touching buckets' first entities or similar). Each card is clickable.
- Single-focal mode (after drilldown OR `selectedNodeIds.size === 1`): existing `EntityDetailPanel` for `focalNodeId`.

### D-5 — Drilldown behavior: clicking a card or halo node collapses to single-focal mode

When the user clicks (a) a card in the sidebar list, or (b) a halo node in the graph:
- `selectedNodeIds` collapses to `new Set([clicked id])`
- `selectedBucketKeys` collapses to empty (clears the timeline halo)
- `focalNodeId` becomes the clicked id
- `selectionSource` becomes `'graph'` (graph click) or `'history'` (card click)
- Sidebar switches to `EntityDetailPanel` for the focal

Rationale: keeping the halo persistent across drills would compound visual noise. The operator's "we get to the corresponding insight and its content" reads as a transition — drilling SHOULD reset the multi-set. This is captured as Claude's discretion in `<discretion>` for the user's review.

1. **Drill collapses the halo** (D-5). The operator's "we get to the corresponding insight" reads as a transition, but they didn't explicitly say "and the halo disappears."
