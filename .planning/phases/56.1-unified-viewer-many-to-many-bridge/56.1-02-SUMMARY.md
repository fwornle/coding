## What Was Built

### 1. `pickAllResolvable(entityIds, visibleIds, relations): Set<string>` (NEW export)

Drop-in multi-resolution sibling of the existing `pickFirstResolvable`. Mirrors the predicate (`resolveToVisibleAncestor` over each id), but accumulates every non-null resolution into a `Set<string>` instead of stopping at the first. Empty `Set` is returned when no id has any visible ancestor (the sidebar-only mode the timeline strip uses to render a bucket selection without a graph target).

Properties verified by tests:
- Returns the union of resolved ancestor ids across `entityIds[]`
- Dedup-safe: two raw ids that resolve to the same ancestor collapse to one Set entry
- Order-independent: the returned Set membership is identical for any permutation of `entityIds`
- Empty input or all-orphan input produces empty Set (no `null` return path)

Consumers (per CONTEXT.md D-2 + D-3):
- **Plan 05 forward:** `LslTimelineStrip.onTickClick` writes `selectedNodeIds` halo via `pickAllResolvable(bucket.entityIds, visibleIds, relations)`.
- **Plan 04 reverse:** `useNodeToBucketsIndex` pre-builds the `nodeId → Set<bucketKey>` reverse map by calling `pickAllResolvable` per bucket and inverting.

PATTERNS contract #7 (pre-index integrity) locks both call sites as the only legal non-test callers — any other on-demand call from inside a click handler is a smell (acceptance grep documented in 56.1-PATTERNS.md §3).

### 2. WR-02 fix in `computeAncestryPath` — diamond/multi-parent BFS

**Before:** `parentOf: Map<string, string>` (first-parent-wins) plus a linear walk along the first-parent chain. Secondary parent edges were silently hidden from `edges`; secondary ancestors were silently hidden from `nodeDepths`. `resolveToVisibleAncestor` returned `null` when only the secondary parent led to a visible ancestor — the WR-02 root cause carried over from Phase 56 round 4 review.

**After:** Single BFS pass records every parent edge in both directions during the walk AND depth-tracks every BFS-visited node inline. The previous split-pass design (BFS to collect `parentOf` → linear walk to emit edges + depths) is gone; the new design is monolithic and exposes the diamond topology faithfully.

Diamond topology test cases verified:
- Both `P1||N` + `N||P1` AND `P2||N` + `N||P2` keys present in `edges`
- Both `P1` and `P2` appear in `nodeDepths` at depth 1
- `resolveToVisibleAncestor('N', visible={'P2'}, relations)` returns `'P2'` (pre-fix returned `null`)
