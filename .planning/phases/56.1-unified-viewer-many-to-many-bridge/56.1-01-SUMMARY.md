## Decisions Made

- **Focal derivation: insertion-order `lastAdded ?? last`.** A single `deriveFocal(prev, next, explicit)` helper handles both axes uniformly. Explicit override wins (including explicit `null`); else `next.size === 0 → null`; else the last id in `next` that wasn't in `prev` (i.e. the "newly added" id); else the last iteration-order element of `next` (when next ⊆ prev). Reasoning matches 56.1-CONTEXT.md D-1 + 56.1-PATTERNS.md §1 "Focal-derivation rule" comment block.
- **`asSet()` identity-passes Set inputs.** When the caller hands us a Set, we use it as-is (no defensive copy). Combined with the `sameSetMembership` guard, this means identical content from the same caller can preserve the caller's own reference too. Only arrays produce a fresh Set.
- **`clearSelection()` writes fresh `new Set()`, NOT null.** Set fields stay Sets so consumers can always call `.has()` / `.size` without null-guard noise. Phase 56.1-PATTERNS §1 invariant #4 explicitly calls this out.
- **Comment-only mentions of deleted field names allowed.** The header comment in `viewer-store.ts` documents the evolution from Phase 56 by name (`Promotes Phase 56's one-to-one selection (selectedNodeId)`). The verify gate filters out comment lines (`grep -v '^[[:space:]]*//'`), so this is compliant. The source-grep sentinel test in `viewer-store.test.ts` uses the same filter.
- **No new dependencies.** Native `Set`/`Map` only, per 56.1-PATTERNS.md "Available Libraries" confirmation. No `npm install` step needed (avoids the package-legitimacy gate).
