## Decisions Made

- **Items useMemo predicate.** CONTEXT.md `<discretion>` #4 confirmed in code: `selectionSource === 'timeline' && selectedBucketKeys.size > 0 && lslFilterEntityIds !== null` → entities filtered by `lslFilterEntityIds`; `selectionSource === 'graph' && selectedNodeIds.size > 1` → entities filtered by `selectedNodeIds`; otherwise empty array. The third branch is defensive — SidePanel's mode-switch shouldn't mount BucketCardList outside multi-mode, but the empty branch covers any race in which the store flips between commits.
- **Wave-2 ↔ Plan-05 deferral.** The graph-multi branch (`selectedNodeIds.size > 1`) renders one card per multi-selected graph node — coherent in isolation. Plan 05 may upgrade this to "buckets touching focal" by wiring `useNodeToBucketsIndex.get(focalNodeId)` when that hook ships. Subscribing to `focalNodeId` already (with `void` to silence the unused warning) keeps the hook dep list stable so the Wave-3 refactor adds no new subscription that surprises tests.
- **`apiClient` + `system` props kept.** Same surface as HistorySidebar / EntityDetailPanel for parity. Currently unused inside BucketCardList (entities come from `useGraphData(apiClient, system)`) but `void`-d to silence the unused warning. Plan 05's `useNodeToBucketsIndex` will consume both.
- **`entityType` label fallback.** `e.entityType` is preferred (HistorySidebar reads the same way via the index signature on `Entity`); falls back to `e.ontologyClass` then `'Entity'`. Future entity-shape refactors that consolidate the two fields won't break this code.
- **Newest-first sort.** Same sort idiom as HistorySidebar (`b.createdAt.localeCompare(a.createdAt)`). Cards with missing `createdAt` sort to the bottom which is the expected behaviour for "stale" entries.
- **Close-button gate unchanged.** Plan task 2 step 3 only mandates the onClick migration. The `{entity && ...}` gate stays — when multi-mode lacks a focal (e.g. user clicked a timeline tick with no resolvable nodes), the close button hides; user clears via bg-click or Esc. Future enhancement could broaden the gate to `(entity || isMultiMode || isSingleFocalMode) && ...` but that's out of scope.
- **Plan-04 sole-owner of SidePanel.tsx in Wave 2.** Plan 03's `<files>` deliberately excludes SidePanel.tsx so Wave-2 file scopes are disjoint and parallel-safe at merge. Plan 04 IS the consolidated SidePanel migration — every SidePanel edit lives in this plan (subscription + entity useMemo + close-button + mode-switch).
