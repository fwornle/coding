## Area discussions (1 question per area, dependency order)

### Area 1 — Multi-selection data model

**Question:** How should the store represent the multi-selection set relative to the existing Phase 56 single-selection fields?

**Options:**
- Additive: keep selectedNodeId as 'focal', add multiSelectionNodeIds:Set as the halo (RECOMMENDED)
- Promote: replace selectedNodeId with selectedNodeIds:Set, focal becomes 'first or last clicked'
- Mode-switched: single-mode vs multi-mode are two distinct store states

**Operator selection:** **Promote** — `selectedNodeIds:Set` replaces `selectedNodeId`; focal is derived from "most recently added". Breaks Phase 56 PATTERNS contract #5's literal field name (acceptance grep references `selectedNodeId`) but is conceptually cleaner — single mode for everything.

**Implications captured in D-1:** Every consumer that reads `selectedNodeId` needs renaming; `setSelection` signature changes; 6 Phase 56 contracts need re-statement (not breaking — input shape evolves).

### Area 2 — 'Touched' predicate

**Question:** When a timeline tick is clicked, which graph nodes get added to selectedNodeIds? (Same rule runs in reverse.)

**Options:**
- Resolved ancestor only — reuse pickFirstResolvable (RECOMMENDED)
- Full ancestry-chain overlap (most permissive)
- Direct entity membership only (strictest)

**Operator selection:** **Resolved ancestor only** — consistent with Phase 56's phantom-id resolution. Symmetric in both directions. Need to extend `pickFirstResolvable` into `pickAllResolvable` to return the full set (not just first).

### Area 3 — Reverse-lookup mechanism

**Question:** When a graph node is selected, how do we find ALL buckets across all sessions that touched that node, to multi-highlight on the timeline?

**Options:**
- Client-side pre-indexed Map<nodeId→Set<bucketKey>>, built when sessions load (RECOMMENDED)
- Client-side scan per click
- New server endpoint /api/coding/lsl/buckets-touching?nodeId=X

**Operator selection:** **Pre-indexed Map** — built at data-load (rebuild on sessions/visibleIds/relations change). O(1) lookup on click. No backend work. Memory cost trivial for typical session counts.

### Area 4 — Graph multi-highlight visual

**Question:** When a tick lights up N nodes in the graph (and reverse), how should focal vs halo be visually distinguished?

**Options:**
- Two-tier — lighter halo ring on touched + bright/red ring on focal (RECOMMENDED)
- Uniform halo — ring all N the same color, no special focal
- Numbered badges showing '1 of N touched'

**Operator selection:** **Two-tier** — halo `ring-blue-300/60`, focal `ring-red-500`. Path-to-CK trace drawn from focal only (not every halo — visual noise at N=5+). Same two-tier on timeline.
