>

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px (`text-sm`) | 400 (`font-normal`) | 1.5 |
| Label | 12px (`text-xs`) | 500 (`font-medium`) | 1.4 |
| Heading | 20px (`text-xl`) | 600 (`font-semibold`) | 1.2 |
| Display | 24px (`text-2xl`) | 700 (`font-bold`) | 1.2 |

Only **2 weights** used across the viewer: 400 (body, labels-as-prose) and 600 (headings, panel titles, primary CTA text, badges that emphasize identity such as the ontology-class chip on a node detail card). 500 appears once as the label weight by Tailwind default and is treated as a sub-token of 400, not a third weight choice. 700 is reserved for the page-level `<h1>` display and the active-system breadcrumb on the NavBar (one instance per route).

**Numeric typography:** Use `font-variant-numeric: tabular-nums` (matches dashboard `.item-progress-badge` at `index.css:12`) on all numeric labels (node count, edge count, "showing X of Y" filter counts, RCA progress N/Total). Implemented as Tailwind class `tabular-nums`.

**Monospace:** `font-mono` only for: entity IDs in the detail panel "raw" footer, file paths inside `MarkdownText`, code blocks rendered by the MarkdownViewer panel. Inherited shadcn defaults (`ui-monospace, SFMono-Regular, ...`).

**Accent reserved for (explicit list):**

1. Primary CTA button label (Search submit, "Expand neighbors" if surfaced as a button, "Reset filters")
2. Active route indicator in NavBar (`/viewer/coding` link state when current)
3. Selected node ring (2px ring around the currently-clicked graph node)

### Ontology-driven node color contract (D-45-03)

Node fill color is **NOT** drawn from the four-role palette above. It is per-ontology-class, sourced live from `/api/v1/ontology/classes`:

```
GET /api/v1/ontology/classes
  returns: [
    { name, level, parent,
      display?: { color: '#hex', icon: '<emoji or lucide-NodeName>', shape: 'circle' or 'square' or 'diamond' }
    }
  ]
```

**Fallback formula** (deterministic, no rebuild needed for new classes):

```
function classColor(className: string, theme: 'light' | 'dark'): string {
  // FNV-1a 32-bit hash, modulo 360 for HSL hue
  let h = 2166136261;
  for (let i = 0; i < className.length; i++) {
    h = (h ^ className.charCodeAt(i)) * 16777619;
    h = h >>> 0;
  }
  const hue = h % 360;
  // Saturation + lightness fixed per theme; same hue stays distinguishable on both backdrops.
  return theme === 'dark'
    ? `hsl(${hue}, 65%, 60%)`   // brighter on dark backdrop
    : `hsl(${hue}, 55%, 45%)`;  // muted on light backdrop
}
```

Rationale for fixed S/L (not configurable): the deterministic hue is the identifier; varying saturation per class would re-introduce drift between systems with overlapping class names.

**State color overlays on top of the per-class fill (computed at render time, NOT stored on the class):**

| State | Treatment |
|-------|-----------|
| Default | per-class fill, 1px stroke = `hsl(var(--border))`, 100% opacity |
| Hovered | per-class fill, 2px stroke = `hsl(var(--ring))`, 100% opacity, cursor=pointer |
| Selected | per-class fill, 3px stroke = `hsl(var(--primary))`, 100% opacity, 4px outer glow `0 0 4px hsl(var(--primary)/0.3)` |
| Search-match (substring hit, not selected) | per-class fill, 2px stroke = `hsl(45, 100%, 50%)` (fixed amber — the only hardcoded non-token color in the viewer, justification: search highlight must NOT collide with the class hue space), 100% opacity |
| Filter-dimmed (level/class filter excludes it but it remains rendered for context) | per-class fill, 1px stroke `hsl(var(--border))`, **opacity 0.25** |
| Filter-hidden (advanced filter sets `display: none`) | not rendered |

Edges follow the same rule: default `hsl(var(--border))` at opacity 0.5; highlighted (incident on selected node) at opacity 1.0 with stroke `hsl(var(--primary)/0.6)`; dimmed (incident only on dimmed nodes) at opacity 0.1.

## Layout Contract

**Three regions, three side-by-side panes (desktop at 1024px and above):**

1. **NavBar** — sticky top, height 64px (`h-16`), bg `hsl(var(--secondary))`. Contents: viewer wordmark (left), three route links `Coding / OKB / CAP` (center), theme toggle and `?` keyboard-help button (right). Active link styled with accent underline and `font-bold` (700) — the one display-weight exception per the Typography section.
2. **Filter Rail** — left, width 256px (`w-64`), bg `hsl(var(--card))`, sticky below NavBar, full remaining viewport height. Houses: Search input at top (with `/` shortcut hint), Level checkboxes (L0 / L1 / L2 / L3), Class multi-select. Collapsible via Radix Collapsible into a 48px (`w-12`) icon-strip below `lg` breakpoint.
3. **Graph Canvas** — center, `flex-1`, bg `hsl(var(--background))`, padding `md` (16px). Renders the force-directed graph plus a floating zoom-control cluster at bottom-right (zoom in, zoom out, fit-to-viewport — lucide `ZoomIn`, `ZoomOut`, `Maximize`).
4. **Side Panel** — right, width 384px (`w-96`) by default; expands to 480px (`w-[30rem]`) when a "wide" panel is active (MarkdownViewer or RCA). Tabbed Radix `<Tabs>` at the top with three tabs: **Entity**, **Markdown** (B's signature, hidden for non-B systems), **RCA** (C's signature, hidden for non-C systems). Tab visibility derived from the `:system` route param — `coding` shows Entity only; `okb` shows Entity + Markdown; `cap` shows Entity + RCA.
5. **Footer** — bottom, height 32px (`h-8`), bg `hsl(var(--muted)/0.3)`, text-xs. Renders: `Showing {visible} of {total} nodes . {edges} edges`.

**Responsive breakpoints (Tailwind default plus one custom):**

| Breakpoint | Width | Layout |
|------------|-------|--------|
| below `md` (under 768px) | mobile / narrow | Graph fills 100vw; filter rail collapses to a drawer (`<Dialog>` via Radix), opened by a `Filter` icon button in NavBar; side panel transforms into a sheet from the bottom 70vh on node tap |
| `md` to `lg` (768-1023px) | tablet | Filter rail stays as a 48px icon strip (hover-expandable); graph + side panel side-by-side; side panel `w-80` (320px) |
| `lg` and above (1024px and above) | desktop | Full three-pane layout as described above |
| `2xl` and above (1536px and above) | wide desktop | Side panel can expand to 576px (`w-[36rem]`) for MarkdownViewer when the user drags the resize handle (deferred — drag-resize is v2; MVP uses fixed widths) |

## Interaction Contract

### Pointer

| Gesture | Result |
|---------|--------|
| Click on graph node | Select node, populate side panel `Entity` tab, switch side panel to active state |
| Click on empty graph background | Deselect node, side panel shows empty state |
| Double-click on graph node | Expand neighbors — fetch 1-hop neighbors via `GET /api/v1/entities/{id}/neighbors?depth=1` (route name TBD by planner; the intent is locked here) and merge into the rendered graph. If the node has no unrendered neighbors, the action is a no-op plus a 1500ms toast `No additional neighbors`. |
| Click on edge | Select edge, side panel shows relation type, source name, target name, provenance fields (`createdBy`, `lastConfirmedBy`, `confirmationCount`) |
| Mouse drag on graph background | Pan |
| Scroll wheel over graph | Zoom in / out, centered on cursor |
| Pinch (touchpad) | Zoom in / out, centered on midpoint |
| Hover on node | Cursor `pointer`; node shows hovered stroke (2px ring); after 400ms a `<Tooltip>` shows `{node.name}` and `{node.ontologyClass}` (no description — keep tooltip lightweight) |
| Click filter checkbox | Update filter; graph re-renders within 200ms; footer "showing X of Y" updates |

### Keyboard

| Key | Result |
|-----|--------|
| `/` | Focus the search input (matches dashboard convention; matches Vim / GitHub muscle memory). The keystroke does NOT enter `/` into the input — `preventDefault` then focus. |
| `Esc` | Blur search input if focused; otherwise close any open `<Dialog>`; otherwise deselect the currently-selected node |
| `Tab` / `Shift+Tab` | Roving focus across NavBar, Filter rail controls, Graph canvas (focusable), Side panel controls |
| `Enter` on focused node (graph canvas keyboard mode) | Same as click |
| `Space` on focused node | Same as double-click (expand neighbors) |
| `f` | Toggle filter rail collapse (icon-strip or expanded) |
| `?` | Open keyboard-shortcut help dialog (Radix Dialog) — content lists all shortcuts above |

The `?` help dialog header reads `Keyboard shortcuts`; body is a two-column table; close button reads `Close`.

### Routing (D-45-02)

| Route | Behavior |
|-------|----------|
| `/` | Redirect to `/viewer/coding` (default landing — System A is the live home of this repo) |
| `/viewer/coding` | Mount `<UnifiedViewer system="coding" />` with `ApiClient(baseUrl=http://localhost:12436)` |
| `/viewer/okb` | Mount with `baseUrl=http://localhost:3848` |
| `/viewer/cap` | Mount with `baseUrl=` from `VITE_BACKEND_CAP_URL` (production hint: `https://okm.cc.bmwgroup.net`) |
| `/viewer/{unknown}` | 404 page: `<UnknownSystem />` showing the three valid system names as `<Link>`s |
| URL `?node=<id>` | If present on initial load, pre-select that node after the first `/api/v1/entities` response lands |
| URL `?search=<q>` | If present on initial load, prefill the search input with `q` |

**System switch contract:** when `:system` changes via React Router, the store + ApiClient are fully reset (no leaked entities, no leaked selection, no leaked filter state). Implemented by keying `<UnifiedViewer key={system}/>` so React tears down and remounts.

## State Contract

Every viewer surface that fetches data MUST render one of these mutually-exclusive states:

| State | Trigger | Visual treatment |
|-------|---------|------------------|
| **Initial load** | First mount, before ontology + entities fetch | Centered `<Loader2 className="animate-spin">` (lucide) plus label `Loading {system} graph...` in `text-sm text-muted-foreground` |
| **Ready** | At least one entity in the store | Render graph + side panel populated |
| **Empty (no data)** | `/api/v1/entities` returns `[]` | Centered illustration (use lucide `Network` icon at 64px in `text-muted-foreground/40`) plus heading `No entities yet` plus body `This {system} knowledge graph is empty. Ingest data via {system}'s pipeline to see nodes appear here.` |
| **Empty (filter)** | Filters reduce visible count to 0 | Heading `No matches` plus body `No entities match the current filter.` The action is a `<Button variant="ghost" size="sm">Clear filters</Button>`. |
| **Empty (search)** | Search query reduces visible count to 0 | Heading `No matches for "{query}"` plus body `Try a different search term or clear it to see all entities.` |
| **Error (unreachable)** | Fetch throws or non-2xx network failure | Top-of-canvas banner (full-width, `bg-destructive/10 border border-destructive/30 text-destructive text-sm rounded-md p-3`) — verbatim style copied from `integrations/system-health-dashboard/src/pages/digests.tsx:304-308`. Body: `Cannot reach {system} API at {baseUrl}. Check that the service is running and accessible.` plus `<Button size="sm">Retry</Button>`. |
| **Error (CORS)** | Fetch fails specifically with CORS or `TypeError: Failed to fetch` | Same banner style, body: `Browser blocked the request to {baseUrl} (CORS). The {system} service must allow this origin or be reached through a proxy. See unified-viewer/README#cors.` |
| **Error (ontology fetch fail)** | `/api/v1/ontology/classes` returns non-200 | Less-severe inline notice on the side panel: `Ontology metadata unavailable — node colors will use hash-based fallback.` (`bg-amber-500/10 border-amber-500/30 text-amber-700 dark:text-amber-400`). Graph still renders. |

Loading states for individual side-panel tabs use the same `Loader2` spinner with a tab-scoped label (`Loading entity detail...`, `Loading markdown...`, `Loading RCA data...`).

## Panel Contracts

### Entity Detail Panel (right side, default tab — always present)

**Header:** Entity name in `text-xl font-semibold` plus ontology-class badge using `<Badge variant="outline">` with `style={{ borderColor: classColor(className) }}` to surface the class hue without filling the chip.

**Sections (top-to-bottom):**

1. **Description** — `<MarkdownText>` (port the dashboard's lightweight inline renderer at `integrations/system-health-dashboard/src/components/markdown-text.tsx`) renders `entity.description` (CommonMark + redaction-token styling — NO Mermaid, NO GFM tables in MVP).
2. **Identity** — Two-column key / value: `Class`, `Level`, `Parent`, `Created`, `Last confirmed`. Values right-aligned, `font-mono` for IDs, `tabular-nums` for dates.
3. **Provenance** — Key / value: `Created by`, `Confirmation count`, `Last segment`. From DATA-02 entity fields.
4. **Neighbors** — Scrollable list (max-height 240px) of incident relations: `{arrow-right} {target.name}` or `{arrow-left} {source.name}` plus relation type. Click a neighbor to focus + select it in the graph.
5. **Raw** — Collapsible (`<Collapsible>`) showing the raw JSON payload in a `<pre class="font-mono text-xs bg-muted p-3 rounded-md">`. Default collapsed.

**Empty state:** when no node is selected, render `Click any node to see its details.` in `text-sm text-muted-foreground` with a `MousePointer` lucide icon at 32px above the text.

### RCA Lookup Panel (C's signature — semantics to be confirmed during planning)

**Visible on:** `/viewer/cap` only. Tab label: `RCA`.

> **PLANNER NOTE — D-45-04 says "RCA lookup panel (C's signature — semantics defer to researcher's reverse-engineering of VOKB)." Research reading of `_work/rapid-automations/integrations/operational-knowledge-management/viewer/src/components/RcaOperationsPanel.tsx` (562 lines) reveals that VOKB's `RcaOperationsPanel` is an INGESTION OPERATIONS panel — it lists `.data/rca/*` directories, triggers ingestion pipelines (`raas` / `kpifw` / `e2e`), and streams SSE progress. > **Option A — Port the actual VOKB panel (ingestion ops plus SSE progress).** Read-only mode unless ingestion endpoints are exposed by C's REST contract; if not, the panel becomes read-only progress monitor only.
>
> **Option B — Build a *new* "RCA lookup for selected node" panel.** Walks back from a selected `Finding` entity to its `RootCause` parent plus sibling `Symptom`s using `/api/v1/entities/{id}/neighbors` filtered by class. No SSE.
>
> **The visual contract below covers BOTH options** so a planner pick does not require re-running this UI phase.

**Header:** `RCA` title plus a small status indicator (`Connected` / `Idle` / `Ingesting`) using a colored dot (`hsl(var(--primary))` for active, `muted-foreground` for idle).

**Sections (Option A — ingestion ops port):**

1. **Directory list** — Two grouped lists (`KPI-FW`, `RaaS`, `E2E`) of available `.data/rca/*` dirs. Each row: timestamp (yyyy-mm-dd) plus finding count plus per-row `Ingest` button. Verbatim layout port of `RcaOperationsPanel.tsx:422-507` but rebuilt in shadcn primitives (`<Button size="sm" variant="default">` instead of raw `<button className="bg-blue-500 ...">`).
2. **Pipeline stages** — Five horizontal pill stages (`Extract`, `Dedup`, `Store`, `Synthesize`, `Resolve`). Use `<Badge variant="outline">` per stage with state-driven className: pending = `text-muted-foreground`, active = `bg-primary text-primary-foreground animate-pulse`, done = `bg-muted text-foreground`. Replaces VOKB's raw color tiers (`TIER_COLORS` at `:59-63`) with semantic shadcn tokens to keep the design system coherent.
3. **Progress bar** — `<Progress value={progressPct}>` (Radix Progress already in `package.json`).
4. **Completion summary / error list** — `<Card>` with `border-l-4 border-l-emerald-500` for success or `border-l-destructive` for error.

**Sections (Option B — selected-node RCA walk):**

1. **Walk header** — `Root cause for {selectedNode.name}` plus breadcrumb of the walk path (`Symptom`, `Finding`, `RootCause`).
2. **Cause tree** — Indented list (Radix Accordion); each level shows class badge plus name plus confirmation count.
3. **Evidence** — Tabbed below the tree: `Observations`, `Files touched`, `Sources`.

**Empty state:** `Select a Finding node to see its root-cause analysis.` (Option B) or `No RCA pipeline runs available.` (Option A — text matches VOKB at `:543-547`).

### Display-hints overlay (D-45-03) — research finding

D-45-03 hint: `.data/ontologies/{coding,okb,cap}.display.json`, merged server-side.

Planner-actionable contract (no UI-side work — this is a Phase 44 contract extension landing in Phase 45):

```
GET /api/v1/ontology/classes
  backend:
    1. Load `${ontologyDir}/${system}.json` (existing OntologyRegistry behavior)
    2. If `${ontologyDir}/${system}.display.json` exists, JSON.parse and merge per-class
    3. Per-class merge:
       if (display[className]) {
         out[className].display = {
           color:  display[className].color  ?? undefined,  // viewer falls back to hash()
           icon:   display[className].icon   ?? undefined,
           shape:  display[className].shape  ?? 'circle',
         }
       }
    4. If display.json is missing, ALL classes are returned without `display`
       -> viewer uses hash fallback exclusively (no degradation in behavior,
          just generic hues)
```

The overlay file is OPTIONAL. Missing file is NOT an error. Malformed JSON triggers a server warning + classes are returned without `display`.

### Graph library — research recommendation (CONTEXT Open Question #1)

For the design-contract phase, the library choice does not change the visual spec (the contracts above are renderer-agnostic). However, three rendering-relevant constraints from the contracts narrow the planner's search:

1. **MUST honor the per-state stroke contract** (default 1px to hover 2px to selected 3px) — d3-force and sigma.js both expose this through their custom-renderer hooks; reagraph exposes through `nodeStyle` callback; cytoscape.js exposes via stylesheet.
2. **MUST honor the deterministic-hue formula** as the per-node `fill` — all four candidates support a per-node fill callback.
3. **MUST support the "filter-dimmed at 0.25 opacity" overlay without re-running the force simulation** — this requires the lib to expose opacity as a separate visual attribute from the layout. d3-force, reagraph, and sigma.js all support this; cytoscape.js requires a class-based style refresh.

Final recommendation deferred to planner's technical research; design contract is satisfied by any of the four.

### CORS / auth (CONTEXT Open Question #5) — UI-level contract

Whatever the planner picks (direct CORS, backend proxy through A or B, OAuth on C), the viewer's UI contract for failure is fixed (see State Contract section above): the destructive banner with `{baseUrl}` interpolated plus `Retry` action. No system-specific UI for C's OAuth flow is in MVP scope; if OAuth pops a redirect, that is a backend-proxy concern.
