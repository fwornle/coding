# Phase 999.1: Extract Shared LLM Adapter Library — Context

**Gathered:** 2026-06-14
**Status:** Closed — already-shipped (no planning needed)

<domain>
## Phase Boundary

Phase 999.1 was originally framed (during v6.0 backlog capture, predating Phase 42.2 and the rapid-llm-proxy v2 work) as two goals:

1. **Extract `lib/llm/`** into a shared submodule used by `coding` and `rapid-automations/OKM`.
2. **Add a direct HTTP path for Claude Max** (keychain OAuth → `api.anthropic.com`, no `claude` CLI spawn) to eliminate the 12-15s per-call latency the CLI subprocess imposed.

By 2026-06-14 the codebase had moved past both goals through out-of-band work, so this phase is closed without planning. The CONTEXT below records what shipped, where, and the audit trail.

</domain>

<decisions>
## Implementation Decisions

### Goal 1 — Shared LLM library

- **D-01:** The shared library exists as the standalone npm package **`@rapid/llm-proxy`**, not as a `lib/llm/` directory inside the `coding` repo. The package lives at `~/Agentic/_work/rapid-llm-proxy/` (its own git repo) and is consumed by both:
  - **`coding`** — via the thin wrapper `src/llm-proxy/llm-proxy.mjs` (delegates to `@rapid/llm-proxy/proxy-bridge/server.mjs`) and via direct `LLMService` imports in `lib/km-core/src/dedup/LLMSemanticMatcher.ts` and `integrations/mcp-server-semantic-analysis/src/agents/{semantic-analysis,git-staleness-detector,wave-controller,llm-with-process}.ts`.
  - **`rapid-automations/OKM`** — via `@rapid/llm-proxy` imports in `src/index.ts`, `src/ingestion/{extractor,deduplicator}.ts`, and `src/api/server.ts` (plus a test-only shim at `src/llm/{index,llm-service}.ts` that re-exports `LLMService` for legacy import paths).
- **D-02:** The "git submodule under `lib/llm/`" framing is **rejected**. The npm-package form was preferred in practice because it keeps the proxy independently versioned, lets both consumers pin to a published `.tgz` (`rapid-llm-proxy-2.0.0.tgz`), and avoids the cross-tree git-submodule cost when the two consumers update on independent schedules.
- **D-03:** `lib/llm/` was **never created** in `coding`. The Phase 50 discussion log explicitly flagged that the path referenced in CLAUDE.md was the Phase 999.1 placeholder and never materialized — that placeholder is now obsolete.

### Goal 2 — Direct HTTP path for Claude Max

- **D-04:** Shipped in `@rapid/llm-proxy` commit `c043362` (2026-05-19) as `completeClaudeCodeDirect()` in `proxy-bridge/server.mjs:873`. Reads the Max-subscription OAuth bearer from macOS keychain and POSTs directly to `https://api.anthropic.com/v1/messages` with `anthropic-version: 2023-06-01` + `anthropic-beta: oauth-2025-04-20`.
- **D-05:** End-to-end measurement on 2026-05-19 — `~0.9s` and `9 input / 4 output` tokens for a "say OK" probe vs `~7.5s / 14.9K` tokens via the CLI path. The latency win is the ~8x speedup the phase goal called out; the token win was a bonus (CLI auto-injected ~16-22K tokens of Claude Code system prompt + tool definitions for caching, billed on every call).
- **D-06:** Fallback contract — the CLI path remains as the AUTH_ERROR (401/403) and 429 fallback. The CLI rotates the keychain blob during its own completion, so the next direct call picks up the refreshed token. No bespoke OAuth refresh logic in the proxy. Forced-CLI escape hatch: `LLM_PROXY_DISABLE_CLAUDE_DIRECT=1`.
- **D-07:** Subsequent commits hardened the fallback path:
  - `c46b1d5 feat(claude-code): fall back to CLI on 429, unify dropdown, fresh model IDs` — extends fallback to per-model-rate-limit 429s (Max-OAuth and CLI endpoints have separate rate-limit buckets).
  - `7b1d197 fix(env): proxy-side env hygiene to prevent stale HTTPS_PROXY outages` — corp-vs-public network swap robustness.
  - `513fb5b feat(diagnostics): log build SHA at startup and expose via /health` — operational visibility.

### Multi-provider routing (was implicit in the phase scope)

- **D-08:** Provider matrix shipped in `@rapid/llm-proxy` v2 covers `anthropic` (direct HTTP), `claude-code` (direct HTTP + CLI fallback), `copilot` (direct HTTP), plus `groq` and `openai` for non-Anthropic workloads. The auto-select preference order is `claude-code → copilot → groq → openai → anthropic`.
- **D-09:** Per-process routing via `processOverrides` is the coding-side knob — see `scripts/configure-wave-analysis-routing.sh` for the mandatory routing that pins `wave-analysis-*` calls to copilot for ~30x speedup (and avoids the silent mock-mode degradation that happens when claude-code CLI responses get truncated). This is documented in CLAUDE.md as a required first-start configuration and is **out of scope** for any future "extract LLM library" phase — it's coding-application routing policy, not library API.

### Claude's Discretion

- The `coding/src/llm-proxy/llm-proxy.mjs` thin wrapper stays in place. It's three lines that resolve the `@rapid/llm-proxy` package directory and launch its `proxy-bridge/server.mjs`. There is no "lift it into lib/" win.
- The OKM test-only shim at `src/llm/{index,llm-service}.ts` (re-exports `LLMService` from `@rapid/llm-proxy` for legacy import paths) is acceptable technical debt — Phase 43 D-G5.1 already documented why it stays.

### Folded Todos

None — no open todos in `.planning/todos/pending/` referenced this phase.

<code_context>
## Existing Code Insights

### Reusable Assets (already in production)
- **`@rapid/llm-proxy/proxy-bridge/server.mjs`** — single HTTP entrypoint (`POST /api/complete`) serving all five providers, with shape normalization, token-usage telemetry, auto-routing, `processOverrides` consumption, AUTH/RATE_LIMITED fallback, and a `/health` endpoint that includes the build SHA.
- **`LLMService` class** (exported from `@rapid/llm-proxy`) — high-level SDK used by `git-staleness-detector`, `semantic-analysis-agent`, OKM `ingestion/extractor`, OKM `ingestion/deduplicator`, etc. Carries auto-select + metrics.
- **`llm-with-process.ts`** (`integrations/mcp-server-semantic-analysis/src/agents/`) — the explicit `/api/complete` POSTer that the SDK's `LLMService.complete()` doesn't reach because it can't forward the `process` tag through the SDK layer. Coding-specific tool, but lives where it's used; no extraction win.

### Established Patterns
- **Package-per-concern, not folder-per-concern, when the consumer is cross-repo.** `@rapid/llm-proxy` and `@fwornle/km-core` are both shaped this way. `lib/<category>/` is reserved for coding-only internal libs (LSL, knowledge-api, vkb-server).
- **Direct HTTP > CLI subprocess for proxy-style providers.** Copilot pioneered this in `@rapid/llm-proxy` (~2-5s); claude-code adopted the same shape (~0.9s) in 2026-05.
- **Fallback to a known-good path, not to a different subscription.** When the Max direct API rate-limits, fall back to the Max CLI (same subscription, different rate-limit bucket), NOT to copilot or anthropic. Cross-provider fallback would clobber operator intent.

### Integration Points
- `/api/complete` (proxy-bridge HTTP server) is THE coding-side LLM integration boundary. Everything else (LLMService SDK, llm-with-process wrapper, `processOverrides` config, `LLM_CLI_PROXY_URL` env) wraps that boundary in different ways for different callers.

If embeddings become a routed concern (e.g., per-process embedding model selection), the proxy would be the right home. Not urgent.
