In wave-controller.ts, add a retry mechanism when QA fails:

1. **Add `private retryWaveWithFeedback()` method**:
   ```typescript
   private async retryWaveWithFeedback(
     waveNumber: number,
     qaErrors: string[],
     originalEntities: KGEntity[],
     retryFn: () => Promise<WaveResult>,
   ): Promise<{ result: WaveResult; retried: boolean }>
   ```

   This method:
   - Logs: `[WaveController] QA rejected wave ${waveNumber} output, retrying with feedback (errors: ${qaErrors.length})`
   - Calls `retryFn()` to re-execute the wave
   - Returns `{ result: retryResult, retried: true }`
   - If retry throws, logs warning and returns original result with `retried: false`

2. **Add `private buildQAFeedbackContext()` method**:
   ```typescript
   private buildQAFeedbackContext(qaErrors: string[]): string
   ```
   Returns a formatted string:
   ```
   ## QA Feedback (retry attempt)
   The previous output was rejected by quality assurance. Fix these issues:
   ${qaErrors.map(e => `- ${e}`).join('\n')}

   Ensure all entities have at least 3 specific observations with code references.
   ```

3. **Modify each wave's post-analyze QA check** (added in Plan 01) to trigger retry:

   For each wave (1, 2, 3), after the QA validation call:
   ```typescript
   if (!qaReport.passed && qaReport.score < 60) {
     // Build feedback context for agents
     const feedbackContext = this.buildQAFeedbackContext(qaReport.errors);

     log('[WaveController] QA retry triggered', 'info', {
       wave: N,
       score: qaReport.score,
       errors: qaReport.errors.length,
     });

     this.updateProgress({ currentWave: N, totalWaves: 4, subPhase: 'qa_retry', message: `Wave ${N}: Retrying with QA feedback` });
     await new Promise(r => setTimeout(r, 50));

     // Re-execute the wave with feedback
     // For Wave 1: pass feedbackContext to a new Wave1ProjectAgent instance
     // For Wave 2/3: pass feedbackContext via the agent input
     const { result: retryResult, retried } = await this.retryWaveWithFeedback(
       N,
       qaReport.errors,
       waveNResult.agentOutputs.flatMap(o => o.entities),
       async () => { /* re-execute wave N */ },
     );

     if (retried) {
       // Re-validate with QA
       const retryEntities = retryResult.agentOutputs.flatMap(o => o.entities);
       const retryQA = await this.qaAgent.validateWaveOutput(`wave${N}_retry`, retryEntities);
       log('[WaveController] QA retry result', 'info', {
         wave: N,
         passed: retryQA.passed,
         score: retryQA.score,
         improved: retryQA.score > qaReport.score,
       });

       // Use retry result if it improved, otherwise keep original
       if (retryQA.score > qaReport.score) {
         waveNResult = retryResult;
         // Re-capture agent metrics for the retry
       }
     }
   }
   ```

4. **For the retry execution functions**, use the existing wave execution methods but with feedback:
   - Wave 1: Create a new Wave1ProjectAgent, but since its `execute()` takes manifest + existingEntities (no feedback param), we need a lightweight approach. Add `qaFeedback?: string` as an optional field on the existing input types in a backward-compatible way. For now, log the feedback and re-execute -- the prompt hardening in Plan 03 is what actually uses the feedback to improve output quality.
   - Wave 2/3: Same approach -- re-execute the agent tasks with the same inputs.

   **Important**: The retry re-runs the wave agents, which makes new LLM calls. This is the mechanism for improvement. The QA feedback is logged so operators can see what was wrong. Future enhancement (out of scope for this plan) could inject feedback into the LLM prompts.

5. **Cap at 1 retry per wave** -- use a simple boolean `hasRetried` per wave to prevent loops. If the retry also fails QA, proceed with whichever result scored higher (original or retry) and log a warning.
