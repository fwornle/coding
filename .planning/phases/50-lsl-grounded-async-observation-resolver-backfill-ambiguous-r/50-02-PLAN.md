<objective>
Wire `ObservationWriter` to the LSL-window primitive from Plan 1, doing two
independent but co-located changes:

1. **Detector B (capture-time stamp).** When `_buildPriorContext` returns an
   empty context AND the user's current message contains an unresolved
   pronoun, stamp `metadata.needs_lsl_resolution = true` on the persisted
   observation row. The resolver CLI from Plan 1 will pick these up first
   on its next pass (cheap pre-filter — no regex re-scan of all rows).

2. **`_buildPriorContext` migration (Should #7).** Replace the existing
   30-minute wall-clock SQL window on the observations DB with a call to
   `getLSLWindow(observation, { maxPrompts: 3 })` from `lib/lsl/window.mjs`.
   Both the inline tier (this writer call) and the async tier (Plan 1's
   resolver) now agree on what "recent context" means: 3 user prompts,
   not 30 minutes.

Purpose: close the inline-tier weakness called out in CONTEXT.md lines 84-90.
The current 30-min observation-DB window misses any antecedent that wasn't
itself captured as a prior observation — exactly the failure mode that
motivates the entire Phase 50 backfill effort. Migrating it to the LSL
window means the inline tier resolves "do the same again" correctly the
first time, even across multi-hour autonomous tasks.

Output: one source file modified (`ObservationWriter.js`), two new test
files (detector B + prior-context migration), zero regressions to existing
ObservationWriter tests.

**Out of scope (per D-47-Boundary):** the image-attachment writer-path fix
from Phase 47. This plan ONLY touches detector B + `_buildPriorContext`.
Image attachments that cause prompt text to be dropped remain a Phase 47
problem — Plan 1's detector C is the recovery path for already-broken rows.

<!-- Pronoun detection (new for detector B): -->
Unresolved-pronoun heuristic operates on `messages[*].content` where `role === 'user'`. Match patterns (case-insensitive, word boundary):
- `\b(it|that|this|those|these|the same)\b` AND no immediately-following clarifying noun within 5 tokens
- `\b(implement|do|fix|continue|resume) (it|that|this|the same|again)\b` (the canonical pronoun trigger)
- Standalone single-word user message matching `/^(yes|yep|sure|proceed|continue|do it|go ahead|same again|do the same)$/i`

The heuristic does NOT need to be exhaustive — false positives just mean an
extra `needs_lsl_resolution` flag on a row that the resolver will re-evaluate
with the LLM. False negatives are the failure mode (row goes unflagged) —
err on the side of over-flagging when ambiguous.
