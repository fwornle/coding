`lib/lsl/window.mjs` exports:
- `getLSLWindow(observation, { maxPrompts = 3, maxWallClockMs = 24*60*60*1000, maxBytes = 30 * 1024, project } = {})`
- Returns `{ exchanges, sourceFile, byteCount, windowSpanMs }`
- `exchanges` is an array of `{ role: 'user'|'assistant', content: string, timestamp: ISO string }` ordered chronologically (oldest first)
- `sourceFile` is the LSL filename (e.g. `2026-05-22_2100-2200_c197ef.md`) that contained the most recent exchange in the window
- `byteCount` is the total byte size of `exchanges[*].content` after trimming
- `windowSpanMs` is the actual elapsed time (ms) between the oldest and newest exchange in the window

`lib/lsl/scan-and-convert.mjs` exports:
- `scanTranscriptsForUnconverted(searchPaths, { since, project } = {})`
- Returns `Array<{ path: string, mtime: number, projectHint: string, parentSession: string|null }>`
- `convertTranscriptsToObservations(transcripts, { dryRun = false, tag } = {})`
- Returns `Array<{ transcriptPath: string, observationsWritten: number, skipped: number }>`

CLI flag contract for `scripts/resolve-observations-from-lsl.mjs`:
- `--dry-run` — list candidates and show planned rewrites, do not UPDATE the DB (default: false)
- `--limit N` — process at most N rows (default: 50, hard cap per CONTEXT.md threat model section "LLM cost runaway")
- `--id <uuid>` — process exactly one row by id
- `--since <ISO-timestamp>` — only process rows with `created_at >= since`
- `--force` — re-process rows already stamped with `lsl_resolved_at` OR `lsl_resolution_skipped`
- `--mode=ambiguous|images-only|all` — which detector class to scan (default: `all`)
- `--project <name>` — scope candidate selection AND LSL window to a single project (default: `coding`). Cross-project rewrites are rejected as a class because `getLSLWindow` walks `.specstory/history/` for the named project only — selecting a row whose `metadata.project` ≠ `--project` value would call the resolver with mismatched LSL evidence.

Audit-trail metadata fields (all stored as JSON keys in the `metadata` column — no schema migration):
- `lsl_resolved_at` (ISO timestamp)
- `lsl_resolution_source` (e.g. "2026-05-22_2100-2200_c197ef.md:1234-1450")
- `lsl_resolution_window` ({ prompts: N, span_ms: <actual elapsed> })
- `lsl_resolution_confidence` (0.0–1.0 from LLM)
- `lsl_resolution_needs_review` (boolean, true when confidence 0.4–0.7)
- `lsl_resolution_skipped` ("low_confidence" or "no_antecedent")
- `lsl_resolution_attempted_at` (ISO timestamp — idempotency marker for skipped rows)
- `pre_resolution_summary` (the original `summary` string, preserved for rollback)
</interfaces>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| LSL filesystem → resolver | LSL files contain verbatim user input; treat as untrusted relative to the resolver process |
| Resolver → LLM proxy | Outbound HTTP to host port 12435; user content embedded in prompt; LLM-side prompt injection vector |
| LLM proxy → resolver | Inbound JSON response; trust boundary because the LLM is reading untrusted LSL content |
| Resolver → observations DB | Outbound writes; per-row UPDATE with audit trail; rollback-able via `pre_resolution_summary` |

| T-50-01-PI | Tampering | LSL content embedded in LLM prompt | mitigate | Resolver prompt template includes explicit SECURITY block instructing LLM to ignore instructions embedded in `<lsl_window>`; user-supplied content is wrapped in literal `<lsl_window>` tags that the system prompt names as untrusted.
