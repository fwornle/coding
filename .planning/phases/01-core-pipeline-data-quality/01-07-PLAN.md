Two changes: one in the coordinator to read the parameter, one in the YAML to declare it.

**Change 1 -- coordinator.ts (line 2639):**

Find:
```typescript
{ analysisDepth: 'surface' }
```

Replace with:
```typescript
{ analysisDepth: parameters.analysisDepth || 'surface' }
```

This reads `analysisDepth` from the workflow parameters. If not provided, defaults to `'surface'`. Users can pass `analysisDepth: 'deep'` in workflow parameters to get thorough analysis.

Also add the analysisDepth value to the log statement at line 2631. Find:
```typescript
log(`Batch ${batch.id}: Running semantic analysis`, 'info', {
  commits: commits.commits.length,
  sessions: sessionResult.sessions.length
});
```

Replace with:
```typescript
log(`Batch ${batch.id}: Running semantic analysis`, 'info', {
  commits: commits.commits.length,
  sessions: sessionResult.sessions.length,
  analysisDepth: parameters.analysisDepth || 'surface'
});
```

This satisfies the requirement that analysisDepth is "visible in workflow execution logs".

**Change 2 -- batch-analysis.yaml (in the parameters section, around line 73):**

After the `operatorWeights` parameter block, add:
```yaml
  analysisDepth:
    default: surface
    description: "Depth of semantic code analysis. 'surface' analyzes diffs and signatures (fast, lower token cost). 'deep' includes full file contents, imports, and dependencies (thorough, higher token cost). 'comprehensive' adds cross-file relationship analysis."
```

This documents the parameter and sets the default. The parameter value flows through to `executeBatchWorkflow(workflowName, parameters)` which is where the coordinator reads it.

**Important constraints:**
- Do NOT change the semantic-analysis-agent.ts -- it already supports all three depth levels correctly
- Do NOT change the default in semantic-analysis-agent.ts (`options.analysisDepth || 'deep'`) -- the coordinator now explicitly passes the depth value, so the agent's internal default is only used when called from other contexts
- Do NOT add TypeScript type validation for the string -- the semantic-analysis-agent already handles unknown values by defaulting to 'deep' internally
- The YAML parameter name must match exactly what the coordinator reads: `analysisDepth` (camelCase, matching existing parameter naming convention in the YAML)
