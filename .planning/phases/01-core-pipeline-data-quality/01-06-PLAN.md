<objective>
Replace the hardcoded `const synthesizedContent: any = null;` constants in four observation creation methods with actual batch LLM synthesis calls that produce code-specific, architecturally meaningful observations.

Purpose: Plans 01-01 through 01-05 fixed the parser, naming, fence-stripping, and pattern enrichment. However, four observation methods still short-circuit LLM synthesis by assigning null to their synthesis variable. This means observations contain only basic fallback content from raw input data (e.g., `"Impact: moderate. Files: a.ts, b.ts"`) instead of LLM-synthesized analysis. The gap: OBSV-01 requires observations to be LLM-synthesized via batch processing, and OBSV-02 requires observations to capture architectural patterns with code-specific analysis language. Both are unmet because the synthesis calls were intentionally skipped due to per-item LLM call slowness. The fix: use a single LLM call per method invocation that synthesizes content from the available decision/pattern/entity data.

Output: Patched observation-generation-agent.ts with LLM synthesis in all four methods.
</objective>
