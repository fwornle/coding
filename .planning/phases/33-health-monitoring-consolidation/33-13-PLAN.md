<objective>
**Closes G3 — statusline LSL session-id mismatch.** Coordinator's `lsl[<sid>]` keys (e.g., `claude-60474-1777723363`, derived from ETM's `lsl_heartbeat` payload `session_id` from `process.env.CLAUDE_SESSION_ID`) don't match what the tmux statusline display layer expects (e.g., `coding-claude-85122` per the gap inventory). Result: statusline shows `[LSL🔴]` even when coordinator says `lsl_by_project: healthy`.

**This plan REQUIRES a design decision before code can be written.** Three options on the table (per gap inventory):

(a) **ETM normalizes session_id** to match what statusline derives (e.g., `coding-claude-<pid>` form).
(b) **Statusline derives session_id** the same way ETM emits it (use `CLAUDE_SESSION_ID` directly).
(c) **Coordinator does fuzzy-match** on partial keys (e.g., substring match on `claude-<pid>` portion).

Each has tradeoffs:
- (a) requires modifying ETM in every project (small files but breaks D-09 which says "use CLAUDE_SESSION_ID directly")
- (b) requires modifying the statusline display layer to read `CLAUDE_SESSION_ID` (cleanest if the display already has access to that env)
- (c) makes the coordinator's lookup semantics fuzzy and harder to reason about (rejected by RESEARCH.md preference for exact keys)

**A 4th option that may dissolve the problem entirely:** the statusline display layer (whatever it is) might be ABLE to read `lsl_by_project[projectName]` instead of `lsl[<sid>]` — the rollup is already form-agnostic. If the display only needs "is THIS project healthy?", lsl_by_project is sufficient. The statusline DAEMON (`scripts/statusline-health-monitor.js`) already does this (line 93-95: reads `lsl_by_project`, computes `projectsHealthy / projectsTotal`).
