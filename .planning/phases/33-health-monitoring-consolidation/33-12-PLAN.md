<objective>
**Closes G7 (NEW from 33-08) — plist `EnvironmentVariables` doesn't propagate `launchctl setenv`.** SPEC AC #13 (injection: forced throw → unknown) currently FAILS not because the coordinator's injection mechanism is broken — `scripts/health-coordinator.js:64,176-378` correctly reads `process.env.HEALTH_COORDINATOR_INJECT_THROW` — but because macOS launchd does NOT consistently merge `launchctl setenv`-set domain variables into a child process whose plist contains an explicit `<key>EnvironmentVariables</key>` dict.

`ps eww -p $(pgrep -f health-coordinator.js)` confirms `HEALTH_COORDINATOR_INJECT_THROW` is missing from the process env even after `launchctl setenv HEALTH_COORDINATOR_INJECT_THROW db_health` and `launchctl kickstart -k`.

**Fix (option (a) from gap inventory):** Declare the injection env vars in the plist's `EnvironmentVariables` dict with empty-string defaults. Once a key is declared in the plist's dict, `launchctl setenv <KEY> <value>` overrides the empty default and the value reaches the child process at next kickstart.

**Closes:** G7
**SPEC ACs unblocked:** #13 (forced injection → unknown)
**Out of scope:** G1, G2, G3, G4, G8 — separate plans.

**Type:** `autonomous: false` because the plist file lives at `~/Library/LaunchAgents/`, the bootstrap step requires `launchctl bootout` + `launchctl bootstrap` cycle (the user MAY want to verify the coordinator is alive between bootout and bootstrap — there's a brief window when the coordinator is down). However, the actual edit + reload is mechanical; the human-verify gate is for **AFTER** to confirm AC#13 passes.
</objective>
