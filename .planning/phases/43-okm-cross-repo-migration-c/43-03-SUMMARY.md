## Decisions Made

- **Delete the file rather than edit-then-commit.** The plan's literal workflow was:
  1. Use `node` one-liner to remove the `@fwornle/km-core` key
  2. `git add package.json` + commit

  But step 2 only makes sense if `package.json` is already tracked. It wasn't — `git ls-files package.json` returned empty. Editing then `git add` would create a *first* commit of a near-empty `{}` package.json into a Python project root, which is pollution. Outright deletion of the untracked garbage achieves the same end state — zero `@fwornle/km-core` reference at the root — with a cleaner tree.
- **Also delete `package-lock.json` and `node_modules/`** even though the plan only mentions `package.json`. The lockfile and `node_modules/` were direct artifacts of the same npm install. Leaving them as dangling orphans would have re-introduced the dep on the next `npm install` if anyone ran it. The plan's spirit (eliminate orphan npm state from rapid-automations root) is best served by removing the whole cluster.
