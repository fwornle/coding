## Step 3 — The four acceptance checks

For each check below — record **PASS** or **FAIL** with a one-line note. If FAIL, note what you saw + whether you triaged with a D-G6.1 bug-fix-only edit (see § Bug-fix scope below).

### ✅ Check 1 — Graph renders

- Wait for the graph to draw (may take a few seconds for ~1665 entities).
- **PASS criteria:** force-directed graph appears; node count badge/header shows a number in the ballpark of 1665 (kpifw + raas entities also show — total ~2000+ depending on filter defaults).
- **FAIL signals:** blank canvas, infinite spinner, browser console shows `Failed to fetch http://localhost:8090/api/entities`, or node count = 0.

### ✅ Check 2 — Entity click opens detail panel

- Click any visible node.
- **PASS criteria:** detail panel/sidebar opens showing:
  - `description` text
  - `metadata` (project, runId, embeddingModel, etc.)
  - `provenance` (extracted from RCA / doc source)
  - `legacyId` (one of the OKM original IDs preserved by Plan 07's JSON-replay)
- **FAIL signals:** click does nothing, panel opens but description/metadata/legacyId fields are blank/missing.

### ✅ Check 3 — RCA lookup returns results

- Find the RCA lookup affordance (search box, "RCA Lookup" button, or right-click → "Show RCAs for this entity").
- Trigger a lookup against any known entity (one with a real description, not a stub).
- **PASS criteria:** related RCA(s) appear, with at least one populated field per result.
- **FAIL signals:** "No results" for an entity you know should have RCAs, OR a 500 from the backend on `/api/rca/...`.

### ✅ Check 4 — Both `layer=evidence` and `layer=pattern` entities visible

The OKM ontology splits entities across `evidence` and `pattern` layers. Plan 10's PLAN.md flags a hardcoded `entity.layer === 'evidence'` filter as the canonical D-G6.1-allowed bug.

- Look for a layer filter, color legend, or filter dropdown.
- **PASS criteria:** evidence-layer entities AND pattern-layer entities are both visible in the graph (use the filter to toggle if needed; default view may show only one layer).
- **FAIL signals:** only one layer shown; toggling the filter does nothing; pattern-layer entities completely absent from API responses (check via `curl http://localhost:8090/api/entities?layer=pattern&limit=1`).

## Bug-fix scope — D-G6.1 in-place fix allowance

Per Plan 10's `<objective>` and D-G6.1: if a check FAILS due to a **real regression** introduced by the post-cutover code (not your local env quirk), you MAY apply a minimal in-place fix to `integrations/operational-knowledge-management/viewer/*` files. Examples of what's in-scope:

- A hardcoded `entity.layer === 'evidence'` filter that should be removed or layer-agnostic.
- A field-name mismatch in the detail panel (e.g., panel reads `entity.description_text` but km-core stores `entity.description`).
- An API URL that points at the legacy `.data/exports/` path instead of the post-cutover surface.

**What's NOT in scope** (file as Phase 45 work instead):
- Refactoring the viewer architecture, replacing D3 with sigma.js, adding new features.
- Backend changes beyond the bug-fix-only allowance (don't touch `src/api/routes.ts` unless it's a regression directly causing a Check failure).

If you apply a fix, capture:
- The file path + 1-line description of the fix
- The OKM commit SHA you used to land it
