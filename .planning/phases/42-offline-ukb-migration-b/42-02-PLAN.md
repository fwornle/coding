<objective>
Fix the dashboard "Race condition detected (0/0 steps)" log spam and the "stuck at running" terminal-state inconsistency (SC#3 + SC#4). Per RESEARCH §2, the actual race is between two concurrent writers to the progress JSON file:

  - **Writer 1:** `coordinator.writeProgressFile` (65 call sites in coordinator.ts) — does a full `fs.writeFileSync(progressPath, JSON.stringify(progress, null, 2))` with NO read-modify-write.
  - **Writer 2:** `createProgressFileSubscriber` (workflow-state-machine.ts:117-162) — already does field-preserving read-modify-write that preserves `stepPaused` / `mockLLM` / `singleStepMode` / etc.

The asymmetry causes the bug: when coordinator writes a "fat" progress object with `stepsDetail`, the state-machine subscriber's next transition write overwrites it with the lean state-machine shape, clobbering `stepsDetail`. The dashboard polls during that window and sees `totalSteps: 0`, triggering the warning at `system-health-dashboard/server.js:1071`.

RESEARCH recommendation: fix #3 (field-preserving merges in coordinator) is the cheapest land. This plan adopts that recommendation.

Purpose: SC#3 (no race-condition log spam after a ukb full run); SC#4 (dashboard reflects terminal state consistently).

Output: 1 modified source file (coordinator.ts) + 1 new test file. NO changes to workflow-state-machine.ts, NO changes to dashboard/server.js (the dashboard log will simply stop firing because coordinator no longer creates the inconsistency).
</objective>
