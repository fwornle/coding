<objective>
Eliminate the `process='unknown'` bucket from semantic-analysis container LLM traffic by introducing a frozen per-sub-step process tag registry, relaxing the `createLLMWithProcess` factory to honor per-call-site overrides, threading `process` through `SemanticAnalyzer.analyzeContent` via a strangler swap of its SDK direct path, and tagging all wave-1/2/3/4 + ontology-classify call sites. Lands the operator-facing `processOverrides` defaults extension and the zero-unknown acceptance gate.

Purpose: Without per-sub-step process tags, operators cannot pin a provider/model per sub-step via the settings UI — every wave-4 LLM call shows up as `process='unknown'` in `.data/llm-proxy/token-usage.db` and the routing knob has no name to bind against. Closes the deferred 42.2-06 wave-4 InsightGenerationAgent gap.

Output:
- New module `integrations/mcp-server-semantic-analysis/src/agents/process-tags.ts` exporting `PROCESS_TAGS` as a frozen `as const` object (9 keys — see registry decision below).
- 2-line factory change in `llm-with-process.ts` (relaxed `Omit` typing + `req.process ?? processTag` body).
- Per-call `process:` overrides at 7 wave-1/2/3 call sites + 5 wave-4 call sites in `insight-generation-agent.ts` + 1 site in `ontology-classification-agent.ts`.
- `SemanticAnalyzer.analyzeContent` strangler-routed through `llmWithProcessComplete` when `options.process` is set.
- `scripts/configure-wave-analysis-routing.sh` extended with the new sub-step entries.
- `scripts/verify-zero-unknown.mjs` acceptance gate (D-10).
- Submodule + outer-repo pointer-bump commit pair (Phase 42.2-04 precedent).

**Registry decision (planner-final after Task 2 inventory):** the registry ships with **9 keys**, NOT 10. `WAVE3_RELATION_DISCOVERY` is dropped because the only "relation discovery" path is the `pred` step in `kg-operators.ts:edgePrediction` (~line 567+), which is purely score-based math (cos / Adamic-Adar / Common-Ancestors) — there is no LLM call to tag.
