<objective>
Replace the 24 hardcoded `llmModel:` literals in `integrations/system-health-dashboard/src/components/workflow/constants.ts` with live-derived model badges sourced from the existing `/api/token-usage/recent` endpoint. Each `WORKFLOW_NODE` gains a `processTag` field referencing the frozen registry from Plan 52-01; the trace modal renders the actual provider/model that served the most-recent N=10 calls for that tag, with a documented preference-order fallback when the bucket is empty. Additionally, the operator settings dialog (`token-usage-settings-dialog.tsx`) auto-lists rows from the `PROCESS_TAGS` registry per D-11 — adding a new tag to the registry surfaces it in the settings UI automatically, with no UI file edit required.

Purpose: Today the dashboard's badge ("Groq: llama-3.3-70b-versatile") is a cosmetic lie. Routing already goes claude-code → copilot → groq → openai → anthropic, but operators reading the dashboard can't confirm that without running ad-hoc SQL against `.data/llm-proxy/token-usage.db`. After this plan, the dashboard is the live source-of-truth — operators see what actually happened, refreshed every 30 seconds. The settings dialog auto-listing closes the registry-as-source-of-truth loop per D-11: the same `PROCESS_TAGS` registry feeds both the call-site tagging (Plan 52-01) AND the operator settings UI (this plan), so the two surfaces can never drift.
