<objective>
Make client-disconnect / request-abort actually propagate to the worker (D-01/D-03 / WLIFE-04). Replace the Phase-62 non-destructive `worker.cancel()` seam (the protocol interrupt that live-HANGS — 62-HUMAN-UAT.md test 6) with SIGTERM + dispose + synchronous pool-drop, so a dead client never pins a concurrency-1 worker. Distinguish the two abort cases: an IN-FLIGHT request aborts via SIGTERM+dispose+drop (next same-key request lazily respawns a cold worker); a QUEUED request aborts by dequeue+reject-RETRYABLE only, leaving the worker and its (different) in-flight request alive.

Purpose: This is the headline WLIFE-04 behavior and the direct fix for the live-proven cancel hang. It reuses Plan 01's `_disposeAndDrop` synchronous-drop helper so the SIGTERMed worker is spliced before any concurrent `acquire` can pick it.
Output: a rewritten `complete()` abort handler with in-flight-vs-queued discrimination, plus unit coverage. The server.mjs side needs NO change (the abort signal already reaches `complete` — verify only).
</objective>
