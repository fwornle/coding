## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client inline score edit → PATCH /api/experiments/scores/:taskId | untrusted client value crosses to the server; must be re-validated server-side |
| stored reconciliation summary → DOM | status strings rendered in the badge/tooltip; already-persisted, verbatim |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-86-05-01 | Tampering | client-forged score override becoming source of truth | mitigate | Inline edit dispatches the EXISTING `saveOverride` PATCH; server `handleScoreOverride` re-validates ranges + writes corrected_* (client range mirror is UX-only; grep-gate forbids a client `applyOverride`). Non-2xx reverts the optimistic value. (V5 input validation) |
| T-86-05-02 | Tampering | reconciliation status recomputed/altered client-side | mitigate | Badge reads the summary verbatim via fetchReconciliation; no client recompute of matched/flagged (Phase-83 owns semantics). null → no badge, never a fabricated status. |
| T-86-05-03 | Information disclosure | quarantined-run count leaking a wrong/inflated number | accept | Count is a client-side `runs.filter(r=>r.pending).length` over already-fetched rows; low-value internal metric, no PII. |
| T-86-05-04 | Tampering (XSS) | server validation message / reconciliation string rendered unescaped | mitigate | React default escaping; render error/status strings as text, no `dangerouslySetInnerHTML`. |
| T-86-05-SC | Tampering | npm installs | mitigate | No packages installed — shadcn Input/Button/Tooltip/Badge already present. No slopcheck surface. |
</threat_model>
