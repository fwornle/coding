<objective>
Fix two OpenCode-specific BLOCKING gaps from Phase 51 verification:

**CR-01** (REVIEW.md line 64-90, VERIFICATION.md line 35-42): `lib/lsl/adapters/opencode-sqlite.mjs:231` reads `searchPaths.limit` where `searchPaths` is an Array. Arrays have no `.limit` property → the value is always `undefined` → `limit` always defaults to 100. The dispatcher's `--limit` flag (and the `SUB_AGENT_SWEEP_LIMIT` launchd env var) is dead for OpenCode. Users running `--limit 500` to backfill a larger window only ever retrieve the 100 oldest rows from SQLite. The dispatcher does its own `discovered.slice(0, limit)` AFTER the adapter returns, but the SQL `LIMIT ?` bind is hard-pinned to 100 — the data is truncated at the DB query, not at the post-process layer.

**CR-02** (REVIEW.md line 94-132, VERIFICATION.md line 43-50): `scripts/sub-agent-live-opencode.mjs:232-250` writeHeartbeat() omits the `registry_rows` array that scripts/sub-agent-live-claude.mjs:140 and scripts/sub-agent-live-copilot.mjs:226 both include. `lib/lsl/registry-reader.mjs:145` enumerates running sub-agents EXCLUSIVELY from `hb.registry_rows`. Net result: `live_registrations.opencode.running` in /health/state is permanently 0, regardless of real OpenCode activity. Operators looking at the dashboard to verify OpenCode capture will see `running: 0` and assume the daemon is broken when it's actually working. Per REVIEW line 132, also include `project` in each registry_row so the reader's project filter is correct rather than permissive.

Both bugs are OpenCode-only and touch non-overlapping but adjacent files. Bundled into a single plan per planner discretion (both small fixes; both add testing; both share read_first context).

Purpose: Restore OpenCode's `--limit` flag (data-loss bug — quiet truncation) AND restore OpenCode's visibility on /health/state (observability bug — silently 0).
Output: opencode-sqlite.mjs limit plumb-through + dispatcher forwarding + opencode daemon registry_rows emit + two new tests.
</objective>
