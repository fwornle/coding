<objective>
Fix CR-03 (REVIEW.md line 136-173, VERIFICATION.md line 51-58): `lib/lsl/live/claude-fs-watch.mjs:515-527` mutates a stale Map reference after `registry.upsert({})` replaces the slot.

The broken code does three operations in sequence:
1. `const cur = opts.registry.get('claude', subHash)` — get reference to current row
2. `opts.registry.upsert({agent:'claude', sub_hash: subHash})` — upsert with empty payload, which internally does `const merged = { ...existing, ...row }; this._rows.set(key, merged);` — creating a NEW object and replacing the Map slot
3. `cur.observations_written = (cur.observations_written || 0) + 1` — mutates the ORIGINAL `cur` object, which is no longer in the Map. The new Map slot still has whatever `observations_written` the prior row had (typically 0).

The downstream effect is `observations_written` on the live row stuck at 0 forever. The `markCompleted` at line 544 reads `row.observations_written` — since `0 || exchangesEmitted` falls through to `exchangesEmitted`, the FINAL value is right by accident, but any tooling that reads `observations_written` mid-tail (heartbeat, registry-reader extensions in Plan 51-10, future health-coordinator queries) sees permanent 0. Provably broken if read mid-tail.

Additionally, the comment block at lines 520-522 claims `observations_written` "is preserved by upsert (it is not in the overrideable list)". No such allowlist mechanism exists in `lib/lsl/registry.mjs` — the spread `{ ...existing, ...row }` preserves any field whose key is not in `row`. The comment is misleading and must be removed.

Purpose: Restore mid-tail observability of `observations_written` for the Claude live watcher; remove the misleading comment that fabricated a registry mechanism. Lock the fix with a regression test that fails on the broken code path.
Output: claude-fs-watch.mjs fix + optional registry.mjs helper + regression test.
</objective>
