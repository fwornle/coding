## Accomplishments (shipped)

### T-42-1-01 — Extend persistWithKmCore return type (`36a1647b9`)

Extended the return type with `anchorEdgesAdded`, `anchorEdgesFailed`,
`anchorEdgesSkipped` counters. Caller (`persistWaveResult`) logs them at the
wave-persistence-complete boundary so operators can see the anchor pass
outcome in `.data/workflow-logs/wf_<id>.log`.

### T-42-1-02 — Port `findBestParent` into wave-controller (`7f81313e8`)

Mirror of `persistence-agent.ts:1043-1071` with the same semantics:
substring-match on lowercased names, longer-match wins, ties broken toward
SubComponent, falls back to `Coding` Project if present, returns null
otherwise. JSDoc explicitly forbids deletion when persistence-agent.ts is
retired.

### T-42-1-03 — Add `ensureProjectAnchor` helper (`303c74539`)

Idempotent Coding-bootstrap. Queries km-core for an existing Project; mints
one if absent. Wrapped in try/catch so any failure is fail-soft (the
post-sweep anchor pass continues even if bootstrap fails). **This fail-soft
behaviour is what allowed the SC#6 failure mode to surface in production
without crashing the wave-analysis run** — exactly the design intent.

### T-42-1-04 — Slot the anchor pass into `persistWithKmCore` (`8ea18924d`)

Runs LAST (after entity and relation sweeps). Two-layer idempotency check:

- Layer (a): in-wave `relationships` array — any `contains` / `parent-child`
  edge marks its `.to` as already-anchored.
- Layer (b): `kmCoreAdapter.queryIncomingRelations` — entities with existing
  anchor edges in the store are also marked already-anchored.
- Layer (c): km-core's `storeRelationship` upsert semantics tolerate
  duplicates at the storage layer.

The loop walks remaining entities, calls `findBestParent`, and inserts a
`contains` edge from the chosen parent. Every `storeRelationship` is wrapped
in try/catch so a single failed edge doesn't abort the sweep.

### T-42-1-05 — 6 unit tests (A–F) (`042f4db46`)

In a new `describe('Phase 42.1 — project-anchor parity')` block. Tests A–F
cover: anchor pass adds edges to Coding for unparented entities; respects
existing in-wave anchors; bootstraps Coding when absent; uses findBestParent
to attach to Components / SubComponents; fail-soft on storeRelationship
errors; idempotent on second-run (queryIncomingRelations layer).

### T-42-1-06 — SC#6 orphan-count check in e2e verifier (`b4e25e6bd`)

Reads `.data/knowledge-graph-migrated/exports/general.json`, builds an
incoming-anchor index keyed by node id, sweeps in-scope nodes (excluding
`entityType in {Project, System}` AND `legacyId?.id === 'unknown'`), reports
`{ inScopeEntities, excludedLegacyEntities, orphans, sample }`.

## Blocked (gating PARTIAL → COMPLETE)

### T-42-1-07 — Production rebuild + SC#6 verifier

- Submodule build: clean (zero TS errors).
- Docker rebuild: clean (build #67).
- `docker-compose up -d coding-services`: clean (healthy in 5s).
- Sticky debug state reset: clean.
- `wave-analysis` workflow run via MCP `execute_workflow`: completed
  successfully (154 LLM calls, 396 K tokens, 131 min wall time, 4/4 waves
  green from the runner's POV).
- SC#6 verifier: **fail** — 68 in-scope orphans out of 106 in-scope entities.

**Why it failed:** the ontology loader inside the container could not find
the expected `upper/` and `lower/` subdirectories under `.data/ontologies/`.
With zero ontology classes loaded, `ensureProjectAnchor` threw "Unknown
ontology class: Project" on every wave. The fail-soft design then let the
wave continue, but the in-wave `Coding -contains-> X` relations failed
silently while their `.to` entries still populated the `alreadyAnchored`
bookkeeping. The anchor pass loop therefore skipped every entity rather than
falling back to `findBestParent`. Result on every wave:
`added=0 skipped=N failed=0`.

Full cascade and root-cause analysis: `.planning/forensics/report-20260524-130355.md`.

## Routing decision

Phase 42.1's code is correct. It cannot be validated end-to-end until the
ontology blocker resolves. Phase 42.1.1 — Ontology Layout Resolution — has
been inserted to fix this. Once 42.1.1 lands and SC#6 is re-run and passes,
this plan promotes from PARTIAL to COMPLETE.

## Follow-up tickets (do not block 42.1.1)

1. **Anchor pass bookkeeping bug** — move `alreadyAnchored` population to
   AFTER the relations sweep verifies persistence success. Today the
   bookkeeping trusts the in-wave `relationships` array regardless of whether
   each relation actually persisted. Hardening only — won't trigger unless
   the upstream bootstrap fails again.
2. **Verifier-vs-registry API mismatch (SC#5)** — `registry.hasClass is not
   a function`. Fix verifier or registry to align surfaces.
3. **Plan template Task 7 Step 8 recipe** — references `docker logs
   coding-services | grep 'Project-anchor pass:'` but the semantic-analysis
   `log()` function writes to `.data/workflow-logs/wf_<id>.log`, not to
   docker stdout. Update the recipe so future forensics on similar phases
   don't get misled.

## Lessons for downstream phases

- **Fail-soft design surfaced a hidden upstream failure.** The
  `ensureProjectAnchor` try/catch did exactly what it was supposed to do:
  let the run complete and recorded the failure to the wave log. Without it,
  the wave-analysis would have crashed on every wave and the broader symptom
  (orphans + duplicate Components) would have stayed invisible. Keep this
  design pattern.
- **SC verifiers should not assume the underlying registry is loaded.** SC#5
  here had a different API drift, but the broader lesson is that an empty
  registry is a real production state and SCs should report it explicitly
  rather than dying with a TypeError.
- **The "fix one bug" plan revealed a class of upstream gaps.** Phase 42.1
  was scoped narrowly (port findBestParent), but the production verify
  exposed Phase 38's ontology layout, Phase 40's dedup type-dependency, and
  the verifier's registry API. None of these are 42.1's responsibility but
  all of them block 42.1's success criterion. Adjust the test recipe so
  future single-bug-fix plans surface these blockers earlier.
