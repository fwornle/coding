<domain>
## Phase Boundary

**The single load-bearing fix.** Phase 42-07 Phase B1 replaced `persistence-agent.persistEntities()` with `persistWithKmCore` in `wave-controller.ts` and inadvertently dropped the legacy `findBestParent` lookup + post-sweep `contains`-relation insertion that ran after every batch. As a result, every `ukb full` run now stores new wave-emitted entities WITHOUT any incoming `contains` edge from the `Coding` Project anchor (or any matching Component/SubComponent).

**Empirical evidence (2026-05-24 overnight run):**
- 16:59 backup ‚Üí 07:31 live store: +64 entities, +1 relation, **0 new edges to the Coding project anchor**.
- The +1 relation is between two of the new entities, not to a project.
- 113 fresh-wave entities have `metadata.team='coding'` (set by legacy `mapEntityToSharedMemory`); 802 migrated legacy entities have neither team nor anchor edges. None of them are connected to the `Coding` Project node.

**Comparable paths that DO attach anchors correctly:**
- Legacy `persistence-agent.ts:1043-1110` ‚Äî `findBestParent()` + `updateEntityRelationships()` post-sweep adds a `contains` edge for every recent entity lacking a parent.
- Online learning `src/live-logging/ObservationConsolidator.js` (around L290 / L367) ‚Äî `_ensureProjectAnchor()` mints the project node if absent + POSTs `has_insight` / parent relations for every new insight.

**The gap is exactly one location:** `wave-controller.ts:persistWithKmCore` does an entity sweep + relation sweep but has no post-sweep anchor pass.

</domain>

<decisions>
## Implementation Decisions

### Location & approach (locked)
- The fix lands in `integrations/mcp-server-semantic-analysis/src/agents/wave-controller.ts`, inside (or immediately after) `persistWithKmCore`.
- Port `findBestParent` semantics from `persistence-agent.ts:1043-1071` ‚Äî DO NOT call the legacy method directly (persistence-agent.ts is scheduled for deletion in a follow-up phase). Either inline as a private method on WaveController, or extract to a small helper module if the planner judges it cleaner (e.g., `src/agents/parent-resolver.ts`).
- Anchor pass runs AFTER the entity sweep AND the relation sweep complete ‚Äî so it can read the just-stored entities + the just-stored relations to decide which entities need a `contains` edge.

### `findBestParent` algorithm (locked ‚Äî mirror the legacy)
1. Filter candidates to entities where `entityType ‚àà {Component, SubComponent}`.
2. For each candidate, check if `entityName.toLowerCase().includes(candidate.name.toLowerCase())`.
3. Among matches, prefer the LONGEST candidate name; among same-length matches, prefer SubComponent over Component.
4. If no Component/SubComponent matches, fall back to the `Coding` Project entity (lookup by `name === 'Coding' AND entityType === 'Project'`).
5. If `Coding` doesn't exist in the store, MINT it (idempotent cold-start, matching `_ensureProjectAnchor` semantics).
6. Return the parent name (the relation `from` endpoint). Skip the entity entirely if no parent and no Project (this should never happen post-Coding-bootstrap).

### Edge-skipping rules (locked)
- Skip entities where `entityType ‚àà {Project, System}` ‚Äî these ARE the anchors / scaffolding; don't add `contains` edges to them.
- Skip entities that ALREADY have an incoming `contains` or `parent-child` relation. Idempotent re-runs must not duplicate edges.
- Skip entities whose `parentEntityName` field (set by `canonical-mapper`) already names an existing entity ‚Äî that entity is the intended parent; the existing relation sweep handles it.

### Relation insertion (locked)
- Use `this.kmCoreAdapter.storeRelationship(parentName, entityName, 'contains', { source: 'wave-analysis', runId: this.runId })`.
- Wrap each call in try/catch matching the existing `persistWithKmCore` pattern (T-42-06-03 mitigation). Increment a `anchorEdgesAdded` counter on success; `anchorEdgesFailed` counter + stderr line on failure. No throw.

### `Coding` Project bootstrap (locked)
- Check for `Coding` (entityType=Project) in the store via `kmCoreAdapter.queryEntities({ entityType: 'Project' })`. If not found:
  - Mint a minimal Project entity: `{ name: 'Coding', entityType: 'Project', ontologyClass: 'Project', layer: 'evidence', description: 'Coding project root entity (anchor for wave-analysis emissions).' }`.
  - Stamp `metadata.subsystem='wave-analysis'`, `metadata.provenance` via the canonical-mapper helpers (so the bootstrap entity has the same provenance shape as other wave entities).
  - Persist via `kmCoreAdapter.storeEntity(entity, { team: this.team })`.
- This is idempotent: subsequent `ukb full` runs find the existing `Coding` entity and skip the mint.

### Logging & counters (locked)
- The existing `persistWithKmCore` returns `{ entitiesStored, entityErrors, relationshipsStored, relationshipErrors, relationshipsSkipped }`. Extend the return to include `anchorEdgesAdded, anchorEdgesFailed, anchorEdgesSkipped` so the wave-controller summary log surfaces the new step's outcome.
- Log at `info` level: `[WaveController] Project-anchor pass: added=X skipped=Y failed=Z`. Log at `debug` level the per-entity parent assignment for traceability.

### Claude's Discretion
- Whether `findBestParent` becomes a private method on WaveController or a separate helper module ‚Äî planner decides based on testability and current code organization.
- How to test the `Coding`-bootstrap path without requiring a clean store ‚Äî propose either (a) a unit test that mocks `kmCoreAdapter.queryEntities` to return empty, or (b) an integration test that wipes the store before the run.
- Whether to add an early-exit when `entities.length === 0` ‚Äî minor optimization; planner decides.
- Whether to include the `metadata.parentEntityName` shortcut check or always run full `findBestParent` even when the field is present ‚Äî planner decides; the legacy code path runs `findBestParent` regardless.

</decisions>
