<approach_decision_for_research>
## Open question RESOLVED — L2 classification is DETERMINISTIC (name+description keyword heuristic with parent-edge consistency), NOT LLM-driven.

The 60-09-CONTEXT.md open question was: LLM-semantic classification vs path/keyword heuristic vs hybrid, and exactly how an entity maps to e.g. RapidLlmProxy vs DockerizedServices. Decision: deterministic keyword heuristic over the closed 10-class vocabulary, applied to name + description, parent-consistent. Rationale (load-bearing):

1. The L2 vocabulary is a closed set of 10 SUBSYSTEM identities, and entity names already encode subsystem membership. Live probe of Component-class entities returns names that are literally or near-literally the L2 classes: LiveLoggingSystem, KnowledgeManagement, DockerizedServices (exact), SemanticAnalysis->BatchSemanticAnalysis, ConstraintSystem->ConstraintMonitor, LLMAbstraction->RapidLlmProxy. Detail entities matching Observation|Digest|Insight|Online map to the OnlineObservation/OnlineDigest/OnlineInsight set. This is a lookup problem, not a semantic-judgment problem.

2. Entities carry NO usable path/tags signal. A live probe of Detail-node attributes shows filePath:null, path:null, tags:null, parent:null -- the only signals available are name, description, and parent-child edges. So a "path heuristic" is impossible, and the only signal an LLM could use (name+description) is exactly what the keyword heuristic uses. An LLM call adds cost, latency, non-determinism, and hallucination risk for zero added signal.

3. A migration over ~1140 entities must be cheap, idempotent, and re-runnable. LLM-driven backfill at ~1140 entities is both slow and non-deterministic across runs (a re-run could flip classifications), defeating idempotency. The deterministic mapper yields the SAME result every run -- the migration can be re-applied safely.

4. SC#5 requires NON-ZERO per-L2 counts and working L2 selection -- NOT that every entity gets an L2. Most generic SubComponent/Detail entities (e.g. CachingMechanism, TranscriptAdapter) legitimately belong to no specific L2 subsystem. The mapper returns null (leave the L1 class unchanged) when there is no confident keyword match -- no forced L2, no hallucinated L2. This preserves the existing no-forced-L2 invariant (Phase 57 D-10) and avoids polluting L2 buckets with mis-assigned entities.

5. The existing LLM L2-refinement prompt (Phase 57-04) has already proven ineffective -- it ships in the pipeline yet produced 0 L2-classified entities. Replacing the prompt-step outcome with a deterministic mapper is the correct fix, not more prompt tuning.

### Exact mapping rule (the closed vocabulary + keyword sets)

classifyL2(name, description, l1Parent) returns an L2 class name or null. It only returns an L2 whose parent matches l1Parent (consistency invariant -- an entity whose L1 is Detail can only be refined to OnlineObservation/OnlineDigest/OnlineInsight; SubComponent only to EtmDaemon; Component only to the 6 Component-parented L2s). Matching is case-insensitive substring/token on name first, then description. The implementer derives the keyword sets from the L2 class description fields in .data/ontologies/coding.lower.json and from the L1 entity-name probe; baseline keyword sets to encode:
