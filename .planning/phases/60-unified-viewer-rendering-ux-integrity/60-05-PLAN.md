<objective>
G5 (D-16..D-20) — Drop the hardcoded `CODING_SCHEMA` (Phase 55 stopgap) including the `Typed Views`
(LSL Pipeline / Patterns / Other) group. Build L1→L2 groups dynamically from the response of
`GET /api/v1/ontology/classes?withDisplay=true`. Closes LOWERONTO-03.

Purpose: Phase 57 shipped 10 L2 classes in `.data/ontologies/coding.lower.json` extending L1 parents
(Component / SubComponent / Detail). The viewer is still rendering the old Phase 55 stopgap, so those
L2 classes are invisible to the operator. The endpoint already returns `{name, level, parent, display}` —
the work is purely on the client.

**Scope guard (per checker W-1):** The VOKB tab at `/viewer/okb` currently consumes `OntologyFilter`
with NO `groupingSchema` prop and relies on the default `groupingSchema = VOKB_SCHEMA` to render its
Upper Ontology / Failure Model groups. Dropping the prop entirely would silently degrade VOKB to flat
rows. This plan keeps the `groupingSchema?` prop as optional and reshapes the default-fallback
semantics: present → legacy hardcoded path (VOKB tab); omitted → new API-driven path (coding tab).
Only `CODING_SCHEMA` + its `Typed Views` group are dropped.

Output: Refactored `OntologyFilter.tsx` with dual-mode rendering; tests covering API-driven L1/L2
hierarchy AND VOKB_SCHEMA legacy regression; FilterRail no longer passes `CODING_SCHEMA` for the
coding tab but continues to pass nothing (relying on VOKB_SCHEMA default) for the VOKB tab.
</objective>
