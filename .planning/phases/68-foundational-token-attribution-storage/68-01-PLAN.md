## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| caller → logCall row fields | Row field values (agent/task_id/etc.) originate from callers (proxy write path, future adapters) and are written to SQLite via a prepared statement |
| startup → SQLite schema | initTokenDb mutates the on-disk schema of a shared DB at every proxy boot |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-68-01-01 | Tampering | logCall SQL insert | mitigate | Values bound via better-sqlite3 prepared statement placeholders (no string interpolation) — injection-safe by construction; preserve the parameterized insertStmt, never build SQL from row fields. |
| T-68-01-02 | Denial of Service | initTokenDb migration | mitigate | Each ALTER is PRAGMA-guarded + wrapped in try/catch with a non-fatal stderr warning; a failed/locked ALTER never aborts startup or throws into proxy init. |
| T-68-01-03 | Denial of Service | logCall hot path | mitigate | logCall stays best-effort: the existing top-level try/catch swallows any insert/bind error so a malformed new-field value cannot fail the LLM request. |
| T-68-01-SC | Tampering | npm installs | accept | No new npm/pip/cargo packages are introduced (only better-sqlite3 already present); no package-legitimacy checkpoint required. |
</threat_model>
