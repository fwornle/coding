<objective>
Plan 44-13 (commit `b58616713`) cut ObservationWriter's WRITE path to km-core only. Plan 44-14 (commit `16360d48c`) cut obs-api's READ + 14 endpoint surface to km-core. But ObservationConsolidator (3456 lines, 47 SQLite call sites, 19 FROM queries, 3 INSERTs) was DEFERRED.

The divergence has user-visible impact: today's gap-recovery backfill (commit `df4622b87`) added 44 obs to km-core that the consolidator literally cannot see, because the consolidator's query path goes through `.observations/observations.db` SQLite. New digests + insights pipelines stop fully reflecting reality.

Today's bridge script (`scripts/bridge-obs-from-kmcore.mjs`) is a band-aid: it mirrors missing km-core obs into SQLite so the consolidator can pick them up. That works for one-shot recovery but is the wrong architecture going forward — every km-core write would need a parallel SQLite mirror to avoid divergence. This plan eliminates the band-aid by cutting the consolidator itself to km-core.

After this plan + 44-13 + 44-14, **the SQLite file `.observations/observations.db` is fully unused** and can be archived (the deferred Plan 44-12 § "fully unused observations.db" claim finally becomes true).

`autonomous: false` because Task N (operator-checkpoint) gates the archive of the SQLite file and the validation that no other consumer is left holding a handle to it.

## Migration mapping

| SQLite query pattern | km-core replacement |
|----------------------|---------------------|
| `SELECT ... FROM observations WHERE date(created_at)=? AND agent=?` | `findByOntologyClass('Observation', { predicate: e => e.metadata?.created_at?.startsWith(dayISO) && e.metadata?.agent === agent })` — or new bounded helper |
| `SELECT COUNT(*) FROM observations WHERE digested_at IS NULL` | `countByOntologyClass('Observation', { predicate: e => !e.metadata?.digested_at })` |
| `UPDATE observations SET digested_at=? WHERE id IN (...)` | `kmStore.mergeAttributes(obsId, { metadata: { ...meta, digested_at: nowISO }})` — OR `addRelation(digestId, 'digests', obsId)` for relational tracking (decide in Task 1) |
| `INSERT INTO digests (...) VALUES (...)` | `kmStore.putEntity({ id, entityType:'Digest', ontologyClass:'Digest', ... })` |
| `INSERT INTO insights (...) VALUES (...)` | `kmStore.putEntity({ id, entityType:'Insight', ontologyClass:'Insight', ... })` |

## Idempotency design — TWO options to decide in Task 1

**Option A**: Add `metadata.digested_at` to Observation entities. Consolidator filters on it. Pure metadata; no graph relations.

**Option B**: Add `Digest --digests--> Observation` graph relation. Consolidator filters Observations that have no inbound `digests` relation. Cleaner provenance; more graph traversal cost.

**Recommendation**: Option A. Lower-cost, mirrors the existing SQLite column 1:1, easier to backfill (the bridge script could even pre-stamp). Option B is a Phase 45 graph-viewer concern.

## What this plan does NOT touch

- The legacy `/api/observations|digests|insights` endpoints — already cut in Plan 44-14
- The dashboard at :3032 — already reads from km-core via typed views
- ETM's writer path — already cut in Plan 44-13
- The snapshot/restore path — already cut in Plan 44-15
- The OKM (system C) consolidator — out of scope (system C has its own pipeline)

After this plan, the **only** intentional remaining SQLite consumer is... none. The file can be archived.
