<objective>
A-1 architectural close-out: cut the WRITE path of A's observation pipeline from
SQLite to km-core in lock-step with Plan 44-07's read-path cutover. The
architectural gap surfaced during the Plan 44-10 catchup-sync (documented in
`44-CONTEXT-amendment-2.md`): the dashboard reads km-core but
`src/live-logging/ObservationWriter.js` keeps writing to SQLite, so new
observations are invisible until a manual `--resume` migration runs. This
plan closes the gap.

Output: ObservationWriter writes directly to km-core via a shared adapter
(`lib/km-core/src/adapters/legacy-ingest.ts`) that the migration script also
imports â€” eliminates the duplicated field-map maintenance between the two
ingest paths. After this plan, the dashboard shows new observations within
~2s of ETM writing them; the legacy SQLite file is unreferenced.

Per CONTEXT R-4: hard cutover, no dual-write window, no feature flag. SQLite
write code is REMOVED in the same commit the km-core write code is added.

This plan is `autonomous: false` because the cutover affects the live
ObservationWriter the operator depends on for transcript capture. Tasks 1+2
land the adapter + writer changes; Task 3 is the human-verify checkpoint
where the operator (1) restarts obs-api on the new code, (2) confirms via
the dashboard that new observations appear in real-time, and (3) archives
`.observations/observations.db` to the backup directory.
</objective>
