## New km-core API: countByOntologyClass

```ts
// lib/km-core/src/store/GraphKMStore.ts

/**
 * Efficient count over the ontology-class index without materializing
 * the entity array. Default uses the index size; with `predicate`,
 * walks and counts. Used by dashboard COUNT queries that don't need
 * the row data, only the cardinality.
 */
async countByOntologyClass(
  ontologyClass: string,
  opts?: { predicate?: (e: Entity) => boolean }
): Promise<number>;
```

Cost model: O(1) without predicate (index size lookup); O(N_class) with
predicate. The dashboard staleness-clock case (`MAX(createdAt)`) is a
separate helper at Task 1 — `lastModifiedByClass(class)` returns the
max-createdAt ISO across the ontology-class index.

## Replay-style Artifacts patch

Both server endpoints (`/patch-artifacts/recent` and
`/patch-artifacts/historical`) use the same mutation logic the writer's
`_maybePatchArtifacts` uses in Plan 44-13. To avoid duplication, factor
the logic into a small util:

```js
// scripts/lib/artifacts-patch-util.mjs (NEW)

export function patchArtifactsInPlace(entity, modifiedFiles) {
  const oldSummary = entity.properties?.summary || '';
  if (!/Artifacts:\s*none/i.test(oldSummary)) return false;
  const artifactsList = modifiedFiles.map(f => `edited ${f.split('/').pop()}`).join(', ');
  entity.properties.summary = oldSummary.replace(/Artifacts:\s*none/i, `Artifacts: ${artifactsList}`);
  const meta = typeof entity.properties.metadata === 'string'
    ? JSON.parse(entity.properties.metadata || '{}')
    : (entity.properties.metadata || {});
  meta.modifiedFiles = Array.from(new Set([...(meta.modifiedFiles || []), ...modifiedFiles]));
  entity.properties.metadata = JSON.stringify(meta);
  return true;
}
```

The writer's `_maybePatchArtifacts` in Plan 44-13 also calls this util
(Plan 44-13 Task 2 step 4 — small follow-up nit). Single source of
truth for the patch shape.

## Dashboard COUNTs — what stays, what moves

`GET /api/consolidation/status` (line 857) returns a mixed payload:
some stats are pure entity-class counts (totalObs, totalDigests,
totalInsights), others are consolidator-pipeline state (undigested,
pendingPast, pendingToday, lowQuality — these depend on the
`digested_at` column on `observations` rows, which is consolidator
bookkeeping).

In this plan:
- totalObs, totalDigests, totalInsights → km-core
  `countByOntologyClass`
- undigested/pendingPast/pendingToday/lowQuality → stay on the
  consolidator's `this.db` (deferred to Plan 44-15)
- lastObservationAt/lastDigestAt/lastInsightAt → km-core
  `lastModifiedByClass` helper (Task 1)

The endpoint signature stays identical — consumers (dashboard +
health-coordinator [📚] statusline badge) see the same JSON shape.
