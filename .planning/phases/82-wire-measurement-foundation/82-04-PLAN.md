<objective>
Upgrade `insertTokenRowDeduped` in the coding-side token DB from first-writer-wins to merge-on-cache. Today a cache-less tap row permanently shadows the richer `cladpt` transcript row (the exact root cause of the claude-cells-unrouted workaround this milestone removes).

Purpose: With Plan 02 the /v1/messages tap now writes cache data, but the stop-time transcript adapter (cladpt) and the tap can produce the same `(user_hash, tool_call_id)` key. First-writer-wins drops whichever arrives second — losing cache/reasoning fidelity. This plan makes a dedup HIT ENRICH a cache-less existing row in place instead of dropping the incoming richer row, while still dropping a genuinely duplicate row (no double-count).
Output: An in-place merge branch in `insertTokenRowDeduped` (coding repo) + a unit test covering positive merge, negative drop, and the empty-tool_call_id guard.
</objective>
