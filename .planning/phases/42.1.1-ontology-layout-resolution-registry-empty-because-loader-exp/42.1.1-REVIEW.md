# Phase 42.1.1: Code Review Report

**Reviewed:** 2026-05-24
**Depth:** standard
**Files Reviewed:** 5
**Status:** issues_found

## Summary

Phase 42.1.1 adds a layout-tolerant `ontologyPathResolver` and rewires
`OntologyConfigManager.validatePaths()` + `injectOntology()` to use it. The
resolver itself is small, single-purpose, and well-covered by tests. The
canonical alias table matches CONTEXT.md exactly, the `path.basename(team)`
sanitisation correctly blocks the documented T-42.1.1-01 traversal vector, and
the ancestor walk terminates safely at filesystem root.

The main concerns surface at the integration boundary:

- One Critical: `injectOntology()` rewrites `this.config.{upper,lower}OntologyPath`
  to the resolver-canonical path but does not re-register file watchers, so a
  runtime ontology swap can leave the polling watcher tracking the old
  (possibly non-existent two-tier) path forever — silent on-disk update misses.
  This existed pre-phase as a latent issue but is GUARANTEED to misbehave now
  because the whole point of the resolver is that the resolved path *differs*
  from what callers pass in.
- Several Warnings: error-type information loss in `injectOntology`'s re-throw
  path, misleading `type: 'upper'` event when only lower/team changed,
  inconsistent emitted event prefixes vs. error grep contract, no
  cache-invalidation hook when files are deleted, redundant `team` in the
  upper-kind cache key, `getStatus()` return type doesn't match nullable
  `upperOntologyPath`.
- A few Info-level cleanups (unused import in resolver test, redundant
  re-export, jsdoc/code drift comments).

No security defects beyond the team-sanitisation that is already in place. No
secrets, no injection vectors, no path traversal that the basename normalisation
doesn't already block.

## Critical Issues

### CR-01: `injectOntology()` rewrites config paths but does not re-register file watchers (BLOCKER)

**File:** `src/ontology/OntologyConfigManager.ts:340-379`

**Issue:**
When `injectOntology()` runs through the new resolver, lines 350 and 369 reassign
`this.config.upperOntologyPath` / `this.config.lowerOntologyPath` to
`layout.resolvedPath` — the canonical absolute path the resolver picked. But
the existing file watchers in `this.watchers` are keyed by the PREVIOUS resolved
path (set during `startWatching()` after `initialize()`). The injection path
never calls `stopWatchingFile(oldPath)` / `watchFile(newPath, ...)`, so:

1. The old watcher polls a path that may not exist anymore (or may be the wrong
   layout) — its inner `fs.statSync` swallows ENOENT silently in the empty
   `catch (err)` at line 470, so failures are invisible.
2. The new resolved path is never watched — runtime edits to the real on-disk
   ontology file never fire `ontologyChanged` events.
3. Hot-reload silently breaks for any ontology that was injected at runtime.

Before this phase the bug was latent because `injectOntology` assigned the
caller's path verbatim, and `watchFile` already gated on `this.watchers.has()`
to avoid double-registration on the same path — so the same path was watched
once and stayed correct. The whole *point* of the new resolver is that the
resolved path differs from what the caller passed (two-tier in, flat out), so
the desync is now the rule, not the exception.

**Fix:**
```typescript
async injectOntology(options: { ... }): Promise<void> {
  const previousConfig = { ...this.config };
  const previousUpper = this.config.upperOntologyPath;
  const previousLower = this.config.lowerOntologyPath;

  if (options.upperOntologyPath) {
    try {
      const layout = resolveOntologyPath({ kind: 'upper', team: options.team ?? this.config.team, configHint: options.upperOntologyPath });
      this.config.upperOntologyPath = layout.resolvedPath;
      process.stderr.write(`[OntologyConfigManager] upper ontology injected: ...\n`);
    } catch (err) { /* ... */ }
  }
  // (same for lower)

  // Re-wire watchers when paths actually changed
  if (this.config.hotReload) {
    if (previousUpper !== this.config.upperOntologyPath) {
      this.stopWatchingFile(previousUpper);
      this.watchFile(this.config.upperOntologyPath, () => this.emit('ontologyChanged', { type: 'upper', timestamp: new Date() }));
    }
    if (this.config.lowerOntologyPath && previousLower !== this.config.lowerOntologyPath) {
      if (previousLower) this.stopWatchingFile(previousLower);
      this.watchFile(this.config.lowerOntologyPath, () => this.emit('ontologyChanged', { type: 'lower', team: this.config.team, timestamp: new Date() }));
    }
  }

  // (rest unchanged)
}
```

## Warnings

### WR-01: `injectOntology()` loses `OntologyPathNotFoundError` type on re-throw

**File:** `src/ontology/OntologyConfigManager.ts:355-358, 374-377`

**Issue:**
The catch arms wrap the resolver's structured error in a plain `Error`:
```typescript
throw new Error(`Upper ontology not found: ${err.message}`);
```
Callers can no longer `instanceof OntologyPathNotFoundError`, lose the
`kind` / `team` / `probedPaths` fields, and lose the original stack. The
"`Upper ontology not found:` prefix" mentioned in the inline comment is an
asserted contract, but the planner already exports `OntologyPathNotFoundError`
through the barrel — wrapping defeats the structured-error story.

**Fix:**
Use the ES2022 `cause` option and a thin subclass (or just re-throw the original):
```typescript
} catch (err) {
  if (err instanceof OntologyPathNotFoundError) {
    const wrapped = new Error(`Upper ontology not found: ${err.message}`, { cause: err });
    (wrapped as any).probedPaths = err.probedPaths;
    (wrapped as any).kind = err.kind;
    throw wrapped;
  }
  throw err;
}
```
Or — preferred — re-throw the original `OntologyPathNotFoundError` and let
callers grep on `err.message`; the prefix is informational, not load-bearing.

### WR-02: `injectOntology()` emits `type: 'upper'` even when only lower/team/validationMode changed

**File:** `src/ontology/OntologyConfigManager.ts:389-395`

**Issue:**
```typescript
this.emit('ontologyInjected', {
  type: 'upper',  // <-- always 'upper'
  previousValue: previousConfig,
  newValue: this.config,
  timestamp: new Date(),
} as ConfigChangeEvent);
```
The hard-coded `type: 'upper'` is wrong when the caller only passed
`lowerOntologyPath` / `team` / `validationMode`. Downstream listeners that
discriminate on `type` will route the event incorrectly. This existed before
the phase, but the phase doubled the surface area touching this method, so it
is in scope for the rewire.

**Fix:**
Derive `type` from what actually changed, or emit two distinct events:
```typescript
const changed: ('upper' | 'lower' | 'team')[] = [];
if (options.upperOntologyPath) changed.push('upper');
if (options.lowerOntologyPath) changed.push('lower');
if (options.team) changed.push('team');
for (const t of changed) {
  this.emit('ontologyInjected', { type: t, previousValue: previousConfig, newValue: this.config, timestamp: new Date() } as ConfigChangeEvent);
}
```

### WR-03: Switching `team` without paths leaves stale `lowerOntologyPath`

**File:** `src/ontology/OntologyConfigManager.ts:362-383`

**Issue:**
If a caller invokes `injectOntology({ team: 'newteam' })` with no
`lowerOntologyPath`, the team is updated (line 382) but
`this.config.lowerOntologyPath` still points at the previous team's resolved
JSON. Subsequent reads (e.g. `getStatus()`, `getConfig()`) report a
team/path mismatch. The hot-reload watcher continues polling the old team's
file.

**Fix:**
When `options.team` changes the team but no `lowerOntologyPath` is supplied,
either (a) reject with a clear error, or (b) re-resolve the lower path using
the new team and a default config-hint:
```typescript
if (options.team && !options.lowerOntologyPath) {
  const layout = resolveOntologyPath({ kind: 'lower', team: options.team, configHint: this.config.lowerOntologyPath });
  this.config.lowerOntologyPath = layout.resolvedPath;
}
this.config.team = options.team;
```

### WR-04: Resolver cache is never invalidated when files are deleted/moved on disk

**File:** `src/ontology/ontologyPathResolver.ts:265-282`

**Issue:**
`resolveOntologyPath` caches the first existing probed path indefinitely.
The `OntologyConfigManager` watcher fires `ontologyChanged` events when an
ontology file changes, but the resolver cache is a module-level `Map` external
to the manager — no event hook clears it. If an operator moves
`development-knowledge-ontology.json` from flat to two-tier (or vice versa)
while the process is running, the resolver continues returning the stale
absolute path and the registry silently loads from the wrong location until
the process restarts.

CONTEXT acknowledges unbounded cache growth as acceptable, but does NOT
acknowledge staleness. The two are different threat models.

**Fix:**
Either (a) add the resolver `__clearCache()` call into the
`ontologyChanged` watcher callback in `OntologyConfigManager.watchFile`, or
(b) re-probe via `fs.existsSync(cached.resolvedPath)` before returning a
cache hit and evict on miss:
```typescript
const cached = _cache.get(cacheKey);
if (cached && fs.existsSync(cached.resolvedPath)) {
  return cached;
}
if (cached) _cache.delete(cacheKey);
```
Option (b) costs one extra `probeExists` per cache hit but preserves the
test-9 contract if the probe counter is reset between calls.

### WR-05: Upper-kind cache key includes `team`, creating useless per-team entries

**File:** `src/ontology/ontologyPathResolver.ts:262-264`

**Issue:**
```typescript
const cacheTeam = opts.team ? path.basename(opts.team) : '';
const cacheKey = `${opts.kind}::${cacheTeam}::${ontologyDir}`;
```
For `kind='upper'`, `buildProbeList()` does not use `team` at all (lines
201-221). Including the team in the cache key means
`{kind:'upper', team:'a'}` and `{kind:'upper', team:'b'}` create two cache
entries that resolve to the same path — wasteful and confusing. With 7
teams, the upper-resolution cache holds 7 identical entries instead of 1.

**Fix:**
Omit team from the cache key when kind is `upper`:
```typescript
const cacheTeam = opts.kind === 'lower' && opts.team ? path.basename(opts.team) : '';
const cacheKey = `${opts.kind}::${cacheTeam}::${ontologyDir}`;
```

### WR-06: `getStatus()` return type declares `upperOntologyPath: string` but underlying field can be unset

**File:** `src/ontology/OntologyConfigManager.ts:516-538`

**Issue:**
`getStatus()`'s return type declares `upperOntologyPath: string` (line 519,
not optional) but `validatePaths` only resolves when
`this.config.upperOntologyPath` is truthy (line 146). If a caller constructs
the manager without an upper path (e.g. `enabled: false` test fixture),
`getStatus()` returns an object where the runtime value is `undefined` but
the TypeScript type claims it is `string`. Downstream consumers that call
`.length` or string methods will crash at runtime.

This existed before this phase but was masked because `normalizeConfig`
preserves the input value verbatim — it doesn't enforce the type contract
either. The phase now sets `upperOntologyPath` to a resolved absolute path on
success, which makes the inconsistency more visible to forensic readers.

**Fix:**
Make the field optional in the return type:
```typescript
getStatus(): {
  initialized: boolean;
  enabled: boolean;
  upperOntologyPath?: string;
  lowerOntologyPath?: string;
  // ...
}
```

### WR-07: `Test 9` cache-hit assertion depends on test execution order, not isolation

**File:** `src/ontology/ontologyPathResolver.test.ts:245-265`

**Issue:**
Test 9 calls `__resetProbeCounter()` but never calls `__clearCache()`. Since
each test uses `mkdtempSync` with a unique tmpdir, the cache keys *currently*
do not collide across tests, so Test 9 happens to work. But the contract is
brittle: any future refactor that adds a fixed-path test before Test 9
(e.g. asserting the `fallback` path with no `configHint`) would prime the
cache and break Test 9's "n1 >= 1" assertion silently (`n1` would be 0).

The `__clearCache()` export already exists for exactly this purpose
(line 129 of `ontologyPathResolver.ts`) and is not used by any test.

**Fix:**
Either call `__clearCache()` at the top of Test 9 (or in a `beforeEach`
hook), or document the cross-test cache assumption explicitly in the test
header. The minimal fix:
```typescript
it('Test 9: cache hit — second call adds zero new probes', () => {
  __clearCache();
  __resetProbeCounter();
  // ...rest unchanged
});
```

## Info

### IN-01: Unused `node:fs` import in resolver test

**File:** `src/ontology/ontologyPathResolver.test.ts:25-30`

**Issue:**
`mkdirSync` is imported but only used inside `writeOntology()`. `existsSync`
is imported but only used in Test 6. Both are referenced, so the import is
correct — but `rmSync` is imported at the top and only used in 8 cleanup
lines that all wrap it in a `try`/`catch`-less call. If `rmSync` fails (e.g.
permission), the tmpdir leaks. Not a bug; just a hygiene note. Consider
`t.after(() => rmSync(basePath, { recursive: true, force: true }))` via the
`node:test` context arg so cleanup runs even on assertion failure.
