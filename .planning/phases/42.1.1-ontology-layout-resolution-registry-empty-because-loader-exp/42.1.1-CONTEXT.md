# Phase 42.1.1: Ontology Layout Resolution — Context

**Gathered:** 2026-05-24
**Status:** Ready for planning
**Source:** Inline discuss equivalent (forensic-report-driven; user decisions captured in `/gsd-plan-phase 42.1.1` invocation)

<domain>
## Phase Boundary

Resolve the empty-ontology-registry blocker surfaced by Phase 42.1's SC#6 failure. The blocker
is purely a path-resolution / filesystem-layout mismatch:

- Ontology JSON files live FLAT under `.data/ontologies/*.json` (Phase 38 flattening, never
  reversed).
- Multiple callers + the central validator in `src/ontology/OntologyConfigManager.ts` hard-code
  the two-tier shape `.data/ontologies/{upper,lower}/<file>.json`.
- Result: `validatePaths()` throws "Upper ontology not found …" → registry initialises with
  zero classes for the `coding` team → `ensureProjectAnchor` fails → 68 in-scope orphans + 7
  duplicated L2 Components in production.

This phase fixes the path-resolution layer ONLY. It explicitly does NOT touch:
- The anchor-pass bookkeeping bug (Anomaly 3 of the forensic report) — separate hardening
  ticket, won't reproduce once the registry loads classes.
- The verifier-vs-registry API drift `registry.hasClass is not a function` (Anomaly 5) —
  separate verifier-fix ticket.
- The misleading `docker logs` recipe in Task 7 Step 8 (Anomaly 6) — docs-only update.

Phase 42.1's code (findBestParent + ensureProjectAnchor + post-sweep anchor pass) is correct
and already merged; it just cannot be validated end-to-end until this phase lands.

**Single deliverable:** ontology registry loads the full set of classes for the `coding` team
on the next `wave-analysis` run, against the current flat on-disk layout, without moving any
ontology JSON files.

</domain>

<decisions>
## Implementation Decisions

### Remediation Path (LOCKED — user choice during plan-phase invocation)

- **Path (a) — fix the loader to accept the flat layout.** Smallest blast radius. Loader code
  is updated to find ontology JSONs in BOTH layouts. No file moves, no config edits.
- Explicitly rejected: **Path (b)** (migrate files into `upper/` + `lower/` subdirs) — touches
  on-disk state, requires deployment-time migration, breaks any external consumer that already
  reads the flat layout.
- Explicitly rejected: **Path (c)** (update ontology-paths config) — requires changing per-file
  paths in 4+ caller sites without addressing the underlying brittleness; one new caller would
  re-introduce the bug.

### Scope (LOCKED — user choice during plan-phase invocation)

- Defer the 3 cleanup tickets identified by the forensic report (Anomalies 3, 5, 6). They do
  not block 42.1.1 verification. Anomaly 5 in particular is tempting to include, but keeping
  the phase single-purpose makes pass/fail unambiguous.

### Layout-Resolution Strategy (LOCKED)

- The fix introduces a single shared **path resolver** in `src/ontology/` that:
  1. Accepts a `{team, kind: 'upper'|'lower'}` request.
  2. Probes the two-tier path first (`.data/ontologies/{kind}/<canonical-filename>.json`).
  3. Falls back to the flat path (`.data/ontologies/<canonical-filename>.json`, plus the
     historical aliases `upper.json` for kind=upper and `<team>-ontology.json` for kind=lower).
  4. Returns the first existing absolute path, or throws a clear error listing every path it
     probed so future layout drift surfaces immediately.
- `OntologyConfigManager.validatePaths()` calls the resolver instead of `fs.existsSync` on the
  raw config string. Result: the same `upperOntologyPath` / `lowerOntologyPath` config values
  resolve under either layout — no caller has to know which layout is on disk.
- All 4 known callers (`tools.ts`, `ontology-classification-agent.ts`,
  `insight-generation-agent.ts`, `persistence-agent.ts`) keep their existing path-construction
  code so external behaviour is preserved. The resolver runs inside the loader as a
  transparent layer.

### Filename Aliases the Resolver MUST Recognise (LOCKED)

| Kind  | Two-tier (preferred)                                                  | Flat fallback(s)                              |
| ----- | --------------------------------------------------------------------- | --------------------------------------------- |
| upper | `.data/ontologies/upper/development-knowledge-ontology.json`          | `.data/ontologies/development-knowledge-ontology.json`, then `.data/ontologies/upper.json` |
| lower | `.data/ontologies/lower/<team>-ontology.json`                         | `.data/ontologies/<team>-ontology.json`       |

The `upper.json` flat-fallback alias reflects what is actually on disk today; without it the
fix does not unblock production.

### Verification Boundary (LOCKED)

- Unit test: resolver returns correct absolute path for every combination of {layout x kind x
  team} the project currently uses. Add fixtures for both layouts under `tests/fixtures/`.
- Integration test: `OntologyConfigManager.initialize()` succeeds against the real flat layout
  in `.data/ontologies/` and reports non-zero class count for team `coding`.
- E2E smoke test: a one-wave `wave-analysis` dry run (or equivalent harness) shows
  `ensureProjectAnchor` minting the `Coding` Project node with no "Unknown ontology class:
  Project" error.
- SC#6 pass for Phase 42.1 (re-run its verifier) is the **promotion gate**, not a 42.1.1
  acceptance gate — keep them separate so 42.1.1 can land independently.

### Claude's Discretion

- Resolver function signature, file location within `src/ontology/`, internal caching of probe
  results, log message wording. Default to the smallest, most idiomatic shape that fits the
  existing module conventions.
- Whether the resolver emits a structured `layoutDetected: 'two-tier' | 'flat'` field on
  startup so operators can see which path served the registry. Recommended, not mandatory.
- Test framework choice should match the surrounding module conventions (likely Vitest based
  on `lib/km-core`).

</decisions>

<specifics>
## Specific Ideas

- The resolver's "no path found" error message MUST list every probed path so future ontology
  layout drift produces an immediately actionable error (Phase 42.1 spent half a day chasing
  the symptom because the original error was just "Upper ontology not found: <one path>").
- The resolver SHOULD log the resolved layout once at startup (`upper: flat fallback
  matched .data/ontologies/upper.json`) so post-incident forensics can confirm which layout
  the registry loaded from.
- The integration test must use the **actual** `.data/ontologies/` directory (or a fixture
  copy of it), not a synthetic two-tier mock. Mock-based tests would have passed during
  Phase 42.1 and missed the bug entirely.

</specifics>
