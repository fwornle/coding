<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `.data/ontologies/coding.display.json` → km-core in-memory overlay | overlay JSON is read at obs-api boot via `loadDisplayOverlay()`. A malicious overlay file could inject arbitrary strings into the wire response. Bounded by Zod validation on load (Task 1). |
| km-core `/api/v1/ontology/classes?withDisplay=true` → unified-viewer frontend | display field values flow into the SVG node renderer. Color/icon strings are NOT executed as code; shape/borderStyle/pulseRule are enum-validated. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-55-02-01 | Tampering | overlay JSON could specify unknown `pulseRule` literals → renderer mis-behavior | mitigate | Zod `z.union([z.null(), z.literal(...), z.literal(...), z.literal(...)])` on load (Task 1); test rejects `lastUpdatedWithin:90s` |
| T-55-02-02 | Information disclosure | enriched response contains all overlay entries even if frontend doesn't render them | accept | overlay is per-system; coding.display.json contains only coding-tab classes; no cross-system data leak |
| T-55-02-03 | Spoofing | malicious icon string contains script tags | mitigate | icons render via React text node (escaped by default), NOT `dangerouslySetInnerHTML` (matches Phase 45 Plan 04 XSS-clean port) |

Package legitimacy: no new packages.
</threat_model>
