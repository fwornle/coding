Create `src/agents/wave2-component-agent.ts` -- the Wave 2 agent that receives an L1 parent and produces L2 SubComponent entities. See RESEARCH.md Pattern 2 for the skeleton.

Per CONTEXT.md locked decisions:
- Wave 2 starts from **manifest-defined L2 sub-components** but agents can **discover additional sub-nodes** from code-graph-rag analysis
- Wave 2 agents each read files belonging to their assigned L1 component, scoped via **code-graph-rag queries**
- Discovered entities flagged with a 'discovered' flag/tag
- Each wave agent produces: entity node + observations + parent-child relationships
- Fresh agent instances per wave -- no cross-wave state contamination

**Class structure:**

```typescript
import * as fs from 'fs';
import * as path from 'path';
import { log } from '../logging.js';
import { LLMService } from '../../../../lib/llm/dist/index.js';
import { isMockLLMEnabled, getMockDelay } from '../mock/llm-mock-service.js';
import type { KGEntity, KGRelation } from './kg-operators.js';
import type { Wave2Input, WaveAgentOutput, ChildManifestEntry } from '../types/wave-types.js';

export class Wave2ComponentAgent {
  private repositoryPath: string;
  private team: string;
  private llmService: LLMService;
  private llmInitialized: boolean = false;

  constructor(repositoryPath: string, team: string) { ... }

  async execute(input: Wave2Input): Promise<WaveAgentOutput> { ... }

  private buildL2Entity(name: string, description: string, observations: string[], parentEntity: KGEntity, discovered: boolean): KGEntity { ... }
  private buildRelationships(l2Entities: KGEntity[], parentEntity: KGEntity): KGRelation[] { ... }
  private async identifyChildNodes(fileContents: string[], parentEntity: KGEntity): Promise<ChildManifestEntry[]> { ... }
  private async readComponentFiles(files: string[], maxFiles: number): Promise<string> { ... }
}
```

**execute() method flow:**

1. **Log start**: `[Wave2Agent] Starting analysis for ${input.l1Entity.name}`
2. **Read component files**: Read up to 8-10 files from `input.componentFiles` (already scoped by WaveController via CGR). Truncate each to 200 lines. If `componentFiles` is empty, fall back to scanning directories matching component keywords.
3. **Process manifest-defined L2 children** from `input.manifestChildren`:
   - For each manifest child, ask LLM for a detailed description and observations specific to that sub-component
   - Create L2 entity with `discovered: false`
4. **Discover additional L2 children** via LLM:
   - Send file contents + parent context to LLM asking "What other significant sub-components exist within {componentName} that are NOT already in this list: {manifestChildNames}?"
   - For each discovered child, create L2 entity with `discovered: true`
5. **Build entities**: For each L2 child (manifest + discovered):
   - name: child name (PascalCase)
   - type: "SubComponent"
   - level: 2
   - parentId: input.l1Entity.name
   - hierarchyPath: `${input.l1Entity.hierarchyPath}/${childName}`
   - observations: 3-7 specific observations from LLM analysis
   - significance: 7
6. **Build relationships**: parent-child edges from L1 parent to each L2 entity
7. **Identify L3 children**: For each L2 entity, ask LLM what Detail-level nodes should exist. Build ChildManifestEntry[] for Wave 3.
8. **Return WaveAgentOutput** with all L2 entities, relationships, and child manifest for Wave 3.

**LLM prompt for L2 analysis:**
```
You are analyzing the {l1Entity.name} component to identify its sub-components (L2 nodes).

## Parent Component
Name: {l1Entity.name}
Description: {l1Entity.observations[0]}

## Known Sub-Components (from manifest)
{manifestChildren.map(c => `- ${c.name}: ${c.description}`).join('\n')}

## Source Files
{fileContents}

## Task
1. For each known sub-component, provide:
   - 3-7 specific observations about its architecture, patterns, and behavior
   - These must be specific to THIS sub-component, not generic statements

2. Identify additional sub-components NOT in the known list:
   - Only suggest sub-components that represent real, distinct architectural areas
   - Each must have clear file/directory evidence in the source code

3. For ALL sub-components (known + discovered), suggest what Detail-level (L3) entities exist within them.

## Output (JSON)
{
  "subComponents": [
    {
      "name": "...",
      "description": "...",
      "observations": ["...", "..."],
      "discovered": false,
      "suggestedL3Children": [
        { "name": "...", "description": "..." }
      ]
    }
  ]
}
```

**CRITICAL field mapping**: Ensure all L2 entities have:

- `level: 2`
- `parentId: input.l1Entity.name`
- `hierarchyPath` includes full path from project root

Create `src/agents/wave3-detail-agent.ts` -- the Wave 3 agent that produces L3 Detail entities. This is the simplest agent: it receives an L2 parent and discovers Detail-level knowledge nodes via pure LLM analysis.

Per CONTEXT.md locked decisions:
- Wave 3 is **entirely LLM-driven** -- agents analyze code within their L2 scope and identify Detail-level entities
- No manifest seeds for L3 (pure discovery)
- Discovered entities flagged
- Wave 3 agents read files relevant to their assigned L2 sub-component via code-graph-rag queries

**Class structure:**

```typescript

import * as path from 'path';

  private repositoryPath: string;
  private team: string;
  private llmService: LLMService;
  private llmInitialized: boolean = false;

  constructor(repositoryPath: string, team: string) { ... }

  async execute(input: Wave3Input): Promise<WaveAgentOutput> { ... }

  private buildL3Entity(name: string, description: string, observations: string[], l2Entity: KGEntity, l1Entity: KGEntity): KGEntity { ... }

```

**execute() method flow:**

1. **Log start**: `[Wave3Agent] Starting detail analysis for ${input.l2Entity.name}`
2. **Read scoped files**: Read up to 5-8 files from `input.scopedFiles`. If empty, log a warning and return a minimal output (some L2 nodes may have no specific files).
3. **Call LLM** to identify Detail-level entities within this L2 scope:
   - Provide L2 parent context (name, description, observations)
   - Provide L1 grandparent context for broader orientation
   - Provide file contents
   - Ask LLM to identify 2-8 Detail nodes that represent specific implementation aspects, key classes, important patterns, configuration areas, or notable behaviors
4. **Build L3 entities** for each discovered detail:
   - name: detail name (PascalCase, descriptive)
   - type: "Detail"
   - level: 3
   - parentId: input.l2Entity.name
   - hierarchyPath: `${input.l2Entity.hierarchyPath}/${detailName}`
   - observations: 2-5 detailed observations specific to this detail
   - significance: 6
5. **Build relationships**: parent-child edges from L2 to each L3
6. **Return WaveAgentOutput** with L3 entities, relationships, and empty childManifest (Wave 3 is the last wave -- no children to suggest).

**LLM prompt for L3 analysis:**
```
You are analyzing the {l2Entity.name} sub-component to identify its detail-level knowledge nodes.

## Hierarchy Context
Project: Coding
Component (L1): {l1Entity.name} - {l1Entity.observations[0]}
SubComponent (L2): {l2Entity.name} - {l2Entity.observations[0]}

## Source Files
{fileContents}

## Task
Identify 2-8 Detail-level (L3) nodes that represent specific, notable aspects of this sub-component. Good L3 nodes are:
- Specific classes or modules with distinct behavior (e.g., "BatchScheduler", "LLMRetryPolicy")
- Key algorithms or processing patterns (e.g., "StreamingResponseParser", "DAGDependencyResolver")
- Important configuration or integration points (e.g., "ProviderFallbackConfig", "MemgraphConnection")
- Notable architectural decisions or design patterns (e.g., "EventDrivenPipeline", "SharedMemoryStore")

BAD L3 nodes (do NOT suggest these):
- Generic labels like "Configuration", "Utils", "Types", "Helpers"
- Nodes that just restate the L2 name with "Manager" or "Service" suffix
- Nodes with no specific code evidence

For each L3 node, provide:
- name: PascalCase, specific and descriptive
- description: 1-2 sentences about what it does
- observations: 2-5 specific observations about architecture, behavior, or design decisions
  Each observation must reference something visible in the source files.

## Output (JSON)
{
  "details": [
    {
      "name": "...",
      "description": "...",
      "observations": ["...", "..."]
    }
  ]
}
```
