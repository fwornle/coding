- **CR-01 (gap 1) — fuzzy matcher matches non-wire rows.** `FUZZY_CANDIDATES_SQL` (`reconcile.mjs`) selects EVERY same-model `token_usage` row with no restriction to the pre-loop wire snapshot and no exclusion of already-matched rows. `reconcileBatches` computes `wireRowIds` (pre-loop PK snapshot) and `matchedWireRowIds` (matched PKs) for the `unmatched_wire` diff but never passes them into `reconcileRow`, so `fuzzyMatch` can consume a fallback row inserted seconds earlier in the same loop. Failure mode (defeats golden property 2): in a proxy-down cell, turn 1 fallback-inserts; turn 2 fuzzy-matches turn 1's fallback row (same model, within the 2-min window), is reported `matched`, and turn 2's tokens are silently dropped. The wire-verify gate only asserts `summary.fallback > 0`, which turn 1 alone satisfies. Repairs D-02 (fallback loud-not-silent) + D-04/D-05.
- **CR-02 (gap 2) — unmatched_wire structurally 0 for copilot.** `snapshotWireRowIds` queries `WHERE user_hash = 'copadt'`, but copilot BYOK wire rows are written by `/api/complete` -> `logTokenCall` (proxy `server.mjs:2668`) which passes NO `user_hash`, so `logCall` (`token-usage.ts:926`) defaults them to the machine `USER_HASH`. Only the `/v1/messages` Anthropic tap stamps `adapterUserHash(agent)`; copilot traffic never hits `/v1/messages`. The copadt-keyed snapshot is therefore always empty and `unmatched_wire` is vacuously 0 for every copilot span — the same defect class old-CR-02 fixed for claude, via a different route. Repairs D-04 (wire authoritative / meaningful metric) + D-12 (reconciliation.json summary meaningful).

**Chosen fix for CR-02 = Option A (stamp the adapter hash at the proxy source), NOT Option B (task_id-only snapshot on the coding side).** Rationale: Option A fixes the ROOT CAUSE (provenance) so copilot becomes symmetric with claude (both snapshot by adapter hash), keeps the coding side unchanged (lower risk — `snapshotWireRowIds` already queries `adapter.userHash`), and fixes downstream user_hash attribution/exports for Phase 84/86. Option B (coding-side branch: snapshot copilot by `task_id` only) leaves the machine-hash provenance bug latent and makes the snapshot logic asymmetric. Both keep the metric meaningful without double-counting; Option A is the durable correct design and is the primary fix both 83-REVIEW.md and 83-VERIFICATION.md recommend.
