Per D-11: No filename distinction for mastracode LSL files — agent identity is captured inside the file (header/metadata).

Create src/live-logging/MastraTranscriptReader.js that:

1. Extends EventEmitter (same pattern as StreamingTranscriptReader)
2. Watches a mastra transcript file for new content using fs.watch or fs.watchFile
3. Parses the native mastra format:
   - Mastra lifecycle hooks produce events with: type (onStepFinish, onToolCall, etc.), timestamp, content
   - The reader should handle JSON-lines or structured JSON output from mastra hooks
   - Each lifecycle event maps to a transcript entry with role (user/assistant/tool) and content
4. Emits 'message' events with objects shaped like: `{ role: string, content: string, timestamp: Date, metadata: { agent: 'mastra', hookType: string } }`
5. Emits 'exchange' events when a complete user->assistant turn is detected
6. Has a static `extractExchangesFromBatch(messages)` method following StreamingTranscriptReader pattern
7. Handles file rotation (new file created when session restarts)
8. Constructor accepts: `new MastraTranscriptReader(transcriptPath, options)`
   - transcriptPath: path to the mastra transcript file
   - options: { encoding, pollInterval }

The transcript file location should be derived from the mastra plugin config:
- Default: `.observations/transcripts/` directory (alongside LibSQL DB)
- The reader watches for new `.jsonl` or `.ndjson` files in this directory

Export as default: `export default MastraTranscriptReader;`

IMPORTANT: This is a new file. Follow the exact code style of StreamingTranscriptReader.js — same EventEmitter pattern, same logging style, same error handling approach. Use process.stderr.write() for logging (not console.log per project constraints).
