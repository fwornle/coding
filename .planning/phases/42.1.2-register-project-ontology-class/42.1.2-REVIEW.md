# Phase 42.1.2: Code Review Report

**Reviewed:** 2026-05-25
**Depth:** standard
**Files Reviewed:** 3
**Status:** issues_found

## Summary

Phase 42.1.2 registers a new `Project` class in `.data/ontologies/upper.json`, restores the positive `isValidClass('Project')` assertion in the layout test (parameterised across all 7 teams), and adds a `node:test` integration smoke for `WaveController.ensureProjectAnchor`. The submitted code is functionally sound for its stated contract — the JSON is valid, the new class declaration mirrors sibling classes (`File`, `Service`, `Feature`) in shape, the layout test's per-team loop is correctly gated against the only canary class (`Component`) that is genuinely `coding`-team-specific, and the wave-controller smoke faithfully exercises the cold/warm/idempotent paths of `ensureProjectAnchor` without dragging in real km-core. No console.* usage; both tests use `process.stderr.write`. No source production files touched.

Three issues warrant attention before merge:

1. **Ontology gap (WR-01):** `Project` is declared with zero relationships — neither the new `classes.Project.relationships` block nor the `_legacy.relationships` block contains any edge with `Project` as source or target. The description claims "every entity ultimately roots back to a Project", but `part_of` only chains `File → Service → Feature`. Wave-analysis emissions that need to express `Service part_of Project` (or any Project-rooted edge) will silently produce edges the schema does not validate.

2. **Schema drift (WR-02):** The Project class is added only to the new top-level `classes` block, not to `_legacy.entities`. This is internally consistent with the intent to retire `_legacy`, but if any code path still reads `_legacy.entities` (the legacy loader path) for validation, it will report Project as unknown. Plan 01 SUMMARY should explicitly confirm no caller still reads `_legacy.entities`; absent that, this is a latent split-brain risk.

3. **Idempotency-test fragility (WR-03):** The flag-flip in Test 3 of `wave-controller-ensure-project-anchor.test.ts` (`codingMinted` toggled between the two `await ensureProjectAnchor` calls, outside the spy) couples the test to runtime ordering rather than to the warm/cold contract itself. A regression that performed the storeEntity asynchronously after `ensureProjectAnchor` returned would falsely pass.

The Info items below capture documentation and minor style observations that should not block merge.

## Warnings

### WR-01: `Project` class declares zero relationships — anchor edges have no schema home

**File:** `.data/ontologies/upper.json:149-165`
**Issue:** The new `Project` class sets `"relationships": {}` and there is no edge anywhere in the file (active `classes` block has no top-level `relationships` key; `_legacy.relationships` enumerates `part_of`, `fulfills`, `depends_on`, etc. but none references `Project` in either `sourceEntityClass` or `targetEntityClass`). The class description states "Top-level anchor for a team's wave-analysis emissions — every entity ultimately roots back to a Project", which strongly implies a `part_of` (or analogous) edge with `Project` as a valid target. Without that edge declared, any wave-analysis output that asserts `Service part_of Project` (or `Feature part_of Project`) is emitting an undeclared relation. If downstream validation is strict, those edges will be dropped; if it is lenient, they will pass through unvalidated — both outcomes contradict the class's stated role.

The wave-controller body that motivates this phase (`ensureProjectAnchor` at `wave-controller.ts:2262`) mints the Project node but the *anchor pass* that follows it (referenced at `wave-controller.ts:2256-2257` — "the loop can rely on `findBestParent` returning 'Coding' as a fallback") will produce edges INTO the Project node. Those edges need a schema entry.

**Fix:** Extend `_legacy.relationships.part_of.targetEntityClass` to include `Project` (and source to include `Feature`), or add a dedicated `anchored_in` / `rooted_in` relation:

```json
"part_of": {
  "facet": "structure",
  "description": "Composition hierarchy: File → Service → Feature → Project",
  "sourceEntityClass": ["File", "Service", "Feature"],
  "targetEntityClass": ["Service", "Feature", "Project"],
  "cardinality": "many-to-one"
}
```

Alternatively, add a new edge purpose-built for the anchor relationship and document it in the Project class's `relationships` block. Either way, the gap should be closed in this phase since the test suite (Test C in `OntologyConfigManager.layout.test.ts`) only asserts that `Project` is a *valid class*, not that it can participate in any *valid relation* — the gap will not be caught by current tests.

### WR-02: `Project` added to active `classes` block but absent from `_legacy.entities` — split-brain risk

**File:** `.data/ontologies/upper.json:149-165` and `.data/ontologies/upper.json:504-1075`
**Issue:** The file carries two parallel definitions: the new `classes` block (lines 7-490) and a `_legacy` block (lines 491-1246) that retains the older `entities` map with `requiredProperties` and `examples` keys. The Project class is added only to `classes`; `_legacy.entities` is unchanged. If any consumer still parses `_legacy.entities` (for validation, documentation generation, or fallback), it will continue to report `Project` as an unknown class — silently breaking the Phase 42.1.2 layer-2 closure for that consumer.

The phase plan and CONTEXT.md presumably establish that `_legacy.entities` is dead, but the *evidence* for that claim is not visible in the diff. A grep across the repo for `_legacy` keys consumed by ontology-loading code would be the safest gate.

**Fix:** Either (a) verify no code path reads `_legacy.entities` and add an `OVERRIDE_LEGACY_DEAD` comment in `upper.json` documenting that the legacy block is documentation-only, or (b) add the `Project` entry to `_legacy.entities` for parity, using the legacy shape:

```json
"Project": {
  "facet": "structure",
  "description": "Top-level anchor for a team's wave-analysis emissions...",
  "properties": { ... },
  "requiredProperties": ["name"],
  "examples": [
    "Coding (the canonical anchor for team=coding wave-analysis runs)",
    "RaaS (anchor for team=raas)",
    "Agentic (anchor for team=agentic)"
  ]
}
```

Option (b) is the safer default until a forensic grep confirms `_legacy.entities` is unread.

### WR-03: Idempotency test couples to ordering instead of asserting the contract directly

**File:** `integrations/mcp-server-semantic-analysis/src/agents/wave-controller-ensure-project-anchor.test.ts:259-309`
**Issue:** Test 3 ("idempotency") flips a closure variable (`codingMinted`) between the two `ensureProjectAnchor` invocations, using `harness.storeEntityCalls.length > originalLength` as the proxy for "did the first call mint?". This works for the current synchronous-storeEntity implementation but is fragile against two plausible regressions:

1. A future ensureProjectAnchor that schedules the storeEntity via `setImmediate` / `queueMicrotask` and returns early would leave `storeEntityCalls.length === 0` at the post-await check, falsely setting `codingMinted = false` — and the second call would *also* mint, producing 2 storeEntity calls. The final `assert.equal(storeEntityCalls.length, 1)` would then *fail* — surfacing the regression — but the failure message ("idempotency: storeEntity called exactly once across two ensureProjectAnchor invocations") would mislead the reader because the *real* regression is the flag never flipped, not a contract violation.

2. If `originalLength` is captured before the harness is fully wired, the comparison becomes meaningless. (`buildHarness` returns `storeEntityCalls` as an empty array, so `originalLength === 0` always — the conditional is currently equivalent to `if (storeEntityCalls.length > 0)`.)

**Fix:** Move the flag-flip *inside* the fake adapter's `storeEntity` stub so the warm-vs-cold state machine lives with the adapter:

```ts
let codingMinted = false;
const fakeAdapter = {
  async queryEntities(filter: QueryFilter = {}): Promise<StoredEntity[]> {
    queryEntitiesCalls.push({ filter });
    return codingMinted ? [{ name: 'Coding', entityType: 'Project' }] : [];
  },
  async storeEntity(entity, options) {
    storeEntityCalls.push({ entity, opts: options });
    codingMinted = true;  // flip atomically with the mint
    return { id: `fake-id-${storeEntityCalls.length}` };
  },
};
```

Then drop the `originalLength`/conditional bookkeeping in the test body — both invocations just `await harness.ensureProjectAnchor(...)` back-to-back. This couples the warm-state-after-mint observable to the *same* operation that produces it, which is what the contract actually says.

### IN-01: Repo-root walk depth is silently load-bearing across Phase 42.1.1 → 42.1.2

**File:** `src/ontology/OntologyConfigManager.layout.test.ts:37`
**Issue:** `REPO_ROOT = resolve(__dirname, '..', '..', '..', '..')` assumes the test runs from `dist/ontology/` four levels below repo root. The file header (lines 32-34) acknowledges this with "FOUR `..` segments. Two segments resolves to dist/.data/ontologies which does not exist (silent-pass regression guard)" — the `before()` sanity gate at lines 63-77 protects against the silent-pass failure mode. Good defense.

However, the sanity gate asserts existence of `upper.json` and `coding-ontology.json` but does NOT assert the `Project` class is actually declared in `upper.json`. If a future ontology refactor removed the `Project` class from `upper.json` but kept the file, Test C would fail with `isValidClass('Project') === false` — but the failure would point at the registry rather than the ontology file. A `before()` line like:

```ts
const upper = JSON.parse(readFileSync(join(ONTOLOGY_DIR, 'upper.json'), 'utf-8'));
assert.ok(
  upper.classes?.Project,
  `sanity-gate: upper.json must declare 'Project' class (Phase 42.1.2 invariant)`,
);
```

would isolate the file-state failure from the registry-loading failure. Cheap to add.

**Fix:** Optional — add the Project-class existence assertion to the `before()` block.

### IN-03: Test C's per-team registry construction is N+1 — consider documenting the cost

**File:** `src/ontology/OntologyConfigManager.layout.test.ts:190-245`
**Issue:** Test C runs `resetInstance` → `getInstance` → `initialize` → `new OntologyRegistry({ ontologyDir })` for each of 7 teams. Each `initialize()` reads and validates the upper + lower JSON from disk, and each registry construction re-parses them. Total: 14 file reads + 7 registry constructions per test run. Not a correctness issue — but for a CI signal that's expected to run on every PR touching the ontology, this is the kind of cost that creeps up. Worth a brief comment in the test header noting the per-team I/O so a future contributor doesn't add an 8th team without considering test latency.

Performance is explicitly out of v1 scope, so flagging this as Info only. Skip if not concerned.

**Fix:** Optional — add to the test header comment block:

```
COST NOTE: Each TEAMS iteration performs 2 file reads + 1 registry construction.
Currently 7 teams × ~50ms/team ≈ 350ms. Acceptable for CI but watch for growth.
```

### IN-04: Description em-dash uses `—` (U+2014) — verify upstream consumers handle UTF-8

**File:** `.data/ontologies/upper.json:150, 155, 161`
**Issue:** The new `Project` class description contains three em-dashes (U+2014) and one Unicode apostrophe in `team's`. Existing classes in `upper.json` already use em-dashes liberally (e.g., line 9 for `File`, line 167 for `Contract`), so this is consistent with prior art and not a *new* risk. Flagging only because if any tool serialises the ontology to ASCII-only contexts (e.g., legacy LLM prompts with strict tokenisation), the em-dashes will be re-encoded.

**Fix:** None required — purely informational. The constraint system's `no-evolutionary-names` and similar regex-based checks should be UTF-8 safe; verify if a constraint trips.
