**Gathered:** 2026-05-24

<domain>
## Phase Boundary

Add a `Project` class definition to the on-disk ontology JSON so that `km-core`
`OntologyRegistry.isValidClass('Project')` returns `true` for every team after
`OntologyConfigManager.initialize()`. This unblocks `ensureProjectAnchor('Coding')` at
runtime (`integrations/mcp-server-semantic-analysis/src/agents/wave-controller.ts:2262`),
which stores an entity with `entityType: 'Project'` and currently throws because no
ontology declares the `Project` class.

This is **layer 2 of the SC#6 root cause** surfaced by Phase 42.1.1 Test C (see
`42.1.1-01-SUMMARY.md` § Known Residuals). Layer 1 — loader path resolution — landed in
42.1.1; the registry now loads all 8 ontology domains for the `coding` team but none of
them declares `Project`.

**Single deliverable:** `Project` class registered in `.data/ontologies/upper.json` so
every team's km-core registry recognises it as a valid class, plus a smoke/integration test
that proves `ensureProjectAnchor` no longer throws.

**Explicitly NOT in scope:**
- The broader hierarchical knowledge restructuring (Project → Component → SubComponent →
  Detail with declared edges + properties) — MEMORY.md flags that as its own future project.
- Adding `contains` / `owns` / other outgoing relationships from `Project` — anchor-only is
  sufficient for `ensureProjectAnchor`.
- Per-team Project subtypes (CodingProject extends Project, etc.) — not needed today.
- Phase 42.1 SC#6 wave-analysis re-run — that is Phase 42.1's promotion gate, not this
  phase's acceptance gate (continues the 42.1.1 boundary at CONTEXT.md line 93-94).

</domain>

<decisions>
## Implementation Decisions

### Target File (LOCKED)

- **Add `Project` class to `.data/ontologies/upper.json`** — system-level definition.
- Rationale: `Project` is a universal concept; every team's wave-analysis emissions need
  an anchor. coding-ontology.json (and every other `<team>-ontology.json`) implicitly
  `extends: "upper"` (see coding-ontology.json `meta.extends`), so adding the class once
  in upper.json makes it visible to all 7 teams (`agentic`, `cluster-reprocessing`,
  `code-entities`, `coding`, `raas`, `resi`, `ui`) on the next registry load — zero
  per-team duplication, zero drift risk.
- Explicitly rejected:
  - **coding-ontology.json only** — would scope Project to one team; other teams would
    need their own Project class added later (drift + duplication risk).
  - **Both (upper minimal + coding-ontology CodingProject extends Project)** — premature
    abstraction; we have no current need for per-team Project subtypes.

### Class Shape (LOCKED)

- **Standard shape: `name` + `repoRoot` with `facet: "structure"` annotations.**

```json
"Project": {
  "description": "Top-level anchor for a team's wave-analysis emissions — every entity ultimately roots back to a Project. Minted by ensureProjectAnchor when the registry first sees a team's wave-analysis run.",
  "relationships": {},
  "properties": {
    "name": {
      "type": "string",
      "description": "Project name (PascalCase or single-word — e.g. 'Coding', 'RaaS', 'Agentic')",
      "required": true,
      "facet": "structure"
    },
    "repoRoot": {
      "type": "string",
      "description": "Absolute filesystem path to the project's root directory (matches scout-codebase + findBestParent heuristics)",
      "facet": "structure"
    }
  }
}
```

- Rationale: `name` matches what `ensureProjectAnchor` already stores (`name: 'Coding'`).
  `repoRoot` is a forward-looking property that `findBestParent` and other agents can
  lean on for path-prefix matching against entity-emitted file paths. `facet: "structure"`
  matches upper.json's existing convention (Structure / Behavior / Operations / Resilience
  / Quality / Evolution).
- Explicitly rejected:
  - **Minimal (just name)** — leaves `repoRoot` to a future migration; cheap now, costly
    if a downstream agent needs it next phase.
  - **Rich (name + repoRoot + team + ecosystem + lifecycle)** — premature; no current
    consumer needs `ecosystem` or `lifecycle`.

### Relationships (LOCKED)

- **Anchor-only: `relationships: {}` (empty).**
- Rationale: matches the existing convention in upper.json (File, Service, Feature,
  Component all declare `relationships: {}`). Reverse edges from Component / Service /
  Feature into Project are already implied by the anchor-pass logic in
  `wave-controller.persistWithKmCore`. Declaring outgoing edges from `Project` would
  either:
  - Reference classes defined in lower ontologies (e.g. `contains: ['Component']`) →
    upper-to-lower coupling that breaks the layered ontology model and may trigger
    "unknown class reference" warnings in teams whose lower ontology lacks `Component`.
  - Bundle hierarchical-restructure design work that MEMORY.md explicitly marked as a
    separate project.
- Explicitly rejected:
  - **`contains: ['Component']`** — upper-to-lower coupling, breaks layering.
  - **Full hierarchy `contains: [Component, Service, Feature]`** — bundles future
    restructure work; not justified by this phase's goal.

### Verification Boundary (LOCKED)

Two test gates; both run as `node --test` in
`integrations/mcp-server-semantic-analysis/` after `npm run build`:

1. **Unit-level smoke (extends 42.1.1 Test C):** for every team currently in
   `.data/ontologies/` (`agentic`, `cluster-reprocessing`, `code-entities`, `coding`,
   `raas`, `resi`, `ui`), construct a fresh km-core `OntologyRegistry` over the
   post-`OntologyConfigManager.initialize()` `ontologyDir` and assert
   `registry.isValidClass('Project') === true`. The 42.1.1 Test C comment line
   "Project=NOT loaded (see SUMMARY known-residuals)" must be replaced by a positive
   assertion.

2. **Integration smoke (new test, possibly Test F in
   `OntologyConfigManager.layout.test.ts` or a new
   `wave-controller-ensure-project-anchor.test.ts`):**
   - Stub or fake `kmCoreAdapter` with `queryEntities`/`storeEntity` spies.
   - Construct a `WaveController` (or minimum surface) wired with the stub.
   - Call `ensureProjectAnchor('test-run-id')`.
   - Assert: no throw; `kmCoreAdapter.storeEntity` called once with `name: 'Coding'`,
     `entityType: 'Project'`, `ontologyClass: 'Project'`, `team: 'coding'`; idempotency
     proven by calling twice and asserting `storeEntity` called once total (second call
     finds the existing entity via `queryEntities`).

**Phase 42.1 SC#6 wave-analysis re-run is OUT OF SCOPE** — same boundary as
`42.1.1-CONTEXT.md` line 93-94. Once this phase lands, the operator runs the Docker
rebuild and re-invokes Phase 42.1's verifier separately. This phase does not own SC#6.

<specifics>
## Specific Ideas

- The `Project` class lives in upper.json's `classes` map. Suggested insertion point:
  alphabetically near `Process` / `Port`, or grouped with other "anchor / hierarchy"
  classes if upper.json has visual grouping conventions worth respecting.
- The 7-team iteration list for the unit test must match the 7 lower ontologies that
  currently live in `.data/ontologies/`: `agentic-ontology.json`,
  `cluster-reprocessing-ontology.json`, `code-entities-ontology.json`,
  `coding-ontology.json`, `raas-ontology.json`, `resi-ontology.json`, `ui-ontology.json`.
  This is exactly the team set 42.1.1 Test A parameterised over — re-use the same
  TEAMS literal.
- `ensureProjectAnchor` stores `entityType: 'Project'` AND `ontologyClass: 'Project'` —
  both are the same string. If km-core treats them differently, the integration test
  should assert the recorded `entityType` matches the ontology class name exactly.
- `ensureProjectAnchor` is idempotent: it queries existing `entityType: 'Project'`
  entities first and returns early if `name === 'Coding'` is already stored. The test
  should exercise both the cold path (no existing entity) AND the warm path (entity
  already exists from a previous call) to lock in idempotency.

</specifics>
