## Decisions Made

- **Component canary gated to `team === 'coding'`** — the original Test C asserted `Component` globally (single team only — `coding`). Generalising to all 7 teams would require either dropping the canary (loses defense-in-depth) or asserting per-team-canary-classes (large new lookup table — out of scope). Gating the existing canary to `coding` keeps the original defense-in-depth signal intact for the only team that legitimately has `Component` while keeping the Project primary assertion universal.
- **`await import('@fwornle/km-core')` hoisted above the loop** — the existing test imported the module inside the body; for the 7-iteration loop this would re-resolve the module 7× unnecessarily. ES-module imports are cached so the runtime cost is negligible, but hoisting is cleaner and matches the canonical km-core test-style.
- **`OntologyConfigManager.resetInstance()` called inside the loop AND once after the loop** — `resetInstance()` is the documented way to break singleton state between team iterations. Without it, the second iteration's `getInstance()` returns the first team's manager and `initialize()` becomes a no-op against stale config. The trailing reset matches the existing test-cleanup pattern (Tests A/B/D/E all end with `resetInstance()`).
- **Per-team forensic log line uses the EXACT template the verify grep gate counts** — `[Test C] team=<team> Project=valid (Phase 42.1.2 layer-2 closed)`. The grep gate uses `grep -F` (fixed-string), so the template must match character-for-character. The grep gate's `wc -l | awk '$1 == 7'` is the count-assertion; if a team were silently skipped (e.g. test failed early and rethrew), the count would be < 7 and the gate would fail.
