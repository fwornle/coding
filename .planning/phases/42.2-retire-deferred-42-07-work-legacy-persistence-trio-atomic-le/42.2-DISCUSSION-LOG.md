## LevelDB dir-swap safety

| Option | Description | Selected |
|--------|-------------|----------|
| Copy-verify-rename, keep backup | Snapshot legacy → verify migrated count + spot-check legacyIds → atomic rename pair → rewire km-core `dbPath` back to canonical. Backup retained until SC#1-5 passes. Safe + rollbackable. | ✓ |
| Hard-link bridge then swap | Hard-link migrated files into new dir, atomic rename. Saves disk space. LevelDB internals may not love hard-links across `.ldb` files. | |
| Direct swap, no backup | Stop services → mv legacy out → mv migrated in → restart. Lowest disk overhead; no rollback if SC fails. | |
| Keep two dirs forever, route reads | Don't swap; update km-core to read from both. Avoids swap risk but leaves permanent two-store oddity. | |

**User's choice:** Copy-verify-rename, keep backup (recommended option)
**Notes:** Rollback path explicit (see D-DirSwap in CONTEXT.md). LevelDB stores typically <1GB, so `cp -R` is acceptable. Backup deletion is deferred to follow-up cleanup phase or manual operator step.
