# Phase 42.2 Plan 03: Retire QdrantSyncService Summary

QdrantSyncService retired in full. Outer-repo `scripts/sync-graph-to-qdrant.js` now constructs a km-core `GraphKMStore` (with mandatory `ontologyDir`) and calls `syncQdrantFromStore` (Phase 42-04 D-52a canonical maintenance op). DatabaseManager's vestigial `qdrantSync` field deleted outright. `src/knowledge-management/QdrantSyncService.js` removed from disk. Monorepo-wide grep gate clean — zero `QdrantSyncService` references survive in live code.

## What was built

- **`scripts/sync-graph-to-qdrant.js` (full rewrite, 252 → 239 lines).** Old script orchestrated `DatabaseManager` + `GraphDatabaseService` + `QdrantSyncService` and called `syncService.syncAllEntities({ teams, batchSize })`. New script:
  - Imports `GraphKMStore` from `@fwornle/km-core` and `syncQdrantFromStore` from `@fwornle/km-core/maintenance` (Phase 42-04 deliverable, exposed via the `./maintenance` sub-path in km-core's package.json `exports` map).
  - Wraps `@qdrant/js-client-rest`'s `upsert(collection, { wait, points })` to match km-core's structural `QdrantClient.upsert(collection, points)` interface.
  - Constructs the store with `ontologyDir: '.data/ontologies'` (MANDATORY per CLAUDE.md "km-core scripts" — Phase 41 lesson, commits `87bc2f567` / `fd35c5350`).
  - Probes Qdrant connectivity via `getCollections()` BEFORE opening the LevelDB store (fail-loud early, avoid LOCK file cleanup on the failure path).
  - Honors `--dry-run` by iterating the store and reporting entity/embedding counts without invoking Qdrant.
  - Passes a `log` callback to `syncQdrantFromStore` that emits per-batch progress lines to stderr.
  - Exit code: 0 on success, 1 on per-entity errors observed (operator gate signal), 2 on missing dataDir, 3 on Qdrant unreachable.
  - Operator CLI surface preserved: `--batch-size`, `--dry-run`, plus new `--collection` / `--db-path` / `--export-dir` / `--ontology-dir`. `--teams` honored as a deprecated no-op (km-core is team-agnostic; full-store sync). Env vars: `QDRANT_HOST`, `QDRANT_PORT`, `QDRANT_URL`, `COLLECTION_NAME` (overridden by `--collection`).
  - Zero `console.*` calls — all diagnostics via `process.stderr.write` per no-console-log constraint.

- **`src/databases/DatabaseManager.js`.** Four edits, all deletions:
  - Field declaration `this.qdrantSync = null` → replaced with deletion-marker comment.
  - Dynamic import `const { QdrantSyncService } = await import(...)` → removed.

  - Shutdown block `if (this.qdrantSync) { await this.qdrantSync.shutdown(); }` → removed.
  - Net: 5 `console.log`/`console.warn` lines REMOVED (no-console-log net improvement). No new `console.*` lines added.

## Why these decisions

- **Delete outright vs thin-shim** (D-Qdrant step 2 — planner discretion): The pre-existing `DatabaseManager.qdrantSync` exposed bidirectional sync semantics (`syncAllEntities` for forward sync, plus event listeners for inverse delete propagation). `syncQdrantFromStore` is one-way only: store → Qdrant. A thin-shim that delegates only the forward path would silently drop the reverse-sync API, breaking any caller that depended on it. There are NO such callers in the current tree (grep verified) — the reverse path was vestigial — so the honest answer is full deletion, not a misleading wrapper. The new CLI is the operator's rebuild path.

- **Single outer-repo commit vs submodule + pointer-bump pair**: The orchestrator's prompt text mandated a "dual-commit dance" for files in `integrations/mcp-server-semantic-analysis/`. The plan explicitly overrode this for `src/knowledge-management/` files because `ls -la` confirms `integrations/mcp-server-semantic-analysis/src/knowledge-management` is a symlink (`lrwxr-xr-x ... -> ../../../src/knowledge-management`). The file is tracked by the outer repo only; the submodule view sees it through the symlink. `git ls-files` in the submodule returns zero hits for `QdrantSyncService`. Matches Phase 42.1.2-02's single-outer-repo-commit pattern. Confirmed via state-decision `[Phase 42.1.2-02]: ... 'integrations/mcp-server-semantic-analysis/src/ontology' is a '120000' symlink (target '../../../src/ontology'); the file is tracked by the OUTER repo. Single outer-repo commit ... Future plans touching src/ontology/ should inherit this — Plan 03 must re-evaluate the symlink check before assuming the submodule tracks any new file under that path.` Same logic applies to `src/knowledge-management/`.

- **Legacy `--teams` flag kept as no-op**: D-Qdrant says "Operator CLI surface stays identical." Existing operator scripts may pass `--teams coding,ui,resi`. Dropping the flag would break them with `unknown arg`. Honoring it as a no-op with a stderr deprecation warning preserves the contract while documenting that km-core is team-agnostic.
