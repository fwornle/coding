### LevelDB Dir-Swap Recipe (LOCKED — D-DirSwap)

Copy-verify-rename with backup retained until SC#1–5 passes:

1. **Stop coding-services container** (or at minimum, halt any process writing to either dir — required for atomic rename and to prevent torn writes).
2. **Snapshot the legacy store:** `cp -R .data/knowledge-graph .data/knowledge-graph.legacy-pre-42.2.bak` (cheap LevelDB cold copy; expected <1GB).
3. **Verify migrated dir health:** count entities in `.data/knowledge-graph-migrated/` (expect ≥802); spot-check 5 random legacyIds for shape conformance (top-level `legacyId`, `ontologyClass`, `descriptionSegments`, provenance). Fail-closed if either check fails — abort the swap.
4. **Atomic rename pair:**
   ```bash
   mv .data/knowledge-graph .data/knowledge-graph.legacy-pre-42.2.bak.old
   mv .data/knowledge-graph-migrated .data/knowledge-graph
   ```
   (Two `mv` calls on the same filesystem are individually atomic; the brief window between them is acceptable because services are stopped.)
5. **Rewire km-core `dbPath`:** revert the 42-07 Phase A line at `integrations/mcp-server-semantic-analysis/src/agents/wave-controller.ts:507` from `.data/knowledge-graph-migrated` back to the canonical `.data/knowledge-graph`. (Submodule commit + outer-repo pointer bump.)
6. **Restart container** and run a small no-mock `ukb` sanity check (NOT full production — that's Wave 3's job) to confirm km-core reads + writes land in the canonical dir.
7. **Backup retention:** keep `.data/knowledge-graph.legacy-pre-42.2.bak.old` until SC#1–5 gate (Wave 3) reports PASS. Deletion is deferred to a follow-up cleanup phase (or manual operator step after operator-gate confirms). The `.legacy-pre-42.2.bak` snapshot (from step 2) is the immediate rollback path; the `.bak.old` (from step 4) is the post-rename safety net.

If anything fails between steps 4 and 6: `mv .data/knowledge-graph .data/knowledge-graph-migrated; mv .data/knowledge-graph.legacy-pre-42.2.bak.old .data/knowledge-graph` to roll back.
