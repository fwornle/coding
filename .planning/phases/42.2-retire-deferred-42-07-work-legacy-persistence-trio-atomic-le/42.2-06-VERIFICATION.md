## SC#6 root cause analysis (the orphan finding)

The verifier reads `general.json` (graphology export). The 18 reported orphans all match the pattern:
- `validUntil: null`
- `legacyId: null` (NOT `legacyId.id === 'unknown'` — different shape from the migrated cohort)

km-core's `iterate()` doesn't surface these even with `{includeSuperseded: true}`. The backfill dry-run scans 802 entities (all legacy with `legacyId.id === 'unknown'`); zero entities with `legacyId: null` are visible.

**Interpretation:** These 18 orphans were emitted by an earlier wave-analysis flow (UUIDv7 timestamps 10:13–11:39 UTC today, the run + a previous test pass) and JSON-snapshotted, but the corresponding leveldb writes either failed silently or were never invoked. They exist in `general.json` only and are effectively "ghost entities" — visible to graphology export consumers but not to the km-core runtime.

A fresh wave-analysis after the `findEntityByName` fix would:
1. Bootstrap `Coding` Project via `ensureProjectAnchor` (already works pre-fix)
2. Persist new Components/SubComponents via `kmCoreAdapter.storeEntity` → writes to leveldb
3. Run anchor pass → call `storeRelationship('Coding', entity)` → `findEntityByName('Coding')` now succeeds (post-fix) → contains edge added → entity NOT orphaned

The fix is structurally correct and proven at unit level. SC#6's full clean reading requires either (a) a fresh wave-analysis that overwrites general.json with anchored entities, or (b) deeper investigation into the JSON/leveldb sync that's outside Plan 06 scope.
