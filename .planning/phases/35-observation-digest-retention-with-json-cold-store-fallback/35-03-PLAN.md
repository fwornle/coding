<objective>
Create the `ColdStoreReader` module — a read-only reader over `.data/observation-export/observations.json` and `digests.json` that returns rows in a `[from, to)` range, with a day-bucketed LRU cache so repeated range queries don't re-parse the same file. Parsing a 1,667-row JSON once per dashboard refresh is fine; parsing on every paginated scroll is not — hence the LRU.

  <action>
Create `src/live-logging/ColdStoreReader.js` implementing the API in the `<interfaces>` block above. The module MUST:

1. **Export a single class `ColdStoreReader`** with two public methods: `readObservations({ from, to })` and `readDigests({ from, to })`. Plus a `_stats()` introspection method (underscore-prefixed — for test-only use).

2. **Constructor:**
   - `constructor({ exportDir, cacheSize } = {})`
   - `this.exportDir = exportDir || path.resolve('.data/observation-export')`
   - `this.cacheSize = Number.isFinite(cacheSize) && cacheSize > 0 ? cacheSize : 16` (per CONTEXT.md G2 — 16 day-bucket entries; ~16d of history hot in memory at peak).
   - `this._cache = new Map()` — keyed by ``${file}:${YYYY-MM-DD}``, value `{ rows: Array, parsedAt: Date.now() }`. Map preserves insertion order, so LRU eviction is the natural "evict the oldest key" pattern.
   - `this._statsObj = { observationsParsed: 0, digestsParsed: 0, cacheHits: 0, cacheMisses: 0 }`.
   - **DO NOT parse anything in the constructor.** Lazy — first parse on first read.

3. **`readObservations({ from, to })`:**
   - Normalize `from` and `to` to ISO-8601 strings. If `from` is missing, default to a very old date (e.g. `'1970-01-01T00:00:00.000Z'`). If `to` is missing, default to `new Date().toISOString()`.
   - Derive the set of YYYY-MM-DD day-buckets touched by the range: parse `from`/`to` as Dates, iterate by day. Cap at e.g. 366 days (sanity guard — refuse > 1 year of cold-store data in one request, return `[]` + warn).
   - For each day-bucket, look up in `this._cache`. On hit, increment `cacheHits` and use the cached rows. On miss:
     - `cacheMisses++`. Read `path.join(this.exportDir, 'observations.json')` ONCE per call (not per day-bucket — the JSON file is a single flat array of all observations across all days). Catch `ENOENT` or `SyntaxError`: write a single stderr warning `[ColdStoreReader] observations.json unreadable: ${err.message}` and return `[]`. Increment `this._statsObj.observationsParsed`.
     - Bucket the parsed array by `createdAt.slice(0,10)`. Insert each bucket into `this._cache` keyed by ``obs:${date}``. If the cache exceeds `this.cacheSize`, evict the oldest key (the first key in Map iteration order; `this._cache.delete(this._cache.keys().next().value)`).
   - Filter the combined rows from all hit buckets: keep rows where `row.createdAt >= from && row.createdAt < to`. (String comparison works correctly on ISO-8601.)
   - Sort the result by `createdAt` ascending (or descending — match what 35-04 expects; ascending is the natural cold-store order and the consumer can re-sort).
   - Return the filtered array.

4. **`readDigests({ from, to })`:**
   - Same shape but using `digests.json` and `date` (YYYY-MM-DD) as the primary key.
   - Normalize: if `from`/`to` is an ISO-8601 string, slice to `YYYY-MM-DD`. If already `YYYY-MM-DD`, use as-is.
   - Bucket by `row.date`. Filter `row.date >= fromDate && row.date < toDate`. Cache key prefix `'dig:'`.

5. **`_stats()`:** return `{ ...this._statsObj, cacheKeys: this._cache.size }`. Read-only.

6. **CRITICAL — file-level rule: this module MUST NOT contain any `fs.writeFile`, `fs.appendFile`, `fs.writeFileSync`, `fs.appendFileSync`, `fs.createWriteStream`, or any other write operation.** Only reads (`fs.readFileSync` or `fs.readFile`). The test in Task 2 enforces this with a source-code grep.

7. **Imports:** `node:fs`, `node:path`. Nothing else.

8. **No console.log/error/warn** — use `process.stderr.write(...)` for warnings, per project rule.

9. **JSDoc:** match `ObservationWriter.js` style. Top-of-file `@module`, class JSDoc, method-level `@param`/`@returns`.

**Notes for the executor:**
- The LRU is simple: Map insertion order is LRU order. When evicting, delete the first key. The cache stores parsed-and-bucketed rows, not the raw JSON string — parsing is O(n) once, bucketing is O(n) once, then range queries are O(buckets × bucket_size) which for 7d retention + 30d window is at most 30 × 100 = 3000 row comparisons.
- DO NOT add an async variant. Synchronous `readFileSync` is correct here: the JSON files are small (~1MB at current scale, ~10MB extrapolated), parsing is fast, and obs_api's existing handlers are also synchronous against SQLite. Async would add complexity without throughput benefit at this scale.
- DO NOT add a "watch the file for changes" hook. The 10s exporter cadence means the cache can be at most 10s stale for the most-recent bucket; 35-04 handles freshness by always preferring SQLite for the hot window.
- DO NOT export any helper functions. Class only. Keep the public surface minimal.
- If the parsed JSON is not an array (corrupted shape), warn-and-return-[] — same as ENOENT.

<risks_and_mitigations>
- **Risk:** `readFileSync` on a 10MB+ JSON file blocks the obs_api event loop. **Mitigation:** at current scale (1,667 obs ≈ 1MB), parse is sub-50ms — fine. If the JSON ever crosses ~50MB, the LRU still keeps repeat queries cheap, and the consumer (35-04) reads it at most once per range query (not per row). A future plan can swap to streaming `JSONStream` if profiling warrants it. Out of scope for this phase.
- **Risk:** The exporter's `_mergeWithExisting` (`ObservationExporter.js:109-135`) preserves pre-prune rows in JSON BUT may interleave them with currently-hot rows. **Mitigation:** the consumer (35-04) filters cold-store output by `createdAt < retention_boundary` before merging — the reader just returns whatever's in the range, and the consumer is responsible for de-duplication. Documented in 35-04.
- **Risk:** Day-bucket cache key collisions across observations vs digests. **Mitigation:** cache keys are prefixed `obs:` and `dig:` — spelled out in the action.
- **Risk:** The 366-day sanity cap rejects legitimate long-range queries. **Mitigation:** the dashboard's date pickers max at "all time" but practical queries are ≤ 90d. A 1-year cap is generous; the operator can override by composing two queries if needed.
</risks_and_mitigations>
