## Concurrency model (per checker WARNING 3)

The pruner does **NOT** acquire `ObservationWriter._writeLock` (the writer's per-instance mutex at `ObservationWriter.js:67-68`). Three reasons this is safe:

1. **Atomicity is provided by better-sqlite3 transactions.** The two `DELETE` statements run inside a single `db.transaction(() => { ... })`. better-sqlite3 wraps this in `BEGIN IMMEDIATE … COMMIT`, which acquires an exclusive write lock for the duration of the deletes. The writer cannot interleave a partial-row INSERT against either table while a prune transaction is in flight — SQLite serializes them at the file level.

2. **Working sets are temporally disjoint.** The writer's hot path operates on rows with `created_at > datetime('now', '-4 hours')` (the dedup window at `ObservationWriter.js:626-631`). The pruner only ever deletes rows with `created_at < datetime('now', '-${retentionDays} days')`, and `retentionDays >= 1` (24h) is enforced by both 35-01 (`ObservationWriter` constructor throws on `< 1`) and this plan's pruner constructor (defense-in-depth). The two sets cannot overlap — the pruner never deletes a row the writer is still reading or might re-write.

3. **Future "read-after-write" patterns require coordination.** If a future change in `ObservationWriter` introduces a SELECT-then-UPDATE flow against the same row (e.g. updating `digested_at`), and that flow operates on rows older than 4h (currently doesn't), the pruner could delete a row between the SELECT and the UPDATE. That scenario would need either (a) shifting to a `WHERE updated_at IS NULL` filter on the pruner, or (b) introducing a shared lock primitive. **For Phase 35, this is not a concern.** The writer's only timestamp-touch is `digested_at`, set inside `Consolidator.run()` which operates synchronously on a snapshot of IDs — no read-then-write window the pruner can punch through.

The pruner's `prune()` is therefore safe to call from `obs_api`'s `setInterval` without coordinating with `ensureWriter`.
