# Phase 35 Plan 06: paginable-total accounting for `_mergeObservations` + `_mergeDigests` (gap closure)

## Goal

Fix the misleading `total` returned by the merge helpers so the dashboard's last-page navigation doesn't land on empty offset space. The fix is local to `scripts/observations-api-merge.mjs`: both `_mergeObservations` and `_mergeDigests` must report a `total` that matches the number of rows actually paginable (SQLite total in range + cold rows that fit on page 0).

## Why this isn't part of the deferred merged-set-pagination work

The phase PLAN.md (Open questions section) explicitly defers true server-side pagination over the merged set ("interleaved by timestamp, correct counts on every page") to a follow-up phase. This plan does NOT implement that. It only corrects the bookkeeping of `total` to reflect the existing offset=0-only contract. No merge algorithm change.

## Tasks

### Task 1: Fix `_mergeObservations` total accounting (feat)

Locate the return statement in `_mergeObservations` (around `scripts/observations-api-merge.mjs`). It currently returns:

```js
{
  data: pageRows,
  total: sqliteTotalInRange + coldFiltered.length,
  _metadata: { ... fromColdStore: true, coldRows: coldFiltered.length, ... },
}
```

Change `total` calculation to:

```js
// Cold rows only contribute to offset=0, so total must reflect what pagination actually walks:
// SQLite's full range count + however many cold rows fit on page 0 after SQLite consumed its slots.
const sqliteOnThisPage = pageRows.filter(r => r._origin === 'sqlite').length;
const coldCapacityOnPage0 = Math.max(0, limit - sqliteOnThisPage);
const paginableCold = Math.min(coldFiltered.length, coldCapacityOnPage0);
const paginableTotal = sqliteTotalInRange + paginableCold;
```

Then `total: paginableTotal`. Keep `_metadata.coldRows` reporting the unfiltered `coldFiltered.length` (so operators can still see how many historical rows exist).

Apply the analogous change to `_mergeDigests`.

### Task 2: Two Jest cases (test)

In `tests/scripts/observations-api-server.merge.test.js`, add:

```js
test('Phase 35-06: paginable total reflects offset=0 cold contribution, not raw cold count', () => {
  // Seed 5 SQLite rows + 100 cold rows. Limit = 10, offset = 0.
  // Expected: total = 5 + (10-5) = 10 (cold can fill 5 slots; the rest are not paginable).
  // NOT: total = 5 + 100 = 105.
});

test('Phase 35-06: paginable total = sqliteTotal when offset > 0 (cold absent on subsequent pages)', () => {
  // Seed 50 SQLite rows + 100 cold rows. Limit = 10, offset = 10.
  // Expected: total = 50 (SQLite-only, cold not paginable past page 0).
});
```

### Task 3: Operator checkpoint (`autonomous: false`)

After commits land on main: `launchctl kickstart -k gui/$(id -u)/com.coding.obs-api` to reload the merge module. The dashboard backend forwards transparently (35-05's byte-pipe), so no docker-compose restart is needed for the backend fix to take effect on the UI. Frontend bundle is unaffected.

## Verification gate

After kickstart + a 30-day-window dashboard probe, the last-page request must return non-empty rows. Concretely:

```bash
# Before fix: total = 1110, lastPage at offset=1100 returns []
# After fix:  total ≈ 724, lastPage at offset≈700 returns the oldest-in-SQLite rows
FROM_18D=$(date -u -d '18 days ago' +%Y-%m-%dT%H:%M:%SZ)
RESPONSE=$(curl -fs "http://localhost:3032/api/observations?from=${FROM_18D}&limit=50&offset=0")
TOTAL=$(echo "$RESPONSE" | jq -r '.total')
LIMIT=50
LAST_OFFSET=$(( (TOTAL / LIMIT) * LIMIT ))
[ "$LAST_OFFSET" -ge "$TOTAL" ] && LAST_OFFSET=$(( LAST_OFFSET - LIMIT ))
curl -fs "http://localhost:3032/api/observations?from=${FROM_18D}&limit=${LIMIT}&offset=${LAST_OFFSET}" | jq '.data | length'
# Must be > 0
```

## Scope guardrails

- Do NOT touch `scripts/observations-api-server.mjs` — the merge module's exported function signature is unchanged.
- Do NOT touch frontend code — the frontend's pagination math is correct; it's the server's reported `total` that was wrong.
- Do NOT change the offset=0-only-cold contract (Option B from the original 35-04 checker WARNING 1). Total math just needs to align with it.
- Do NOT remove or rename `_metadata.coldRows` — that's a different observability signal and downstream tooling may rely on it.
