<risks_and_mitigations>
- **Risk:** A running obs-api instance picks up the new config on next writer construction, which it caches per-process. If the operator forgets to kickstart, the in-process `this.retentionDays` stays at the old default-7 anyway (no behavior change for this plan — retentionDays isn't read by anything yet). **Mitigation:** no restart required at the end of this plan. The phase-level rollout checklist in `PLAN.md` covers it.
- **Risk:** A sibling project's `ObservationWriter` import (if any exists) breaks because the constructor now throws on a degenerate config. **Mitigation:** grep verifies — `grep -rln "new ObservationWriter" --include='*.js' --include='*.mjs'` should show only callers within this repo. The pre-existing fallback path in `loadConfig()` returns the safe default-7, so any caller that doesn't override `configPath` keeps working unchanged.
- **Risk:** The test creates temp files on each run. **Mitigation:** `fs.mkdtempSync` + `afterAll` cleanup, and the test specifies `tmpDir/test-*.db` paths that never actually get opened.
- **Risk:** Jest's `--experimental-vm-modules` flag has historically been flaky for dynamic ES-module imports. **Mitigation:** the existing `tests/live-logging/EmbeddingClassifier.test.js` proves the pattern works in this repo's Jest setup. If it does fail, the executor can fall back to importing via `require('../../src/live-logging/ObservationWriter.js').ObservationWriter` (Jest handles the ESM-CJS interop transparently for some module shapes; if not, the dynamic import inside `beforeAll` is the documented escape hatch).
