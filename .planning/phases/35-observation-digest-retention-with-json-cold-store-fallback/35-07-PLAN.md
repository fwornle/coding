# Design decisions

design_decisions:
  - decision: "In-memory full union per request (not keyset pagination)"
    rationale: |
      Cold store is bounded by JSON file size (currently 862 obs + ~1000 digests in
      `.data/observation-export/`). Total rows-in-range for typical queries is <5000.
      Per-request memory cost is small (a few MB of JSON). Keyset pagination is more
      complex and the perf win is not worth it at this scale. If the project grows to
      >100k cold rows per range, revisit.
  - decision: "ColdStoreReader.readObservations called on every request (no caching of the union)"
    rationale: |
      ColdStoreReader already has a day-bucketed LRU cache internally (35-03). The
      per-request union construction is O(N log N) for the sort but N is bounded as
      above. Keeping this simple — the caller doesn't need to track which pages have
      been served.
  - decision: "Remove `coldOnFirstPageOnly` from _metadata (or set false)"
    rationale: |
      Field name describes the old contract. Either remove it entirely or set it to
      `false` for back-compat with any code that checks it. Document the choice in
      SUMMARY.

# Phase 35 Plan 07: True merged pagination (deferred follow-up)

## Goal

Replace the offset=0-only-cold contract from 35-04 with full-union pagination so cold rows appear interleaved by timestamp across all pages of `/api/observations` and `/api/digests`. Resolves the user-reported "widening the date range doesn't reveal cold rows" UX issue.

## Tasks

### Task 1: Rewrite `_mergeObservations` and `_mergeDigests` (feat)

In `scripts/observations-api-merge.mjs`:

```js
// New shape (rough sketch — fill in details):
export function _mergeObservations(sqliteRows, coldRows, retentionBoundary, opts) {
  const { limit, offset, sqliteTotalInRange, coldTotalInRange } = opts;

  // Filter cold rows strictly older than retention (sqlite is already in-range).
  const safeCold = coldRows.filter(r => r.createdAt < retentionBoundary);

  // LOAD-BEARING Set-based id dedup (invariant #5).
  const sqliteIds = new Set(sqliteRows.map(r => r.id));
  const safeColdNoOverlap = safeCold.filter(r => !sqliteIds.has(r.id));

  // Tag origins.
  const taggedSqlite = sqliteRows.map(r => ({ ...r, _origin: 'sqlite' }));
  const taggedCold = safeColdNoOverlap.map(r => ({ /* reshape... */, _origin: 'cold' }));

  // Full union, sort DESC by timestamp, slice by [offset, offset+limit).
  const union = [...taggedSqlite, ...taggedCold].sort(
    (a, b) => (b.timestamp || '').localeCompare(a.timestamp || '')
  );
  const page = union.slice(offset, offset + limit);

  return {
    data: page,
    total: sqliteTotalInRange + safeColdNoOverlap.length,
    _metadata: {
      fromColdStore: taggedCold.length > 0,
      sqliteRows: page.filter(r => r._origin === 'sqlite').length,
      coldRows: page.filter(r => r._origin === 'cold').length,
      retentionBoundary,
    },
  };
}
```

**Important:**
- The caller (`observations-api-server.mjs`) passes the FULL sqlite-in-range (not just the current page's sqlite slice) plus the FULL cold-in-range. The merge function does the union+sort+slice.
- This means the server's existing SQL must be widened to `SELECT ... WHERE created_at BETWEEN from AND to ORDER BY created_at DESC` without `LIMIT` (or with a generous safety limit like 100000), letting the merge module do the pagination.
- Alternatively, fetch SQLite with the original LIMIT/OFFSET for pages whose offset is comfortably inside the sqlite-only range, and fall back to full-union when offset crosses the retention boundary. Pick whichever is simpler — favor correctness.

Apply analogous changes to `_mergeDigests`.
