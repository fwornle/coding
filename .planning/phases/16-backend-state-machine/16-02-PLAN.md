    Import `{ dispatch, getState }` from `../workflow-state-machine` at the top of wave-controller.ts.

    **Event mapping strategy:**

    The existing 30 updateProgress calls use `{ currentWave, subPhase, message }` patterns. The Phase 15 transition events are: start, step-complete, pause, resume, fail, cancel, complete. These are high-level state transitions, not substep-granularity events.

    The wave-controller's updateProgress calls are mostly logging/tracking substep progress within a single "running" state. The state machine should NOT have 30 transitions per run — that would pollute the transition graph with noise.

    **Correct approach — two layers:**

    1. **State machine transitions** (via dispatch): Use for actual state changes only:
       - Workflow start (handled by Plan 01 in workflow-runner/tools.ts)
       - Step completion: `dispatch({ type: 'step-complete', stepName: 'wave1', nextStep: 'wave2', duration })` — called once per wave completion
       - Pause at single-step boundary: `dispatch({ type: 'pause', reason: 'single-step-boundary' })`
       - Failure: `dispatch({ type: 'fail', error, step })` — on catch blocks
       - Completion: `dispatch({ type: 'complete', summary })` — at end of executeWaveAnalysis

    2. **Substep progress tracking** (via direct state mutation on RunProgress): The 30 updateProgress calls mostly track substep position within a running state (subPhase: 'analyze', 'classify', 'persist', etc.). These update `progress.currentSubstepId` and `progress.currentSubstepIndex` on the running state. Create a helper method `updateSubstep(substepId: string, wave: number, message?: string)` that:
       - Calls `getState()` to get current state
       - Asserts state.status === 'running'
       - Mutates `state.progress.currentSubstepId`, `state.progress.currentSubstepIndex`, `state.progress.currentWave`, `state.progress.totalWaves`, `state.progress.lastUpdate`
       - Does NOT call dispatch — this is intra-step bookkeeping, not a state transition
       - Notifies subscribers manually (import the subscriber notification mechanism, or use a lighter mechanism like having the singleton expose an `updateProgress(fn: (state: RunningState) => void)` helper)
