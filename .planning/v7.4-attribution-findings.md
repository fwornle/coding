# v7.4 Measurement — attribution findings & planning input

Source: the `exp-dash-start-control-claude-opus-4-8` dogfood measurement (2026-06-29).
See `.data/experiments/exp-dash-start-control-claude-opus-4-8.log.md` for the run.

## What the experiment exposed
A measurement of a live Claude Code (Opus 4.8) session reported **1,244,689 tokens / 186 calls**,
model **claude-haiku-4.5**, score 0.933. Inspecting the attributed rows:

| Process (all swept into the span window) | rows | tokens |
|---|---|---|
| consolidator-mentions | 30 | 731,649 |
| consolidator-insight | 28 | 414,459 |
| observation-writer | 106 | 94,480 |
| consolidator-digest | 1 | 3,828 |
| health-coordinator | 21 | 273 |

**None of it is the Opus session.** It is the observation/digest/insight consolidator +
health daemons that happened to run during the 21-minute window, on haiku/sonnet via the proxy.

## Findings
- **A — Timeline drops untagged rows (FIXED).** `readTimeline` deduped parents by `tool_call_id`;
  all untagged rows share `''` so 185/186 were dropped (timeline 8.9k vs aggregate 1.24M). Fixed:
  dedup only by a non-empty id; each empty-id row is its own turn. (`lib/experiments/timeline-read.mjs`
  + regression test.)
- **B — Model attribution is inconsistent.** Table shows the **dominant** model; the timeline shows the
  **first row's** model; neither matches the session. No single canonical model/agent per Run.
- **C — Time-window over-attribution (root cause).** `TELEM-03`'s rule — "in-window → task_id" — attributes
  *every* proxy row in the window to the task, regardless of which process/agent produced it. For an
  interactive foreground session with concurrent background daemons this is wrong on both sides:
  it (1) includes unrelated daemon/consolidator traffic and (2) misses the foreground agent's own
  tokens (Claude Code talks to Anthropic directly; its usage must come from the LSL transcript adapter,
  stamped with the active task_id).
- **D — ETM under-captures long agentic prompt-sets + collapses their timestamps (NEW, 2026-06-29).**
  Discovered when the operator's morning work showed as "no observations this morning." Root cause: ETM's
  unit of observation is the **typed-prompt prompt-set, snapshotted once near its start and stamped at the
  prompt-set start time.** A GSD/agentic session is **one typed prompt → many hours of agent work that the
  operator steers via AskUserQuestion** (no new typed prompts). Evidence: transcript `e0af5b8b` had its last
  typed prompt at 2026-06-28T21:00:43Z, then ran live until 2026-06-29T06:08Z with **5 operator AskUserQuestion
  decisions this morning** (05:30–06:03Z) and real deliverables (dashboard control, dogfood run, Fix A timeline
  patch, this findings doc). ETM produced **only 8 observations, all the opening investigation burst, all stamped
  21:00:43Z** — zero observations for ~9 h of subsequent work, and morning work mis-timed to the previous night.
  Two compounding defects from one assumption: (1) **under-capture** — no re-capture as a prompt-set continues;
  (2) **timestamp-collapse** — observations stamped at prompt-set start, not at the real event time. The
  AskUserQuestion decision points (and significant tool-activity boundaries) are natural observation units the
  pipeline ignores. This is the *same blindness* to interactive foreground agentic work that breaks token
  attribution (C/ATTR-03).

## Proposed requirements (for /gsd-plan-phase)
- **ATTR-01 (C):** Attribute a token row to a measurement only when it belongs to the measured task's
  process/agent lineage — not by time-window overlap alone. Background processes (consolidator-*,
  health-coordinator, observation-writer, and other non-task daemons) are excluded from, or segregated
  within, a foreground interactive measurement. Define the lineage signal (process allowlist/denylist,
  agent/session id, or an explicit task_id stamp at the call site).
- **ATTR-02 (B):** One canonical model/agent source-of-truth per Run, shown consistently in the runs
  table, the score drawer, and the timeline (no dominant-vs-first-row divergence).
- **ATTR-03 (capture foreground tokens):** The interactive agent's (Claude Code) own session tokens are
  captured and attributed to the active task_id (LSL transcript adapter → token_usage with task_id),
  so the recorded model reflects reality (Opus), not concurrent proxy traffic.
- **OBS-01 (gap d):** Observations/digests/insights produced during a measurement are tagged with the
  active `task_id` (ETM reads `getActiveMeasurement()`), exposed in the observation view, and
  queryable/correlatable to the Run — so "what happened during measurement X" is reconstructable.
- **OBS-02 (finding D — continuous capture + true timestamps):** ETM captures observations *throughout* a
  long-running agentic prompt-set, not only at its start: re-capture on meaningful boundaries (each
  AskUserQuestion decision, and/or significant tool-activity batches / periodic flush) so multi-hour GSD
  sessions yield observations as work progresses. Each observation is stamped with its **real event time**
  (the exchange/decision timestamp), not the prompt-set start. Acceptance: a session whose only typed prompt
  is at T0 but which runs for hours with operator decisions at T0+n yields observations dated at ~T0+n, not all
  at T0.
