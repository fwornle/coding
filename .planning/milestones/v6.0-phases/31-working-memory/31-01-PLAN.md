Create `src/retrieval/working-memory.js` with a single exported function `buildWorkingMemory(codingRoot)`.

**Module header:** Follow `keyword-search.js` JSDoc module pattern with `@module working-memory`.

**Constants (top of file):**
- `WM_BUDGET = 300` -- firm 300-token ceiling per D-05
- `VKB_TIMEOUT = 2000` -- 2-second abort timeout per RESEARCH.md pitfall 1
- `VKB_BASE = 'http://localhost:8080'` -- VKB API base URL

**Imports:**
- `import { readFileSync } from 'node:fs';`
- `import { countTokens } from 'gpt-tokenizer';`

**Function: `fetchKGStructure()`** (async, internal)
- Fetch Project entities: `GET ${VKB_BASE}/api/entities?team=coding&type=Project` with `AbortSignal.timeout(VKB_TIMEOUT)`
- Fetch Component entities: `GET ${VKB_BASE}/api/entities?team=coding&type=Component` with `AbortSignal.timeout(VKB_TIMEOUT)`
- Use `Promise.all` for parallel fetches
- Check `response.ok` before parsing JSON
- Return `{ project: projectData.entities?.[0] || null, components: componentData.entities || [] }`
- On any error: log to `process.stderr.write('[WorkingMemory] VKB fetch failed: ...\n')` and return `{ project: null, components: [] }`

**Function: `parseStateFrontmatter(codingRoot)`** (sync, internal)
- Read `${codingRoot}/.planning/STATE.md` via `readFileSync`
- Extract YAML frontmatter between `---` markers using regex: `/^---\n([\s\S]*?)\n---/`
- Parse simple key-value pairs with regex: `new RegExp('^' + key + ':\\s*(.+)$', 'm')`, strip quotes
- Return object with: `milestone`, `milestoneName` (from `milestone_name`), `status`, `stoppedAt` (from `stopped_at`), `lastActivity` (from `last_activity`)
- Also parse the `### Blockers/Concerns` section from the body: extract lines starting with `- ` after that heading, return as `concerns` array
- On any error: return `null`

**Function: `assembleMarkdown(kgData, stateData)`** (internal)
- Build markdown string under `## Working Memory` header
- **Structure subsection:** If `kgData.project` exists, output `**Project:** {entity_name}`. For each component in `kgData.components`, output `- **{entity_name}**: {first_observation_content_truncated_to_60_chars}`. Extract description from `component.observations?.[0]?.content`, strip any `[LLM]` prefix. If no observations, just output the component name.
- **State subsection:** If `stateData` exists, output `**Milestone:** {milestone} {milestoneName}`, `**Status:** {status}`, `**Current:** {stoppedAt}`. If `stateData.concerns` has items, output `**Known Issues:**` followed by the first 3 concerns as bullet points.
- If both kgData and stateData are empty/null, return empty string.

**Function: `truncateToFit(fullMarkdown, kgData, stateData, maxTokens)`** (internal)
- Per D-07 truncation strategy:
  - Step 1: Rebuild markdown with component names only (no descriptions). Check token count.
  - Step 2: If still over, rebuild with only Project node + state section (drop all components). Check token count.
  - Step 3: If still over, rebuild with state section only (drop KG entirely). Truncate state text if needed.
- Return the truncated markdown string.

**Function: `buildWorkingMemory(codingRoot)`** (async, EXPORTED)
- Call `fetchKGStructure()` and `parseStateFrontmatter(codingRoot)` in parallel using `Promise.all` (wrap sync call in `Promise.resolve`)
- Call `assembleMarkdown(kgData, stateData)` to build full markdown
- Count tokens via `countTokens(markdown)`
- If tokens > `WM_BUDGET`, call `truncateToFit(markdown, kgData, stateData, WM_BUDGET)`
- Recount tokens after truncation
- Return `{ markdown, tokens }` where `tokens` is the actual token count
- Entire function wrapped in try/catch: on error, log to `process.stderr.write('[WorkingMemory] Failed: ...\n')` and return `{ markdown: '', tokens: 0 }` (fail-open per D-03 and pitfall 1)

**CRITICAL constraints:**
- NEVER use `console.log` -- use `process.stderr.write` for all logging (no-console-log constraint)
- Plain JS (not TypeScript) -- matches existing `src/retrieval/` modules per STATE.md locked decision
- Use `countTokens` from `gpt-tokenizer` for all token counting (not word-count approximation)
- No caching of any kind per D-03 and D-04
  </action>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| retrieval-service -> VKB API | HTTP call to localhost:8080; VKB runs in Docker, retrieval runs host-side |
| retrieval-service -> STATE.md | Filesystem read of a markdown file controlled by the project |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-31-01 | D (Denial of Service) | VKB API fetch in buildWorkingMemory | mitigate | AbortSignal.timeout(2000) prevents hung requests; fail-open returns empty WM if VKB unresponsive |
| T-31-02 | I (Information Disclosure) | STATE.md content in retrieval response | accept | STATE.md contains only project planning metadata (milestone, phase names), no secrets. Same trust domain as the agent conversation. |
| T-31-03 | T (Tampering) | VKB API response manipulation | accept | VKB runs on localhost inside Docker; same-host trust boundary. No cross-network exposure. |
| T-31-04 | D (Denial of Service) | STATE.md readFileSync blocking event loop | accept | File is <5KB, readFileSync completes in <1ms. Async read unnecessary for this size. |
| T-31-05 | S (Spoofing) | VKB API endpoint identity | accept | localhost-only communication; no authentication needed for same-host service |
</threat_model>
