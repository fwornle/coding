## Root Cause Hypothesis

Two separate root causes, both confirmed and now refined after walking the migrated store and the proxy-provider call chain:

**For observation (b) ‚ÄĒ entities lack project linkage (REFINED twice; latest revision dominates):**

The visible orphan-in-viewer symptom is caused PRIMARILY by Anomaly 6: Phase 42-07 Phase B1 replaced the legacy `persistence-agent.persistEntities()` with `persistWithKmCore`, dropping the `findBestParent` + post-sweep `contains`-relation pass that attached every new entity to the `Coding` Project anchor. The new entities are now stored without any incoming `contains` edge ‚Üí dashboard sees them as orphans. Empirical confirmation: +64 entities since the 16:59 backup, +1 relation between two of those entities, **0 new edges to Coding**.

A secondary, parallel architectural bug exists (Anomaly 1): metadata tagging is broken across THREE writers ‚ÄĒ `canonical-mapper`, `km-core-adapter`, and the migration script. Wave entities accidentally get `metadata.team='coding'` via the legacy `mapEntityToSharedMemory`, but km-core's Exporter buckets on `metadata.domain` (not `team`), so `coding.json` stays empty and all entities fall into the catch-all `general.json` bucket. This is also a real bug, but it would not by itself produce the visible "orphan nodes" symptom ‚ÄĒ the LevelDB still holds the data; only the per-domain export file is mis-bucketed.

Phase 42.1 must fix BOTH: Anomaly 6 (the load-bearing symptom) and Anomaly 1 (the latent architectural drift). Anomaly 6 is the smaller, more urgent fix.

**For observation (a) ‚ÄĒ LLM workload as `'unknown'` (REFINED):**

Three issues stacked, but only one is the real bug:

1. **The real bug:** `proxy-provider.ts:complete()` builds the HTTP body to `/api/complete` with only `messages, maxTokens, temperature` (+ optional model). It does NOT forward `request.agentId` to the wire. The proxy server then reads `body.process` (not present) and defaults to `'unknown'`. Every SDK-style caller (all wave agents, semantic-analyzer, semantic-analysis-agent, git-staleness-detector) falls through this gap.
2. **Hidden by the real bug:** the submodule's `attachTokenLogger` writes to `.observations/token-usage.db` (0 bytes ‚ÄĒ likely Docker path mismatch or silent init failure). Even if it worked, the dashboard wouldn't see it because the dashboard reads from `.data/llm-proxy/token-usage.db`.
3. **By contrast,** direct-POST callers (`health-coordinator`, `observation-writer`) bypass the SDK and explicitly set `body.process` in their POST ‚ÄĒ that's why they attribute correctly. The CLAUDE.md `/api/complete` shape doc (`{process, messages, taskType?}`) describes the direct-POST contract, not the SDK contract.

Fix (1) alone resolves the observation. Fixes (2) and (3) are cleanup, not load-bearing.
