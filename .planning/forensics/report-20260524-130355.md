### Anomaly 1 — Ontology files at the wrong filesystem layout — HIGH confidence

**Evidence:** `wf_1779622455132_gxccxm.log` repeatedly emits

```

```

Reality on host AND inside the container:

```

.data/ontologies/coding-ontology.json             (present, flat layout)
.data/ontologies/upper/                           (does NOT exist)
.data/ontologies/lower/                           (does NOT exist)
```

The ontology loader expects a two-tier directory layout
(`upper/<file>.json`, `lower/<file>.json`) per `km-core/ontology-registry`'s
contract. The actual on-disk layout from Phase 38 is flat — every ontology
file lives directly under `.data/ontologies/`. The mismatch causes the
registry to load zero classes for the coding team.

### Anomaly 2 — `ensureProjectAnchor` throws "Unknown ontology class: Project" on every wave — HIGH confidence

**Evidence:** `wf_1779622455132_gxccxm.log` (one occurrence per wave, four
total):

```
[WaveController] ensureProjectAnchor failed (runId=wave-analysis-2026-05-24T11-34-15-780Z): Unknown ontology class: Project
```

This is the direct downstream of Anomaly 1. `ensureProjectAnchor` tries to mint
the `Coding` Project node when absent — but km-core's `storeEntity` validates
`ontologyClass` against the registry, and with zero classes loaded, every
upsert with `entityType: "Project"` is rejected.

The error is caught (per the plan's fail-soft design) so the wave continues,
but the Coding anchor never exists in the km-core store.

### Anomaly 3 — Anchor pass loop skips all entities (added=0 on every wave) — HIGH confidence

Mechanism: `wave-controller.ts:2436-2440` (Layer a of the idempotency check)
pre-populates `alreadyAnchored` from the in-wave `relationships` array — any
relation with `type in {contains, parent-child}` adds its `.to` to the
already-anchored set BEFORE checking whether the relation was actually
persisted to km-core. With `Coding` missing from km-core, those relations
likely failed silently at storage time, but the bookkeeping had already marked
the entities as anchored. The for-loop at line 2459 therefore skips every
entity rather than falling back to `findBestParent`.

This is a **bookkeeping bug** in the anchor pass, surfaced only when the
Project anchor bootstrap fails. Under normal conditions (Coding exists) the
relations would persist and the bookkeeping would be correct.

## Root Cause

**Single dominant cause:** ontology files for the coding team are present on
disk but in a **flat** `.data/ontologies/*.json` layout, while the
ontology-registry loader expects a **two-tier subdirectory** layout
(`.data/ontologies/upper/*.json` + `.data/ontologies/lower/*.json`). The
mismatch produces an empty/incomplete registry for the coding team, which
cascades into:

1. `ensureProjectAnchor` cannot mint `Coding` (Anomaly 2)
2. In-wave `contains` relations from `Coding` fail to persist but are still
   recorded as "already-anchored" in the bookkeeping (Anomaly 3)
3. Dedup loses the type signal it needs to fuse same-named L2 Components
   (Anomaly 4)

The wave-analysis pipeline silently degrades and produces orphaned and
duplicated entities, which is exactly what the user observed in the VKB
viewer.

The verifier-vs-registry API drift (Anomaly 5) and the misleading
`docker logs` instruction (Anomaly 6) are separate but worth addressing in the
same remediation pass — they obscured the diagnosis.
