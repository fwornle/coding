# v7.2 — LLM Proxy Performance: Claude CLI Worker Pool

**Status:** Research seed for queued milestone v7.2. Do NOT activate planning until v7.1 (KM Unification) ships.

**Anchor date:** 2026-05-19 (date this seed was written; numbers below were measured then)

---

## Context — what v7.1 (this turn, day-of) did NOT fix

The `claude-code` provider in `rapid-llm-proxy/proxy-bridge/server.mjs` now has a two-tier dispatch:

| Tier | When | Where the bearer goes | Latency | Tokens (`say OK`) |
|---|---|---|---|---|
| Direct API (new) | First attempt | `api.anthropic.com/v1/messages`, OAuth `Authorization: Bearer` from keychain | **0.9s** | 9 in / 4 out |
| CLI fallback | 401/403/429 from direct | `claude -p` subprocess, OAuth via keychain (CLI-managed) | **10-14s** | ~9 in + ~16-22K `cache_creation` |

The fallback fires whenever Anthropic rate-limits the bare OAuth bearer endpoint. Probed 2026-05-19: haiku is consistently accepted on the direct path, but sonnet and (almost certainly) opus return HTTP 429 with an empty error body within milliseconds. The Claude Code CLI session endpoint uses a different per-model rate-limit bucket — same Max subscription, but the CLI path goes through.

So `observation-writer` pinned to claude-code/sonnet works (we just verified backfill of 6 stuck rows), but at ~14s per call. That's the gap this milestone closes.

---

## Where the 10-14s actually goes

Measured on host directly (`time env -u ANTHROPIC_API_KEY claude -p ... say OK`):

| Phase | ms | Notes |
|---|---|---|
| Node + CLI boot | 300-500 | Cold spawn every call |
| CLI internal setup | 700-1000 | Auth check, args parse, plugin registry load |
| API request prep + TLS | 200-400 | Includes injecting the ~16-22K system prompt |
| Anthropic processing of cache_creation | 1500-2500 | First call burns cache_creation tokens to populate the prefix cache |
| Network roundtrip + response parse | 200-400 | |
| CLI shutdown | 100-200 | |
| **Total typical** | **~3-5s** if cached, **~7-14s** cold | Empirically ~14s in production via Docker/proxy-bridge |

Two main levers:

1. **Spawn cost (~500-700ms)** is per-call and unavoidable with the current `execFileAsync` model.
2. **cache_creation (~1500-2500ms)** is per-call ONLY because the cache TTL is 5 minutes and our calls are spaced further apart, so each call re-creates the cache instead of reading it.

A persistent worker keeps both warm.

---

## Approach — persistent worker pool

The `claude` CLI supports a streaming JSON session via:

```
claude -p --input-format stream-json --output-format stream-json
```

Each worker is a long-lived subprocess; the proxy writes JSON-Lines `{type: "user", message: ...}` events to stdin and reads `{type: "assistant", message: ...}` events from stdout. Auth + system prompt are loaded once at worker boot, then reused for every request that worker serves.

### Design constraints

- **Pool size**: 2-3 workers — enough for concurrent observation-writer + consolidator + health probe without queue depth, but not so many that we burn idle RAM. Workers crash-recover individually.
- **Worker lifecycle**: spawn lazily on first claude-code fallback. Idle-evict after N minutes (configurable; default 30) to free RAM after busy periods.
- **Affinity**: route by model (haiku/sonnet/opus) — each worker is pinned to one model since the CLI's `--model` is a startup flag. So actually 2-3 workers PER MODEL, but lazy-spawned: the haiku pool stays cold if no haiku calls hit fallback.
- **Concurrency per worker**: 1. The CLI doesn't multiplex; the second request would interleave or block.
- **Cancellation propagation**: client-disconnect → abort the in-flight request via the stream-JSON protocol (send `{type: "cancel"}` if supported, else SIGTERM the worker and respawn).
- **Crash recovery**: worker exits unexpectedly → mark dead, drop in-flight call as RETRYABLE, respawn lazily next call. Don't auto-restart immediately (avoid spin loops on persistent CLI breakage).

### Expected speedup

| Phase | Cold per-call (today) | Warm worker (proposed) |
|---|---|---|
| Spawn | 500-700ms | **0** (already running) |
| CLI setup | 700-1000ms | **0** (one-time at worker boot) |
| Anthropic cache_creation | 1500-2500ms | **~300ms cache_read** (5x cheaper, faster) |
| API processing | 200-400ms | 200-400ms |
| **Total** | ~10-14s | **~2-3s** |

Roughly 5× speedup on the fallback path. The direct OAuth path stays the primary route for haiku at 0.9s; the worker pool kicks in only for sonnet/opus (current rate-limit profile) or transient OAuth 401s.

## Risks & open questions

1. **stream-JSON protocol stability** — Anthropic may version the protocol. Need to pin a CLI version and detect schema drift at worker boot.
2. **Per-model pool isolation** — does `--model haiku` on a stream-JSON worker accept a request asking for a different model mid-stream? Probably no; verify before assuming 1-worker-per-model.
3. **System prompt drift** — if the CLI's auto-injected system prompt changes between worker boot and request time (CLI upgrade, plugin reload), cache invalidation needs to be detected. A simple heuristic: invalidate the worker if `claude --version` drifts from boot.
4. **Idle eviction timing** — too aggressive → cold-spawn penalty returns. Too lazy → idle RAM and stale workers. 30min default is a guess; needs tuning against real traffic.
5. **Worker stderr noise** — the CLI emits warnings ("no stdin data received in 3s") that the current `execFile` path swallows. Persistent workers will spam these; need a stderr drain that logs once-per-minute, not once-per-line.

---

## Out of scope for v7.2

- Cross-provider fallback on 429 (claude-code → copilot). Already deliberately excluded: pinning to claude-code expresses user intent, not just "anything that works".
- A general-purpose work queue / scheduler. The use case is small (2-3 concurrent fallback calls); a fixed pool is sufficient.
- Worker pools for other CLI-based providers. claude-code is the only one where CLI spawn dominates latency.

---

## Acceptance criteria (for plan-phase to refine)

- A sonnet `say OK` probe routed through `claude-code` provider via the worker pool completes in **≤ 3s steady-state** (warm pool, cache hit). Cold-spawn first call may still take ~10s.
- Pool survives at least 1 worker crash (e.g. SIGKILL one PID) without dropping subsequent requests.
- Idle eviction after N minutes verified by watching `ps` and seeing the worker exit, then a fresh request spawns a new one within bounds.
- `LLM_PROXY_DISABLE_WORKER_POOL=1` escape hatch reverts to the current per-call `execFile` path.
- Dashboard's `claude-code` latency column shows the speedup (median latency for claude-code/sonnet drops from ~14s to ≤3s within 24h of rollout).
