# v7.1 Research Seed — Knowledge Management Unification

Three knowledge-management systems exist in the coding org. Each covers the same
abstract stages (ingest → persist → consolidate → surface) but has independently
evolved its backends, ontology model, and consolidation cadence. This doc maps
them stage-by-stage to identify a shared core (**KM-Core**) and what stays
per-system configuration. It is the PRD seed for milestone v7.1.

## Systems compared

| Tag | System | Path | Role |
|-----|--------|------|------|
| **A** | Online Learning | `coding/integrations/observation-*`, `scripts/observation-*`, `src/live-logging/*` | Push-driven, transcript-derived observations → daily digests → weekly insights |
| **B** | Offline UKB | `coding/integrations/mcp-server-semantic-analysis` | Wave-analysis over git + code → hierarchical KG (Project→Component→SubComponent→Detail) |
| **C** | OKM | `_work/rapid-automations/integrations/operational-knowledge-management` | RCA/runbook/doc capture from Confluence/CodeBeamer/MkDocs → temporal KG |

## Proposed KM-Core (what converges)

**1. Storage** — Graphology + LevelDB + git-tracked JSON export already used by B and C. Extract into `km-core/storage/GraphKMStore` with pluggable graph schema. (A stays on SQLite for now — different workload: hot transactional writes vs. graph mutation.)

**2. Entity shape** — Adopt C's uniform entity (`ontologyClass`, `layer`, `confidence`, `validFrom`, `validUntil`, `provenance[]`). Kill B's `type`↔`entityType` split (memory item: `persistence-agent.ts:583` is the current divergence point).

**3. Ontology registry** — Adopt C's `OntologyRegistry` model (upper + N lower JSONs auto-discovered from `ontology/`). B's component-manifest becomes one type of lower ontology; A grows a minimal "online-learning" upper that types observations/digests/insights as ontology classes.

**4. Consolidation framework** — C's 4-stage abstraction (`extract → dedup → store → synthesize`) becomes the ingest-time framework. A's digest+insight roll-up and B's per-wave agents both implement these stages. Each system picks which stages run and on what cadence (per ingest vs. cron vs. wave). **Plus** C's post-hoc `resolveEntities()` graph-maintenance operation becomes a shared KM-Core API — runs across the entire graph by `ontologyClass`, catches duplicates that escaped per-batch dedup. All three systems benefit (A: long-running insight corpus, B: cross-wave duplicate components, C: already has it).

**5. Dedup pipeline** — C's layered approach (exact → embedding → LLM semantic) is the most efficient. A keeps embedding-cosine but plugs into the layered pipeline. B's name-Jaccard becomes the "exact-name" stage.

**6. Provenance + temporal validity** — Add `validFrom`/`validUntil`/`supersedes` to KM-Core entities. Backfill A's observations with `validFrom = createdAt`, B's entities with `validFrom = first-seen`.

**7. Query API** — Common REST contract: `/api/entities[:id]`, `/api/search`, `/api/clusters`, `/api/snapshots`, `/api/ontology/*`. A's `/api/observations|digests|insights` become typed views over `/api/entities?ontologyClass=…`.

**8. Snapshot/restore** — C's git-on-`.data/exports/` pattern works everywhere; A and B add it.

**9. Viewer** — Single web viewer parameterized by ontology config. Today B has sigma.js, C has D3; pick one (D3 has the richer feature set in C) and unify.

## What stays per-system

- **Ingest adapters** — transcript monitor (A), git+code-graph (B), Confluence/CodeBeamer/MkDocs (C). These are domain-specific and shouldn't converge.
- **Ontology files** — each system defines its own domain ontology JSON.
- **LLM provider/task-tier config** — per-system YAML, same shape, different content.
- **Domain eval logic** — `ObservationSanitizer` (A), `QualityAssuranceAgent` (B), schema validation (C). Each enforces domain-specific invariants the others don't care about.
- **A's SQLite hot path** — different workload (high-rate transactional writes from ETM) than a KG. Keep as-is; expose its content as KM-Core entities via a thin adapter.

## Open questions for the discuss phase

1. **Repo layout** — Does KM-Core live in `coding/lib/km-core/`, become a sibling repo, or get extracted to a published package? (C is in a separate repo; B is local; cross-repo refactor needs a story.)
2. **A's SQLite future** — Keep the SQLite hot path indefinitely, or eventually migrate observations into the graph too? (Affects how aggressive the unification is.)
3. **Viewer choice** — D3 (OKM's, richer) vs. sigma.js (VKB's, faster on large graphs)? Or keep both behind a common config?
4. **Backwards compat** — `.data/observation-export/*.json` and `.data/knowledge-export/coding.json` have established commit hygiene (two-commit pattern, OKB-baseline guard). Do we preserve those paths or break them in v7.1?
5. **Migration cutover** — Do B and C migrate to KM-Core in parallel or sequentially? Risk: parallel doubles review burden; sequential leaves the laggard on duplicate code for a long window.

## Memory-flagged constraints to honor

- Phase 10 runtime issues still open in B (embeddings not reaching GraphDB, KG-Ops sub-step visibility, persist-node expansion). Address these in B's KM-Core migration phase rather than as separate fixes.
- Wave-analysis race condition (`workflow-runner.ts:469–530`) — refactor target for the wave-controller pass.
- Numpy/QdrantSync non-fatal error in `embedding_generator.py` — decide in v7.1 whether KM-Core embeds Qdrant or treats it as optional sidecar.
