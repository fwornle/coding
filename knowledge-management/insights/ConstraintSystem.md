# ConstraintSystem

**Type:** Component

The ConstraintSystem is a multi-layered enforcement framework that validates code actions and file operations during Claude Code sessions. It operates through a hook-based architecture where the UnifiedHookManager (lib/agent-api/hooks/hook-manager.js) intercepts agent tool events (pre-tool, post-tool, etc.) and routes them through registered handlers loaded from user-level (~/.coding-tools/hooks.json) and project-level (.coding/hooks.json) configuration files. Violations detected during these checks are captured, persisted, and surfaced through a dashboard for monitoring.

# ConstraintSystem — Technical Reference

## What It Is

The ConstraintSystem is a multi-layered enforcement framework that validates code actions and file operations during Claude Code sessions. Its core interception logic lives in `lib/agent-api/hooks/hook-manager.js` (UnifiedHookManager), with configuration loading in `lib/agent-api/hooks/hook-config.js` (HookConfigLoader), violation persistence in `scripts/violation-capture-service.js` (ViolationCaptureService), and content staleness checking in `integrations/mcp-server-semantic-analysis/src/agents/content-validation-agent.ts` (ContentValidationAgent). Shared violation state flows through `.mcp-sync/` — a filesystem sync directory that bridges live sessions with the ConstraintDashboard's historical view.

Within the broader Coding project, ConstraintSystem sits alongside LiveLoggingSystem, LLMAbstraction, and SemanticAnalysis as a peer component under the root Coding node. Where LiveLoggingSystem intercepts transcript output and LLMAbstraction mediates provider calls, ConstraintSystem intercepts *tool invocations* — the actions agents take — and enforces behavioral rules around them.

## Architecture and Design

![ConstraintSystem — Architecture](images/constraint-system-architecture.png)

The system follows a hook-based interception architecture. UnifiedHookManager acts as the single choke point for all agent tool lifecycle events, maintaining a priority-sorted `Map` of handlers per event type (`pre-tool`, `post-tool`, etc.). The priority field is numeric — lower numbers execute first — and duplicate handler IDs are resolved by `findIndex` replacement rather than accumulation. This gives the system a deterministic execution order without requiring a separate scheduler.

Configuration is resolved through a two-level merge in HookConfigLoader: user-global config at `~/.coding-tools/hooks.json` loads first, then project-local `.coding/hooks.json` overrides it. `DEFAULT_CONFIG` provides fallback structure so the system degrades gracefully when configs are absent. A deliberate design decision is that `validateConfig()` emits *warnings* rather than throwing on malformed entries — structural rules (type and path required, agents must be array, priority must be number) are enforced diagnostically, allowing partial configs to load rather than failing entirely. This trades strict correctness for operational resilience: a misconfigured handler doesn't abort the entire hook chain.

Agent-scoped filtering adds a second dimension of control. Each handler carries an optional `agents` array (typed in the `HookHandler` typedef in `hook-manager.js`) that restricts execution to named agents; an empty array means all-agents applicability. This allows the same hook configuration file to serve multi-agent environments — relevant given that the broader Coding project configures agents like Claude, Copilot, and Mastra under `config/agents/` — without requiring separate config files per agent.

## Implementation Details

**UnifiedHookManager** (`lib/agent-api/hooks/hook-manager.js`) is the runtime core. It holds handlers in a priority-sorted `Map` keyed by event type. On each tool event, it retrieves the relevant handler list, filters by agent scope, and executes in priority order. The deduplication-by-replacement strategy (using `findIndex`) means re-registering a handler ID updates it in place while preserving sort order.

**HookConfigLoader** (`lib/agent-api/hooks/hook-config.js`) implements the merge strategy. The two-level hierarchy — user then project — mirrors a pattern common across developer tooling (think `.gitconfig` vs repo-level config). The DEFAULT_CONFIG fallback means the system is always in a valid structural state even before any user config exists.

**ViolationCaptureService** (`scripts/violation-capture-service.js`) writes to two targets simultaneously: `.mcp-sync/session-violations.jsonl` (append-only, session-scoped) and `.mcp-sync/violation-history.json` (rolling JSON array capped at 1000 entries). The dual-write design separates concerns — the JSONL log is suitable for streaming/tailing during a live session, while the JSON history file is what ConstraintDashboard reads for cross-session analytics. `calculateStatistics()` operates directly on the in-memory violations array rather than re-reading from disk, computing severity breakdowns, 24-hour windowed counts, most common constraint ID, and per-session averages on each update. This means statistics are always current relative to the in-memory state but do not require disk I/O on each metric query.

**ContentValidationAgent** (`integrations/mcp-server-semantic-analysis/src/agents/content-validation-agent.ts`) contributes a staleness detection layer via git history. The `useGitBasedDetection` flag enables a path that compares recorded entity observations against recent commits, with a 60-second TTL cache (`GIT_CACHE_TTL_MS`) for batched commit fetches. This avoids redundant git subprocess calls when validating multiple entities in a single analysis pass — important given that ContentValidationAgent operates inside the SemanticAnalysis MCP server's batch pipeline.

## Integration Points

The ConstraintSystem's primary integration surface is the `.mcp-sync/` directory, which serves as a filesystem-based message bus between live session components and the ConstraintDashboard. ViolationCaptureService writes; ConstraintDashboard reads `violation-history.json`. This loose coupling means the dashboard does not need a live connection to the session — it can run as a separate process (consistent with the DockerizedServices architecture, where the constraint monitor API and dashboard are co-located under supervisord alongside the semantic analysis MCP server).

ContentValidationAgent is a child of ConstraintSystem but executes within the SemanticAnalysis MCP server's agent pipeline. Its git-cache integration means it shares infrastructure with however the SemanticAnalysis component manages its batch-analysis orchestration, bridging the two sibling components at the implementation level.

HookConfigLoader's two-level config merge gives it an implicit dependency on filesystem conventions: `~/.coding-tools/hooks.json` for user-level and `.coding/hooks.json` for project-level. Any tooling that provisions agent environments needs to be aware of these paths as the canonical config locations for the hook system.

## Usage Guidelines

**Config authoring:** Handler entries in `hooks.json` must include `type` and `path` fields; `agents` must be an array (not a string); `priority` must be a number. Violations of these rules produce warnings, not errors — so malformed handlers silently drop out of the execution chain. Developers should treat hook config validation warnings as errors during setup.

**Priority assignment:** Lower numeric priority executes first. When ordering matters (e.g., a security check must run before a logging handler), assign the security handler a lower priority number explicitly. Gaps in priority numbers (e.g., 10, 20, 30) give room to insert handlers later without renumbering.

**Agent scoping:** Use the `agents` array to target handlers at specific agents when deploying in multi-agent environments. Leave it empty (not omitted) to signal all-agents applicability — the typedef in `hook-manager.js` treats empty array as the all-agents sentinel.

**Violation cap:** The rolling 1000-violation cap in `violation-history.json` means long-running projects with frequent violations will lose oldest history. If historical retention beyond 1000 entries matters, the JSONL session logs serve as the archival source.

**Git-based staleness:** ContentValidationAgent's 60-second TTL cache is scoped to batched validation runs. If triggering validation programmatically with high frequency, be aware that staleness signals may lag up to 60 seconds behind the actual git state. The `useGitBasedDetection` flag allows this path to be disabled if git availability cannot be assumed in the execution environment.

- [LiveLoggingSystem](./LiveLoggingSystem.md) -- [LLM] The TranscriptAdapter abstract class in `lib/agent-api/transcript-api.js` enforces a strict interface contract that all agent-specific adapters must satisfy. Subclasses must implement five methods: `getAgentType()` (returns a string identifier like 'claude' or 'copilot'), `getTranscriptDirectory()` (returns the filesystem path where native transcripts are stored), `readTranscripts()` (reads raw agent-native files), `convertToLSL()` (transforms them into the unified LSLSession/LSLEntry format), and `getCurrentSession()` (returns the active session for live ingestion). The LSLMetadata type includes a `timeWindow` field (formatted as e.g. '0800-0900') that the adapter is responsible for populating, meaning the adapter layer also owns the hourly-bucketing logic that drives file routing downstream.
- [LLMAbstraction](./LLMAbstraction.md) -- LLMAbstraction is a multi-layered abstraction over LLM providers that enables provider-agnostic model calls through three distinct execution paths: mock mode (for testing), local inference via Docker Model Runner (DMR), and public cloud providers (Anthropic, OpenAI, Groq) routed through a rapid-llm-proxy. The system supports per-agent and global mode switching stored in `.data/workflow-progress.json`, allowing runtime toggling between modes without code changes. Provider selection follows a priority chain from per-agent overrides to global mode to legacy flags.

The architecture centers on a proxy-mediated request pattern where most LLM calls route through a local rapid-llm-proxy daemon (default port 12435) via `/api/complete`, enabling centralized token tracking, tier-based routing, and telemetry attribution. The `llm-with-process.ts` module exists specifically to inject a `process` tag into proxy requests — a gap in the SDK's `LLMService.complete()` that caused all wave-analysis calls to appear as `process='unknown'` in token-usage telemetry. DMR provider uses an OpenAI-compatible API at `localhost:${DMR_PORT}/engines/v1` for fully local inference.

Key patterns include: environment-variable-driven URL resolution with multiple fallback levels, singleton client instances with health-check caching, YAML-based provider configuration with env-var expansion, and SDK-shape response normalization ensuring downstream consumers work unchanged regardless of which provider path was taken.
- [DockerizedServices](./DockerizedServices.md) -- [LLM] The multi-service container architecture uses supervisord (docker/supervisord.conf) to manage multiple long-running processes within a single Docker container defined in docker/Dockerfile.coding-services. This is an intentional design choice that trades container isolation purity for operational simplicity — rather than running separate containers for the semantic analysis MCP server, constraint monitor API, and dashboard service, all are co-located under supervisord's process supervision. The entrypoint script handles startup orchestration sequencing before handing off to supervisord. A new developer should be aware that this means a crash in one supervised service does not terminate the container, supervisord will attempt restarts, but it also means a single container failure takes down all co-located services simultaneously. The bind mount pattern `CODING_ROOT=/coding` connects the host repository filesystem into the container, making the container dependent on the host directory layout at runtime rather than baking code into the image.
