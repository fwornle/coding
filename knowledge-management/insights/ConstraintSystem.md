# ConstraintSystem — Technical Insight Document

## What It Is

The ConstraintSystem is the component within the broader **Coding** project responsible for intercepting agent tool/prompt events, evaluating them against configurable rules, and persisting any resulting violations for downstream monitoring. It is physically organized across three distinct locations in the repository, each owning a separate concern:

- **Hook interception layer** — `lib/agent-api/hooks/`, anchored by `hook-config.js` (the `HookConfigLoader` class) and `hook-manager.js` (the `UnifiedHookManager` class).
- **Violation persistence layer** — `scripts/violation-capture-service.js` (the `ViolationCaptureService` class) writing to `.mcp-sync/session-violations.jsonl` and `.mcp-sync/violation-history.json`.
- **Monitoring and integration layer** — `integrations/mcp-constraint-monitor/`, which contains both the monitor itself and a dedicated `dashboard/` sub-project, plus the formal hook-envelope specification at `integrations/mcp-constraint-monitor/docs/CLAUDE-CODE-HOOK-FORMAT.md`.

The semantic side of detection is documented separately in `docs/semantic-constraint-detection.md` and `docs/semantic-detection-design.md`, and is expected to interoperate with `integrations/mcp-server-semantic-analysis/`. The component sits alongside sibling components such as LiveLoggingSystem, LLMAbstraction, DockerizedServices, KnowledgeManagement, CodingPatterns, and SemanticAnalysis under the Coding root, and it directly contains the **HookConfigLoader** child component as its configuration entry point.

## Architecture and Design

The ConstraintSystem follows a layered architecture in which configuration loading, runtime dispatch, and violation persistence are cleanly separated. Configuration is owned by `HookConfigLoader`, which implements a **two-level hierarchy** — user-wide defaults at `~/.coding-tools/hooks.json` and project-specific overrides at `{project}/.coding/hooks.json`. The `mergeConfigs()` method applies project configuration on top of user configuration, with the project layer always taking precedence. This precedence rule enables teams to enforce stricter or more specific constraints per repository without requiring developers to alter their global setup, but it intentionally provides no "lock" mechanism — a project file can fully override anything a user has set globally.

![ConstraintSystem — Architecture](images/constraint-system-architecture.png)

Runtime dispatch is centralized in `UnifiedHookManager`, which stores registered handlers in a `Map<string, HookHandler[]>` keyed by `HookEvent` enum values (`pre-tool`, `post-tool`, `pre-prompt`, `post-prompt`, `startup`, `shutdown`, `error`). Handlers are sorted within each event bucket by a numeric `priority` field — lower values execute earlier — producing a deterministic, explicit execution order. The manager also de-duplicates handlers by their `id` field via `registerHandler()`, so re-registering the same ID replaces the previous binding rather than appending a duplicate. This combination of priority ordering and ID-keyed identity is the core dispatch contract on which all constraint enforcement depends.

Persistence is delegated to `ViolationCaptureService`, which implements a **dual-format write strategy**: an append-only JSONL log for streaming/audit consumers and a bounded JSON history (capped at 1000 entries) for fast random access to recent violations. The `.mcp-sync/` prefix signals that this data is intended to flow upward into the MCP (Model Context Protocol) ecosystem, where the `mcp-constraint-monitor` integration and its dashboard consume it. Statistics are computed entirely in-memory by `calculateStatistics()`, keeping the service dependency-free at the cost of O(n) scans.

A pervasive design choice across both `HookConfigLoader.validateConfig()` and `UnifiedHookManager.loadConfig()` is **warn-and-continue error handling**: invalid handlers (missing `type`/`path`, non-numeric `priority`, non-array `agents`) and unrecognized hook event names produce warnings rather than exceptions. This makes the system tolerant of forward-compatible configuration during rolling upgrades but introduces a class of silent misconfiguration risks.

## Implementation Details

**HookConfigLoader** (`lib/agent-api/hooks/hook-config.js`) is the sole entry point for reading configuration files. Its `mergeConfigs()` method performs the layered merge that gives project-level overrides full precedence over user-level defaults, while `validateConfig()` enforces structural rules on each handler descriptor: every handler must declare `type` and `path`, `priority` must be numeric, and `agents` must be an array. Violations of these rules are logged as warnings; the loader does not throw or return an error sentinel.

**UnifiedHookManager** (`lib/agent-api/hooks/hook-manager.js`) implements the dispatch mechanics. Its internal `Map<string, HookHandler[]>` is keyed by `HookEvent` enum values and each value list is sorted by numeric `priority` after every registration so dispatch is a simple ordered traversal. `registerHandler()` performs ID-based de-duplication: if a handler with the same `id` already exists in a list, it is replaced rather than appended. `loadConfig()` iterates the merged configuration and silently skips any hook event name it does not recognize, again logging a warning rather than raising.

**ViolationCaptureService** (`scripts/violation-capture-service.js`) maintains an in-memory `violations` array as the authoritative working set, with two persistence projections: line-oriented append-only JSONL at `.mcp-sync/session-violations.jsonl` (suitable for `tail`-style streaming and complete audit history) and a structured, bounded JSON history at `.mcp-sync/violation-history.json` (capped at 1000 entries, with older entries presumably evicted). The `calculateStatistics()` method computes 24-hour rolling counts, per-severity breakdowns, the most-common constraint ID, and average violations per session by linearly scanning the in-memory array — there is no index, no precomputed counter, and no incremental update path.

The data envelope traveling from hook interception through to the monitoring layer is formally specified in `integrations/mcp-constraint-monitor/docs/CLAUDE-CODE-HOOK-FORMAT.md`. This versioned document establishes the stable interface between the hooks subsystem and downstream consumers including the dashboard sub-project under `integrations/mcp-constraint-monitor/dashboard/`. Semantic detection capabilities, designed but documented separately in `docs/semantic-constraint-detection.md` and `docs/semantic-detection-design.md`, are expected to integrate via the sibling SemanticAnalysis component and the `integrations/mcp-server-semantic-analysis/` server.

## Integration Points

The ConstraintSystem's primary integration surface is the **HookConfigLoader child**, which owns both configuration file locations and is the sole bridge between on-disk JSON and the runtime `UnifiedHookManager`. Any subsystem that needs to introduce constraint behavior interacts with the system either by writing a handler entry into one of the two `hooks.json` files or by calling `UnifiedHookManager.registerHandler()` programmatically.

Downstream, the persistence layer integrates with the broader MCP ecosystem via the `.mcp-sync/` directory. The structured JSON history is consumed by the **mcp-constraint-monitor** integration (which includes a status-line integration documented at `integrations/mcp-constraint-monitor/docs/status-line-integration.md`) for fast lookup of recent violations, while the JSONL log is the source of truth for complete audit trails. The dashboard sub-project under `integrations/mcp-constraint-monitor/dashboard/` consumes the same files for visualization.

The component also has implicit integration relationships with several siblings under the Coding root. The **SemanticAnalysis** sibling supplies semantic-intent classification consumed by the constraint detection design documents. The **CodingPatterns** sibling codifies the singleton guard pattern that any stateful manager inside the ConstraintSystem (such as the `UnifiedHookManager`) is expected to follow — a new developer instantiating these managers should go through the designated accessor rather than calling `new` directly. The **KnowledgeManagement** sibling does not have a direct code linkage evident in the observations, but the `.mcp-sync/` data flow positions violations as a candidate input for graph-based knowledge tracking.

The hook-envelope format is the most critical cross-cutting integration contract. Because it spans three locations — hook interception code in `lib/agent-api/hooks/`, the `CLAUDE-CODE-HOOK-FORMAT.md` specification, and the monitoring/dashboard consumers under `integrations/mcp-constraint-monitor/` — any change to the envelope must be propagated across all three. The system currently manages this through documentation rather than a shared schema or TypeScript type definition, which is a notable maintainability boundary.

## Usage Guidelines

**Handler IDs must be globally unique across configuration levels.** Because `UnifiedHookManager.registerHandler()` de-duplicates by `id`, a project-level handler can silently shadow a user-level handler that shares its ID. Developers introducing new constraint handlers should namespace their IDs (e.g., prefix with the package or feature name) to avoid accidental shadowing.

**Assign handler priorities deliberately, especially for handlers with side effects.** The sorted-by-priority execution order means that handlers performing file writes, process signals, or other irrecoverable side effects should be ordered after pure-validation handlers that might reject the event. Acting on an event that a higher-priority rule would have rejected produces incorrect behavior; the priority field is the only mechanism to control this.

**Do not assume configuration errors will surface as exceptions.** Both `validateConfig()` and `loadConfig()` use warn-and-continue handling. A misspelled `type`, a non-numeric `priority`, an unrecognized hook event name, or a non-array `agents` field will produce only a log warning, and the affected handler will silently fail to register. Teams should write integration tests that assert specific handlers are present in the `UnifiedHookManager` registry after configuration load, rather than relying on the loader to fail loudly.

**Choose persistence consumers deliberately based on access pattern.** Use `.mcp-sync/session-violations.jsonl` when you need the complete historical record or streaming reads — it is append-only and grows without bound. Use `.mcp-sync/violation-history.json` when you need fast, random access to recent violations — it is capped at 1000 entries and is not a complete record. The two are not equivalent replicas and treating them as such will produce subtle bugs.

**Be aware of the in-memory scan cost in `calculateStatistics()`.** Every call performs a full linear scan over the in-memory `violations` array, and the 24-hour rolling count requires timestamp comparison on every entry. If you are wiring this into a frequently-polled UI surface such as a status line, consider rate-limiting calls or caching the result, as the current implementation has no incremental counter mechanism.

**Recognize the security boundary in the two-level configuration model.** A project's `.coding/hooks.json` can fully override any user-level setting — there is no "lock" or "user-only" mechanism. Compliance constraints that must not be bypassed cannot be enforced at the user level alone; they require either external validation or alternative enforcement outside this system.

**Coordinate changes to the hook envelope across three locations.** Modifying the hook data format requires synchronized updates to the hook interception code in `lib/agent-api/hooks/`, the `CLAUDE-CODE-HOOK-FORMAT.md` specification, and the monitoring/dashboard consumers under `integrations/mcp-constraint-monitor/`. Because there is no shared schema artifact, this coordination is currently a manual, documentation-driven process.

- [LiveLoggingSystem](./LiveLoggingSystem.md) -- [LLM] The TranscriptAdapter class in lib/agent-api/transcript-api.js establishes a strict plugin contract for integrating new agent conversation sources into the LSL pipeline. It requires five abstract methods: getAgentType(), getTranscriptDirectory(), readTranscripts(), convertToLSL(), and getCurrentSession(). This design cleanly separates concerns — the base class owns the lifecycle and dispatch logic, while subclasses own the format-specific parsing. Claude Code transcripts live at ~/.claude/projects/<project>/conversation.jsonl, a JSONL file that grows append-only during a session; the Copilot CLI adapter targets a different directory structure. The convertToLSL() method is the normalization seam, responsible for mapping each agent's native message structure into the unified LSL typed-entry format with types: user, assistant, tool_use, tool_result, system, and error. A new developer adding a third agent integration (e.g., Cursor, Aider) only needs to subclass TranscriptAdapter and implement these five methods — no changes to the core LSL infrastructure are required. The getCurrentSession() method is particularly important: it must return the 'live' session object so the polling loop knows which entries are part of the current conversation versus a historical one, which implies adapters must implement their own session-boundary detection logic appropriate to their agent's format.
- [LLMAbstraction](./LLMAbstraction.md) -- [LLM] The LLMAbstraction layer implements a three-tier mode resolution hierarchy in `getLLMMode()` within `llm-mock-service.ts` that prioritizes specificity over generality: a per-agent override stored in `llmState.perAgentOverrides` takes precedence over the `llmState.globalMode`, which in turn takes precedence over a legacy `mockLLM` boolean field. This design is significant because it allows the system to run mixed-mode configurations in production — for example, running a lightweight code-indexing agent against the local Docker Model Runner while routing a more complex reasoning agent to a cloud API like Anthropic or OpenAI, all within a single workflow execution. The legacy `mockLLM` boolean is retained explicitly for backward compatibility, meaning older configuration files or callers that predate the `llmState` structure continue to function without migration. New developers should be aware that if they observe unexpected behavior when `globalMode` is set but `mockLLM` is also present in a persisted state file, the `mockLLM` boolean is the lowest-priority fallback and will only activate when neither of the higher-priority fields are populated.
- [DockerizedServices](./DockerizedServices.md) -- [LLM] The ProcessStateManager (PSM) singleton implements a deliberate decoupling between service identity and process identity across both `scripts/api-service.js` and `scripts/dashboard-service.js`. Each script follows an identical structural pattern: spawn a child process via Node's `child_process` module, register the resulting process handle with the PSM via `psm.registerService()`, wire up `SIGTERM`/`SIGINT` forwarding so that signals delivered to the wrapper propagate to the child, and call `psm.unregisterService()` in the exit handler. This indirection means that the rest of the system (including `scripts/health-coordinator.js`) can query the PSM registry without holding direct references to OS-level process IDs. The practical consequence for developers is that a service restart — where a new child process replaces the old one — does not require the health coordinator or any consumer of PSM state to be aware of the PID change; only the wrapper scripts update the registry. This pattern also cleanly isolates the restart/retry logic in `lib/service-starter.js` from signal-handling responsibilities, since the wrapper owns the process lifecycle signals while the starter owns the retry policy. A new developer should note that adding a new containerized service almost certainly means creating a new wrapper script that replicates this boilerplate rather than centralizing it, which is a potential maintenance concern as the number of services grows.
- [KnowledgeManagement](./KnowledgeManagement.md) -- The KnowledgeManagement component provides knowledge graph storage, query, and lifecycle management for the Coding project. The component supports multiple entity types (System, Project, Component, SubComponent, Pattern, Detail) with ontology classification, bi-temporal staleness tracking, embedding vectors, and hierarchical parent/child relationships. It integrates with the MCP semantic analysis server via PersistenceAgent and GraphDatabaseAdapter, which route writes through the VKB API when the server is running or fall back to direct LevelDB access to avoid lock conflicts.
- [CodingPatterns](./CodingPatterns.md) -- [LLM] The project-wide singleton guard pattern is formally codified in `docs/puml/psm-singleton-pattern.puml` and manifests consistently wherever stateful managers are instantiated. The pattern follows a strict guard-and-return idiom: a module-level variable holds the single instance (initialized to null or undefined), and every access point checks that variable before constructing a new object. If an instance already exists, the existing reference is returned immediately without re-running any constructor or initialization logic. This prevents race conditions in async service environments where multiple subsystems might attempt to spin up the same stateful manager concurrently — a real concern in Node.js applications that use event-driven concurrency without explicit locking primitives. For new developers, the implication is that any class described as a 'manager' or 'session' object in this codebase should be assumed to follow this pattern: do not call `new` directly on these classes from arbitrary call sites; instead, always go through the designated factory or accessor function that enforces the singleton contract. The PlantUML diagram in `docs/puml/psm-singleton-pattern.puml` is authoritative and should be consulted before introducing any new singleton-style manager to ensure the guard logic is structurally consistent with the rest of the project.
- [SemanticAnalysis](./SemanticAnalysis.md) -- [LLM] The `BaseAgent<TInput, TOutput>` abstract class defined in `src/agents/base-agent.ts` establishes a rigid, five-method execution contract that every agent in the pipeline must implement: `process()`, `calculateConfidence()`, `detectIssues()`, `generateRouting()`, and `applyCorrections()`. This is not a loose interface — each method is called sequentially within a standardized envelope, meaning an agent cannot skip confidence calculation or issue detection even if it has nothing meaningful to report for those phases. The resulting `AgentResponse` envelope carries not just the domain output but also metadata (timestamps, model usage), routing suggestions for downstream agents, and a corrections list for self-healing. For a new developer, this means that implementing a new agent is less about writing a single processing function and more about correctly filling all five lifecycle slots; an agent that returns empty stubs for `detectIssues()` or `generateRouting()` will still compile and run, but the orchestrating pipeline likely depends on those fields being populated to make branching decisions. The generic type parameters `<TInput, TOutput>` allow the base class to be reused across wildly different domains — from raw git commit arrays (SemanticAnalysisAgent) to ontology classification batches (OntologyClassificationAgent) — without sacrificing static type safety on the input/output contracts.
