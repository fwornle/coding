# ConstraintSystem

**Type:** Component

The ConstraintSystem is a monitoring and enforcement layer that validates code actions and file operations against configured rules during Claude Code sessions. It operates through a hook-based architecture where constraint checks are triggered at key lifecycle points (pre-tool, post-tool, etc.) and violations are captured, persisted, and surfaced to dashboards. The system integrates with Claude Code's native hook mechanism via configuration files at user-level (~/.coding-tools/hooks.json) and project-level (.coding/hooks.json), with project config overriding user config.

## What It Is

The ConstraintSystem is a monitoring and enforcement layer implemented across `lib/agent-api/hooks/`, `scripts/violation-capture-service.js`, and `integrations/mcp-constraint-monitor/`. It validates code actions and file operations against configured rules during Claude Code sessions, capturing violations and surfacing them to dashboards. The system sits as one of seven major components under the Coding parent, alongside siblings like SemanticAnalysis and LiveLoggingSystem, and decomposes into four child components: HookConfigurationLayer, ConstraintRuleEngine, ViolationPersistenceStore, and ConstraintDashboard.

## Architecture and Design

The ConstraintSystem is built around Claude Code's native hook mechanism, inserting itself at lifecycle points (pre-tool, post-tool, etc.) via JSON configuration files at two levels: user-global (`~/.coding-tools/hooks.json`) and project-local (`.coding/hooks.json`). This two-tier configuration mirrors the approach seen in CodingPatterns, where MCP integrations use structured configuration files to declare capabilities. Project configuration takes override precedence over user configuration, enabling per-repository constraint customization without disrupting global defaults.

The `UnifiedHookManager` (`lib/agent-api/hooks/hook-manager.js`) is the central dispatch layer. It maintains per-event handler arrays sorted by numeric priority, supporting handler deduplication by ID. Critically, handlers carry an `agents[]` array field that gates execution by agent identity — a design that makes the system multi-agent-aware from the start, consistent with the broader project's support for multiple agent runtimes (Claude Code, Copilot CLI, etc.) seen in LiveLoggingSystem's `TranscriptAdapter` pattern.

Violation persistence follows a dual-write strategy in `ViolationCaptureService` (`scripts/violation-capture-service.js`): a JSONL stream at `.mcp-sync/session-violations.jsonl` for live consumption and a capped JSON history file at `.mcp-sync/violation-history.json` (1000-record ceiling) for dashboard display via ConstraintDashboard. This separation between streaming state and display state is a deliberate architectural boundary — the ViolationPersistenceStore child component is explicitly decoupled from in-memory hook execution state, as noted in the dashboard's dedicated sub-directory (`integrations/mcp-constraint-monitor/dashboard/`).

The ConstraintRuleEngine extends beyond literal rule matching. Per `integrations/mcp-constraint-monitor/docs/semantic-constraint-detection.md`, it employs meaning-aware matching — a semantic layer that parallels the SemanticAnalysis sibling component's coordinator-based pipeline. The HookConfigurationLayer's schema contract is documented in `integrations/mcp-constraint-monitor/docs/CLAUDE-CODE-HOOK-FORMAT.md`, which defines exactly what JSON Claude Code emits at each lifecycle point, giving the rule engine reliable structured input.

## Implementation Details

`HookConfigLoader` (`lib/agent-api/hooks/hook-config.js`) merges user and project configs, defaulting to a `DEFAULT_CONFIG` with a 30,000ms timeout and `stopOnError: false`. The `validateConfig()` method enforces structural invariants — `type` and `path` are required per handler, `priority` must be numeric, `agents` must be an array — but logs warnings rather than throwing, making validation non-blocking. This is a deliberate trade-off: misconfigured handlers degrade gracefully rather than halting the session, consistent with the `stopOnError: false` default posture.

`ViolationCaptureService` structures each violation record with `constraint_id`, `severity`, `pattern`, tool name, sanitized tool params (with `REDACTED` substitution for fields matching password/token/key/secret/auth), and repository context from `process.cwd()`. The sanitization step is an explicit security boundary — sensitive credential-like values are scrubbed before persistence, preventing secrets from leaking into JSONL logs or dashboard history. The `calculateStatistics()` method derives real-time metrics — last-24h counts, severity breakdowns, most-common constraint ID, average violations per session — directly from the in-memory violations array, avoiding a separate aggregation store at the cost of recomputing on each call.

## Integration Points

The ConstraintSystem hooks into Claude Code's native lifecycle via configuration files, making its activation dependent on those files being present and correctly formed. It writes to `.mcp-sync/` paths shared with other components (the `mcp-` prefix aligns with the MCP server pattern documented under CodingPatterns), suggesting the sync directory is a shared inter-component communication surface. The JSONL stream in particular implies a consumer — likely the ConstraintDashboard or an external monitoring process — polling or tailing that file.

The `agents[]` field on each `HookHandler` creates a direct dependency on agent identity resolution, tying the ConstraintSystem to whatever runtime provides agent identity — the same concern that drives LiveLoggingSystem's pluggable `TranscriptAdapter` design. Adding a new agent type to the broader Coding platform requires ensuring its identity string is registered in relevant hook handler configurations.

## Usage Guidelines

Developers configuring constraints should prefer project-level `.coding/hooks.json` over user-level config for repository-specific rules, relying on override precedence to keep project rules authoritative. All handlers must include `type` and `path` fields; missing these will produce warnings but the handler will be silently skipped — a failure mode that can be difficult to detect without monitoring validation logs.

The `stopOnError: false` default means constraint violations do not block tool execution by default. Teams that want enforcement (not just monitoring) must explicitly override this default in their hook configuration. The 1,000-record cap on `violation-history.json` is a fixed ceiling; high-violation environments will silently drop older records, so external archiving of the JSONL stream is advisable for audit purposes.

When adding new handler logic, the numeric `priority` field controls execution order within an event — lower or higher values (the sort direction should be confirmed against `hook-manager.js` source) determine which handlers run first. Handler deduplication by ID means re-registering the same handler ID is safe and idempotent, but changing handler behavior while reusing an ID requires ensuring the old registration is fully replaced.

### Siblings
- [LiveLoggingSystem](./LiveLoggingSystem.md) -- [LLM] The `TranscriptAdapter` abstract base class in `lib/agent-api/transcript-api.js` defines a pluggable adapter contract that decouples the LSL pipeline from specific agent implementations. The five abstract methods — `getAgentType()`, `getTranscriptDirectory()`, `readTranscripts()`, `convertToLSL()`, and `getCurrentSession()` — establish a clear interface that concrete adapters (e.g., for Claude Code and Copilot CLI) must implement. This pattern means the core LSL infrastructure never directly references agent-specific transcript formats or filesystem layouts. A new developer adding support for a third agent (say, Cursor or Aider) would subclass `TranscriptAdapter` and implement only these five methods without touching the converter, file manager, or validation layers. The `watchTranscripts()` method is notably NOT abstract — it is a concrete polling implementation shared by all adapters, using `setInterval` with a default 1000ms interval and a `lastEntryCount` integer cursor that advances only when new entries are detected, then fires registered callbacks with only the delta entries. This means the watch mechanism is O(n) in total entry count per poll cycle, which could become a performance concern for very long sessions.
- [LLMAbstraction](./LLMAbstraction.md) -- LLMAbstraction is a multi-layered abstraction over LLM providers that enables provider-agnostic model calls across public APIs (Anthropic, OpenAI, Groq), a local Docker Model Runner (DMR), and a mock mode for testing. The system routes requests through a rapid-llm-proxy daemon (port 12435) that handles provider selection, tier-based routing, and per-call telemetry attribution. Mode selection is dynamic and per-agent, stored in workflow-progress.json, supporting global and per-agent overrides across three modes: 'mock', 'local' (DMR), and 'public' (cloud APIs).
- [DockerizedServices](./DockerizedServices.md) -- [LLM] The DockerizedServices component uses a dual-probe health checking architecture implemented in lib/utils/service-probe.js that strictly separates HTTP-based and TCP-based service verification. probeHttpHealth() issues HTTP/HTTPS requests and maps 2xx/3xx response codes to the 'running' state, while probeTcpPort() opens a raw net.Socket connection to verify port reachability for non-HTTP protocols like Memgraph's Bolt protocol. A critical architectural invariant (documented as SPEC R6) enforces that neither probe ever returns the string 'healthy'—only 'running', 'stopped', or 'unknown' are valid return values. This distinction matters because health-coordinator.js in scripts/health-coordinator.js consumes these probes on a 5-second polling interval and must be able to uniformly handle all service types (Next.js dashboard, Node.js API, Memgraph, Redis) without special-casing the protocol. New developers adding services must use probeTcpPort() for any non-HTTP service and must not introduce a fourth status string, or the health-coordinator's state machine will behave incorrectly.
- [KnowledgeManagement](./KnowledgeManagement.md) -- [LLM] The primary persistence mechanism in KnowledgeManagement is a single-key LevelDB strategy implemented in `src/knowledge-management/GraphDatabaseService.js`. Rather than storing each graph entity as a separate LevelDB key (which would enable partial reads and atomic per-entity updates), the entire Graphology in-memory graph is serialized as one JSON blob stored under the key `'graph'`. This blob contains all nodes, edges, and metadata. Writes are deferred using an `isDirty` flag — mutations to the graph set `isDirty = true`, and `_persistGraphToLevel()` is only called when a flush is explicitly triggered. This design optimizes for read-heavy, batch-write workloads but creates a risk of data loss if the process crashes between mutations and the next flush. New developers should be aware that any code path that modifies graph nodes or edges must ensure the flush cycle is triggered, or changes will silently remain only in memory.
- [CodingPatterns](./CodingPatterns.md) -- [LLM] The MCP (Model Context Protocol) server pattern is applied uniformly across all three major integrations: mcp-constraint-monitor, mcp-server-semantic-analysis, and code-graph-rag. Each integration follows a consistent directory layout with a README.md describing the tool interface, a docs/ subdirectory containing architectural detail, and configuration files that declare the tool's capabilities to AI agent hosts. This pattern means new integrations can be scaffolded by copying this structure and registering new tool definitions. The mcp-server-semantic-analysis integration further documents this in docs/api/README.md and docs/architecture/README.md, separating API surface from internal architecture. Developers adding a new MCP tool should expect to create at minimum: a README with tool description, an API reference doc, and a configuration file declaring tool schemas. The consistency of this pattern across integrations suggests it is an enforced project convention rather than an organic coincidence.
- [SemanticAnalysis](./SemanticAnalysis.md) -- [LLM] The SemanticAnalysis pipeline is structured around a coordinator pattern where specialized agents each extend BaseAgent<TInput,TOutput> defined in src/agents/base-agent.ts. This base class implements a strict template-method execute() that sequences six steps in order: process(), calculateConfidence(), detectIssues(), generateRouting(), applyCorrections(), and buildMetadata(). Every agent—CodeGraphAgent, SemanticAnalysisAgent, OntologyClassificationAgent, ContentValidationAgent—inherits this contract and returns a uniform AgentResponse envelope. This design means a new developer adding an agent only needs to implement the domain-specific process() logic; confidence scoring, issue detection, and metadata construction are guaranteed to run in a consistent order regardless of which agent is invoked. The tradeoff is that the template method imposes overhead steps even when an agent's output is trivially simple, and agents cannot short-circuit the sequence without throwing exceptions.
