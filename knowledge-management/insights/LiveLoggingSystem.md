# LiveLoggingSystem — Technical Insight Document

## What It Is

The LiveLoggingSystem is the structured transcript capture and normalization layer for the Coding project. Its core contract is defined in `lib/agent-api/transcript-api.js` (the `TranscriptAdapter` abstract class), with conversion logic in `lib/agent-api/transcripts/lsl-converter.js`, configuration validation in `scripts/validate-lsl-config.js`, and semantic classification in `integrations/mcp-server-semantic-analysis/src/agents/ontology-classification-agent.ts`. The system's purpose is to capture raw, agent-specific transcript files from multiple AI coding agents, normalize them into a typed Unified LSL (Log Session Log) format, redact sensitive content, and route the resulting output to time-windowed log files organized by user, project, and session hour.

Within the Coding parent hierarchy, LiveLoggingSystem sits alongside SemanticAnalysis, ConstraintSystem, and LLMAbstraction as a peer infrastructure component. Where SemanticAnalysis consumes classification signals and ConstraintSystem enforces behavioral rules, LiveLoggingSystem is the data collection foundation: without its normalized transcript output, neither downstream component has reliable input to work from.

## Architecture and Design

![LiveLoggingSystem — Architecture](images/live-logging-system-architecture.png)

The LiveLoggingSystem is built around a strict **adapter pattern** enforced by `TranscriptAdapter` in `lib/agent-api/transcript-api.js`. Every agent integration must implement five methods — `getAgentType()`, `getTranscriptDirectory()`, `readTranscripts()`, `convertToLSL()`, and `getCurrentSession()` — making the class the single contract point for all agent-specific logic. The TranscriptAdapterRegistry child component governs registration and lookup of these adapters, meaning the system is open to new agent types (Claude Code, Copilot, etc.) without requiring changes to the core pipeline.

**Live capture uses polling, not filesystem events.** The `watchTranscripts()` method drives a `setInterval` loop that periodically calls `readTranscripts()` and diffs the result against previously seen entries. This is an explicit portability decision: `fs.watch` exhibits well-documented inconsistencies inside Docker containers and on network-mounted filesystems (both realistic deployment contexts for this project, given DockerizedServices). The trade-off is introduced latency between an agent writing a transcript entry and the LSL system capturing it — a latency that is configurable but structurally unavoidable in a polling model.

**Session routing is time-windowed, not session-ID-keyed.** The SessionWindowManager child component produces hourly labels (e.g., `'0800-0900'`) that become routing keys in the `LSLMetadata` struct. Combined with `userHash`, `projectPath`, and `agentType`, this creates a directory structure that is browsable by humans without exposing raw session IDs or Unix usernames. The `userHash` itself is derived by SHA-256 hashing the `USER` environment variable and truncating to 6 hex characters — anonymizing filesystem paths in shared or auditable environments, though 6 hex characters (16M possibilities) offers limited collision resistance in large teams and should be reconsidered as `userHashLength` is configurable up to 12.

**Configuration validation is fail-fast and centralized.** The `LSLConfigValidator` class in `scripts/validate-lsl-config.js` enforces numeric bounds (`userHashLength` 6–12, `maxFileSize` 1MB–100MB, `batchSize` 10–1000) and is designed to catch misconfiguration at startup or CI time rather than allowing silent runtime degradation. The separation of redaction patterns into a YAML file (`.specstory/config/redaction-config.yaml`, managed by the RedactionEngine child component) from structural config (`.specstory/config/lsl-config.json`) reflects a deliberate ownership boundary: security and ops teams can update redaction rules — and document their rationale via YAML comments — without touching numeric runtime parameters.

## Implementation Details

### The LSL Entry Type System

The `LSLConverter` class in `lib/agent-api/transcripts/lsl-converter.js` enforces a six-type entry model: `user`, `assistant`, `tool_use`, `tool_result`, `system`, and `error`. Entry type drives serialization behavior rather than being purely documentary. Most consequentially, `tool_result` entries are conditionally excluded from output when the `includeToolResults` option is false — they are stripped from both the `toMarkdown()` human-readable path and the `toJSONL()`/`fromJSONL()` machine-consumption path. This means the inclusion decision is a first-class API concern, not a post-hoc filter.

Content truncation is applied uniformly at `maxContentLength: 10000` characters across all entry types. This guards against tool outputs (large file reads, verbose shell stdout) bloating log files. The `redactSecrets` option triggers pattern-based scrubbing before any content is written, which is architecturally critical: tool results can capture API keys or environment variable dumps from shell commands, so redaction must occur prior to serialization, not as a display-layer concern.

### Buffered Async Write Pattern

The async logging subsystem in `integrations/mcp-server-semantic-analysis/src/logging.ts` demonstrates the buffered write pattern that the broader LSL system mirrors. An in-memory buffer capped at 50 entries (`MAX_BUFFER_SIZE`) is drained by a `setInterval` running every 100ms (`FLUSH_INTERVAL_MS`). A capacity-triggered immediate flush prevents unbounded memory growth under burst conditions. Output is anchored to `CODING_REPO/.data/logs/semantic-analysis-YYYY-MM-DD.log` using the `CODING_REPO` environment variable, deliberately avoiding current-working-directory-relative paths that would scatter logs when the CLI or MCP server is invoked from arbitrary locations. Date-based filename rotation eliminates the need for an external rotation daemon, though sessions spanning midnight will have their entries split across two files — a known limitation of this approach.

### Classification Pipeline

The `OntologyClassificationAgent` in `integrations/mcp-server-semantic-analysis/src/agents/ontology-classification-agent.ts` applies a five-layer classification strategy to LSL entries: `heuristic` → `llm` → `hybrid` → `auto-assigned` → `unclassified`, in priority order. The `minConfidence: 0.6` threshold means any LLM classification below 60% confidence either escalates to the next layer or falls through to `unclassified` rather than emitting a low-confidence label. The system is designed to be **sparse-but-reliable** rather than exhaustive-but-noisy — an explicit design choice suited to downstream consumers that act on classification labels rather than merely display them. Per-entry token usage tracking enables cost attribution at the log-entry level, not just per-session.

## Integration Points

The LiveLoggingSystem's primary upstream dependency is the agent transcript filesystem — whichever path `getTranscriptDirectory()` returns for a given `TranscriptAdapter` implementation. It has no network dependencies for capture; the polling loop reads directly from disk. Downstream, its JSONL output feeds SemanticAnalysis, which consumes normalized entries for its batch classification pipeline. The ConstraintSystem similarly benefits from the normalized LSL stream as a source of behavioral evidence, though the exact integration boundary is governed by those components' own intake interfaces.

The RedactionEngine child component depends on `.specstory/config/redaction-config.yaml` being present and parseable at startup. The SessionWindowManager child component is a pure computation dependency — it requires no external resources, only the current timestamp. The TranscriptAdapterRegistry's relationship to LLMAbstraction is indirect: agent type identifiers returned by `getAgentType()` (e.g., `'claude-code'`) align with the agent profiles defined in `config/agent-profiles.json` as documented in CodingPatterns, enabling cross-component agent identity resolution.

Configuration for the LSL system lives in two files maintained separately from runtime code: `.specstory/config/lsl-config.json` (numeric and structural parameters) and `.specstory/config/redaction-config.yaml` (secret scrubbing patterns). Both are validated by `LSLConfigValidator` in `scripts/validate-lsl-config.js`.

## Usage Guidelines

**Implementing a new agent adapter** requires subclassing `TranscriptAdapter` in `lib/agent-api/transcript-api.js` and implementing all five mandatory methods. `getAgentType()` must return a stable string identifier — changing it mid-deployment would break log file routing and session continuity. `getTranscriptDirectory()` should return an absolute path; relative paths create the same invocation-directory scattering problem that the `CODING_REPO` anchor in the logging subsystem was designed to solve.

**Redaction must be treated as a hard requirement**, not an optional optimization. Tool results routinely capture environment variable dumps, API key echoes from shell commands, and other sensitive data. The `redactSecrets` option should default to enabled in all production configurations. New redaction patterns belong in `.specstory/config/redaction-config.yaml` with inline YAML comments explaining the pattern's rationale — the YAML format exists precisely to support this documentation practice.

**The `userHashLength` configuration trade-off** deserves explicit attention in large-team deployments. At 6 hex characters, the SHA-256 truncation provides only 16M distinct values. Teams with hundreds of developers should increase `userHashLength` toward the validator-allowed maximum of 12 to reduce collision probability. Note that changing this value mid-deployment renames all existing log directories, breaking historical file references.

**Cross-midnight sessions** will have their log entries split across two date-keyed files due to the date-based filename rotation in the async logging subsystem. Tools that need to reconstruct full sessions spanning midnight must be prepared to merge entries from consecutive log files. This is a known architectural limitation of the rotation approach and is the cost of avoiding an external log rotation daemon.

**Batch size and flush interval tuning** should be validated against `LSLConfigValidator` bounds (`batchSize` 10–1000) before deployment. Small batch sizes increase I/O overhead; large batch sizes increase data loss exposure if the process is killed between flushes. The 50-entry in-memory buffer and 100ms flush interval in the semantic analysis logging subsystem represent a reasonable baseline, but high-throughput agent sessions may require tuning toward the upper bound.
