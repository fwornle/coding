# LiveLoggingSystem — Technical Insight Document

## What It Is

LiveLoggingSystem (LSL) is a structured session logging pipeline for AI agent activity, implemented primarily across `lib/agent-api/transcript-api.js`, `lib/agent-api/transcripts/lsl-converter.js`, and `scripts/validate-lsl-config.js`. It captures, converts, routes, and stores agent conversation transcripts — including tool invocations and their results — in a format suitable for both human review and programmatic ingestion. Configuration lives under `.specstory/config/` (specifically `lsl-config.json` and `redaction-config.yaml`), co-locating operational config with the SpecStory project metadata it supports.

Within the broader Coding parent hierarchy, LiveLoggingSystem sits alongside components like LLMAbstraction, ConstraintSystem, and SemanticAnalysis. Where LLMAbstraction handles routing and telemetry for model calls, and ConstraintSystem hooks into lifecycle points to enforce rules, LiveLoggingSystem is responsible for durable, structured observation of what agents actually do — the persistent record layer.

## Architecture and Design

![LiveLoggingSystem — Architecture](images/live-logging-system-architecture.png)

The central architectural decision is the **pluggable adapter pattern** embodied by `TranscriptAdapter` in `lib/agent-api/transcript-api.js`. This abstract base class defines a five-method contract — `getAgentType()`, `getTranscriptDirectory()`, `readTranscripts()`, `convertToLSL()`, and `getCurrentSession()` — that completely decouples the LSL pipeline from any specific agent implementation. Concrete adapters for Claude Code and Copilot CLI subclass this base without any changes required in the converter, file manager, or validation layers. This is a deliberate extension boundary: a developer adding support for a new agent (Cursor, Aider, etc.) touches only one file.

The sixth method, `watchTranscripts()`, is intentionally **not abstract**. It is a shared concrete polling implementation, reflecting the design judgment that the watch mechanism is agent-agnostic — what varies is how transcripts are read and converted, not how they are observed for changes. This is a meaningful architectural line: variation points are abstract, stable shared behavior is concrete.

![LiveLoggingSystem — Relationship](images/live-logging-system-relationship.png)

The system's two child components — SessionWindowing and RedactionSystem — each address a cross-cutting concern that would otherwise pollute the core pipeline. SessionWindowing partitions log output using the `LSLMetadata.timeWindow` field (`HHMM-HHMM` format), ensuring that no single log file grows unbounded and that time-range <USER_ID_REDACTED> are efficient without requiring a log rotation daemon. RedactionSystem, driven by `.specstory/config/redaction-config.yaml`, applies category-based pattern scrubbing before any content is written, keeping secret detection rules versioned in the repository rather than hardcoded in the converter.

## Implementation Details

The `LSLEntryType` enum defines exactly six log entry categories: `USER`, `ASSISTANT`, `TOOL_USE`, `TOOL_RESULT`, `SYSTEM`, and `ERROR`. The separation of `TOOL_USE` and `TOOL_RESULT` as distinct entry types is significant — it allows consumers to reconstruct full agentic reasoning chains including matched call/response pairs, which is directly relevant to downstream consumers like the SemanticAnalysis pipeline or external vector databases.

`LSLConverter` in `lib/agent-api/transcripts/lsl-converter.js` supports two output formats: `toMarkdown()` for human-readable review and `toJSONL()` for streaming ingestion. The complementary `fromJSONL()` enables lossless round-trips, meaning logs can be re-processed (re-indexed, re-formatted) without loss of structure. Two constructor options govern content fidelity: `maxContentLength` truncates runaway tool results, and `redactSecrets` triggers pattern-based scrubbing sourced from `redaction-config.yaml` via the RedactionSystem child component.

`LSLConfigValidator` in `scripts/validate-lsl-config.js` enforces an **eight-stage tiered validation sequence** with deliberate early-exit semantics — environment checks before directory structure before schema validation before security checks. This ordering prevents misleading downstream errors (e.g., checking `operationalLogger.batchSize` is meaningless if the config file doesn't exist). Two optional post-validation passes are controlled by `options.generateRepairs` and `options.generateOptimizations`: the repair pass auto-corrects out-of-range values (e.g., bringing `userHashLength` back into the 6–12 range), while the optimization pass emits non-blocking suggestions. CI runs validation in strict read-only mode; developer setup scripts can enable both flags.

The multi-user routing strategy combines `userHash` (6–12 characters, enforced by `LSLConfigValidator`) with `timeWindow` to construct unique log file paths. The 6-character minimum is sized for teams up to a few hundred users (6 hex characters = 16M possibilities). The `fileManager.maxFileSize` constraint (1MB–100MB) acts as a secondary safety valve alongside hourly time-window rotation, giving operators predictable storage accounting.

The `watchTranscripts()` polling mechanism uses a `lastEntryCount` integer cursor rather than OS-level inotify/FSEvents or byte-offset tracking. This is an explicit portability trade-off: the implementation works identically across operating systems and handles in-memory array accumulation correctly, but it is O(total entries) per poll cycle rather than O(new entries). The default 1000ms interval is configurable. Crucially, registered callbacks receive only the delta slice — downstream consumers (dashboards, streaming writers) are shielded from the internal re-scan cost and always receive incremental updates.

## Integration Points

LiveLoggingSystem connects to the broader Coding hierarchy at several points. Its configuration co-location in `.specstory/config/` ties it to the same project metadata layer used by SpecStory story files. The JSONL output format of `LSLConverter` is designed for streaming ingestion by log aggregators or vector databases — making it a natural upstream data source for the SemanticAnalysis component, which uses a coordinator pattern over specialized agents that could consume structured LSL logs as input.

The ConstraintSystem's hook-based architecture (firing at pre-tool and post-tool lifecycle points) and LSL's `TOOL_USE`/`TOOL_RESULT` entry taxonomy are structurally complementary: constraint violations captured by ConstraintSystem could be written as `ERROR` entries into LSL logs, though the observations don't confirm this integration explicitly.

The RedactionSystem child component is sourced from `.specstory/config/redaction-config.yaml` and operates as a category-based filter (e.g., `api_keys`, `pii`) applied inside `LSLConverter` before any write. The SessionWindowing child component governs file partitioning through the `HHMM-HHMM` timeWindow key in `LSLMetadata`, meaning log consumers reconstructing full sessions must join across multiple windowed files.

## Usage Guidelines

**Adding a new agent adapter:** Subclass `TranscriptAdapter` and implement the five abstract methods. Do not modify `watchTranscripts()` — the polling mechanism is shared and stable. Ensure `getTranscriptDirectory()` returns a path consistent with the directory structure validated by `LSLConfigValidator`.

**Configuration management:** All config lives in `.specstory/config/lsl-config.json` and `.specstory/config/redaction-config.yaml`. Run `LSLConfigValidator` with `options.generateRepairs = true` during dev setup to auto-correct out-of-range values. In CI, run without repair flags to enforce strict validation. Keep `userHashLength` within 6–12 characters — values outside this range will be auto-repaired but indicate a misconfiguration.

**Performance considerations:** The polling watch mechanism is O(total entries) per tick. For long-running sessions with thousands of tool calls, consider increasing the poll interval to reduce CPU overhead. The `maxContentLength` constructor option on `LSLConverter` is the primary lever for controlling file size when tool results contain large payloads; rely on `maxFileSize` and hourly time-window rotation as secondary safeguards, not primary ones.

**Output format selection:** Use `toJSONL()` for any programmatic consumer (analytics pipelines, vector databases, re-processing workflows). Use `toMarkdown()` only for human-readable artifacts. The `fromJSONL()` round-trip capability means JSONL is the canonical durable format — Markdown output should be treated as derived.

**Scalability assessment:** The current design handles multi-user deployments cleanly through hash-based file routing and is sized for teams of hundreds of users. The primary scalability constraint is the O(n) polling mechanism for long sessions — this is architectural rather than incidental and would require a redesign (byte-offset tracking or FSEvents) to resolve for very long-running sessions.
