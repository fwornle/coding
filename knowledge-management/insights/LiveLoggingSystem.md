# LiveLoggingSystem

**Type:** Component

[LLM] LSLConfigValidator in scripts/validate-lsl-config.js acts as the configuration schema enforcement layer for both the primary lsl-config.json (at .specstory/config/lsl-config.json) and the companion redaction-config.yaml (at .specstory/config/redaction-config.yaml). The lsl-config.json schema enforces four numeric range constraints: fileManager.maxFileSize (1MB–100MB), fileManager.rotationThreshold (1MB–80MB), operationalLogger.maxLogSizeMB (1–100), and operationalLogger.batchSize (10–1000). The rotation threshold being constrained to at most 80MB when maxFileSize tops at 100MB ensures a meaningful gap between when rotation is triggered and when a file actually hits its hard limit — preventing a misconfiguration where threshold equals or exceeds maxFileSize. The batchSize constraint (10–1000) on the operational logger suggests writes are batched before flush, and too small a batch (below 10) would cause excessive I/O while too large a batch (above 1000) would risk data loss on crash. The redaction-config.yaml schema requires redaction.enabled and redaction.categories as mandatory fields, with globalSettings and alerts as optional sections — the mandatory structure ensures the system cannot silently skip redaction configuration, though a developer must note that the enabled flag being false is a valid state that disables redaction entirely. The use of YAML for redaction config versus JSON for the primary config suggests these files have different intended editors: YAML for human-authored sensitive policy, JSON for machine-generated or tooling-managed operational config.

# LiveLoggingSystem — Technical Insight Document

## What It Is

The LiveLoggingSystem (LSL) is implemented across several modules under `lib/agent-api/` and `scripts/`, with its configuration externalized to `.specstory/config/`. The core runtime lives in `lib/agent-api/transcript-api.js` (the `TranscriptAdapter` base class and `watchTranscripts()` polling loop) and `lib/agent-api/transcripts/lsl-converter.js` (the `LSLConverter` with dual serialization paths). Configuration enforcement is handled by `LSLConfigValidator` in `scripts/validate-lsl-config.js`, which validates both `lsl-config.json` and `redaction-config.yaml`.

As a sibling of LLMAbstraction, DockerizedServices, KnowledgeManagement, CodingPatterns, ConstraintSystem, and SemanticAnalysis under the Coding parent component, LiveLoggingSystem occupies the observability and conversation-capture niche of the project. It exists to ingest, normalize, and persist agent conversation transcripts (from Claude Code, the Copilot CLI, and any future agent integrations) into a unified, queryable form called the LSL format. Two child components, SessionWindowing and RedactionLayer, implement, respectively, the temporal partitioning strategy (hourly buckets in the `timeWindow` field) and the externalized redaction policy (rules declared in `.specstory/config/redaction-config.yaml`).

![LiveLoggingSystem — Architecture](images/live-logging-system-architecture.png)

The system's defining characteristic is its plugin-style ingestion model paired with a strict, schema-validated output format — agents write transcripts in their own native shapes (e.g., Claude Code's JSONL stream at `~/.claude/projects/<project>/conversation.jsonl`), and adapters normalize them into a canonical typed-entry vocabulary (`user`, `assistant`, `tool_use`, `tool_result`, `system`, `error`).

## Architecture and Design

The architecture rests on three deliberate design decisions: (1) a **plugin contract** for transcript sources, (2) a **pull-based polling loop** instead of OS filesystem watchers, and (3) **dual serialization paths** for human and machine consumption. The plugin contract is defined by the `TranscriptAdapter` abstract class in `lib/agent-api/transcript-api.js`, which mandates five abstract methods: `getAgentType()`, `getTranscriptDirectory()`, `readTranscripts()`, `convertToLSL()`, and `getCurrentSession()`. This is a textbook Template Method pattern — the base class owns lifecycle and dispatch logic while subclasses own format-specific parsing. Adding a third or fourth agent (Cursor, Aider, etc.) requires only a new subclass and zero changes to the core LSL infrastructure.

The decision to use `setInterval`-based polling in `watchTranscripts()` rather than `fs.watch` or `chokidar` is a portability-over-latency trade-off. Filesystem watchers behave inconsistently across operating systems, Docker containers (a relevant concern given the sibling DockerizedServices component), and network-mounted filesystems. The polling loop maintains a `lastEntryCount` integer per tick, compares it against the current count from `readTranscripts()`, and dispatches deltas to registered callbacks. The default 1000ms interval caps new-entry latency at roughly one second but trades that for predictable behavior across all environments.

The dual serialization in `LSLConverter` (`lib/agent-api/transcripts/lsl-converter.js`) serves two fundamentally different consumers. `toMarkdown()` produces typed-header human-readable output for developers reading logs in editors or markdown viewers. `toJSONL()` / `fromJSONL()` produce machine-readable output with a critical structural convention: **line 0 is always a session metadata record** (containing `sessionId`, `agent`, `timeWindow`, `projectPath`, `startTime`, `endTime`, `userHash`), and lines 1..N are typed entries. This metadata-first layout means filtering scans (by agent, time window, or project) can read a single line per file rather than the entire file — important when directories grow to hundreds of session logs.

## Implementation Details

The `TranscriptAdapter.convertToLSL()` method is the **normalization seam** of the entire system. Each agent's native message structure — for Claude Code, JSON Lines from the append-only `~/.claude/projects/<project>/conversation.jsonl`; for the Copilot CLI adapter, a different directory structure — is mapped into the unified LSL typed-entry vocabulary. The `getCurrentSession()` method is structurally critical because the polling loop relies on it to identify which entries belong to the live session versus historical sessions. This implies that adapters must implement their own session-boundary detection appropriate to the agent's storage format — there is no generic mechanism in the base class.

Session metadata uses the hourly window convention `'HHMM-HHMM'` (e.g., `'0800-0900'`) in the `timeWindow` field, a design implemented by the child SessionWindowing component. The fixed 60-minute granularity means a conversation that begins at 08:47 and ends at 09:15 is split across two window files; reconstructing the full logical conversation requires joining on `sessionId`. The `timeWindow` string almost certainly participates in file routing (directory paths or filenames embedding the window string), enabling chronological browsing without opening file contents.

![LiveLoggingSystem — Relationship](images/live-logging-system-relationship.png)

User identity is handled by hashing the `USER` environment variable with SHA-256 and taking the first 6 hexadecimal characters as the `userHash` field. The JSON schema in `LSLConfigValidator` permits `userHashLength` between 6 and 12 characters, allowing operators to lengthen the prefix in larger deployments where 6 characters (~16.7 million distinct values) might collide. The hashing happens at config-validation time in `LSLConfigValidator` and is stored in `lsl-config.json`, not recomputed per session write. Notably, this scheme is optimized for **session-correlation** (linking activity by the same user) rather than true anonymization — in a small team with a known set of usernames, brute-forcing the mapping is trivial.

`LSLConfigValidator` enforces four numeric range constraints on `lsl-config.json`: `fileManager.maxFileSize` (1MB–100MB), `fileManager.rotationThreshold` (1MB–80MB), `operationalLogger.maxLogSizeMB` (1–100), and `operationalLogger.batchSize` (10–1000). The rotation threshold cap of 80MB versus the maxFileSize cap of 100MB guarantees a 20MB gap between when rotation triggers and when the file hits its hard limit, preventing misconfiguration where rotation never fires before the hard cap. The batch size range (10–1000) signals that operational logger writes are batched before flush — too small and I/O thrashes; too large and a crash risks losing more entries. The companion `redaction-config.yaml` schema requires `redaction.enabled` and `redaction.categories` as mandatory fields (with `globalSettings` and `alerts` optional), structurally forcing operators to make an explicit decision about redaction rather than silently defaulting.

## Integration Points

The LiveLoggingSystem integrates with its child RedactionLayer through the externalized `.specstory/config/redaction-config.yaml` file. This separation means operators can update redaction patterns without modifying adapter or converter code — a deliberate decoupling of privacy policy from execution logic. The child SessionWindowing component is integrated through the shared `LSLMetadata.timeWindow` field referenced across `transcript-api.js` and `lsl-converter.js`, making the windowing strategy a cross-cutting concern rather than a localized implementation detail.

The configuration files live at well-defined paths: `.specstory/config/lsl-config.json` (machine-managed JSON for operational config) and `.specstory/config/redaction-config.yaml` (human-authored YAML for sensitive policy). The use of two different serialization formats is itself a design signal — JSON for tooling-managed config that may be edited programmatically, YAML for the kind of structured policy that humans review during compliance audits. This mirrors the configuration-hierarchy philosophy seen in the sibling ConstraintSystem, where `HookConfigLoader` uses a two-level user-then-project merge — LSL's split between operational JSON and policy YAML similarly partitions configuration by intended editor and intended lifecycle.

Externally, the system depends on the filesystem layouts of the agents it ingests from (notably `~/.claude/projects/<project>/conversation.jsonl` for Claude Code), the OS environment variable `USER` for the userHash derivation, and the configuration schema enforced by `LSLConfigValidator`. Unlike sibling components such as DockerizedServices (which integrates with a `ProcessStateManager` singleton) or KnowledgeManagement (which integrates with the VKB HTTP API), LiveLoggingSystem has no observed runtime integration with other coding-project components — it is a relatively standalone observability pipeline that produces artifacts on disk for later consumption.

## Usage Guidelines

When adding a new agent integration, subclass `TranscriptAdapter` in `lib/agent-api/transcript-api.js` and implement all five abstract methods. Pay particular attention to `getCurrentSession()` — it must correctly identify the live session object so the polling loop dispatches only current-conversation entries to callbacks. Do not modify the core LSL infrastructure to accommodate the new agent; the contract is designed so the base class never needs changes.

When tuning the polling interval in `watchTranscripts()`, recognize that the default 1000ms is a portability decision. Lowering to 200–500ms is acceptable for low-latency dashboards but adds CPU cost from re-reading and re-parsing a growing JSONL file every tick. Adapters must also correctly handle the **batch dispatch case** — when many `tool_result` entries arrive between two poll ticks, a single dispatch will contain multiple new entries, and they must be delivered in order.

For consumers parsing LSL JSONL files, always treat line 0 as the session metadata record and lines 1..N as typed entries. The `fromJSONL()` deserializer in `LSLConverter` is **not a generic JSONL reader** and cannot be used interchangeably with other JSONL tooling without awareness of this schema. Use the metadata-first convention to your advantage: filter by `agent`, `timeWindow`, or `projectPath` by reading only the first line of each file, then full-parse only the matches.

When reconstructing full logical conversations, expect that the hourly `timeWindow` partitioning will split conversations that cross hour boundaries. Join entries by `sessionId` to reassemble the complete exchange. Operators concerned about privacy should understand that the 6-character `userHash` is not robust anonymization in small known-team contexts — it leaks relative activity patterns even when the actual username is not revealed. Raise `userHashLength` toward 12 in larger deployments, and audit `.specstory/config/redaction-config.yaml` to confirm `redaction.enabled` is `true`, since a `false` value silently disables the entire redaction layer.

Finally, when modifying `lsl-config.json`, keep the rotation threshold meaningfully below `maxFileSize` (the 20MB schema-enforced gap is the minimum sensible margin) and keep `operationalLogger.batchSize` in a range that balances I/O frequency against crash-loss risk for your specific durability requirements.
