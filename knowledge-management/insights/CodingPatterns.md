# CodingPatterns

## What It Is

CodingPatterns is the architectural catch-all component of the Coding project, serving as the canonical reference for cross-cutting conventions, design patterns, and structural decisions that unify the codebase. Unlike sibling components such as ConstraintSystem or LLMAbstraction — which encapsulate specific runtime subsystems — CodingPatterns is a knowledge component: it documents and codifies the recurring design vocabulary used throughout the project.

The patterns it captures span multiple physical locations in the repository: `config/agents/` for agent configuration conventions, `config/teams/` for team topology, `bin/` for CLI entrypoints, `config/llm-providers.yaml` and associated environment variables for LLM routing, and `config/knowledge-management.json` for knowledge base configuration. These locations are not arbitrary — they reflect deliberate structural decisions that CodingPatterns exists to explain and preserve.

## Architecture and Design

The overarching architectural theme across CodingPatterns' child components is **declarative configuration driving agent-agnostic behavior**. Agents (Claude, Copilot, Mastra, OpenCode) are configured via per-agent shell scripts in `config/agents/` (e.g., `claude.sh`, `copilot.sh`) and then unified into `config/agent-profiles.json` — a single aggregation point that runtime systems use for agent selection. This two-level pattern (per-agent scripts → unified profile manifest) cleanly separates environment-specific setup from runtime dispatch, making it straightforward to add a new agent without modifying downstream consumers. AgentConfigConventions captures this pattern, with `docs/architecture/adding-new-agent.md` codifying the registration procedure.

The same structural logic applies to team composition. `config/teams/` holds JSON manifests (`agentic.json`, `coding.json`, `raas.json`, `resi.json`, `ui.json`) that declare team topology independently of agent implementation details. TeamCompositionPatterns mirrors AgentConfigConventions at the group level — both use declarative JSON/shell artifacts in `config/` subdirectories, establishing a consistent configuration idiom across the project.

![CodingPatterns — Architecture](images/coding-patterns-architecture.png)

The CLI layer in `bin/` follows a strict single-responsibility convention: each executable (`ckb`, `vkb`, `llm`, `log-session`, `graph-sync`, `status`) is a standalone tool composable via shell pipelines rather than an imported library. This is an explicit architectural decision documented in CLIToolConventions and referenced in `docs/getting-started.md` — the bin scripts are the *intended* interaction layer, not library imports. This design keeps cognitive overhead low and enables operator-level automation without requiring knowledge of internal module structure.

LLM routing, documented under LLMRoutingPatterns, introduces a tiered proxy pattern driven by three distinct environment variables: `LLM_PROXY_URL`, `RAPID_LLM_PROXY_URL`, and `LLM_CLI_PROXY_URL`. The tier selection reflects different latency and cost profiles — interactive CLI use cases route differently from wave-analysis batch calls. This pattern is visualized in `docs/images/llm-tier-routing.png` and implemented through `config/llm-providers.yaml` combined with the `bin/llm` executable. The LLMAbstraction sibling component implements the runtime mechanics of this routing; LLMRoutingPatterns documents the *convention* of which tier to select and why.

## Implementation Details

Agent configuration follows a two-artifact convention per agent: a shell script in `config/agents/` that handles environment-specific initialization, and an entry in `config/agent-profiles.json` that exposes the agent to runtime selection logic. The shell script owns environment setup (API keys, endpoint URLs, model selection), while the JSON profile owns metadata consumed by orchestration layers. This separation means changing an agent's runtime environment never requires touching the profile manifest, and vice versa.

Knowledge management implements a dual-interface pattern surfaced through two CLI tools: `bin/ckb` (command-line knowledge base) and `bin/vkb` (visual knowledge base, also available as `vkb-cli.js` and `vkb.bat` for cross-platform use). Both tools are backed by `config/knowledge-management.json` for declarative configuration. KnowledgeManagementPatterns documents this interface duality, while the KnowledgeManagement sibling component implements the underlying graph storage (Graphology + LevelDB via GraphKMStore and GraphDatabaseService).

The Docker deployment pattern — a supervisor-based multi-process container defined in `docker/Dockerfile.coding-services` with `docker/supervisord.conf` and `docker/entrypoint.sh` — is captured by the DockerizedServices sibling component but reflects a project-wide architectural decision: operational simplicity over container isolation purity. All coding services co-locate under a single supervisord instance, with `CODING_ROOT=/coding` bind-mounting the host repository into the container at runtime.

Health and constraint monitoring are structured as distinct observability subsystems: `docs/health-system/` and `docs/constraints/` each have their own README files, and `config/health-verification-rules.json` provides declarative rule definitions. HealthMonitoringArchitecture and ConstraintEnforcementPatterns capture these subsystems respectively, with the ConstraintSystem sibling component implementing the runtime hook-based enforcement via `lib/agent-api/hooks/hook-manager.js`.

## Integration Points

CodingPatterns sits directly under the Coding parent as a cross-cutting knowledge component, meaning its documented conventions apply to all seven major components in the hierarchy. LLMAbstraction implements the proxy routing conventions described in LLMRoutingPatterns. DockerizedServices implements the supervisor deployment pattern. KnowledgeManagement implements the dual-interface knowledge store pattern described in KnowledgeManagementPatterns. ConstraintSystem implements the hook-based enforcement pattern captured in ConstraintEnforcementPatterns. LiveLoggingSystem's TranscriptAdapter pattern reflects the same agent-abstraction idiom that AgentConfigConventions documents at the configuration layer.

The `bin/` tools integrate across subsystems: `bin/llm` connects to the LLM routing tier, `bin/ckb` and `bin/vkb` connect to the knowledge graph, `bin/log-session` connects to LiveLoggingSystem, `bin/graph-sync` connects to KnowledgeManagement, and `bin/status` provides cross-system health visibility. The CLI layer is therefore the primary integration surface for operator workflows.

## Usage Guidelines

When adding a new AI agent, the canonical path is: create `config/agents/<agent>.sh` for environment setup, register the agent in `config/agent-profiles.json`, and follow the step-by-step procedure in `docs/architecture/adding-new-agent.md`. Deviating from this two-artifact convention breaks the agent-agnostic dispatch that the rest of the system depends on.

LLM proxy URL selection should be deliberate: use `LLM_CLI_PROXY_URL` for interactive CLI invocations, `RAPID_LLM_PROXY_URL` for low-latency paths, and `LLM_PROXY_URL` as the general fallback. Conflating these tiers routes traffic to inappropriate cost/latency profiles and corrupts telemetry attribution — a failure mode documented explicitly in the LLMAbstraction component's handling of the `process` tag.

New CLI tools belong in `bin/` as standalone executables following the single-responsibility principle. They should not be structured as importable Node modules if their primary use case is operator-level shell composition. Configuration for new subsystems should follow the `config/<subsystem>.json` or `config/<subsystem>.yaml` convention, keeping declarative configuration co-located and discoverable. Team topology changes belong exclusively in `config/teams/` — never embedded in agent scripts or runtime code — to preserve the separation between team structure and agent implementation.

### Siblings
- [LiveLoggingSystem](./LiveLoggingSystem.md) -- [LLM] The TranscriptAdapter abstract class in `lib/agent-api/transcript-api.js` enforces a strict interface contract that all agent-specific adapters must satisfy. Subclasses must implement five methods: `getAgentType()` (returns a string identifier like 'claude' or 'copilot'), `getTranscriptDirectory()` (returns the filesystem path where native transcripts are stored), `readTranscripts()` (reads raw agent-native files), `convertToLSL()` (transforms them into the unified LSLSession/LSLEntry format), and `getCurrentSession()` (returns the active session for live ingestion). This adapter pattern means that adding a new agent source — say, a Cursor or Gemini CLI — requires only implementing this interface without touching any downstream pipeline code. The LSLMetadata type includes a `timeWindow` field (formatted as e.g. '0800-0900') that the adapter is responsible for populating, meaning the adapter layer also owns the hourly-bucketing logic that drives file routing downstream.
- [LLMAbstraction](./LLMAbstraction.md) -- LLMAbstraction is a multi-layered abstraction over LLM providers that enables provider-agnostic model calls through three distinct execution paths: mock mode (for testing), local inference via Docker Model Runner (DMR), and public cloud providers (Anthropic, OpenAI, Groq) routed through a rapid-llm-proxy. The system supports per-agent and global mode switching stored in `.data/workflow-progress.json`, allowing runtime toggling between modes without code changes. Provider selection follows a priority chain from per-agent overrides to global mode to legacy flags.

The architecture centers on a proxy-mediated request pattern where most LLM calls route through a local rapid-llm-proxy daemon (default port 12435) via `/api/complete`, enabling centralized token tracking, tier-based routing, and telemetry attribution. The `llm-with-process.ts` module exists specifically to inject a `process` tag into proxy requests — a gap in the SDK's `LLMService.complete()` that caused all wave-analysis calls to appear as `process='unknown'` in token-usage telemetry. DMR provider uses an OpenAI-compatible API at `localhost:${DMR_PORT}/engines/v1` for fully local inference.

Key patterns include: environment-variable-driven URL resolution with multiple fallback levels, singleton client instances with health-check caching, YAML-based provider configuration with env-var expansion, and SDK-shape response normalization ensuring downstream consumers work unchanged regardless of which provider path was taken.
- [DockerizedServices](./DockerizedServices.md) -- [LLM] The multi-service container architecture uses supervisord (docker/supervisord.conf) to manage multiple long-running processes within a single Docker container defined in docker/Dockerfile.coding-services. This is an intentional design choice that trades container isolation purity for operational simplicity — rather than running separate containers for the semantic analysis MCP server, constraint monitor API, and dashboard service, all are co-located under supervisord's process supervision. The entrypoint script handles startup orchestration sequencing before handing off to supervisord. A new developer should be aware that this means a crash in one supervised service does not terminate the container, supervisord will attempt restarts, but it also means a single container failure takes down all co-located services simultaneously. The bind mount pattern `CODING_ROOT=/coding` connects the host repository filesystem into the container, making the container dependent on the host directory layout at runtime rather than baking code into the image.
- [KnowledgeManagement](./KnowledgeManagement.md) -- The KnowledgeManagement component provides the core knowledge graph infrastructure for the Coding project, encompassing entity storage, querying, and lifecycle management. It uses a Graphology in-memory graph backed by LevelDB for persistence, storing entities with typed attributes (System, Project, Pattern) and relationships. The system supports multiple knowledge stores: a local LevelDB/Graphology store (GraphDatabaseService) and a canonical km-core shape store (GraphKMStore), with migration tooling to move between them.
- [ConstraintSystem](./ConstraintSystem.md) -- The ConstraintSystem is a multi-layered enforcement framework that validates code actions and file operations during Claude Code sessions. It operates through a hook-based architecture where the UnifiedHookManager (lib/agent-api/hooks/hook-manager.js) intercepts agent tool events (pre-tool, post-tool, etc.) and routes them through registered handlers loaded from user-level (~/.coding-tools/hooks.json) and project-level (.coding/hooks.json) configuration files. Violations detected during these checks are captured, persisted, and surfaced through a dashboard for monitoring.
- [SemanticAnalysis](./SemanticAnalysis.md) -- The SemanticAnalysis component is a multi-agent MCP server (`integrations/mcp-server-semantic-analysis`) that orchestrates a batch-analysis pipeline over git history and LSL (vibe) sessions to extract, classify, validate, and persist structured knowledge entities. It coordinates several specialized agents in sequence: git history ingestion, vibe/LSL session ingestion, AST-based code graph construction, semantic LLM analysis, ontology classification, content validation, and insight generation. Each agent is built on a shared `BaseAgent<TInput, TOutput>` abstract class that wraps execution in a standardized `AgentResponse` envelope with confidence scoring, issue detection, routing suggestions, and retry guidance.

The pipeline uses an ontology system backed by `@fwornle/km-core`'s `OntologyRegistry` (accessed via a `LegacyOntologyAdapter` shim) to classify extracted observations into upper/lower ontology classes with configurable heuristic and LLM-assisted classification modes. The `OntologyClassificationAgent` manages lifecycle (initialize → classify → suggest extensions) and attaches `OntologyMetadata` (class, confidence, method, version) to every entity before persistence. Storage was migrated from a legacy `GraphDatabaseAdapter`+`PersistenceAgent` trio to a `KmCoreAdapter` surface in Phase 42.x, with field names preserved for minimal call-site disruption.

Key cross-cutting concerns include: LLM calls routed through `@rapid/llm-proxy`'s `LLMService` with token usage telemetry via `attachTokenLogger`; optional code-graph-rag integration via `CodeGraphAgent` (Tree-sitter AST + Memgraph) that gracefully degrades when the `uv` CLI or Memgraph TCP connection is unavailable; content staleness detection combining reference-pattern regex scanning and git-commit correlation via `GitStalenessDetector`; and trace files written to `logs/` for debugging non-fatally.
