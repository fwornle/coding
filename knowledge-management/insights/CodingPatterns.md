# CodingPatterns

**Type:** Component

[LLM] Graph-based code modeling is a first-class architectural idiom documented via config/graph-database-config.json and implemented in the integrations/code-graph-rag/codebase_rag/ directory. The knowledge graph uses typed edge relationships: CONTAINS_PACKAGE, CONTAINS_FOLDER, CONTAINS_FILE, CONTAINS_MODULE, DEFINES, DEFINES_METHOD, DEPENDS_ON_EXTERNAL. These relationship types form a hierarchical containment spine (CONTAINS_*) overlaid with semantic edges (DEFINES, DEFINES_METHOD) and dependency tracking (DEPENDS_ON_EXTERNAL). The MEMGRAPH_BATCH_SIZE environment variable documented in the project controls how many graph write operations are batched together during ingestion, which is critical for performance when indexing large codebases. New developers extending the graph model should add new edge types to the shared config rather than inventing ad-hoc relationship names, to keep the ontology coherent across integrations. The README.md for code-graph-rag describes this as a 'Graph-Based RAG System for Any Codebases', meaning the graph schema is designed for portability across different source languages and project structures.

# CodingPatterns — Technical Insight Document

## What It Is

CodingPatterns is the architectural conventions layer within the broader `Coding` root hierarchy, capturing the enforced structural idioms that govern how integrations, agents, and cross-cutting concerns are built and composed. It is not a runtime library but a **design contract**: a set of repeatable patterns that every integration in the project must follow. Its primary concern is the `MCPServerPattern` (its direct child), but it also encompasses the agent abstraction strategy, LLM tier routing, hook-based lifecycle extensibility, and knowledge capture/retrieval pipelines.

The patterns are operationalized across three major integration directories — `integrations/mcp-constraint-monitor/`, `integrations/mcp-server-semantic-analysis/`, and `integrations/code-graph-rag/` — supported by centralized configuration files including `config/agent-profiles.json`, `config/hooks-config.json`, `config/llm-providers.yaml`, `config/knowledge-management.json`, and `config/graph-database-config.json`. PlantUML diagrams under `docs/puml/` serve as the authoritative visual specification for each pattern's flow and interface contracts.

## Architecture and Design

![CodingPatterns — Architecture](images/coding-patterns-architecture.png)

CodingPatterns organizes the system around five interlocking idioms, each solving a distinct extensibility problem while remaining decoupled from the others.

**MCP Server Pattern** (formalized as the child component `MCPServerPattern`) is the integration scaffolding convention. Every integration exposes itself to AI agent hosts through a uniform directory structure: a top-level `README.md` describing the tool interface, a `docs/` subdirectory with an `api/README.md` and an `architecture/README.md` separating API surface from internal design, and a configuration file declaring tool schemas. This is not organic — the consistency across all three integrations marks it as an enforced project convention. The `mcp-server-semantic-analysis` integration is the most fully elaborated example, with its `docs/api/README.md` and `docs/architecture/README.md` serving as the reference implementation of the documentation split.

**Agent Abstraction** applies the Strategy design pattern to LLM backends. Concrete agent implementations live as shell scripts under `config/agents/` (`claude.sh`, `copilot.sh`, `opencode.sh`), registered in `config/agent-profiles.json`, with the interface contract documented in `docs/puml/agent-abstraction-architecture.puml`. The orchestration layer never references a specific backend; it only invokes the registered script. The `mcp-server-semantic-analysis` integration further documents runtime agent selection in `docs/architecture/agents.md`.

**LLM Tier Routing** is a cost/capability optimization layer that sits *above* agent selection. Documented in `docs/puml/llm-tier-routing.puml` and configured in `config/llm-providers.yaml`, the routing layer classifies incoming tasks and assigns them to a tier (fast/cheap vs. slow/capable) before any agent backend is invoked. The formal design rationale lives in `docs/TIERED-MODEL-PROPOSAL.md` under `mcp-server-semantic-analysis`. This separates provider economics from task logic — a clean boundary that prevents model-specific strings from leaking into integration code. This pattern directly complements the `LLMAbstraction` sibling component, which implements the runtime proxy (the `rapid-llm-proxy` daemon on port 12435) that executes the routing decisions CodingPatterns defines at the configuration level.

**Hook-Based Lifecycle Extensibility** is the cross-cutting concern pattern. `config/hooks-config.json` is the registration point; `docs/puml/agent-hook-flow.puml` documents the execution flow. The `ConstraintSystem` sibling is the primary consumer of this pattern, using hooks for both passive observation (recording constraint violations) and active enforcement (blocking on policy breaches). The hook data format is specified in `integrations/mcp-constraint-monitor/docs/CLAUDE-CODE-HOOK-FORMAT.md`, and the status surface integration is described in `docs/status-line-integration.md`. This pattern deliberately mirrors how the `ConstraintSystem` registers handlers at `~/.coding-tools/hooks.json` and `.coding/hooks.json` without touching agent scripts.

![CodingPatterns — Relationship](images/coding-patterns-relationship.png)

**Knowledge Capture and Retrieval Pipeline** is the persistent memory idiom, configured in `config/knowledge-management.json` and visualized in `docs/puml/knowledge-capture-flow.puml` and `docs/puml/knowledge-context-injection.puml`. It follows an observe-store-retrieve cycle: information captured during agent sessions is stored in the graph backend and retrieved as injected context in subsequent interactions. The `KnowledgeManagement` sibling implements the storage half (single-key LevelDB via `src/knowledge-management/GraphDatabaseService.js`), while `integrations/code-graph-rag/` operationalizes the retrieval half, exposing the graph over HTTP (`CODE_GRAPH_RAG_PORT`) and SSE (`CODE_GRAPH_RAG_SSE_PORT`).

## Implementation Details

**Graph-Based Code Modeling** is a first-class idiom defined in `config/graph-database-config.json` and implemented in `integrations/code-graph-rag/codebase_rag/`. The ontology uses typed edge relationships that form two layers: a containment spine (`CONTAINS_PACKAGE`, `CONTAINS_FOLDER`, `CONTAINS_FILE`, `CONTAINS_MODULE`) and a semantic overlay (`DEFINES`, `DEFINES_METHOD`, `DEPENDS_ON_EXTERNAL`). This two-layer design allows the graph to answer both structural questions ("what files are in this package?") and semantic questions ("what external dependencies does this module have?") using the same query interface. The `MEMGRAPH_BATCH_SIZE` environment variable controls write batching during ingestion — a performance-critical tuning knob when indexing large codebases. The graph schema is explicitly designed for language portability, as documented in `integrations/code-graph-rag/README.md` ("Graph-Based RAG System for Any Codebases").

The `MCPServerPattern` child component captures the concrete scaffolding mechanics: each integration's `README.md` (`integrations/mcp-constraint-monitor/README.md`, `integrations/mcp-server-semantic-analysis/README.md`, `integrations/code-graph-rag/README.md`) is the primary entry point for tool discovery by AI agent hosts. The configuration file in each integration directory declares tool schemas that the host reads to understand the tool's capabilities — making the README a human contract and the config file a machine contract for the same interface.

The hook system's two-mode operation (passive observation vs. active enforcement) is mechanically significant. Both modes share the same registration point (`hooks-config.json`) and the same data format (`CLAUDE-CODE-HOOK-FORMAT.md`), but differ in whether the handler returns a blocking response. This means a single hook handler can be promoted from observer to enforcer by changing its return behavior, without re-registering or restructuring. The `mcp-constraint-monitor` integration exploits both modes simultaneously.

## Integration Points

CodingPatterns is the definitional layer that the other six siblings implement or consume. **`ConstraintSystem`** is the most direct consumer: it embodies the hook-based lifecycle pattern, registering handlers in the hooks infrastructure CodingPatterns defines. **`KnowledgeManagement`** implements the storage half of the knowledge capture pipeline, with its LevelDB single-key strategy providing the persistence layer the pipeline depends on. **`LLMAbstraction`** executes the provider selection and tier routing decisions at runtime via the `rapid-llm-proxy`, while CodingPatterns holds the configuration contract (`config/llm-providers.yaml`) that drives those decisions. **`SemanticAnalysis`** and **`LiveLoggingSystem`** both benefit from the agent abstraction pattern — the former via its `BaseAgent<TInput,TOutput>` coordinator pattern, the latter via the `TranscriptAdapter` abstraction — though these are sibling-level implementations rather than direct consumers of CodingPatterns' configuration files.

The `DockerizedServices` sibling is the most loosely coupled: it provides the containerized runtime (Memgraph, Redis, Next.js dashboard, Node.js API) that the `code-graph-rag` integration and `KnowledgeManagement` depend on, but its dual-probe health architecture (`lib/utils/service-probe.js`) operates independently of the CodingPatterns conventions.

External integration points are the AI agent hosts that consume MCP tool schemas, the Memgraph instance accessed via Bolt protocol, and the HTTP/SSE endpoints exposed by `code-graph-rag` on `CODE_GRAPH_RAG_PORT` and `CODE_GRAPH_RAG_SSE_PORT`.

## Usage Guidelines

**Adding a new MCP integration** requires creating at minimum: a `README.md` with tool description, a `docs/api/README.md` for the API surface, a `docs/architecture/README.md` for internal design, and a configuration file declaring tool schemas. The `mcp-server-semantic-analysis` integration is the reference implementation of this structure. Deviating from this layout breaks the uniform discovery contract that AI agent hosts depend on.

**Adding a new LLM provider** means writing a shell script under `config/agents/` and registering it in `config/agent-profiles.json`. No integration code should be modified. Task types should be classified into the existing tier taxonomy in `config/llm-providers.yaml` rather than hardcoding model names, so the routing layer can apply fallback and cost optimization rules uniformly.

**Extending the graph ontology** requires adding new edge types to `config/graph-database-config.json` rather than inventing ad-hoc relationship names in integration code. This keeps the ontology coherent and queryable across all integrations. The `MEMGRAPH_BATCH_SIZE` variable should be tuned when ingesting large codebases to avoid write bottlenecks.

**Observing or modifying agent behavior** should be done by registering hook handlers in `config/hooks-config.json`, not by patching agent scripts in `config/agents/`. The data format defined in `integrations/mcp-constraint-monitor/docs/CLAUDE-CODE-HOOK-FORMAT.md` is the authoritative schema for hook payloads. Both passive and enforcement modes are available within the same registration mechanism.

**Persistent memory across agent sessions** should be modeled as graph nodes and edges queryable via the RAG interface rather than file-based caches. Ad-hoc file caches bypass the standardized context-injection machinery documented in `docs/puml/knowledge-context-injection.puml`, and will not be retrievable by the agent in future sessions. Any code path mutating the graph must ensure the `KnowledgeManagement` flush cycle is triggered — the LevelDB `isDirty` deferral pattern means silent in-memory-only mutations are a real risk if flush is not explicitly called.

### Siblings
- [LiveLoggingSystem](./LiveLoggingSystem.md) -- [LLM] The `TranscriptAdapter` abstract base class in `lib/agent-api/transcript-api.js` defines a pluggable adapter contract that decouples the LSL pipeline from specific agent implementations. The five abstract methods — `getAgentType()`, `getTranscriptDirectory()`, `readTranscripts()`, `convertToLSL()`, and `getCurrentSession()` — establish a clear interface that concrete adapters (e.g., for Claude Code and Copilot CLI) must implement. This pattern means the core LSL infrastructure never directly references agent-specific transcript formats or filesystem layouts. A new developer adding support for a third agent (say, Cursor or Aider) would subclass `TranscriptAdapter` and implement only these five methods without touching the converter, file manager, or validation layers. The `watchTranscripts()` method is notably NOT abstract — it is a concrete polling implementation shared by all adapters, using `setInterval` with a default 1000ms interval and a `lastEntryCount` integer cursor that advances only when new entries are detected, then fires registered callbacks with only the delta entries. This means the watch mechanism is O(n) in total entry count per poll cycle, which could become a performance concern for very long sessions.
- [LLMAbstraction](./LLMAbstraction.md) -- LLMAbstraction is a multi-layered abstraction over LLM providers that enables provider-agnostic model calls across public APIs (Anthropic, OpenAI, Groq), a local Docker Model Runner (DMR), and a mock mode for testing. The system routes requests through a rapid-llm-proxy daemon (port 12435) that handles provider selection, tier-based routing, and per-call telemetry attribution. Mode selection is dynamic and per-agent, stored in workflow-progress.json, supporting global and per-agent overrides across three modes: 'mock', 'local' (DMR), and 'public' (cloud APIs).
- [DockerizedServices](./DockerizedServices.md) -- [LLM] The DockerizedServices component uses a dual-probe health checking architecture implemented in lib/utils/service-probe.js that strictly separates HTTP-based and TCP-based service verification. probeHttpHealth() issues HTTP/HTTPS requests and maps 2xx/3xx response codes to the 'running' state, while probeTcpPort() opens a raw net.Socket connection to verify port reachability for non-HTTP protocols like Memgraph's Bolt protocol. A critical architectural invariant (documented as SPEC R6) enforces that neither probe ever returns the string 'healthy'—only 'running', 'stopped', or 'unknown' are valid return values. This distinction matters because health-coordinator.js in scripts/health-coordinator.js consumes these probes on a 5-second polling interval and must be able to uniformly handle all service types (Next.js dashboard, Node.js API, Memgraph, Redis) without special-casing the protocol. New developers adding services must use probeTcpPort() for any non-HTTP service and must not introduce a fourth status string, or the health-coordinator's state machine will behave incorrectly.
- [KnowledgeManagement](./KnowledgeManagement.md) -- [LLM] The primary persistence mechanism in KnowledgeManagement is a single-key LevelDB strategy implemented in `src/knowledge-management/GraphDatabaseService.js`. Rather than storing each graph entity as a separate LevelDB key (which would enable partial reads and atomic per-entity updates), the entire Graphology in-memory graph is serialized as one JSON blob stored under the key `'graph'`. This blob contains all nodes, edges, and metadata. Writes are deferred using an `isDirty` flag — mutations to the graph set `isDirty = true`, and `_persistGraphToLevel()` is only called when a flush is explicitly triggered. This design optimizes for read-heavy, batch-write workloads but creates a risk of data loss if the process crashes between mutations and the next flush. New developers should be aware that any code path that modifies graph nodes or edges must ensure the flush cycle is triggered, or changes will silently remain only in memory.
- [ConstraintSystem](./ConstraintSystem.md) -- The ConstraintSystem is a monitoring and enforcement layer that validates code actions and file operations against configured rules during Claude Code sessions. It operates through a hook-based architecture where constraint checks are triggered at key lifecycle points (pre-tool, post-tool, etc.) and violations are captured, persisted, and surfaced to dashboards. The system integrates with Claude Code's native hook mechanism via configuration files at user-level (~/.coding-tools/hooks.json) and project-level (.coding/hooks.json), with project config overriding user config.
- [SemanticAnalysis](./SemanticAnalysis.md) -- [LLM] The SemanticAnalysis pipeline is structured around a coordinator pattern where specialized agents each extend BaseAgent<TInput,TOutput> defined in src/agents/base-agent.ts. This base class implements a strict template-method execute() that sequences six steps in order: process(), calculateConfidence(), detectIssues(), generateRouting(), applyCorrections(), and buildMetadata(). Every agent—CodeGraphAgent, SemanticAnalysisAgent, OntologyClassificationAgent, ContentValidationAgent—inherits this contract and returns a uniform AgentResponse envelope. This design means a new developer adding an agent only needs to implement the domain-specific process() logic; confidence scoring, issue detection, and metadata construction are guaranteed to run in a consistent order regardless of which agent is invoked. The tradeoff is that the template method imposes overhead steps even when an agent's output is trivially simple, and agents cannot short-circuit the sequence without throwing exceptions.
