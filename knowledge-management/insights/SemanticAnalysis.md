# SemanticAnalysis

**Type:** Component

The SemanticAnalysis component is a multi-agent MCP server (`integrations/mcp-server-semantic-analysis`) that orchestrates a batch-analysis pipeline over git history and LSL (vibe) sessions to extract, classify, validate, and persist structured knowledge entities. It coordinates several specialized agents in sequence: git history ingestion, vibe/LSL session ingestion, AST-based code graph construction, semantic LLM analysis, ontology classification, content validation, and insight generation. Each agent is built on a shared `BaseAgent<TInput, TOutput>` abstract class that wraps execution in a standardized `AgentResponse` envelope with confidence scoring, issue detection, routing suggestions, and retry guidance.

The pipeline uses an ontology system backed by `@fwornle/km-core`'s `OntologyRegistry` (accessed via a `LegacyOntologyAdapter` shim) to classify extracted observations into upper/lower ontology classes with configurable heuristic and LLM-assisted classification modes. The `OntologyClassificationAgent` manages lifecycle (initialize → classify → suggest extensions) and attaches `OntologyMetadata` (class, confidence, method, version) to every entity before persistence. Storage was migrated from a legacy `GraphDatabaseAdapter`+`PersistenceAgent` trio to a `KmCoreAdapter` surface in Phase 42.x, with field names preserved for minimal call-site disruption.

Key cross-cutting concerns include: LLM calls routed through `@rapid/llm-proxy`'s `LLMService` with token usage telemetry via `attachTokenLogger`; optional code-graph-rag integration via `CodeGraphAgent` (Tree-sitter AST + Memgraph) that gracefully degrades when the `uv` CLI or Memgraph TCP connection is unavailable; content staleness detection combining reference-pattern regex scanning and git-commit correlation via `GitStalenessDetector`; and trace files written to `logs/` for debugging non-fatally.

# SemanticAnalysis — Technical Reference

## What It Is

SemanticAnalysis is a multi-agent MCP server implemented under `integrations/mcp-server-semantic-analysis`. It orchestrates a sequential batch-analysis pipeline that ingests git history and LSL (vibe) session data, constructs code graphs, performs LLM-driven semantic analysis, classifies entities against an ontology, validates content, and generates structured knowledge insights. The component sits within the broader Coding project hierarchy as a peer to KnowledgeManagement — which provides the underlying graph storage infrastructure — and DockerizedServices, which co-locates the MCP server alongside other supervised processes under supervisord.

## Architecture and Design

The central architectural decision is a **sequential multi-agent pipeline** where each stage is encapsulated as a discrete agent class extending `BaseAgent<TInput, TOutput>` (`agents/base-agent.ts`). Subclasses implement only `process()`, while `BaseAgent` wraps every call in a standardized `AgentResponse` envelope via `createAgentResponse()`, computing confidence breakdowns, issue detection, routing suggestions, and retry guidance uniformly. This means operational concerns — retryability, confidence reporting, issue surfacing — are handled once in the base class and inherited by every agent in the pipeline (git history ingestion, vibe/LSL ingestion, AST code graph construction, semantic LLM analysis, ontology classification, content validation, and insight generation). The Pipeline child component documents this shared contract explicitly.

A recurring design pattern is **graceful degradation**: the `CodeGraphAgent` (`agents/code-graph-agent.ts`) performs a TCP socket probe (`checkMemgraphConnection`) with a timeout sourced from `loadAgentTuningConfig().code_graph.memgraph_check_timeout_ms`, and runs a `which uv` subprocess check before attempting Tree-sitter/Memgraph indexing. When either dependency is absent, it sets `skipped: true` on its result and allows downstream agents to continue unimpeded. This means the full AST + graph enrichment is opportunistic — the pipeline degrades to semantic-only analysis rather than failing.

![SemanticAnalysis — Relationship](images/semantic-analysis-relationship.png)

The ontology subsystem follows a **strangler-fig migration** pattern. `OntologyClassificationAgent` (`agents/ontology-classification-agent.ts`) constructs `@fwornle/km-core`'s `OntologyRegistry` directly but wraps it in `LegacyOntologyAdapter` to preserve the `OntologyValidator` + `OntologyClassifier` interfaces expected by downstream consumers. This is the same anti-corruption layer documented in the LegacyOntologyAdapter child component: the adapter shields the rest of the pipeline from the evolving `km-core` registry shape while the migration proceeds incrementally. OntologyConfigManager, implemented as a process-level singleton with an explicit `reset()` for test isolation, governs configuration state shared across all agents without requiring re-initialization.

Storage was migrated (Phase 42.2) from a `GraphDatabaseAdapter` + `PersistenceAgent` pair to a single `KmCoreAdapter` surface in `ContentValidationAgent` (`agents/content-validation-agent.ts`), stored in both `this.graphDB` and `this.persistenceAgent` fields typed as `KmCoreAdapter`. The dual-field assignment is a deliberate call-site compatibility shim — it avoids rewriting every consumer while the legacy field names are retired over time. This mirrors how KnowledgeManagement itself manages the coexistence of a local LevelDB/Graphology store and the canonical km-core shape store.

## Implementation Details

**Agent base contract.** Every agent in the pipeline is typed as `BaseAgent<TInput, TOutput>`. The base class's `createAgentResponse()` computes a structured envelope before returning to the pipeline coordinator, meaning confidence scores and issue lists are never left to individual agent authors to construct consistently. This is the foundation the Pipeline child component describes.

**Ontology classification lifecycle.** `OntologyClassificationAgent` manages a three-phase lifecycle (initialize → classify → suggest extensions) as documented in the Ontology child component. It attaches `OntologyMetadata` — carrying `class`, `confidence`, `method`, and `version` — to every entity before it reaches persistence. Classification can operate in heuristic mode or LLM-assisted mode, configurable through `OntologyConfigManager`. Unrecognized entities trigger a "suggest extensions" phase, allowing the ontology to grow organically from observed data.

**Content staleness detection.** `ContentValidationAgent` combines two mechanisms: regex pattern arrays (`filePathPatterns`, `commandPatterns`) that scan observation text for stale file and command references, and `GitStalenessDetector`, which correlates those references against actual git commit history. To prevent redundant git fetches across batch entity validation, the agent caches git commits in `cachedGitCommits` with a 60-second TTL (`GIT_CACHE_TTL_MS = 60000`). This is a pragmatic performance boundary — tight enough to catch commit activity during a session, long enough to avoid repeated subprocess overhead.

**Trace file debugging.** `SemanticAnalysisAgent.analyzeGitAndVibeData()` (`agents/semantic-analysis-agent.ts`) writes non-fatal JSON traces to `logs/semantic-analysis-trace-<timestamp>.json`. The write is guarded by `mkdir({ recursive: true })` and wrapped in an explicit `catch`, ensuring that a filesystem issue (missing directory, permissions) never propagates as a pipeline abort. This is a deliberate separation: observability infrastructure must not become a failure mode.

**LLM instrumentation.** `SemanticAnalysisAgent` calls `attachTokenLogger(this.llmService, 'semantic-analysis-agent')` from `utils/token-usage-logger.ts`, imported as a static ESM import (corrected from a prior CommonJS `require()` in Phase 42). All LLM calls route through `@rapid/llm-proxy`'s `LLMService`, consistent with how LLMAbstraction centralizes token telemetry and tier-based routing for the broader Coding project. The `'semantic-analysis-agent'` attribution tag ensures per-agent token usage is distinguishable in telemetry dashboards.

## Integration Points

**KnowledgeManagement** is the downstream storage target. The migration to `KmCoreAdapter` aligns SemanticAnalysis's persistence surface with KnowledgeManagement's canonical km-core shape store, reducing the divergence between how entities are written and how they are later <USER_ID_REDACTED>. The `@fwornle/km-core` `OntologyRegistry` is a shared dependency between both components.

**LLMAbstraction / `@rapid/llm-proxy`** is the LLM execution layer. SemanticAnalysis routes all model calls through `LLMService` from `@rapid/llm-proxy`, inheriting the proxy's centralized token tracking, provider routing, and telemetry attribution described in LLMAbstraction. The `attachTokenLogger` instrumentation adds a SemanticAnalysis-specific attribution layer on top.

**LiveLoggingSystem** is an upstream data source. The LSL/vibe session ingestion agent consumes LSL sessions produced by LiveLoggingSystem's adapter pipeline. The `convertToLSL()` output format from LiveLoggingSystem becomes direct input to SemanticAnalysis's vibe ingestion stage.

**DockerizedServices** provides the runtime container. The MCP server runs as a supervisord-managed process in the shared coding-services container, meaning it shares fate with co-located services but benefits from supervisord's restart behavior on non-fatal crashes.

**External optional dependencies.** `CodeGraphAgent` depends on `uv` CLI availability and a reachable Memgraph TCP endpoint. Both are probed at runtime with configurable timeouts, and the agent degrades gracefully when absent — this is the only integration point designed to be skippable without pipeline failure.

## Usage Guidelines

**Extending the pipeline with a new agent** requires only implementing `process()` on a new `BaseAgent<TInput, TOutput>` subclass. Confidence scoring, issue detection, and retry envelope construction are inherited. New agents should not attempt to build their own response envelopes.

**Ontology extension** happens through the "suggest extensions" phase of `OntologyClassificationAgent`, not through direct `OntologyRegistry` mutation. When entities don't fit existing classes, the agent surfaces extension proposals that can be reviewed and incorporated into the registry. Direct registry modification bypasses this validation path.

**LLM call instrumentation** must be attached via `attachTokenLogger` with a distinct agent-scoped label. Uninstrumented `LLMService` usage will result in token telemetry appearing under `process='unknown'` in the proxy's attribution layer — a known gap that caused attribution failures before the Phase 42 ESM import fix.

**Storage interactions** should go through `KmCoreAdapter`, not the retired `GraphDatabaseAdapter` or `PersistenceAgent` types. The dual-field assignment in `ContentValidationAgent` (`this.graphDB` and `this.persistenceAgent` both pointing to the same `KmCoreAdapter` instance) is a temporary compatibility shim — new code should reference the adapter directly without relying on the legacy field names persisting.

**The `CodeGraphAgent` skip path** is normal operating behavior, not an error state. Pipeline consumers checking agent results should treat `skipped: true` as a valid non-failure outcome and not attempt to retry graph construction when the underlying infrastructure is absent. The tuning configuration at `loadAgentTuningConfig().code_graph.memgraph_check_timeout_ms` controls how long the pipeline waits before declaring the dependency unavailable — this should be kept short in environments where Memgraph is known to be absent to avoid adding unnecessary latency to every pipeline run.

- [LiveLoggingSystem](./LiveLoggingSystem.md) -- [LLM] The TranscriptAdapter abstract class in `lib/agent-api/transcript-api.js` enforces a strict interface contract that all agent-specific adapters must satisfy. Subclasses must implement five methods: `getAgentType()` (returns a string identifier like 'claude' or 'copilot'), `getTranscriptDirectory()` (returns the filesystem path where native transcripts are stored), `readTranscripts()` (reads raw agent-native files), `convertToLSL()` (transforms them into the unified LSLSession/LSLEntry format), and `getCurrentSession()` (returns the active session for live ingestion). - [LLMAbstraction](./LLMAbstraction.md) -- LLMAbstraction is a multi-layered abstraction over LLM providers that enables provider-agnostic model calls through three distinct execution paths: mock mode (for testing), local inference via Docker Model Runner (DMR), and public cloud providers (Anthropic, OpenAI, Groq) routed through a rapid-llm-proxy. The system supports per-agent and global mode switching stored in `.data/workflow-progress.json`, allowing runtime toggling between modes without code changes. Provider selection follows a priority chain from per-agent overrides to global mode to legacy flags.

The architecture centers on a proxy-mediated request pattern where most LLM calls route through a local rapid-llm-proxy daemon (default port 12435) via `/api/complete`, enabling centralized token tracking, tier-based routing, and telemetry attribution. The `llm-with-process.ts` module exists specifically to inject a `process` tag into proxy requests — a gap in the SDK's `LLMService.complete()` that caused all wave-analysis calls to appear as `process='unknown'` in token-usage telemetry. DMR provider uses an OpenAI-compatible API at `localhost:${DMR_PORT}/engines/v1` for fully local inference.

Key patterns include: environment-variable-driven URL resolution with multiple fallback levels, singleton client instances with health-check caching, YAML-based provider configuration with env-var expansion, and SDK-shape response normalization ensuring downstream consumers work unchanged regardless of which provider path was taken.
- [DockerizedServices](./DockerizedServices.md) -- [LLM] The multi-service container architecture uses supervisord (docker/supervisord.conf) to manage multiple long-running processes within a single Docker container defined in docker/Dockerfile.coding-services. This is an intentional design choice that trades container isolation purity for operational simplicity — rather than running separate containers for the semantic analysis MCP server, constraint monitor API, and dashboard service, all are co-located under supervisord's process supervision. The entrypoint script handles startup orchestration sequencing before handing off to supervisord. A new developer should be aware that this means a crash in one supervised service does not terminate the container, supervisord will attempt restarts, but it also means a single container failure takes down all co-located services simultaneously. The bind mount pattern `CODING_ROOT=/coding` connects the host repository filesystem into the container, making the container dependent on the host directory layout at runtime rather than baking code into the image.

Violations detected during these checks are captured, persisted, and surfaced through a dashboard for monitoring.
