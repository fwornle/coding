# SemanticAnalysis

**Type:** Component

The SemanticAnalysis component is a multi-agent MCP server (`integrations/mcp-server-semantic-analysis`) that orchestrates a batch-analysis pipeline over git history and LSL (vibe) sessions to extract, classify, and persist structured knowledge entities. It coordinates specialized agents—GitHistoryAgent, VibeHistoryAgent, CodeGraphAgent, SemanticAnalysisAgent, OntologyClassificationAgent, ContentValidationAgent, and InsightGenerationAgent—each handling a distinct stage of the pipeline from data ingestion through ontology classification to knowledge-base persistence. The system integrates with km-core's OntologyRegistry for entity classification and uses the LLMService from @rapid/llm-proxy for semantic reasoning throughout the pipeline.

Architecturally, all agents extend a common `BaseAgent<TInput, TOutput>` abstract class (base-agent.ts) that wraps agent logic in a standardized `AgentResponse` envelope providing confidence scoring, issue detection, routing suggestions, and upstream context propagation. The CodeGraphAgent integrates with a separate code-graph-rag MCP server using Tree-sitter AST parsing backed by Memgraph (graph database accessed over TCP at configurable host:port), with graceful degradation when the CLI or Memgraph is unavailable. The OntologyClassificationAgent underwent a Phase 42-03 migration from a legacy ontology-load class to km-core's OntologyRegistry wrapped in a LegacyOntologyAdapter, maintaining backward compatibility with OntologyValidator and OntologyClassifier while enabling atomic reload semantics.

## What It Is

SemanticAnalysis is a multi-agent MCP server implemented under `integrations/mcp-server-semantic-analysis`. It orchestrates a batch-analysis pipeline that ingests git history and LSL (vibe) session data, extracts structured knowledge entities, classifies them against a shared ontology, validates them, and persists the results into the broader knowledge base. Within the Coding project hierarchy, it sits as a peer to components like KnowledgeManagement, LLMAbstraction, and ConstraintSystem, and is itself the parent of four child concepts: Pipeline, Ontology, Insights, and LegacyOntologyAdapter.

The server's raison d'être is to close the loop between raw development activity (commits, agent sessions) and structured, queryable knowledge. Rather than a monolithic analyzer, it delegates every discrete concern to a specialized agent, making the system decomposable and independently evolvable.

## Architecture and Design

![SemanticAnalysis — Architecture](images/semantic-analysis-architecture.png)

The dominant architectural pattern is a **staged, agent-per-concern pipeline**. Seven agents participate in a DAG-style dependency ordering (as the Pipeline child component describes): GitHistoryAgent and VibeHistoryAgent handle data ingestion; CodeGraphAgent provides structural AST/graph context; SemanticAnalysisAgent performs the core LLM-driven reasoning; OntologyClassificationAgent maps outputs to the shared ontology; ContentValidationAgent enforces integrity; and InsightGenerationAgent produces the terminal higher-order insights. Each stage consumes outputs from earlier stages, forming a directed flow from raw data to structured knowledge.

All agents extend `BaseAgent<TInput, TOutput>` (`agents/base-agent.ts`), which enforces a uniform execution contract: `execute()` sequences `process() → calculateConfidence() → detectIssues() → generateRouting() → applyCorrections() → buildMetadata()`, wrapping every result in an `AgentResponse` envelope. This envelope carries confidence scores, detected issues, routing suggestions, and upstream context, giving the orchestrator a consistent surface for branching decisions regardless of which agent produced the result. The design trades flexibility for uniformity — every new agent automatically gains telemetry hooks and routing infrastructure by extending the base class.

The OntologyClassificationAgent exemplifies a **strangler-fig migration** (internally tagged Phase 42-03). Rather than rewriting consumers of the legacy ontology-load class, `LegacyOntologyAdapter` (the child component `LegacyOntologyAdapter`, residing in the `ontology/` directory) wraps km-core's `OntologyRegistry` behind the old interface surface. `OntologyValidator` and `OntologyClassifier` remain untouched, while the underlying implementation delegates to the new registry with atomic reload semantics. This is a deliberate backward-compatibility trade-off: migration risk is absorbed at the adapter boundary rather than distributed across all call sites.

![SemanticAnalysis — Relationship](images/semantic-analysis-relationship.png)

A second, parallel migration appears in ContentValidationAgent (Phase 42.2 Plan 04): previously separate `GraphDatabaseAdapter` and `PersistenceAgent` fields were collapsed into a single `KmCoreAdapter` handle (typed as both `graphDB` and `persistenceAgent`, imported from `storage/km-core-adapter.ts`). The dual typing preserves call-site compatibility while reducing the number of adapter instances in flight. Legacy file cleanup flows through `cleanupEntityFiles` from `storage/legacy-consumer-helpers.ts`, keeping the migration surface explicit and auditable.

## Implementation Details

**BaseAgent lifecycle.** `agents/base-agent.ts` defines the `execute()` template method. Subclasses implement `process()` for domain logic and optionally override `calculateConfidence()`, `detectIssues()`, `generateRouting()`, and `applyCorrections()`. The metadata builder closes the envelope. This means confidence scoring and issue detection are never optional afterthoughts — they are structurally required by the base class contract.

**SemanticAnalysisAgent depth modes.** `agents/semantic-analysis-agent.ts` exposes `analyzeGitAndVibeData()` with three analysis depths. At `'surface'`, file analysis is capped at 5 files and `performCrossAnalysis()` is skipped entirely. `'deep'` and `'comprehensive'` both run cross-analysis and then enrich `codeAnalysis.architecturalPatterns` with LLM-identified patterns, deduplicated by lowercased name. LLM initialization is deferred via `ensureLLMInitialized()` (lazy init), and every run attaches token usage telemetry through `attachTokenLogger(this.llmService, 'semantic-analysis-agent')` from `utils/token-usage-logger.ts`. Ultra-debug tracing writes per-run trace files to `logs/semantic-analysis-trace-{timestamp}.json`, giving operators a full audit trail without requiring a persistent logging infrastructure.

**CodeGraphAgent resilience.** `agents/code-graph-agent.ts` performs a TCP reachability check via `checkMemgraphConnection()` before any graph operation, providing graceful degradation rather than hard failure when Memgraph (the graph database, accessed over TCP at a configurable host:port) or the code-graph-rag MCP server CLI is unavailable. Directory resolution for `codeGraphRagDir` walks through `CODING_REPO`, `CODING_TOOLS_PATH`, and `CODING_ROOT` environment variables with a sibling-directory fallback, logging a warning rather than throwing on absence. This defensive posture reflects the reality that CodeGraphAgent depends on a separately containerized service (visible in the DockerizedServices sibling component, which includes Memgraph and the code-graph-rag service in `docker/docker-compose.yml`).

**ContentValidationAgent staleness detection.** `agents/git-staleness-detector.ts` is integrated into ContentValidationAgent with a 60-second in-memory cache (`GIT_CACHE_TTL_MS = 60000`) for git commits, preventing redundant fetches during batch entity validation. Parallel refresh workers are configurable (default 1, max 20), providing a knob for throughput vs. resource pressure without architectural changes.

**OntologyClassificationAgent classification taxonomy.** Classification results are tracked in `ClassificationProcessResult.summary.byMethod` across five method labels: `'heuristic'`, `'llm'`, `'hybrid'`, `'auto-assigned'`, and `'unclassified'`. This granular tracking supports post-hoc analysis of classification <USER_ID_REDACTED> and guides decisions about when LLM classification adds value over cheaper heuristic approaches.

## Integration Points

SemanticAnalysis has three primary external dependencies. First, it integrates with **km-core's OntologyRegistry** for entity classification, mediated by `LegacyOntologyAdapter` to preserve backward compatibility with existing `OntologyValidator` and `OntologyClassifier` consumers. The Ontology child component represents this classification stage. Second, it consumes **LLMService from `@rapid/llm-proxy`** throughout the pipeline for semantic reasoning — this connects it to the LLMAbstraction sibling component, which manages provider routing across Anthropic, OpenAI, Docker Model Runner, and the rapid-llm-proxy middleware. Third, CodeGraphAgent reaches into the **code-graph-rag MCP server** (backed by Tree-sitter AST parsing and Memgraph) — a dependency on infrastructure managed by the DockerizedServices sibling component.

Within the pipeline, the InsightGenerationAgent (the Insights child component) acts as the terminal consumer of all upstream agent outputs, synthesizing classified and validated entities into higher-order insights rather than raw classifications. The ContentValidationAgent bridges the pipeline to the persistence layer through `KmCoreAdapter` (`storage/km-core-adapter.ts`), which speaks to the same graph and persistence infrastructure used by the KnowledgeManagement sibling.

The Coding parent component provides the overarching hierarchy into which all structured knowledge produced by SemanticAnalysis flows, making SemanticAnalysis the primary mechanism by which raw development activity becomes queryable knowledge within the Coding project.

## Usage Guidelines

**Choosing analysis depth.** Use `'surface'` for rapid, low-cost passes when full cross-analysis is unnecessary (e.g., CI checks on small changesets). Reserve `'deep'` and `'comprehensive'` for scheduled batch runs or when architectural pattern discovery is the goal. The 5-file cap at surface depth is a hard behavioral boundary, not a configuration option, so callers should not expect partial cross-analysis at that depth.

**Memgraph and code-graph-rag availability.** CodeGraphAgent degrades gracefully when Memgraph is unreachable or the CLI directory is absent, but this means graph-structural signals will be absent from analysis. Operators should ensure the DockerizedServices stack is healthy before triggering comprehensive runs if code graph data is required for completeness. Environment variables `CODING_REPO`, `CODING_TOOLS_PATH`, and `CODING_ROOT` must be set correctly for directory resolution to succeed without falling back to the sibling heuristic.

**Staleness cache and parallel workers.** The 60-second git commit cache in ContentValidationAgent is in-memory and per-process; it does not survive restarts. For long-running batch jobs that span the TTL, some redundant git fetches will occur — this is intentional to avoid serving stale data. The parallel worker count (default 1, max 20) should be tuned based on the underlying git repository's fetch latency and available I/O concurrency; leaving it at 1 is safe but conservative.

**Extending the pipeline.** New agents must extend `BaseAgent<TInput, TOutput>` and implement the full `process()` method. The `execute()` lifecycle is non-negotiable — agents should not bypass it to skip confidence scoring or issue detection, as downstream orchestration logic depends on the `AgentResponse` envelope fields being populated. Token usage telemetry should be attached via `attachTokenLogger` from `utils/token-usage-logger.ts` for any agent that invokes LLMService, keeping per-agent cost accounting consistent with the pattern established in SemanticAnalysisAgent.

**Ontology migrations.** Future migrations away from legacy ontology interfaces should follow the strangler-fig pattern demonstrated by `LegacyOntologyAdapter`: introduce the new implementation behind an adapter that preserves the existing interface, validate parity, then incrementally migrate call sites. The Phase 42-03 migration in OntologyClassificationAgent provides a concrete template for this approach, including the atomic reload semantics enabled by OntologyRegistry.

- [LiveLoggingSystem](./LiveLoggingSystem.md) -- [LLM] The LiveLoggingSystem is built around a strict abstract interface defined by the `TranscriptAdapter` class in `lib/agent-api/transcript-api.js`. Every agent-specific adapter must implement five methods: `getAgentType()` (returns a string identifier like `'claude-code'`), `getTranscriptDirectory()` (returns the filesystem path where raw agent transcripts reside), `readTranscripts()` (reads and parses raw transcript files), `convertToLSL()` (transforms raw entries into the unified LSL typed format), and `getCurrentSession()` (returns metadata for the active session). Live capture is achieved not through filesystem watchers (like `fs.watch`) but through a polling loop: `watchTranscripts()` uses `setInterval` to periodically invoke `readTranscripts()` and diff against previously seen entries. This design trades immediacy for portability—`fs.watch` has known cross-platform inconsistencies, especially in Docker containers and network-mounted filesystems, so polling avoids those failure modes at the cost of introducing a configurable latency between an agent writing a transcript entry and the LSL system capturing it.

- [KnowledgeManagement](./KnowledgeManagement.md) -- [LLM] GraphDatabaseService (src/knowledge-management/GraphDatabaseService.js) uses a single monolithic LevelDB key 'graph' to persist the entire in-memory Graphology graph as one JSON blob with the shape { nodes, edges, metadata }. This design means every write operation—whether updating a single node attribute or adding an edge—requires serializing the entire graph and writing it back under that one key via _persistGraphToLevel(). The trade-off is simplicity of implementation and straightforward restore (load the single key, deserialize, reconstruct the Graphology instance) at the cost of write amplification as the graph grows. The service supports both manual and auto-persist modes, allowing callers to batch multiple mutations before triggering a persist, which partially mitigates this overhead. New developers should be aware that concurrent writes are not safe without external serialization since a read-modify-write cycle on the single key is not atomic in LevelDB.
