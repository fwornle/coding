# SemanticAnalysis Component — Technical Insight Document

## What It Is

SemanticAnalysis is a multi-agent knowledge extraction and qualification subsystem implemented primarily under `src/agents/` and `src/ontology/` within the `integrations/mcp-server-semantic-analysis/` integration. It sits as a sibling component to KnowledgeManagement, LiveLoggingSystem, LLMAbstraction, DockerizedServices, CodingPatterns, and ConstraintSystem under the Coding project root, and it owns four child concerns: Pipeline (DAG-based agent sequencing), Ontology and OntologySubsystem (two-tier classification), and Insights (post-persistence knowledge synthesis).

Its purpose is to transform raw development artifacts — git commits, vibe session data, code graph statistics, and source file contents — into classified, validated knowledge entities suitable for ingestion into the broader KnowledgeManagement graph database. The architecture is organized around a fixed, contract-driven agent lifecycle defined in `src/agents/base-agent.ts`, with each specialized agent (`SemanticAnalysisAgent`, `OntologyClassificationAgent`, `CodeGraphAgent`, `ContentValidationAgent`) implementing the same five-method execution envelope while contributing a distinct phase to the pipeline.

![SemanticAnalysis — Architecture](images/semantic-analysis-architecture.png)

The component formalizes its agent architecture in `integrations/mcp-server-semantic-analysis/docs/architecture/agents.md` and its model-cost-vs-depth strategy in `docs/TIERED-MODEL-PROPOSAL.md`. Together these documents describe a system where deliberate separation of extraction, qualification, and classification stages allows each agent to be developed, tested, and tuned independently while still composing into a coherent pipeline.

## Architecture and Design

The architectural foundation is the `BaseAgent<TInput, TOutput>` abstract class in `src/agents/base-agent.ts`, which establishes a rigid five-method execution contract: `process()`, `calculateConfidence()`, `detectIssues()`, `generateRouting()`, and `applyCorrections()`. These are not loose interface suggestions — they are called sequentially within a standardized envelope, meaning an agent cannot opt out of any phase. The resulting `AgentResponse` carries the domain output plus metadata (timestamps, model usage), routing suggestions for downstream agents, and a corrections list for self-healing. The generic type parameters allow this same envelope to handle wildly different domains — from raw git commit arrays consumed by `SemanticAnalysisAgent` to ontology classification batches consumed by `OntologyClassificationAgent` — without sacrificing static type safety.

A key architectural decision is that pipeline topology is not statically hardcoded. The `generateRouting()` lifecycle method lets each agent declare which downstream agents should receive its output and with what priority. This distributed routing model gives the system flexibility (an agent can conditionally route based on what it found) but creates a debugging trade-off: to understand the full execution graph, a developer must trace `generateRouting()` implementations across all agent files rather than reading a single orchestrator file. This complements the child Pipeline component, which sequences agents in a fixed order via `batch-analysis.yaml` with explicit `depends_on` DAG edges — the YAML declares static structure, while routing declarations express dynamic conditional flow.

![SemanticAnalysis — Relationship](images/semantic-analysis-relationship.png)

A recurring architectural motif is **two-source enrichment**: heuristic results are first computed deterministically, then an LLM pass is invoked, and finally the two are merged. `SemanticAnalysisAgent` exemplifies this — it runs regex/AST-based pattern detection into `codeAnalysis.architecturalPatterns`, then calls an LLM to extract `semanticInsights.keyPatterns`, then merges the LLM-identified patterns back into `codeAnalysis.architecturalPatterns`. The final list is therefore a union of deterministic precision and probabilistic nuance. The pipeline as a whole reflects a parallel separation of concerns: `SemanticAnalysisAgent` and `CodeGraphAgent` perform extraction, `ContentValidationAgent` performs qualification, and `OntologyClassificationAgent` performs classification, with `AgentResponse` routing chaining them together.

## Implementation Details

The `SemanticAnalysisAgent` in `src/agents/semantic-analysis-agent.ts` implements an analysis-depth tiering system with three named levels: `surface`, `deep`, and `comprehensive`. The `surface` tier is explicitly capped at no more than five files and skips full file reading, making it suitable for fast-path heuristic checks on large repositories. The `deep` and `comprehensive` tiers unlock full file reading and cross-analysis — the agent reads actual file contents and correlates signals across multiple files to detect emergent patterns. The tiered model selection in `docs/TIERED-MODEL-PROPOSAL.md` aligns model cost with analytical depth: cheaper, faster models drive surface-tier runs, while more capable models are reserved for deep and comprehensive tiers. The agent also writes a per-run JSON trace to `logs/semantic-analysis-trace-{timestamp}.json` capturing the full pipeline input (git commits, vibe session data, code graph statistics), and crucially, trace file failures are caught and suppressed so observability tooling can never abort the analysis pipeline.

The `OntologyClassificationAgent` in `src/agents/ontology-classification-agent.ts` navigates a deliberate TypeScript constraint: the project's `tsconfig.json` sets `moduleResolution: node`, which prevents sub-path exports from `@fwornle/km-core` being resolved. The agent therefore imports from the top-level barrel and wraps an `OntologyRegistry` in a `LegacyOntologyAdapter` shim that translates the current registry API into the older interface expected by `OntologyValidator` and `OntologyClassifier`. Classification then proceeds in two passes — upper ontology assigns broad conceptual categories, lower ontology refines to project-aware labels — reflecting the structure the child Ontology component formalizes. The `ClassificationProcessResult` aggregates per-item LLM token usage into an `llmUsage` block with `modelsUsed` and `providersUsed` arrays, enabling the orchestrator to audit which models were invoked and enforce token budgets across heterogeneous providers.

The `CodeGraphAgent` in `src/agents/code-graph-agent.ts` implements infrastructure-dependency management through graceful degradation. Before any graph operations it performs two independent reachability checks: a TCP socket probe against Memgraph (default port 7687) via `checkMemgraphConnection()`, and a `which uv` shell invocation to confirm the `uv` Python package manager is on PATH. If either fails, the agent sets `skipped: true` and returns normally rather than throwing. The TCP probe is intentionally lightweight — it confirms port reachability without incurring a full Bolt handshake — but the trade-off is that a listening-but-initializing Memgraph can pass the probe and fail later. The `uv` dependency implies the code graph ingestion invokes Python subprocesses for AST parsing or graph construction tooling.

The `ContentValidationAgent` in `src/agents/content-validation-agent.ts` addresses large-repository validation performance through two complementary mechanisms: up to 20 configurable parallel validation workers, and a `GitStalenessDetector` that caches git commit lookups in memory for 60 seconds (`GIT_CACHE_TTL_MS = 60000`). The cache eliminates redundant `git log`/`git show` subprocess invocations when many entities reference the same commits during a single batch run. The 1-minute TTL is a deliberate trade-off — long enough for meaningful deduplication within one pipeline run, short enough that consecutive validation passes don't see stale data. Staleness itself uses git history as ground truth: an entity is stale if its referenced source has been superseded by newer commits since last validation.

## Integration Points

SemanticAnalysis integrates with several sibling and child components through well-defined seams. The OntologySubsystem child centralizes all configuration loading under `src/ontology/` via `OntologyConfigManager`, meaning `OntologyClassificationAgent` consumes entity-type hierarchies through a single managed entry point rather than scattered files. The Ontology child enforces the two-tier upper/lower classification structure that the agent then exploits. The Pipeline child enforces fixed DAG sequencing via `batch-analysis.yaml`, while `BaseAgent.generateRouting()` contributes dynamic, per-run routing on top. The Insights child operates as a post-persistence concern, consuming already-written knowledge graph data rather than raw pipeline input — that boundary is documented in `integrations/mcp-server-semantic-analysis/docs/architecture/agents.md`.

Externally, the component depends on the sibling KnowledgeManagement component as its downstream consumer: classified, validated entities flow into the Graphology+LevelDB graph database via PersistenceAgent and GraphDatabaseAdapter (which in turn route through the VKB API or fall back to direct LevelDB). It also depends on the `@fwornle/km-core` package for `OntologyRegistry`, on Memgraph for code graph storage, and on the `uv` CLI for Python tooling orchestration. The LLMAbstraction sibling is implicitly leveraged whenever an agent calls a model — the tiered model selection strategy in `docs/TIERED-MODEL-PROPOSAL.md` plugs naturally into LLMAbstraction's three-tier mode resolution hierarchy, so a surface-tier run can be routed to a local Docker Model Runner while a comprehensive run is routed to a cloud API.

The `AgentResponse` envelope itself is the integration contract between agents. Its metadata (timestamps, token usage), routing suggestions, and corrections list are how stage N communicates context to stage N+1. Downstream stages that consume `CodeGraphAgent` output must explicitly check the `skipped` flag — a subtle contract that, if missed, will lead to silent absence of code graph data rather than an exception. Similarly, the JSON trace file written to `logs/semantic-analysis-trace-{timestamp}.json` is the integration point for external observability tooling and post-mortem analysis.

## Usage Guidelines

When implementing a new agent, recognize that subclassing `BaseAgent<TInput, TOutput>` is not just about writing a `process()` function — all five lifecycle slots must be filled meaningfully. Returning empty stubs for `detectIssues()` or `generateRouting()` will compile and run, but the orchestrating pipeline depends on those fields to make branching decisions. In particular, `generateRouting()` is where you declare downstream connectivity; if you skip it, your agent's output will be a dead end regardless of what the YAML pipeline definition implies.

When working inside `SemanticAnalysisAgent`, respect the analysis-depth tiering: do not bypass the five-file cap on `surface` or add file reads to that tier, as it exists specifically to keep fast-path runs cheap on large repositories. If you add tracing or logging, follow the existing pattern of suppressing I/O failures so observability never aborts the analysis pipeline. When adding pattern detection, prefer the two-source enrichment motif — implement deterministic heuristics first, then layer LLM-derived patterns on top, then merge — rather than relying solely on one source.

For `ContentValidationAgent` work, route every git query through `GitStalenessDetector` rather than spawning direct `git` subprocesses; failing to do so will degrade batch validation throughput and bypass the 60-second deduplication cache. For `CodeGraphAgent` consumers, always check the `skipped: true` flag before reading code graph data. For `OntologyClassificationAgent`, remember the `moduleResolution: node` constraint — import `@fwornle/km-core` from its top-level barrel only, and route through `LegacyOntologyAdapter` when interacting with `OntologyValidator` or `OntologyClassifier`.

Finally, treat the architectural motifs as authoritative: the singleton guard pattern from the sibling CodingPatterns component applies to any stateful manager you introduce here (consult `docs/puml/psm-singleton-pattern.puml`), and the graceful-degradation pattern from `CodeGraphAgent` should be the default response to optional infrastructure dependencies — set a `skipped` flag and return rather than throwing.

### Summary

1. **Architectural patterns identified**: Contract-driven agent lifecycle (`BaseAgent` five-method envelope), two-source enrichment (heuristic + LLM merge), graceful degradation for optional infrastructure, distributed routing via `generateRouting()`, two-tier ontology classification, tiered analysis depth aligned with tiered model selection, adapter shim (`LegacyOntologyAdapter`) for API translation, and in-memory TTL cache (`GitStalenessDetector`) for subprocess deduplication.

2. **Design decisions and trade-offs**: Rigid five-method contract trades implementation overhead for predictable orchestrator behavior; distributed routing trades single-file readability for conditional flexibility; lightweight TCP probe trades probe accuracy for low overhead; 60-second git cache TTL balances within-run deduplication against cross-run freshness; suppressing trace I/O errors trades observability completeness for pipeline robustness.

3. **System structure insights**: The pipeline is a three-stage flow — extraction (`SemanticAnalysisAgent`, `CodeGraphAgent`), qualification (`ContentValidationAgent`), classification (`OntologyClassificationAgent`) — chained via `AgentResponse` routing, with static DAG structure declared in `batch-analysis.yaml` and dynamic routing layered on top.

4. **Scalability considerations**: Parallelism via up to 20 validation workers, subprocess deduplication via `GitStalenessDetector`, fast-path surface tier for large repos, cheaper models for shallower analysis, and per-batch token-budget enforcement via aggregated `llmUsage` reporting.

5. **Maintainability assessment**: Strong — type-safe generic base class, single-responsibility agents, documented architecture in `agents.md` and `TIERED-MODEL-PROPOSAL.md`, and centralized ontology configuration via `OntologyConfigManager`. The main maintenance risk is the distributed `generateRouting()` model, which obscures the global execution graph and requires cross-file tracing to fully understand pipeline topology.

- [LiveLoggingSystem](./LiveLoggingSystem.md) -- [LLM] The TranscriptAdapter class in lib/agent-api/transcript-api.js establishes a strict plugin contract for integrating new agent conversation sources into the LSL pipeline. It requires five abstract methods: getAgentType(), getTranscriptDirectory(), readTranscripts(), convertToLSL(), and getCurrentSession(). This design cleanly separates concerns — the base class owns the lifecycle and dispatch logic, while subclasses own the format-specific parsing. Claude Code transcripts live at ~/.claude/projects/<project>/conversation.jsonl, a JSONL file that grows append-only during a session; the Copilot CLI adapter targets a different directory structure. The convertToLSL() method is the normalization seam, responsible for mapping each agent's native message structure into the unified LSL typed-entry format with types: user, assistant, tool_use, tool_result, system, and error. A new developer adding a third agent integration (e.g., Cursor, Aider) only needs to subclass TranscriptAdapter and implement these five methods — no changes to the core LSL infrastructure are required. The getCurrentSession() method is particularly important: it must return the 'live' session object so the polling loop knows which entries are part of the current conversation versus a historical one, which implies adapters must implement their own session-boundary detection logic appropriate to their agent's format.

The legacy `mockLLM` boolean is retained explicitly for backward compatibility, meaning older configuration files or callers that predate the `llmState` structure continue to function without migration. New developers should be aware that if they observe unexpected behavior when `globalMode` is set but `mockLLM` is also present in a persisted state file, the `mockLLM` boolean is the lowest-priority fallback and will only activate when neither of the higher-priority fields are populated.
- [DockerizedServices](./DockerizedServices.md) -- [LLM] The ProcessStateManager (PSM) singleton implements a deliberate decoupling between service identity and process identity across both `scripts/api-service.js` and `scripts/dashboard-service.js`. Each script follows an identical structural pattern: spawn a child process via Node's `child_process` module, register the resulting process handle with the PSM via `psm.registerService()`, wire up `SIGTERM`/`SIGINT` forwarding so that signals delivered to the wrapper propagate to the child, and call `psm.unregisterService()` in the exit handler. This indirection means that the rest of the system (including `scripts/health-coordinator.js`) can query the PSM registry without holding direct references to OS-level process IDs. The practical consequence for developers is that a service restart — where a new child process replaces the old one — does not require the health coordinator or any consumer of PSM state to be aware of the PID change; only the wrapper scripts update the registry. This pattern also cleanly isolates the restart/retry logic in `lib/service-starter.js` from signal-handling responsibilities, since the wrapper owns the process lifecycle signals while the starter owns the retry policy. A new developer should note that adding a new containerized service almost certainly means creating a new wrapper script that replicates this boilerplate rather than centralizing it, which is a potential maintenance concern as the number of services grows.

- [CodingPatterns](./CodingPatterns.md) -- [LLM] The project-wide singleton guard pattern is formally codified in `docs/puml/psm-singleton-pattern.puml` and manifests consistently wherever stateful managers are instantiated. The pattern follows a strict guard-and-return idiom: a module-level variable holds the single instance (initialized to null or undefined), and every access point checks that variable before constructing a new object. If an instance already exists, the existing reference is returned immediately without re-running any constructor or initialization logic. This prevents race conditions in async service environments where multiple subsystems might attempt to spin up the same stateful manager concurrently — a real concern in Node.js applications that use event-driven concurrency without explicit locking primitives. For new developers, the implication is that any class described as a 'manager' or 'session' object in this codebase should be assumed to follow this pattern: do not call `new` directly on these classes from arbitrary call sites; instead, always go through the designated factory or accessor function that enforces the singleton contract. The PlantUML diagram in `docs/puml/psm-singleton-pattern.puml` is authoritative and should be consulted before introducing any new singleton-style manager to ensure the guard logic is structurally consistent with the rest of the project.
- [ConstraintSystem](./ConstraintSystem.md) -- [LLM] The ConstraintSystem implements a two-level configuration hierarchy through `HookConfigLoader` (lib/agent-api/hooks/hook-config.js) that distinguishes between user-wide defaults and project-specific overrides. User-level configuration lives at `~/.coding-tools/hooks.json`, making it available across all Claude Code sessions on the machine, while project-level configuration resides at `{project}/.coding/hooks.json`, enabling per-repository constraint customization. The `mergeConfigs()` method is the critical integration point: it applies project configuration on top of user configuration, meaning any handler, constraint rule, or hook binding defined at the project level supersedes or augments what the user has set globally. This design has a meaningful implication for teams: a repository can enforce stricter or more specific constraints than a developer's personal defaults without requiring them to change their global setup. However, because the merge strategy gives project config full precedence, there is no mechanism for user config to 'lock' a setting that cannot be overridden by a project — a security boundary that new developers should be aware of when assessing whether globally-defined compliance constraints can be bypassed by a malicious or misconfigured `.coding/hooks.json`.
