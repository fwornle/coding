# SemanticAnalysis — Technical Reference

## What It Is

SemanticAnalysis is a multi-agent analysis pipeline within the broader Coding project, responsible for extracting semantic meaning, ontological classification, and validation insights from code and git data. The pipeline is implemented across several agent files under `src/agents/`: `base-agent.ts`, `code-graph-agent.ts`, `semantic-analysis-agent.ts`, `ontology-classification-agent.ts`, and `content-validation-agent.ts`. It sits as a peer to siblings like KnowledgeManagement and LLMAbstraction within the Coding root, and exposes its capabilities externally through the MCP server pattern documented in `mcp-server-semantic-analysis` (consistent with the convention enforced across CodingPatterns).

The pipeline contains five principal child components: Pipeline, Ontology, Insights, OntologyRegistry, and KmCoreAdapter, each representing a distinct concern that collectively produce enriched semantic analysis results for downstream consumers.

## Architecture and Design

![SemanticAnalysis — Architecture](images/semantic-analysis-architecture.png)

The central architectural pattern is a **coordinator with template-method agents**. Every agent—`CodeGraphAgent`, `SemanticAnalysisAgent`, `OntologyClassificationAgent`, and `ContentValidationAgent`—extends `BaseAgent<TInput,TOutput>` defined in `src/agents/base-agent.ts`. The base class enforces a strict six-step `execute()` sequence: `process()`, `calculateConfidence()`, `detectIssues()`, `generateRouting()`, `applyCorrections()`, and `buildMetadata()`. This template-method design means the pipeline always produces a uniform `AgentResponse` envelope regardless of which agent runs, and cross-cutting concerns like confidence scoring and metadata construction are guaranteed to execute in a consistent order. The Pipeline child component directly embodies this contract.

The design trades flexibility for consistency. Agents cannot short-circuit the sequence without throwing exceptions, which imposes overhead even for trivially simple outputs. However, it means a new developer only needs to implement `process()` to introduce a new agent—all other lifecycle steps are inherited. This is a deliberate maintainability choice that accepts some runtime inefficiency in exchange for predictable behavior across the agent fleet.

![SemanticAnalysis — Relationship](images/semantic-analysis-relationship.png)

Two additional structural patterns appear at the component boundaries. In `OntologyClassificationAgent`, a **strangler-facade / adapter pattern** is used: a `LegacyOntologyAdapter` wraps km-core's `OntologyRegistry` and exposes the old ontology-loading interface to `OntologyValidator` and `OntologyClassifier`, which remain unmodified. The call chain is `OntologyClassificationAgent → LegacyOntologyAdapter → OntologyRegistry`. This shim is explicitly a migration artifact, not a permanent abstraction. In `ContentValidationAgent`, a **structural aliasing pattern** appears: a single `KmCoreAdapter` instance (from `storage/km-core-adapter.ts`) is assigned to two fields—`graphDB` and `persistenceAgent`—preserving all existing call sites without a large-scale refactor. Both patterns reflect the same philosophy: limit the blast radius of refactoring by containing change to a single class while keeping external interfaces stable.

## Implementation Details

`CodeGraphAgent` (`src/agents/code-graph-agent.ts`) implements a defensive availability probe via `checkMemgraphConnection()`, which opens a raw TCP socket to the Memgraph host with a timeout sourced from `loadAgentTuningConfig().code_graph.memgraph_check_timeout_ms`. On timeout, the agent sets `skipped=true` on its result and returns early rather than throwing. This mirrors the TCP probe pattern used in the sibling DockerizedServices component (`lib/utils/service-probe.js`), where Memgraph's Bolt protocol is also probed via raw TCP. Any new Memgraph query methods added to `CodeGraphAgent` must guard on the `skipped` flag; downstream pipeline stages must actively inspect this flag rather than relying on exception propagation to detect graph unavailability.

`SemanticAnalysisAgent.analyzeGitAndVibeData()` (`semantic-analysis-agent.ts`) writes input payloads to `logs/semantic-analysis-trace-*.json` before LLM synthesis. These writes are non-fatal: filesystem failures are caught, suppressed, and analysis proceeds. LLM access is mediated through `@rapid/llm-proxy`'s `LLMService`—consistent with the LLMAbstraction sibling's routing through the `rapid-llm-proxy` daemon—and token consumption is instrumented via `attachTokenLogger`, ensuring quota tracking even on non-fatal paths. The silent suppression of trace write failures is a known observability gap: no warning is emitted when writes fail, meaning post-hoc debugging may find no trace files with no indication they were never written.

`OntologyClassificationAgent` aggregates `llmUsage` statistics—`totalPromptTokens`, `modelsUsed`, `providersUsed`—across all per-observation LLM calls into its `ClassificationProcessResult`. This rollup enables the coordinator to enforce token budgets at batch granularity. The Ontology and OntologyRegistry child components represent the classification domain logic and registry integration respectively, with the `LegacyOntologyAdapter` shim bridging them during the ongoing migration.

`ContentValidationAgent` (`src/agents/content-validation-agent.ts`) implements a 1-minute TTL git commit cache (`GIT_CACHE_TTL_MS = 60000`) in `cachedGitCommits` to prevent redundant git subprocess invocations across a batch run. This cache is instance-scoped: it only provides benefit when the coordinator reuses the same agent instance across multiple entities. If the coordinator instantiates a new `ContentValidationAgent` per entity, the TTL design is operationally inert.

## Integration Points

SemanticAnalysis integrates with Memgraph (graph database) through `CodeGraphAgent`'s TCP probe and query layer, with km-core's `OntologyRegistry` through `LegacyOntologyAdapter` inside `OntologyClassificationAgent`, and with the storage layer through `KmCoreAdapter` (`storage/km-core-adapter.ts`), which centralizes entity write paths previously split across `GraphDatabaseAdapter` and `PersistenceAgent`. The KmCoreAdapter child component is the canonical write interface, and its dual-field aliasing in `ContentValidationAgent` is the current transition state toward full consolidation.

LLM access across the pipeline flows through `@rapid/llm-proxy`'s `LLMService`, placing SemanticAnalysis as a consumer of the LLMAbstraction sibling's proxy infrastructure. Token instrumentation via `attachTokenLogger` ensures that all LLM calls within the pipeline participate in the same billing and quota tracking infrastructure. The pipeline is also exposed as an MCP tool via `mcp-server-semantic-analysis`, following the same directory and documentation conventions enforced across CodingPatterns.

## Usage Guidelines

**Skipped-state handling is mandatory.** Because `CodeGraphAgent` signals graph unavailability via `skipped=true` rather than exceptions, any coordinator or downstream consumer must explicitly inspect this flag before treating graph data as present. Relying solely on exception handling will silently miss graph outages.

**`LegacyOntologyAdapter` is a migration shim, not a stable abstraction.** Developers should not add new consumers of its interface. New code interacting with ontology data should target `OntologyRegistry` directly. The adapter exists to avoid rewriting `OntologyValidator` and `OntologyClassifier` during the current migration phase and will eventually be removed.

**The dual-field alias in `ContentValidationAgent` is intentional but fragile.** Both `this.graphDB` and `this.persistenceAgent` point to the same `KmCoreAdapter` instance. This is not a bug, but a developer unfamiliar with this decision may introduce a second adapter instance thinking the fields represent separate connections—which would be incorrect and could cause inconsistent state.

**Trace write failures are currently silent.** When `analyzeGitAndVibeData()` cannot write to `logs/semantic-analysis-trace-*.json`, no warning is emitted. Teams relying on trace files for debugging should add monitoring on the presence of these files in production, or contribute a structured warning log on write failure to close the observability gap.

**Instance reuse matters for `ContentValidationAgent`.** The git commit cache only amortizes subprocess cost when the same instance processes multiple entities. Coordinator implementations that instantiate a new agent per entity should be aware the cache provides no benefit in that configuration and may wish to reconsider instance lifecycle if git subprocess cost becomes a bottleneck.

### Siblings
- [LiveLoggingSystem](./LiveLoggingSystem.md) -- [LLM] The `TranscriptAdapter` abstract base class in `lib/agent-api/transcript-api.js` defines a pluggable adapter contract that decouples the LSL pipeline from specific agent implementations. The five abstract methods — `getAgentType()`, `getTranscriptDirectory()`, `readTranscripts()`, `convertToLSL()`, and `getCurrentSession()` — establish a clear interface that concrete adapters (e.g., for Claude Code and Copilot CLI) must implement. This pattern means the core LSL infrastructure never directly references agent-specific transcript formats or filesystem layouts. A new developer adding support for a third agent (say, Cursor or Aider) would subclass `TranscriptAdapter` and implement only these five methods without touching the converter, file manager, or validation layers. The `watchTranscripts()` method is notably NOT abstract — it is a concrete polling implementation shared by all adapters, using `setInterval` with a default 1000ms interval and a `lastEntryCount` integer cursor that advances only when new entries are detected, then fires registered callbacks with only the delta entries. This means the watch mechanism is O(n) in total entry count per poll cycle, which could become a performance concern for very long sessions.
- [LLMAbstraction](./LLMAbstraction.md) -- LLMAbstraction is a multi-layered abstraction over LLM providers that enables provider-agnostic model calls across public APIs (Anthropic, OpenAI, Groq), a local Docker Model Runner (DMR), and a mock mode for testing. The system routes requests through a rapid-llm-proxy daemon (port 12435) that handles provider selection, tier-based routing, and per-call telemetry attribution. Mode selection is dynamic and per-agent, stored in workflow-progress.json, supporting global and per-agent overrides across three modes: 'mock', 'local' (DMR), and 'public' (cloud APIs).
- [DockerizedServices](./DockerizedServices.md) -- [LLM] The DockerizedServices component uses a dual-probe health checking architecture implemented in lib/utils/service-probe.js that strictly separates HTTP-based and TCP-based service verification. probeHttpHealth() issues HTTP/HTTPS requests and maps 2xx/3xx response codes to the 'running' state, while probeTcpPort() opens a raw net.Socket connection to verify port reachability for non-HTTP protocols like Memgraph's Bolt protocol. A critical architectural invariant (documented as SPEC R6) enforces that neither probe ever returns the string 'healthy'—only 'running', 'stopped', or 'unknown' are valid return values. This distinction matters because health-coordinator.js in scripts/health-coordinator.js consumes these probes on a 5-second polling interval and must be able to uniformly handle all service types (Next.js dashboard, Node.js API, Memgraph, Redis) without special-casing the protocol. New developers adding services must use probeTcpPort() for any non-HTTP service and must not introduce a fourth status string, or the health-coordinator's state machine will behave incorrectly.
- [KnowledgeManagement](./KnowledgeManagement.md) -- [LLM] The primary persistence mechanism in KnowledgeManagement is a single-key LevelDB strategy implemented in `src/knowledge-management/GraphDatabaseService.js`. Rather than storing each graph entity as a separate LevelDB key (which would enable partial reads and atomic per-entity updates), the entire Graphology in-memory graph is serialized as one JSON blob stored under the key `'graph'`. This blob contains all nodes, edges, and metadata. Writes are deferred using an `isDirty` flag — mutations to the graph set `isDirty = true`, and `_persistGraphToLevel()` is only called when a flush is explicitly triggered. This design optimizes for read-heavy, batch-write workloads but creates a risk of data loss if the process crashes between mutations and the next flush. New developers should be aware that any code path that modifies graph nodes or edges must ensure the flush cycle is triggered, or changes will silently remain only in memory.
- [CodingPatterns](./CodingPatterns.md) -- [LLM] The MCP (Model Context Protocol) server pattern is applied uniformly across all three major integrations: mcp-constraint-monitor, mcp-server-semantic-analysis, and code-graph-rag. Each integration follows a consistent directory layout with a README.md describing the tool interface, a docs/ subdirectory containing architectural detail, and configuration files that declare the tool's capabilities to AI agent hosts. This pattern means new integrations can be scaffolded by copying this structure and registering new tool definitions. The mcp-server-semantic-analysis integration further documents this in docs/api/README.md and docs/architecture/README.md, separating API surface from internal architecture. Developers adding a new MCP tool should expect to create at minimum: a README with tool description, an API reference doc, and a configuration file declaring tool schemas. The consistency of this pattern across integrations suggests it is an enforced project convention rather than an organic coincidence.
- [ConstraintSystem](./ConstraintSystem.md) -- The ConstraintSystem is a monitoring and enforcement layer that validates code actions and file operations against configured rules during Claude Code sessions. It operates through a hook-based architecture where constraint checks are triggered at key lifecycle points (pre-tool, post-tool, etc.) and violations are captured, persisted, and surfaced to dashboards. The system integrates with Claude Code's native hook mechanism via configuration files at user-level (~/.coding-tools/hooks.json) and project-level (.coding/hooks.json), with project config overriding user config.
