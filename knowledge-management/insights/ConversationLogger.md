* **Singleton** – ensures a single `ConversationLogger` instance across the application.

### Design decisions and trade‑offs
* **Singleton vs. multiple loggers** – simplifies configuration and guarantees a single point of truth, but makes testing harder because the global state must be reset between tests.  
* **Queue for async writes** – improves caller latency and throughput, at the cost of added memory pressure if producers outpace the consumer. Back‑pressure mechanisms are not described, so developers must monitor queue growth.  
* **Configurable log level** – offers flexibility for dev vs. prod, but requires disciplined early configuration to avoid noisy logs in production.

* Because the logger is a singleton, horizontal scaling (multiple Node processes) would require each process to maintain its own logger instance or externalise the queue (e.g., via a message broker). The current design is optimized for a single‑process deployment.
