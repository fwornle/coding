## Usage Guidelines  

* **Always route human edits through the interface** – Developers must never attempt to write directly to LevelDB or any local replica when handling manual edits.  The `EntityAuthoringInterface` guarantees that the edit will be processed by `GraphDatabaseAdapter`, checked by `VkbServerAvailabilityGate`, and persisted via `VkbApiClient`.

* **Respect server availability semantics** – Before invoking a manual write, ensure that the VKB HTTP server is expected to be up.  If the `VkbServerAvailabilityGate` reports the server as unavailable, the write should be deferred or the user should be notified; attempting to force a write will be rejected by the gate.

* **Do not modify routing logic** – The distinction between human‑authored and automated mutations is a core consistency safeguard.  Changing how `GraphDatabaseAdapter` classifies writes could introduce divergent state between the server and any local caches.

* **Maintain API contract fidelity** – Any changes to the VKB server’s REST endpoints (e.g., URL paths, request schemas) must be reflected in `VkbApiClient`.  Since all manual writes flow through this client, mismatches will cause immediate failures and break the single‑source‑of‑truth guarantee.

* **Testing** – Unit tests for `EntityAuthoringInterface` should mock the `VkbServerAvailabilityGate` to simulate both up and down states, verifying that writes are only attempted when the gate permits them.  Integration tests must confirm that successful writes result in the expected LevelDB state on the server.

### Architectural Patterns Identified  

1. **Gateway‑Router Pattern** – `VkbServerAvailabilityGate` acts as a gate; `GraphDatabaseAdapter` routes based on the gate’s outcome.  
2. **Facade/Interface Pattern** – `EntityAuthoringInterface` provides a simplified entry point for human‑authored mutations, hiding the underlying routing and transport complexity.  
3. **Single‑Source‑of‑Truth Enforcement** – By mandating REST writes to a server‑owned LevelDB, the design enforces data centralisation.

### Design Decisions & Trade‑offs  

* **Explicit separation of manual vs. automated writes** – Guarantees consistency for curated data but introduces dual pathways that must be kept in sync.  
* **Mandatory server‑availability check** – Prevents lost writes during outages but adds latency to each manual edit.  
* **No direct local write path** – Simplifies consistency guarantees at the cost of reliance on network connectivity and server uptime.

### System Structure Insights  

The subsystem is layered: UI → `EntityAuthoringInterface` → `GraphDatabaseAdapter` → `VkbServerAvailabilityGate` → `VkbApiClient` → VKB HTTP server → LevelDB.  Each layer has a single, well‑defined responsibility, facilitating clear ownership and testability.

### Scalability Considerations  

* **REST bottleneck** – All manual edits funnel through a single HTTP server; scaling the server (horizontal replication, load‑balancing) will be required as authoring volume grows.  
* **Gate latency** – The health‑check performed by `VkbServerAvailabilityGate` must remain lightweight; otherwise it could become a throughput limiter.

### Maintainability Assessment  

The clear separation of concerns (routing, availability, transport) supports maintainability: changes to one layer (e.g., swapping LevelDB for another store) can be isolated behind the REST API without touching the adapter or gate.  However, the dual‑write model introduces a maintenance overhead to ensure that automated pipeline writes remain compatible with the manually curated data model.
