# DockerizedServices

**Type:** Component

[LLM] The DockerizedServices component uses a dual-probe health checking architecture implemented in lib/utils/service-probe.js that strictly separates HTTP-based and TCP-based service verification. probeHttpHealth() issues HTTP/HTTPS requests and maps 2xx/3xx response codes to the 'running' state, while probeTcpPort() opens a raw net.Socket connection to verify port reachability for non-HTTP protocols like Memgraph's Bolt protocol. A critical architectural invariant (documented as SPEC R6) enforces that neither probe ever returns the string 'healthy'—only 'running', 'stopped', or 'unknown' are valid return values. This distinction matters because health-coordinator.js in scripts/health-coordinator.js consumes these probes on a 5-second polling interval and must be able to uniformly handle all service types (Next.js dashboard, Node.js API, Memgraph, Redis) without special-casing the protocol. New developers adding services must use probeTcpPort() for any non-HTTP service and must not introduce a fourth status string, or the health-coordinator's state machine will behave incorrectly.

# DockerizedServices — Technical Reference

## What It Is

DockerizedServices is the containerization and process-supervision layer of the Coding project, responsible for launching, health-checking, and gracefully terminating a set of heterogeneous services: a Next.js frontend dashboard (port 3030), a Node.js constraint-monitor API backend (port 3031), a Memgraph graph database (Bolt protocol), and Redis. The component is implemented across several key paths:

- `lib/utils/service-probe.js` — dual-protocol health probing
- `lib/service-starter.js` — retry-with-backoff service launch
- `scripts/health-coordinator.js` — polling-based health aggregation
- `scripts/api-service.js` and `scripts/dashboard-service.js` — wrapper/supervisor scripts
- `integrations/mcp-server-semantic-analysis/src/mock/llm-mock-service.ts` — Docker/host path duality (owned by child component DockerLLMModeControl)

Together these files form a self-contained supervision stack that handles the full service lifecycle: spawn, health verification, retry on failure, signal propagation, and graceful shutdown — without relying on an external process manager like systemd or PM2.

## Architecture and Design

![DockerizedServices — Architecture](images/dockerized-services-architecture.png)

The dominant architectural pattern is a **wrapper-as-supervisor** hierarchy. `scripts/api-service.js` and `scripts/dashboard-service.js` are not thin launchers — they are explicit parent processes that own the lifecycle of their children. SIGTERM and SIGINT received by the wrapper (which is the container's PID 1) are forwarded to child processes, enabling Docker's stop signal to cascade gracefully through the entire process tree. This is a deliberate solution to a concrete Docker problem: `child_process.spawn()` in Node.js does not automatically forward signals, so without this pattern child processes would be orphaned when Docker sends SIGTERM to PID 1, running until Docker's forceful SIGKILL timeout fires. The corollary is a hard constraint: the container's `CMD` or `ENTRYPOINT` must point to the wrapper script; pointing directly to the service binary silently breaks the entire signal chain.

The health-checking subsystem, implemented in ServiceProbeLibrary (`lib/utils/service-probe.js`), enforces a strict dual-probe architecture. `probeHttpHealth()` handles HTTP/HTTPS services by mapping 2xx/3xx response codes to `'running'`. `probeTcpPort()` opens a raw `net.Socket` for non-HTTP protocols such as Memgraph's Bolt. A system-wide invariant (SPEC R6) mandates that neither probe ever returns the string `'healthy'` — only `'running'`, `'stopped'`, or `'unknown'` are valid. This constraint exists so that HealthCoordinator (`scripts/health-coordinator.js`), which polls all probes on a 5-second interval, can handle all four service types uniformly without protocol-specific branching.

![DockerizedServices — Relationship](images/dockerized-services-relationship.png)

The ProcessStateManager (PSM) singleton provides a cross-service process registry, but it occupies a deliberately non-critical position in the architecture. Both `api-service.js` and `dashboard-service.js` register their child processes with PSM after spawn, but registration failures are explicitly caught, logged with a message like `'Continue anyway - not critical'`, and silently bypassed. PSM is therefore an observability and coordination layer, not a liveness dependency. This is a conscious trade-off: services remain available even if PSM has an initialization ordering issue or crashes, but lifecycle tracking (graceful shutdown propagation, restart coordination) can degrade silently — making PSM health an operationally important but easy-to-miss concern.

## Implementation Details

Service startup resilience is centralized in `lib/service-starter.js` via `startServiceWithRetry()`. The function wraps every child process launch in a retry loop with configurable `maxRetries` (default 3) and a per-attempt timeout (default 30 seconds). The critical detail is the inter-attempt cleanup sequence: before each retry, the function explicitly sends SIGTERM followed by SIGKILL to the failing child process. SIGTERM gives the child an opportunity to release bound ports; SIGKILL ensures termination even if SIGTERM is ignored. Without this two-signal sequence, a hung process from attempt N would continue holding its TCP port, causing attempt N+1 to fail immediately with `EADDRINUSE`. The module also provides `isPortListening()` (HTTP GET to `/health`) and `isTcpPortListening()` (raw `net.Socket`), mirroring the dual-probe design in `service-probe.js` — the two files share the same conceptual split between HTTP-aware and protocol-agnostic reachability checks.

Port assignment for the constraint monitor subsystem is hardcoded and split across the two wrapper scripts. `scripts/api-service.js` binds `integrations/mcp-constraint-monitor/src/dashboard-server.js` to port 3031 via the `CONSTRAINT_API_PORT` environment variable. `scripts/dashboard-service.js` starts the Next.js frontend on port 3030 via `CONSTRAINT_DASHBOARD_PORT`, and wires the frontend to the backend by injecting `NEXT_PUBLIC_API_BASE_URL=http://localhost:3031` into the Next.js process environment before invoking `npm run dev`. The `localhost`-relative URL is valid for host-networking Docker Compose configurations and local development, but is architecturally incompatible with multi-container Docker Compose where each service has its own network namespace.

DockerLLMModeControl, a child component implemented in `integrations/mcp-server-semantic-analysis/src/mock/llm-mock-service.ts`, handles the Docker/host filesystem duality through the pattern `process.env.CODING_ROOT || repositoryPath`. Inside Docker, `CODING_ROOT` is set to `/coding` (the container volume mount point), so paths like `.data/workflow-progress.json` resolve correctly. On a developer's host machine, the variable is unset and the fallback `repositoryPath` (derived from the module's own location) takes over. This file (`workflow-progress.json`) drives LLM mode switching between `'mock'`, `'local'`, and `'public'` modes — meaning that an incorrect path resolution silently breaks LLM provider selection in containerized deployments. The LLMAbstraction sibling component depends on this mode-switching file for its per-agent routing decisions, making correct `CODING_ROOT` configuration a cross-component concern.

## Integration Points

DockerizedServices sits under the Coding root and acts as the runtime substrate for several sibling components. The ConstraintSystem's dashboard server (`integrations/mcp-constraint-monitor/src/dashboard-server.js`) is launched and supervised by `scripts/api-service.js`. The SemanticAnalysis integration's mock LLM service is path-resolved through the `CODING_ROOT` pattern described above. LLMAbstraction's mode selection depends on `workflow-progress.json`, which DockerLLMModeControl locates using the same environment-aware resolution.

HealthCoordinator (`scripts/health-coordinator.js`) is the aggregation point for all service health state. It consumes ServiceProbeLibrary's two probe functions and polls them every 5 seconds, producing a unified health view across Next.js, Node.js API, Memgraph, and Redis. The strict three-value status contract (`'running'`/`'stopped'`/`'unknown'`) is the interface between ServiceProbeLibrary and HealthCoordinator — any violation of this contract at the probe layer propagates as undefined behavior in the coordinator's state machine.

The PSM singleton is the loose coupling point between `api-service.js`, `dashboard-service.js`, and any component that needs to locate or signal running child processes by name. Processes are registered under the names `'constraint-api-child'` (API) and the dashboard process respectively. Because PSM registration is non-fatal, downstream consumers of PSM data should treat its registry as eventually consistent and must not assume that an absent PSM entry means a service is not running.

## Usage Guidelines

**Adding a new service** requires choosing the correct probe type from ServiceProbeLibrary. HTTP/HTTPS services use `probeHttpHealth()`; any service using a non-HTTP protocol (like Bolt, Redis RESP, etc.) must use `probeTcpPort()`. Introducing a fourth status string beyond `'running'`/`'stopped'`/`'unknown'` will break HealthCoordinator's state machine and violates SPEC R6 — this is a hard constraint, not a convention.

**Containerizing a new service** requires writing a wrapper script following the pattern in `scripts/api-service.js` and `scripts/dashboard-service.js`. The wrapper must: (1) spawn the child via `child_process.spawn()`, (2) register it with PSM non-fatally, (3) forward SIGTERM and SIGINT to the child, and (4) be the container's `CMD`/`ENTRYPOINT` target. Skipping the signal-forwarding step will cause the child to be orphaned on container stop.

**Adding file-path-dependent features** inside Docker must follow the `process.env.CODING_ROOT || repositoryPath` pattern established in `llm-mock-service.ts`. Hardcoding absolute paths or using only `__dirname`-relative paths will break when the code runs inside a container where the volume is mounted at `/coding`. The `CODING_ROOT` variable is the canonical environment signal for "we are inside Docker."

**Network topology** must be considered before migrating to isolated multi-container Docker Compose. The hardcoded `NEXT_PUBLIC_API_BASE_URL=http://localhost:3031` in `dashboard-service.js` assumes same-namespace networking. Isolation would require replacing `localhost` with the Docker Compose service name (e.g., `http://constraint-api:3031`) and rebuilding the Next.js image so the public URL is baked in at build time or injected via a runtime environment mechanism.

**PSM health** should be monitored as an operational concern even though it is non-fatal. Silent PSM degradation means lifecycle events (shutdown propagation, restart coordination) may not execute as expected, producing hard-to-diagnose operational failures where services appear healthy at the probe level but are not correctly supervised at the registry level.

- [LiveLoggingSystem](./LiveLoggingSystem.md) -- [LLM] The `TranscriptAdapter` abstract base class in `lib/agent-api/transcript-api.js` defines a pluggable adapter contract that decouples the LSL pipeline from specific agent implementations. The five abstract methods — `getAgentType()`, `getTranscriptDirectory()`, `readTranscripts()`, `convertToLSL()`, and `getCurrentSession()` — establish a clear interface that concrete adapters (e.g., for Claude Code and Copilot CLI) must implement. This pattern means the core LSL infrastructure never directly references agent-specific transcript formats or filesystem layouts. A new developer adding support for a third agent (say, Cursor or Aider) would subclass `TranscriptAdapter` and implement only these five methods without touching the converter, file manager, or validation layers. The `watchTranscripts()` method is notably NOT abstract — it is a concrete polling implementation shared by all adapters, using `setInterval` with a default 1000ms interval and a `lastEntryCount` integer cursor that advances only when new entries are detected, then fires registered callbacks with only the delta entries. This means the watch mechanism is O(n) in total entry count per poll cycle, which could become a performance concern for very long sessions.

Rather than storing each graph entity as a separate LevelDB key (which would enable partial reads and atomic per-entity updates), the entire Graphology in-memory graph is serialized as one JSON blob stored under the key `'graph'`. This blob contains all nodes, edges, and metadata. Writes are deferred using an `isDirty` flag — mutations to the graph set `isDirty = true`, and `_persistGraphToLevel()` is only called when a flush is explicitly triggered. This design optimizes for read-heavy, batch-write workloads but creates a risk of data loss if the process crashes between mutations and the next flush. New developers should be aware that any code path that modifies graph nodes or edges must ensure the flush cycle is triggered, or changes will silently remain only in memory.

- [SemanticAnalysis](./SemanticAnalysis.md) -- [LLM] The SemanticAnalysis pipeline is structured around a coordinator pattern where specialized agents each extend BaseAgent<TInput,TOutput> defined in src/agents/base-agent.ts. This base class implements a strict template-method execute() that sequences six steps in order: process(), calculateConfidence(), detectIssues(), generateRouting(), applyCorrections(), and buildMetadata(). Every agent—CodeGraphAgent, SemanticAnalysisAgent, OntologyClassificationAgent, ContentValidationAgent—inherits this contract and returns a uniform AgentResponse envelope. This design means a new developer adding an agent only needs to implement the domain-specific process() logic; confidence scoring, issue detection, and metadata construction are guaranteed to run in a consistent order regardless of which agent is invoked. The tradeoff is that the template method imposes overhead steps even when an agent's output is trivially simple, and agents cannot short-circuit the sequence without throwing exceptions.
