# DockerizedServices — Technical Reference

## What It Is

DockerizedServices is the containerization layer for the Coding project, implemented primarily through `docker/docker-compose.yml` and `docker/Dockerfile.coding-services`. It packages a suite of infrastructure services — the semantic analysis MCP, constraint monitor, code-graph-rag, Memgraph, and Redis — into a unified deployment unit. Within the container, process orchestration is delegated to `supervisord`, configured via `docker/supervisord.conf`, which manages multiple co-located service processes as a single deployable unit. Three child components implement the runtime logic: ServiceProbe (`lib/utils/service-probe.js`), ServiceStarter (`lib/service-starter.js`), and LLMMockService (`integrations/mcp-server-semantic-analysis/src/mock/llm-mock-service.ts`).

## Architecture and Design

The central architectural decision in DockerizedServices is the **supervisord-within-container** pattern: rather than one process per container (the conventional Docker approach), multiple services are co-located and managed by supervisord inside a single container. This is explicitly reflected in `docker/supervisord.conf` and the `coding-services-internal.png` architecture diagram, which shows the internal service topology. This trade-off prioritizes deployment simplicity and inter-service locality over the isolation benefits of separate containers — a deliberate choice for a tightly coupled coding infrastructure.

![DockerizedServices — Relationship](images/dockerized-services-relationship.png)

Health monitoring is built on a **strict status contract** enforced by ServiceProbe. SPEC R6 mandates that probes never return `'healthy'` — only `'running'`, `'stopped'`, or `'unknown'`. This is an intentional design guard against silent pass-through failures: `'unknown'` signals configuration errors (the probe itself is misconfigured), while `'stopped'` signals network-level failures (the service is unreachable). The asymmetry between these two failure modes gives the health coordinator enough information to distinguish operational problems from infrastructure problems without ever creating a false-positive `'healthy'` state.

Service startup resilience is handled by ServiceStarter via `startServiceWithRetry()`, which implements exponential backoff with SIGTERM→SIGKILL escalation between attempts. The design classifies services as either **required** or **optional**, enabling graceful degradation when non-critical services fail to start. This mirrors the broader Coding project's pattern of agent-agnostic, fault-tolerant operation visible in sibling components like LLMAbstraction and SemanticAnalysis.

## Implementation Details

ServiceProbe (`lib/utils/service-probe.js`) implements two distinct probe mechanisms — HTTP endpoint checks and TCP port checks — allowing each service to be monitored via its most appropriate protocol. The status vocabulary is strictly three-valued by SPEC R6 invariant: configuration errors yield `'unknown'`, network failures yield `'stopped'`, and successful contact yields `'running'`. This vocabulary flows up to the health coordinator, which tracks service liveness without ever treating absence of failure as confirmation of health.

ServiceStarter (`lib/service-starter.js`) wraps process startup in retry-with-exponential-backoff logic. Between retry attempts, unhealthy PIDs are killed using SIGTERM first, falling back to SIGKILL, before the next attempt begins. The required/optional classification determines whether a startup failure is fatal to the deployment or merely degrades it. Two wrapper scripts, `scripts/api-service.js` and `scripts/dashboard-service.js`, wrap specific child processes — the constraint monitor API on port 3031 and the Next.js dashboard on port 3030 respectively — and register them with `ProcessStateManager` (`process-state-manager.js`) for lifecycle tracking.

LLMMockService (`integrations/mcp-server-semantic-analysis/src/mock/llm-mock-service.ts`) handles a container-specific concern: path translation. It prefers `process.env.CODING_ROOT` over the caller-supplied `repositoryPath` when resolving `.data/workflow-progress.json`, enabling container-to-host path mapping without requiring callers to be container-aware. This file also governs LLM mode control, supporting three modes (`mock`/`local`/`public`) with per-agent overrides. Backward compatibility is preserved via a legacy `mockLLM` boolean flag coexisting alongside the newer `llmState` structure in `workflow-progress.json` — the same file and mode system used by the LLMAbstraction sibling component's DMR provider and direct-fetch wrapper.

## Integration Points

DockerizedServices sits directly under the Coding root component and acts as the runtime substrate for several siblings. The ConstraintSystem's constraint monitor API runs on port 3031 and is wrapped by `scripts/api-service.js`; SemanticAnalysis's MCP server is packaged within the container. KnowledgeManagement's Memgraph and Redis dependencies are also included in the compose deployment, making DockerizedServices a hard runtime dependency for graph query capabilities.

The LLMMockService's path translation via `CODING_ROOT` is the primary seam between container and host filesystem, and it feeds directly into the `workflow-progress.json` state file shared with LLMAbstraction. ProcessStateManager is the internal registry through which `api-service.js` and `dashboard-service.js` expose process lifecycle state to the broader system. ServiceProbe's status vocabulary flows to a health coordinator (external to this component) that consumes `'running'`/`'stopped'`/`'unknown'` for liveness decisions.

## Usage Guidelines

**Never expect `'healthy'` from ServiceProbe** — the SPEC R6 invariant is non-negotiable. Any code consuming probe results must handle exactly three values: `'running'`, `'stopped'`, `'unknown'`. Treating absence of `'stopped'` as health is a design violation.

When adding new services to the container, they must be registered in both `docker/supervisord.conf` (for process management) and given a probe configuration compatible with ServiceProbe's HTTP or TCP mechanisms. Services should be explicitly classified as required or optional in ServiceStarter configuration — defaulting to required is the safe choice, but optional classification should be used for services where graceful degradation is acceptable.

Path-sensitive code running inside the container should prefer `process.env.CODING_ROOT` for filesystem resolution rather than relying on passed-in paths, following the pattern established in LLMMockService. For LLM mode configuration, new agents should use the `llmState` structure in `workflow-progress.json` rather than the legacy `mockLLM` boolean, though both must remain supported for backward compatibility as long as the legacy flag is present.
