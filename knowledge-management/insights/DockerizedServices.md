**Type:** Component

[LLM] The `scripts/health-coordinator.js` operates on a 5-second polling tick cycle, consuming the normalized `{status, latency_ms, error}` envelopes from both `probeHttpHealth()` and `probeTcpPort()` across all registered services. This coordinator sits at the intersection of the PSM registry (which knows which services exist) and the probe utilities (which know how to check them), making it the single authoritative source of aggregate system health at any given polling interval. The 5-second interval represents a design tradeoff: short enough to detect service failures <USER_ID_REDACTED> for operational awareness, but long enough to avoid overwhelming services that may already be under stress. The fact that the coordinator uses the PSM registry rather than a static list of service endpoints means that dynamically registered services (those registered via `psm.registerService()` at runtime by wrapper scripts) are automatically included in health polling without requiring coordinator restarts. This dynamic discovery pattern is important for the DockerizedServices component because the set of running services can change — for example, during staged startup where Memgraph and Redis must be running before the semantic analysis MCP server starts — and the coordinator will naturally begin polling a new service as soon as its wrapper script registers it, while also ceasing to poll it once `psm.unregisterService()` is called on process exit. Developers extending the system with a new service should be aware that the health coordinator will not probe it until the corresponding PSM registration occurs, meaning there is a window between container/process start and PSM registration during which health state is unknown.

# DockerizedServices — Deep Technical Insight

## What It Is

DockerizedServices is the component within the Coding project responsible for spawning, supervising, and health-monitoring the long-running processes that make up the project's runtime surface — primarily the constraint monitor Express API and the Next.js dashboard. Its implementation is distributed across a small set of script and library files: the wrapper scripts `scripts/api-service.js` and `scripts/dashboard-service.js`, the orchestration loop in `scripts/health-coordinator.js`, the retry/lifecycle library `lib/service-starter.js`, the probe utility `lib/utils/service-probe.js`, and the LLM mode helper in `integrations/mcp-server-semantic-analysis/src/mock/llm-mock-service.ts`. The component sits as a sibling of LiveLoggingSystem, LLMAbstraction, KnowledgeManagement, CodingPatterns, ConstraintSystem, and SemanticAnalysis under the root Coding hierarchy.

The component packages together seven child entities that each occupy a distinct responsibility: ServiceProbe (status probes), ConstraintAPIWrapper (API process wrapping), DashboardWrapper (dashboard process wrapping), LLMMockService (LLM mode resolution inside the containerized runtime), ProcessStateManager (the in-process registry), ServiceStarter (retry-aware spawn logic), and HealthCoordinator (the 5-second polling loop). Together they form a coherent local-orchestration layer that lets the rest of the system reason about "services" abstractly rather than about OS-level PIDs.

## Architecture and Design

The dominant architectural idea is a **deliberate decoupling between service identity and process identity**, mediated by the `ProcessStateManager` (PSM) singleton. Each wrapper script — `scripts/api-service.js` and `scripts/dashboard-service.js` — follows an identical structural pattern: spawn a child process via Node's `child_process` module, register the resulting process handle with the PSM via `psm.registerService()`, forward `SIGTERM` and `SIGINT` to the child, and call `psm.unregisterService()` in the exit handler. Because PSM is a singleton (consistent with the project-wide singleton guard pattern documented under CodingPatterns), every consumer — including `scripts/health-coordinator.js` — shares one authoritative registry instance without explicit reference passing. A restart that replaces the underlying child process therefore does not require any consumer to learn the new PID; only the wrapper script touches the registry.

A second architectural pillar is the **layered timeout architecture** in `lib/service-starter.js`'s `startServiceWithRetry()`. Using `Promise.race()`, the function establishes a 30-second ceiling for an overall startup attempt while independently bounding each in-window probe call at 10 seconds. This means a single slow probe cannot consume the entire startup budget. Between retries, the starter escalates from `SIGTERM` to `SIGKILL` against any still-running unhealthy process, which is essential to avoid port-binding conflicts when the next spawn attempt occurs. The exponential backoff between retries ensures that fast-failing services (e.g., a missing environment variable) do not spin tightly.

A third pillar is the **strict status vocabulary** enforced by `lib/utils/service-probe.js`, codified by SPEC R6. The functions `probeHttpHealth()` and `probeTcpPort()` are explicitly forbidden from returning `'healthy'`; the only valid statuses are `'running'`, `'stopped'`, and `'unknown'`, where `'unknown'` is reserved exclusively for configuration errors (malformed URL, invalid port) and transient network failures map to `'stopped'`. Every probe returns the normalized envelope `{status, latency_ms, error}`, giving the health coordinator a consistent shape on every 5-second tick. The complementarity of the two probe functions is also architecturally significant: HTTP-speaking services (API, dashboard) are probed against `/health` endpoints, while raw TCP services (e.g., Memgraph's Bolt protocol) are probed by opening a `net.Socket` directly.

## Implementation Details

The wrapper scripts (`scripts/api-service.js` for ConstraintAPIWrapper and `scripts/dashboard-service.js` for DashboardWrapper) are intentionally boilerplate-heavy. Each one owns process lifecycle signals while delegating retry policy entirely to `lib/service-starter.js`. The dashboard wrapper in particular hardcodes two environment variables before spawning the Next.js process: `NODE_ENV='development'` and `NEXT_PUBLIC_API_BASE_URL='http://localhost:3031'`. The `development` setting means the dashboard permanently runs the Next.js dev server — with hot module replacement, verbose errors, and no build optimization — even inside Docker. This treats the dashboard as a developer-facing tool rather than a production UI and accepts the resulting memory/CPU overhead. The base URL hardcoding co-locates the dashboard with the API on `localhost:3031`, making the two scripts implicitly coupled through a port that is not enforced by any shared constant.

`ServiceStarter` (`lib/service-starter.js`) coordinates spawning and retry. Its JSDoc explicitly distinguishes `isTcpPortListening()` from `isPortListening()` to make clear which check is appropriate per service type. The escalation sequence (SIGTERM → wait → SIGKILL → retry) is the mechanism by which port conflicts are avoided on subsequent attempts, with the explicit trade-off that any in-flight requests to a briefly-running-but-unhealthy service instance are terminated. Because total worst-case time scales with retry count × backoff ceiling — not the per-attempt timeout alone — any external orchestration grace period (e.g., `docker-compose.yml` health check intervals or a Kubernetes liveness probe `initialDelaySeconds`) must be sized against the cumulative envelope, not the 30-second per-attempt ceiling.

`HealthCoordinator` (`scripts/health-coordinator.js`) sits at the intersection of the PSM registry and the probe utilities. On each 5-second tick it iterates the PSM registry and dispatches the appropriate probe per service type, consuming the normalized `{status, latency_ms, error}` envelopes. Because it reads from the live PSM registry rather than a static list, services registered dynamically at runtime — for example during staged startup where Memgraph and Redis must come up before the semantic analysis MCP server — are automatically included in polling without coordinator restarts. Conversely, services that call `psm.unregisterService()` on exit are dropped from polling immediately. There is, however, a small window between process start and PSM registration during which a service's health state is unknown.

`LLMMockService` (`integrations/mcp-server-semantic-analysis/src/mock/llm-mock-service.ts`) participates in DockerizedServices because of how it resolves configuration paths across Docker-internal and host-machine execution. Its `getLLMMode()` implements a three-level priority chain: (1) per-agent override from `llmState.perAgentOverrides`, (2) `llmState.globalMode`, and (3) the legacy `progress.mockLLM` boolean read from `.data/workflow-progress.json`. The progress file path is resolved using the `CODING_ROOT` environment variable (Docker-internal) with a fallback to `repositoryPath`, allowing the same TypeScript to work in either environment without branching. The legacy `mockLLM` flag is retained for backward compatibility — older progress files continue to work without migration. This mirrors the same three-tier resolution scheme described in the sibling LLMAbstraction component.

## Integration Points

DockerizedServices's primary internal integration seam is the PSM registry. The wrapper scripts populate it via `registerService()`/`unregisterService()`, and HealthCoordinator consumes it. ServiceProbe (`lib/utils/service-probe.js`) is the utility consumed by HealthCoordinator, establishing a clear utility-to-orchestrator dependency direction. ServiceStarter is consumed by the wrapper scripts to perform retry-aware startup, while the wrapper scripts themselves own signal handling — a clean separation that keeps retry policy out of the signal layer and lifecycle signals out of the retry layer.

Externally, the component integrates with the sibling LLMAbstraction component through LLMMockService's shared `getLLMMode()` mode-resolution model — the two components implement the same three-tier hierarchy, with DockerizedServices contributing the path-resolution strategy that makes the same TypeScript run under both host and container layouts. It also touches the ConstraintSystem indirectly because ConstraintAPIWrapper is what spawns the constraint monitor Express API process. The dashboard's hardcoded `NEXT_PUBLIC_API_BASE_URL='http://localhost:3031'` is the contract point between DashboardWrapper and ConstraintAPIWrapper, and the matching port is the implicit interface between them.

Outward, the normalized `{status, latency_ms, error}` envelope is the contract any observability tooling can consume — latency telemetry is always present even when a service is unreachable, which is a deliberate choice to keep observability data uniform. Container orchestration (docker-compose, Kubernetes) integrates at the timing boundary: external health-check grace periods must be sized to the cumulative ServiceStarter worst case, not its per-attempt timeout.

## Usage Guidelines

When adding a new containerized service, expect to **create a new wrapper script** that replicates the existing boilerplate from `api-service.js` and `dashboard-service.js` — spawn via `child_process`, `psm.registerService()`, wire `SIGTERM`/`SIGINT` forwarding, call `psm.unregisterService()` on exit. This duplication is a known maintenance concern as service count grows, but it is the established pattern and should be followed for consistency until centralized. Do **not** instantiate `ProcessStateManager` directly — always go through the project's singleton accessor in line with the guard pattern formalized in `docs/puml/psm-singleton-pattern.puml`.

When writing or extending probe code, never return `'healthy'` — the SPEC R6 status vocabulary is exactly `'running'`, `'stopped'`, `'unknown'`. Reserve `'unknown'` strictly for configuration errors (malformed URL, invalid port). Map transient network failures (connection refused, timeout) to `'stopped'`. Always emit the full `{status, latency_ms, error}` envelope so HealthCoordinator and observability consumers receive uniform data. Choose `probeHttpHealth()` for HTTP services with a `/health` endpoint and `probeTcpPort()` for raw-socket services such as Memgraph's Bolt port.

When tuning ServiceStarter behavior, remember that the **30-second outer ceiling and 10-second per-probe inner ceiling are independent** — increasing the per-probe timeout above the outer ceiling has no effect. The cumulative worst case across retries with exponential backoff must be considered when configuring external orchestrator grace periods. Be aware that the SIGTERM→SIGKILL escalation before each retry will terminate any in-flight requests against a briefly-running unhealthy instance; this is intentional but worth knowing when debugging client-side errors during startup churn.

When working with LLMMockService inside Docker, ensure `CODING_ROOT` is set so `.data/workflow-progress.json` resolves correctly; otherwise it will fall back to `repositoryPath`. Be aware that a **per-agent override silently wins over a global mock flag**, which is a frequent source of debugging confusion. Finally, when modifying the dashboard or API ports, recognize that the hardcoded `NEXT_PUBLIC_API_BASE_URL='http://localhost:3031'` requires changes in at least two files plus a Next.js rebuild for production, and that the development-mode Next.js server is the only supported runtime mode today — there is no `next build && next start` path in the current architecture.

### 1. Architectural Patterns Identified
- **Singleton registry** (PSM) decoupling service identity from process identity.
- **Wrapper-script boilerplate pattern** for every supervised service.
- **Layered timeout via `Promise.race()`** separating startup budget from per-probe budget.
- **SIGTERM→SIGKILL escalation** as a pre-retry cleanup step to avoid port reuse conflicts.
- **Normalized probe envelope** (`{status, latency_ms, error}`) as a uniform consumer contract.
- **Strict status vocabulary** governed by SPEC R6.
- **Tick-based polling loop** (5 s) for aggregate health.
- **Three-tier priority chain** for LLM mode resolution shared with sibling LLMAbstraction.
- **Dual-environment path resolution** (`CODING_ROOT` with `repositoryPath` fallback) for Docker/host parity.

### 2. Design Decisions and Trade-offs
- **PSM indirection** simplifies restarts and consumer code, at the cost of a small unknown-state window between process start and registration.
- **Wrapper-per-service duplication** keeps each service's lifecycle explicit and modifiable in isolation, but is a known scaling concern.
- **Killing unhealthy processes before retry** avoids port conflicts but terminates in-flight requests.
- **Hardcoded `NODE_ENV='development'` for dashboard** prioritizes developer ergonomics (HMR, verbose errors) over production performance.
- **Hardcoded `NEXT_PUBLIC_API_BASE_URL='http://localhost:3031'`** prevents distributed deployment but eliminates a class of configuration mistakes for the supported single-host case.
- **Legacy `mockLLM` boolean retained** trades a small code surface for backward compatibility with older progress files.

### 3. System Structure Insights
The component's structure is a clean three-layer arrangement: **wrappers** (ConstraintAPIWrapper, DashboardWrapper) own process and signal lifecycle, **library logic** (ServiceStarter, ServiceProbe) owns retry and probing, and the **registry+coordinator pair** (ProcessStateManager, HealthCoordinator) owns aggregate state and polling. LLMMockService is structurally peripheral but integrated because its path-resolution behavior is what makes the same code execute identically inside and outside Docker.

### 4. Scalability Considerations
The 5-second poll cycle and 30-second per-attempt startup ceiling are the dominant scalability levers; both scale linearly with service count. The wrapper-per-service pattern means new services impose O(1) boilerplate cost per service, which becomes friction as the service count grows. Distributed deployment is not currently supported because of the hardcoded `localhost` API base URL in the dashboard wrapper — scaling beyond a single host requires source-level changes and a Next.js rebuild (since `NEXT_PUBLIC_*` variables are inlined at build time). The dashboard's permanent dev-mode also imposes a memory/CPU footprint heavier than a production build.

### 5. Maintainability Assessment
Maintainability is high in the small — each script and library file has narrow, well-bounded responsibilities, and the SPEC R6 status vocabulary plus normalized envelope eliminate a whole class of consumer bugs. Maintainability concerns at the larger scale center on (a) the duplicated wrapper-script boilerplate, which will need centralization if service count keeps growing, (b) the implicit port coupling between `api-service.js` and `dashboard-service.js` that is not enforced by a shared constant, and (c) the silent precedence of per-agent LLM overrides over global mode, which can mislead debugging. The clean separation between signal handling (wrappers) and retry policy (ServiceStarter), and the PSM-mediated dynamic discovery used by HealthCoordinator, are strong points that make extension straightforward when the boilerplate cost is accepted.
