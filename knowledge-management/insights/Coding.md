## Usage Guidelines  

1. **Prefer the Facade for LLM Calls** – All code that needs language‑model interaction should go through `LLMService`. Directly instantiating provider classes bypasses caching, circuit‑breaking, and budget checks, increasing the risk of throttling or unexpected failures.  
2. **Register Observers Early** – When extending the **ConstraintSystem**, register your observer callbacks during application bootstrap (before the first log classification) to guarantee that no constraint events are missed.  
3. **Leverage the GraphDatabaseAdapter** – Persisting new entities should always use the adapter rather than accessing Graphology or LevelDB directly. This keeps the JSON export sync functional and isolates storage‑specific changes.  
4. **Container‑Oriented Deployment** – Deploy each L1 component as its own Docker service when scaling. Respect the retry‑with‑backoff semantics of `ServiceStarter`; avoid manual restarts that could interfere with the back‑off schedule.  
5. **Async‑First Initialization** – Follow the pattern used by the `Trajectory` component: make any I/O‑heavy setup return a Promise and await it at the top level. This prevents blocking the event loop and allows other services to start in parallel.  
6. **Configuration Management** – Keep environment‑specific values (e.g., graph database URLs, LLM provider credentials) out of source code and in the JSON/YAML config files referenced by each component. This aligns with the design’s emphasis on loose coupling and easier deployment across stages.

- **Facade vs. Direct Provider Calls** – Centralising LLM interactions simplifies provider swapping and adds safety (caching, budget checks) but introduces a single point of latency if the façade becomes a bottleneck.  
- **Microservices containerisation** – Provides isolation and independent scaling, yet adds operational complexity (networking, orchestration).  
- **GraphDatabaseAdapter abstraction** – Enables storage‑engine flexibility and automatic export, at the cost of an additional indirection layer that must be maintained.  
- **Observer pattern for constraints** – Decouples producers and consumers, improving extensibility, but requires careful management of subscription lifecycles to avoid memory leaks.
