# LLMAbstraction

**Type:** Component

LLMAbstraction is a multi-layered abstraction over LLM providers that enables provider-agnostic model calls across public APIs (Anthropic, OpenAI, Groq), a local Docker Model Runner (DMR), and a mock mode for testing. The system routes requests through a rapid-llm-proxy daemon (port 12435) that handles provider selection, tier-based routing, and per-call telemetry attribution. Mode selection is dynamic and per-agent, stored in workflow-progress.json, supporting global and per-agent overrides across three modes: 'mock', 'local' (DMR), and 'public' (cloud APIs).

## What It Is

LLMAbstraction is the provider-agnostic LLM invocation layer within the Coding project, implemented primarily across `llm-mock-service.ts`, `llm-with-process.ts`, and `dmr-provider.ts`. It sits between agent call-sites (such as those in `wave-controller.ts`) and the concrete inference backends: public cloud APIs (Anthropic, OpenAI, Groq), a local Docker Model Runner (DMR) accessible at `localhost:${DMR_PORT}/engines/v1`, and a mock mode for deterministic testing. All production traffic is routed through the `rapid-llm-proxy` daemon on port 12435, while mock and DMR paths bypass or substitute that daemon. Mode selection is dynamic, per-agent, and persisted in `workflow-progress.json`, making it possible to run different agents against different backends within the same workflow execution.

## Architecture and Design

![LLMAbstraction — Architecture](images/llmabstraction-architecture.png)

The central architectural decision in LLMAbstraction is the separation of *mode selection*, *provider routing*, and *proxy communication* into distinct layers, each with its own priority chain. This avoids a single monolithic switch and allows each concern to evolve independently.

**Mode selection** is handled by `getLLMMode()` in `llm-mock-service.ts`, which implements a three-tier priority chain: per-agent override → `llmState.globalMode` → legacy `mockLLM` boolean. The inclusion of the legacy boolean flag is a deliberate backward-compatibility shim, ensuring older call-sites that predate the structured `llmState` approach continue to function without modification. The effective mode is stored in `workflow-progress.json`, which `llm-mock-service.ts` locates using `process.env.CODING_ROOT` when available, normalizing the path difference between Docker execution (where `CODING_ROOT=/coding`) and host execution.

**Proxy URL resolution** is delegated to the child component ProxyURLResolver, implemented in `llm-with-process.ts`. ProxyURLResolver applies a four-step fallback: `RAPID_LLM_PROXY_URL` → `LLM_CLI_PROXY_URL` → `LLM_PROXY_URL` → `http://<CONNECTION_STRING_REDACTED> SDK's own `cli-provider-base.ts`, meaning LLMAbstraction and the SDK resolve the proxy endpoint consistently even when configured independently. The explicit mirroring of SDK convention here is a maintainability decision: operators configuring the proxy URL for the SDK do not need separate configuration for LLMAbstraction.

**The DMR provider** (`dmr-provider.ts`) instantiates an OpenAI-compatible client pointed at the Docker Desktop Model Runner's local endpoint, sourcing `DMR_HOST` and `DMR_PORT` from environment variables. Availability is checked via a health-check interval with cached results, avoiding per-call latency from health probing. This pattern parallels how DockerizedServices handles non-HTTP service verification using `probeTcpPort()` — both components treat local service availability as a polling concern rather than a per-request concern.

## Implementation Details

The most architecturally significant file is `llm-with-process.ts`, which addresses two separate problems: the `process` field injection and metrics continuity.

**Process field injection** exists because the `@rapid/llm-proxy` SDK's `LLMCompletionRequest` type does not expose a `process` field, yet the `rapid-llm-proxy` daemon's `/api/complete` endpoint accepts it for per-step token attribution into `.data/llm-proxy/token-usage.db`. `llmWithProcessComplete()` works around the type gap by constructing the request payload manually, adding the `process` field outside the SDK's type system. This is a deliberate point of SDK divergence, accepted because the attribution capability it enables (per-step telemetry in the SQLite token-usage database) is valuable enough to justify the maintenance cost of keeping the workaround in sync with SDK changes.

**Metrics continuity** is maintained through an optional `MetricsTrackerLike` duck-typed interface accepted by `llm-with-process.ts`. Wave-agent call-sites can pass `llmService.getMetricsTracker()` to this interface, keeping `wave-controller.ts`'s existing instrumentation — specifically `getDetailedCalls()` and `getLLMMetrics()` — functional without requiring SDK changes. The duck-typed approach is intentional: it avoids a hard dependency on the SDK's metrics types while still providing type safety at the call-site.

**Mock path normalization** in `llm-mock-service.ts` handles the structural difference between Docker and host environments by checking `process.env.CODING_ROOT` before resolving the path to `workflow-progress.json`. Inside Docker, `CODING_ROOT=/coding` remaps the path; on the host, the variable is absent and normal path resolution applies. This is a minimal, targeted fix rather than a generalized environment-detection mechanism, which is appropriate given that `workflow-progress.json` is the only file whose location is affected by this boundary.

## Integration Points

LLMAbstraction sits beneath the SemanticAnalysis pipeline's `BaseAgent`-derived agents and the wave execution layer in `wave-controller.ts`, both of which consume LLM calls through the abstraction boundary rather than calling providers directly. The `getMetricsTracker()` / `MetricsTrackerLike` bridge in `llm-with-process.ts` is the primary integration surface with `wave-controller.ts`, preserving telemetry continuity across the abstraction layer.

The `rapid-llm-proxy` daemon on port 12435 is an external process dependency. LLMAbstraction assumes the daemon is running and reachable; it does not start or monitor the daemon itself. This is consistent with how DockerizedServices treats service health — availability checking is a separate concern from service consumption.

The `workflow-progress.json` file is a shared state artifact also read by other Coding components for agent orchestration state. LLMAbstraction reads it narrowly for mode overrides; it does not write to it. Developers modifying the schema of `workflow-progress.json` must account for `getLLMMode()`'s reading of `llmState.globalMode` and per-agent override fields.

The DMR provider's dependency on Docker Desktop's Model Runner means it shares an operational dependency surface with DockerizedServices — both assume Docker Desktop is present and healthy for their local-inference and containerized-service paths respectively.

## Usage Guidelines

**Mode configuration** should use the structured `llmState.globalMode` field in `workflow-progress.json` rather than the legacy `mockLLM` boolean. The boolean is retained for backward compatibility but represents technical debt; new tooling should not introduce new consumers of it.

**Per-agent overrides** are the correct mechanism for running a single agent against a different backend (e.g., running a fast-feedback agent in mock mode while a reasoning agent hits public APIs). These overrides are stored in `workflow-progress.json` alongside the global mode, and `getLLMMode()` will always prefer them.

**Token attribution** requires passing a `process` identifier through `llmWithProcessComplete()`. Call-sites that use the standard SDK path without this wrapper will produce telemetry records in `.data/llm-proxy/token-usage.db` that lack step-level attribution, making post-hoc cost analysis less granular. Prefer `llmWithProcessComplete()` for any agent step where per-step cost tracking matters.

**Adding a new provider** requires implementing the OpenAI-compatible client interface (as `dmr-provider.ts` demonstrates) and registering it in the mode/provider routing layer. The DMR provider's pattern — environment-variable-sourced endpoint, cached health check, OpenAI-compatible client — is the established template. The ProxyURLResolver's four-step chain does not need modification for new providers unless the new provider introduces its own proxy indirection.

**Metrics tracking** at new call-sites should pass `llmService.getMetricsTracker()` to `llm-with-process.ts`'s `MetricsTrackerLike` parameter. Omitting it silently disables `wave-controller.ts` tracer instrumentation for that call-site without causing errors, which can produce incomplete `getDetailedCalls()` output in ways that are non-obvious to diagnose.

### Siblings
- [LiveLoggingSystem](./LiveLoggingSystem.md) -- [LLM] The `TranscriptAdapter` abstract base class in `lib/agent-api/transcript-api.js` defines a pluggable adapter contract that decouples the LSL pipeline from specific agent implementations. The five abstract methods — `getAgentType()`, `getTranscriptDirectory()`, `readTranscripts()`, `convertToLSL()`, and `getCurrentSession()` — establish a clear interface that concrete adapters (e.g., for Claude Code and Copilot CLI) must implement. This pattern means the core LSL infrastructure never directly references agent-specific transcript formats or filesystem layouts. A new developer adding support for a third agent (say, Cursor or Aider) would subclass `TranscriptAdapter` and implement only these five methods without touching the converter, file manager, or validation layers. The `watchTranscripts()` method is notably NOT abstract — it is a concrete polling implementation shared by all adapters, using `setInterval` with a default 1000ms interval and a `lastEntryCount` integer cursor that advances only when new entries are detected, then fires registered callbacks with only the delta entries. This means the watch mechanism is O(n) in total entry count per poll cycle, which could become a performance concern for very long sessions.
- [DockerizedServices](./DockerizedServices.md) -- [LLM] The DockerizedServices component uses a dual-probe health checking architecture implemented in lib/utils/service-probe.js that strictly separates HTTP-based and TCP-based service verification. probeHttpHealth() issues HTTP/HTTPS requests and maps 2xx/3xx response codes to the 'running' state, while probeTcpPort() opens a raw net.Socket connection to verify port reachability for non-HTTP protocols like Memgraph's Bolt protocol. A critical architectural invariant (documented as SPEC R6) enforces that neither probe ever returns the string 'healthy'—only 'running', 'stopped', or 'unknown' are valid return values. This distinction matters because health-coordinator.js in scripts/health-coordinator.js consumes these probes on a 5-second polling interval and must be able to uniformly handle all service types (Next.js dashboard, Node.js API, Memgraph, Redis) without special-casing the protocol. New developers adding services must use probeTcpPort() for any non-HTTP service and must not introduce a fourth status string, or the health-coordinator's state machine will behave incorrectly.
- [KnowledgeManagement](./KnowledgeManagement.md) -- [LLM] The primary persistence mechanism in KnowledgeManagement is a single-key LevelDB strategy implemented in `src/knowledge-management/GraphDatabaseService.js`. Rather than storing each graph entity as a separate LevelDB key (which would enable partial reads and atomic per-entity updates), the entire Graphology in-memory graph is serialized as one JSON blob stored under the key `'graph'`. This blob contains all nodes, edges, and metadata. Writes are deferred using an `isDirty` flag — mutations to the graph set `isDirty = true`, and `_persistGraphToLevel()` is only called when a flush is explicitly triggered. This design optimizes for read-heavy, batch-write workloads but creates a risk of data loss if the process crashes between mutations and the next flush. New developers should be aware that any code path that modifies graph nodes or edges must ensure the flush cycle is triggered, or changes will silently remain only in memory.
- [CodingPatterns](./CodingPatterns.md) -- [LLM] The MCP (Model Context Protocol) server pattern is applied uniformly across all three major integrations: mcp-constraint-monitor, mcp-server-semantic-analysis, and code-graph-rag. Each integration follows a consistent directory layout with a README.md describing the tool interface, a docs/ subdirectory containing architectural detail, and configuration files that declare the tool's capabilities to AI agent hosts. This pattern means new integrations can be scaffolded by copying this structure and registering new tool definitions. The mcp-server-semantic-analysis integration further documents this in docs/api/README.md and docs/architecture/README.md, separating API surface from internal architecture. Developers adding a new MCP tool should expect to create at minimum: a README with tool description, an API reference doc, and a configuration file declaring tool schemas. The consistency of this pattern across integrations suggests it is an enforced project convention rather than an organic coincidence.
- [ConstraintSystem](./ConstraintSystem.md) -- The ConstraintSystem is a monitoring and enforcement layer that validates code actions and file operations against configured rules during Claude Code sessions. It operates through a hook-based architecture where constraint checks are triggered at key lifecycle points (pre-tool, post-tool, etc.) and violations are captured, persisted, and surfaced to dashboards. The system integrates with Claude Code's native hook mechanism via configuration files at user-level (~/.coding-tools/hooks.json) and project-level (.coding/hooks.json), with project config overriding user config.
- [SemanticAnalysis](./SemanticAnalysis.md) -- [LLM] The SemanticAnalysis pipeline is structured around a coordinator pattern where specialized agents each extend BaseAgent<TInput,TOutput> defined in src/agents/base-agent.ts. This base class implements a strict template-method execute() that sequences six steps in order: process(), calculateConfidence(), detectIssues(), generateRouting(), applyCorrections(), and buildMetadata(). Every agent—CodeGraphAgent, SemanticAnalysisAgent, OntologyClassificationAgent, ContentValidationAgent—inherits this contract and returns a uniform AgentResponse envelope. This design means a new developer adding an agent only needs to implement the domain-specific process() logic; confidence scoring, issue detection, and metadata construction are guaranteed to run in a consistent order regardless of which agent is invoked. The tradeoff is that the template method imposes overhead steps even when an agent's output is trivially simple, and agents cannot short-circuit the sequence without throwing exceptions.
