# KnowledgeManagement — Technical Insight Document

## What It Is

KnowledgeManagement is the persistence and knowledge graph layer within the Coding project, responsible for extracting, classifying, storing, and exporting semantic entities derived from codebases and agent sessions. Its core implementation spans `storage/graph-database-adapter.ts` (dual-mode routing), `agents/persistence-agent.ts` (entity classification and storage), `src/knowledge-management/GraphKnowledgeExporter.js` (JSON export side-effects), and migration scripts under `src/migration/`. It maintains a Graphology in-memory graph backed by LevelDB, with entities classified against a strict ontology into canonical types (System, Project, Pattern). As a child of the Coding root, it sits alongside siblings like SemanticAnalysis (which feeds it extracted entities), LiveLoggingSystem (which captures raw sessions), and ConstraintSystem.

## Architecture and Design

The central architectural decision is the **dual-mode routing strategy** in GraphDatabaseAdapter. At initialization, `VkbApiClient.isServerAvailable()` is called exactly once to select between live mode (HTTP API) and direct mode (in-process LevelDB access). This is not a dynamic failover — it's a one-time decision that persists for the adapter's lifetime. The design exists to solve LevelDB's single-writer lock constraint: when the VKB HTTP server is running, all writes serialize through it; direct mode is only safe for single-process scenarios.

![KnowledgeManagement — Architecture](images/knowledge-management-architecture.png)

A strict **ontology classification gate** in PersistenceAgent prevents unclassified entities from entering the graph. `storeEntity()` throws immediately if `entityType` is missing or equals `'Unclassified'` — no fallbacks. Classification uses `OntologyRegistry` from `@fwornle/km-core` with a 5-minute in-process cache (`CLASSIFICATION_CACHE_TTL_MS`) to reduce LLM calls. This is a deliberate <USER_ID_REDACTED>-over-availability trade-off: the system prefers failure over data degradation.

Two parallel graph representations exist: the Graphology/LevelDB entity graph (semantic/ontological: System, Project, Pattern) and the Code Graph RAG graph (structural: CONTAINS_FILE, DEFINES_METHOD, DEPENDS_ON_EXTERNAL, etc., backed by Memgraph via `integrations/code-graph-rag/`). These are independent systems serving different query needs.

## Implementation Details

**Entity Model:** `SharedMemoryEntity` in `agents/persistence-agent.ts` carries bi-temporal staleness fields inspired by the Graphiti framework: `invalidating_commits`, `staleness_score`, `staleness_check_at`, and `staleness_method`. This makes the knowledge graph git-aware — when commits touch source files, associated entities get their staleness incremented rather than deleted, preserving history while signaling re-extraction need. Null `staleness_check_at` should be treated as maximally stale by consumers.

**Export Side-Effects:** GraphKnowledgeExporter listens for `entity:stored` events on the graph database service. In direct mode, the adapter attaches this exporter; in live mode, the VKB server owns it. This is a hidden contract — bypassing the adapter to call `GraphDatabaseService` methods directly means no JSON exports are written, leaving downstream consumers (RAG indexers, other agents) reading stale data.

**Migrations:** `migrate-leveldb-to-kmcore.mjs` enforces a 5% error budget (>5% failures = non-zero exit for CI). It supports `--dry-run` and emits per-domain JSON exports. A separate `migrate-graph-db-entity-types.js` remaps legacy types via `ENTITY_TYPE_MAPPING` to the three canonical types, calling `dbService._persistGraphToLevel()` directly — intentionally bypassing the classification gate. Post-migration verification for stray `Unclassified` nodes is required.

## Integration Points

Child components have well-defined roles: ManualLearning and OnlineLearning are write paths that must route through GraphDatabaseAdapter (and therefore respect the dual-mode routing). CheckpointManager (`src/utils/checkpoint-manager.ts`) stores commit-hash markers so OnlineLearning skips already-processed git history. VkbApiClient's startup availability check is the single point that determines the entire system's write serialization strategy.

The Code Graph RAG integration exposes separate SSE/HTTP ports (`CODE_GRAPH_RAG_SSE_PORT`, `CODE_GRAPH_RAG_PORT`) and <USER_ID_REDACTED> Memgraph for structural relationships, completely independent of the LevelDB entity store. Developers must choose the correct graph layer: structural/containment <USER_ID_REDACTED> → Code Graph RAG; semantic/ontological <USER_ID_REDACTED> → KnowledgeManagement.

SemanticAnalysis (sibling) is an upstream producer that extracts entities which flow into this component's persistence pipeline. The staleness infrastructure connects back to git history, creating a feedback loop where new commits can invalidate previously extracted knowledge.

## Usage Guidelines

**Always go through GraphDatabaseAdapter** for writes. Direct `GraphDatabaseService` calls skip the export side-effect and risk LevelDB lock collisions. Ensure entities pass through `OntologyRegistry` classification before calling `storeEntity()` — the method will throw, not degrade gracefully.

**Understand the dual-mode implications:** if the VKB server is unavailable at adapter startup, the adapter locks into direct mode permanently. Restarting the adapter is required to switch modes. In multi-process deployments, the VKB server must be running before adapters initialize.

**Staleness handling:** treat null `staleness_check_at` as maximally stale. Use `staleness_score` for RAG retrieval ranking — higher scores indicate entities needing re-extraction. The staleness data is persisted in LevelDB and included in JSON exports automatically.

**Migration scripts bypass classification gates.** After running `migrate-graph-db-entity-types.js`, audit the graph for `Unclassified` nodes. The 5% error budget in the km-core migration provides a CI-friendly safety net.

- [CodingPatterns](./CodingPatterns.md) -- [LLM] Agent Lazy-Initialization Pattern: Across the Coding project's agent implementations, a consistent lazy-initialization idiom is applied where LLM client setup is deferred until actual execution rather than performed at construction time. The motivation is to avoid paying the cost of LLM connection setup (which may involve network calls, credential validation, and model loading) when an agent object is instantiated—particularly important in systems where many agent types are registered but only a subset are invoked per workflow. A new developer working on an agent should expect a two-phase lifecycle: a lightweight constructor that stores configuration references, followed by an initialize() or setup() method (or equivalent lazy property) that establishes the actual LLM connection on first use. Violating this convention by eagerly connecting in the constructor would break the startup performance characteristics that the rest of the system assumes.
