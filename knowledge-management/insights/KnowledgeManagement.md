# KnowledgeManagement

**Type:** Component

[LLM] The `CodeGraphAgent` at `integrations/mcp-server-semantic-analysis/src/agents/code-graph-agent.ts` represents a separate, complementary knowledge store that runs alongside the main Graphology/LevelDB graph. It connects to a Memgraph instance (default port 7687) which holds AST-based code entity graphs — a fundamentally different representation from the higher-level System/Project/Pattern entities in the main graph. Before attempting any indexing operation, `CodeGraphAgent` performs a TCP connectivity check against the Memgraph port using a timeout sourced from `loadAgentTuningConfig().code_graph.memgraph_check_timeout_ms`. This pre-flight check means that if Memgraph is unavailable, the agent fails fast rather than timing out mid-operation. New developers should note that the Coding project therefore has two concurrent graph databases with different schemas, different query interfaces (Graphology API vs. Cypher/Bolt for Memgraph), and different entity granularities — the main LevelDB graph for project-level knowledge and Memgraph for code-structure-level knowledge.

# KnowledgeManagement — Technical Reference

## What It Is

KnowledgeManagement is a component within the Coding project responsible for storing, migrating, classifying, and auditing project-level knowledge entities. Its primary implementation lives in `src/knowledge-management/GraphDatabaseService.js`, which manages an in-memory Graphology graph backed by LevelDB persistence. The component owns a canonical taxonomy of three entity types (`System`, `Project`, `Pattern`), a multi-stage migration pipeline for evolving legacy data formats, and a structured audit trail via the UKB trace system at `src/utils/ukb-trace-report.ts`.

KnowledgeManagement sits as a child of the Coding root and contains four sub-components: GraphDatabaseService (the persistence engine), KmCoreStore (the canonical entity store with UUIDv7-keyed entities), ManualLearning (direct human edits to the graph), and OnlineLearning (automated extraction pipelines). It operates alongside sibling components like SemanticAnalysis and ConstraintSystem, which produce or consume the knowledge it stores.

## Architecture and Design

![KnowledgeManagement — Architecture](images/knowledge-management-architecture.png)

The central architectural decision in KnowledgeManagement is the **single-key LevelDB strategy** in `GraphDatabaseService.js`. The entire Graphology graph — all nodes, edges, and metadata — is serialized as one JSON blob stored under the key `'graph'`. This is a deliberate trade-off: it sacrifices the ability to do partial reads or atomic per-entity updates in exchange for simplicity of implementation and optimized performance for read-heavy, batch-write workloads. Both ManualLearning (direct human edits) and OnlineLearning (automated pipeline writes) funnel through this same service, setting an `isDirty` flag on mutation but not triggering persistence immediately. Durability is deferred to an explicit flush cycle via `_persistGraphToLevel()`.

The component operates with **two concurrent graph databases** at different levels of granularity. The main LevelDB-backed Graphology graph holds project-level knowledge: `System`, `Project`, and `Pattern` entities. A separate Memgraph instance (accessed via `CodeGraphAgent` at `integrations/mcp-server-semantic-analysis/src/agents/code-graph-agent.ts`) holds AST-based code-structure entities <USER_ID_REDACTED> via Cypher/Bolt. These two stores have different schemas, different query interfaces, and different entity granularities — they are complementary, not redundant. This mirrors the dual-probe pattern seen in the DockerizedServices sibling component, where different protocols require different access strategies.

The **km-core canonical shape**, introduced by the migration at `integrations/mcp-server-semantic-analysis/scripts/migrate-leveldb-to-kmcore.mjs`, adds structural properties that the legacy 'B-shape' lacked: UUIDv7 identifiers (time-ordered, used uniformly by KmCoreStore), a `layer='evidence'` classification, `legacyId.system='B'` back-references, `metadata.provenance.createdBy` stamping, and `validFrom`/`validUntil` decay fields for temporal expiry without deletion. This shape evolution reflects a maturation from a simple graph store toward an auditable, temporally-aware knowledge base.

## Implementation Details

**Persistence and the Dirty Flag.** `GraphDatabaseService.js` maintains an `isDirty` boolean. Any mutation to the Graphology graph sets this flag; the graph is only written to LevelDB when `_persistGraphToLevel()` is explicitly invoked. This means there is no write-ahead log, no per-operation durability guarantee, and no automatic flush on process exit unless explicitly wired. The single-key design means every flush rewrites the entire graph blob — the cost of a flush scales with total graph size, not with the size of the change set.

**Entity Type Taxonomy.** The migration script at `scripts/migrate-graph-db-entity-types.js` enforces exactly three canonical entity types by iterating all Graphology nodes and calling `graph.setNodeAttribute(nodeId, 'entityType', newType)` for any node outside the canonical set. Legacy types (`TransferablePattern`, `WorkflowPattern`, `TechnicalIssue`) are collapsed into `System`, `Project`, or `Pattern`. This is a destructive normalization — any code still handling legacy types will silently become unreachable after migration.

**Migration Fault Tolerance.** The migration pipeline tests at `src/migration/migrate-leveldb-to-kmcore.test.ts` enforce a **5% error budget**: migrations with ≤5% malformed entities exit with code 0; above that threshold, the process exits non-zero. This allows bulk migrations to proceed through a small population of corrupt legacy records without halting. The implication is that a successful exit code is not a guarantee of completeness — up to 5% of entities may have been silently skipped.

**UKB Trace Audit System.** `src/utils/ukb-trace-report.ts` generates per-batch trace records with three structured sub-reports: `ObservationDerivation` (data provenance — git commits, vibe sessions, code files), `OntologyClassification` (how entities were typed), and `DataLossTracking` (what was dropped or truncated). This three-part structure cleanly separates the questions of origin, classification, and loss. The trace system operates upstream of the Graphology graph — entities can be lost at ingestion time before ever reaching LevelDB, making `DataLossTracking` the first diagnostic surface for missing entities.

**CodeGraphAgent Pre-flight.** Before any Memgraph indexing operation, `CodeGraphAgent` performs a TCP connectivity check against port 7687 using a timeout from `loadAgentTuningConfig().code_graph.memgraph_check_timeout_ms`. This fail-fast design, consistent with the TCP probe pattern in DockerizedServices (`lib/utils/service-probe.js`), prevents silent mid-operation timeouts when Memgraph is unavailable.

## Integration Points

KnowledgeManagement is the knowledge persistence backbone for the broader Coding project. The SemanticAnalysis sibling component feeds entities into this store via OnlineLearning pipelines, which write nodes and edges into the Graphology graph managed by `GraphDatabaseService.js`. The ConstraintSystem sibling may query graph state to inform constraint validation, though the exact interface is not detailed in the observations. LLMAbstraction provides the model calls that underpin the semantic extraction feeding OnlineLearning.

The KmCoreStore sub-component enforces that all entities carry UUIDv7 identifiers — time-ordered, globally unique, requiring no central ID authority. The `legacyId.system='B'` back-reference in migrated entities preserves a bidirectional link between the km-core identity and the legacy B-shape identity, enabling cross-reference resolution tools to trace lineage across migration boundaries.

The Memgraph-backed CodeGraphAgent introduces a dependency on a running Memgraph service (default port 7687), making that service a hard runtime dependency for code-structure knowledge operations. The main LevelDB graph, by contrast, has no external service dependency — it is entirely in-process. This asymmetry means the two graph stores have different availability profiles and different operational requirements.

## Usage Guidelines

**Always trigger the flush cycle after mutations.** Any code path modifying graph nodes or edges through `GraphDatabaseService.js` must ensure `_persistGraphToLevel()` is called before the process can exit or before the next operation that assumes durable state. This applies equally to ManualLearning edits and OnlineLearning pipeline writes. Silent in-memory-only state is the most common source of data loss in this system.

**Respect the canonical entity type taxonomy.** Only `System`, `Project`, and `Pattern` are valid entity types in a post-migration graph. Introducing new types ad-hoc will result in those types being silently overwritten on the next migration run. Any extension to the taxonomy requires explicitly updating the migration script at `scripts/migrate-graph-db-entity-types.js` and defining the mapping for the new type.

**Do not treat a zero-exit migration as a completeness guarantee.** The 5% error budget means up to 5% of entities may have been skipped without surfacing an error. Downstream systems performing graph analytics, cross-reference resolution, or completeness checks should build in tolerance for this potential gap and cross-reference `DataLossTracking` records in the UKB trace reports.

**Use `DataLossTracking` first when debugging missing entities.** Because entities can be lost at ingestion time — before reaching the Graphology graph or LevelDB — the UKB trace at `src/utils/ukb-trace-report.ts` is the correct first diagnostic step. Looking only at the graph or the LevelDB blob will miss losses that occurred upstream during the observation derivation or classification phases.

**Account for dual-graph architecture when querying knowledge.** Project-level knowledge (Systems, Projects, Patterns) lives in the LevelDB-backed Graphology graph. Code-structure knowledge (AST entities, code relationships) lives in Memgraph and is accessed via Cypher/Bolt through `CodeGraphAgent`. A query targeting only one store will produce an incomplete picture. The Memgraph store also has an operational dependency — verify connectivity via the pre-flight TCP check behavior before assuming Memgraph-backed knowledge is available.

### Siblings
- [LiveLoggingSystem](./LiveLoggingSystem.md) -- [LLM] The `TranscriptAdapter` abstract base class in `lib/agent-api/transcript-api.js` defines a pluggable adapter contract that decouples the LSL pipeline from specific agent implementations. The five abstract methods — `getAgentType()`, `getTranscriptDirectory()`, `readTranscripts()`, `convertToLSL()`, and `getCurrentSession()` — establish a clear interface that concrete adapters (e.g., for Claude Code and Copilot CLI) must implement. This pattern means the core LSL infrastructure never directly references agent-specific transcript formats or filesystem layouts. A new developer adding support for a third agent (say, Cursor or Aider) would subclass `TranscriptAdapter` and implement only these five methods without touching the converter, file manager, or validation layers. The `watchTranscripts()` method is notably NOT abstract — it is a concrete polling implementation shared by all adapters, using `setInterval` with a default 1000ms interval and a `lastEntryCount` integer cursor that advances only when new entries are detected, then fires registered callbacks with only the delta entries. This means the watch mechanism is O(n) in total entry count per poll cycle, which could become a performance concern for very long sessions.
- [LLMAbstraction](./LLMAbstraction.md) -- LLMAbstraction is a multi-layered abstraction over LLM providers that enables provider-agnostic model calls across public APIs (Anthropic, OpenAI, Groq), a local Docker Model Runner (DMR), and a mock mode for testing. The system routes requests through a rapid-llm-proxy daemon (port 12435) that handles provider selection, tier-based routing, and per-call telemetry attribution. Mode selection is dynamic and per-agent, stored in workflow-progress.json, supporting global and per-agent overrides across three modes: 'mock', 'local' (DMR), and 'public' (cloud APIs).
- [DockerizedServices](./DockerizedServices.md) -- [LLM] The DockerizedServices component uses a dual-probe health checking architecture implemented in lib/utils/service-probe.js that strictly separates HTTP-based and TCP-based service verification. probeHttpHealth() issues HTTP/HTTPS requests and maps 2xx/3xx response codes to the 'running' state, while probeTcpPort() opens a raw net.Socket connection to verify port reachability for non-HTTP protocols like Memgraph's Bolt protocol. A critical architectural invariant (documented as SPEC R6) enforces that neither probe ever returns the string 'healthy'—only 'running', 'stopped', or 'unknown' are valid return values. This distinction matters because health-coordinator.js in scripts/health-coordinator.js consumes these probes on a 5-second polling interval and must be able to uniformly handle all service types (Next.js dashboard, Node.js API, Memgraph, Redis) without special-casing the protocol. New developers adding services must use probeTcpPort() for any non-HTTP service and must not introduce a fourth status string, or the health-coordinator's state machine will behave incorrectly.
- [CodingPatterns](./CodingPatterns.md) -- [LLM] The MCP (Model Context Protocol) server pattern is applied uniformly across all three major integrations: mcp-constraint-monitor, mcp-server-semantic-analysis, and code-graph-rag. Each integration follows a consistent directory layout with a README.md describing the tool interface, a docs/ subdirectory containing architectural detail, and configuration files that declare the tool's capabilities to AI agent hosts. This pattern means new integrations can be scaffolded by copying this structure and registering new tool definitions. The mcp-server-semantic-analysis integration further documents this in docs/api/README.md and docs/architecture/README.md, separating API surface from internal architecture. Developers adding a new MCP tool should expect to create at minimum: a README with tool description, an API reference doc, and a configuration file declaring tool schemas. The consistency of this pattern across integrations suggests it is an enforced project convention rather than an organic coincidence.
- [ConstraintSystem](./ConstraintSystem.md) -- The ConstraintSystem is a monitoring and enforcement layer that validates code actions and file operations against configured rules during Claude Code sessions. It operates through a hook-based architecture where constraint checks are triggered at key lifecycle points (pre-tool, post-tool, etc.) and violations are captured, persisted, and surfaced to dashboards. The system integrates with Claude Code's native hook mechanism via configuration files at user-level (~/.coding-tools/hooks.json) and project-level (.coding/hooks.json), with project config overriding user config.
- [SemanticAnalysis](./SemanticAnalysis.md) -- [LLM] The SemanticAnalysis pipeline is structured around a coordinator pattern where specialized agents each extend BaseAgent<TInput,TOutput> defined in src/agents/base-agent.ts. This base class implements a strict template-method execute() that sequences six steps in order: process(), calculateConfidence(), detectIssues(), generateRouting(), applyCorrections(), and buildMetadata(). Every agent—CodeGraphAgent, SemanticAnalysisAgent, OntologyClassificationAgent, ContentValidationAgent—inherits this contract and returns a uniform AgentResponse envelope. This design means a new developer adding an agent only needs to implement the domain-specific process() logic; confidence scoring, issue detection, and metadata construction are guaranteed to run in a consistent order regardless of which agent is invoked. The tradeoff is that the template method imposes overhead steps even when an agent's output is trivially simple, and agents cannot short-circuit the sequence without throwing exceptions.
