# KnowledgeManagement

**Type:** Component

The KnowledgeManagement component provides knowledge graph storage, query, and lifecycle management for the Coding project. It centers on a Graphology+LevelDB graph database (GraphDatabaseService) that stores entities as typed nodes with rich metadata, exposed through both a direct-access path and a VKB HTTP API. The component supports multiple entity types (System, Project, Component, SubComponent, Pattern, Detail) with ontology classification, bi-temporal staleness tracking, embedding vectors, and hierarchical parent/child relationships. It integrates with the MCP semantic analysis server via PersistenceAgent and GraphDatabaseAdapter, which route writes through the VKB API when the server is running or fall back to direct LevelDB access to avoid lock conflicts.

## What It Is

The KnowledgeManagement component is implemented primarily in `src/knowledge-management/` (notably `GraphKnowledgeExporter.js`), with integration adapters living in `integrations/mcp-server-semantic-analysis/src/storage/graph-database-adapter.ts` and `integrations/mcp-server-semantic-analysis/src/agents/persistence-agent.ts`, plus a JavaScript HTTP client at `lib/ukb-unified/core/VkbApiClient.js`. Migration tooling lives in `scripts/migrate-graph-db-entity-types.js` and the test harness at `src/migration/migrate-leveldb-to-kmcore.test.ts`. As a child of the `Coding` root component, it serves as the shared knowledge backbone for the seven major project components — sitting alongside siblings such as `SemanticAnalysis`, `LiveLoggingSystem`, `LLMAbstraction`, `DockerizedServices`, `CodingPatterns`, and `ConstraintSystem`.

At its core, the component provides knowledge-graph storage, query, and lifecycle management built on a Graphology in-memory graph backed by LevelDB for persistence. The central service, `GraphDatabaseService`, stores entities as typed nodes (System, Project, Component, SubComponent, Pattern, Detail) with rich metadata including ontology classification, bi-temporal staleness tracking, embedding vectors, and hierarchical parent/child relationships. Access is exposed through two paths: a direct in-process API and an HTTP-based VKB API, with adapters automatically routing between them depending on server availability.

## Architecture and Design

The most striking architectural decision is the **dual-mode routing pattern** implemented in `GraphDatabaseAdapter`. The adapter dynamically imports `VkbApiClient` and probes server liveness via `isServerAvailable()`. When the MCP semantic analysis server is running, all writes are routed through its HTTP API; when it is not, the adapter falls back to direct `GraphDatabaseService` instantiation against LevelDB. This is not a performance optimization — it is a correctness pattern designed to prevent LevelDB lock conflicts that would arise if two processes attempted to open the same database simultaneously. LevelDB's exclusive-lock semantics make this routing essential rather than optional.

A second key pattern is **dynamic import for runtime-only coupling**. Both `VkbApiClient` (loaded from `lib/ukb-unified/core/VkbApiClient.js`) and `GraphKnowledgeExporter` (loaded from `src/knowledge-management/GraphKnowledgeExporter.js`) are JavaScript modules brought into the TypeScript `GraphDatabaseAdapter` via dynamic `import()` calls at `initialize()` time. The `VkbApiClient` path is constructed by stripping `/.data/knowledge-graph` from `dbPath` and appending the relative module path, deliberately avoiding static TypeScript compilation dependencies on JavaScript-only client code. This crosses the TS/JS boundary cleanly while keeping the build graph minimal.

The system enforces a **closed ontology contract** through `PersistenceAgent`: `storeEntity()` in `GraphDatabaseAdapter` throws an error rather than silently defaulting when `entityType` is missing or set to `'Unclassified'`. This hard no-fallback rule pushes the responsibility for classification upstream — by the time anything reaches persistence, ontology classification must have already happened. Combined with the migration script's enforcement of a closed three-member canonical set (System, Project, Pattern) at the graph layer, this prevents drift and unclassified-entity pollution.

The event-driven export sync via `entity:stored` events with a 2-second debounce represents an **eventual-consistency design** between the authoritative LevelDB store and the JSON export mirror under `.data/knowledge-export/`. The debounce window is short enough to feel real-time but long enough to coalesce bursts of writes, trading immediate consistency for I/O efficiency.

## Implementation Details

`GraphDatabaseService` is the in-process service that owns the Graphology graph instance and the LevelDB persistence operations. Its `_persistGraphToLevel()` method serializes the graph under a single `'graph'` key as a structured blob containing nodes, edges, and metadata — a format that test fixtures in `src/migration/migrate-leveldb-to-kmcore.test.ts` deliberately replicate to validate migration paths. The service exposes an `isDirty` flag that must be set before `_persistGraphToLevel()` is called manually, a pattern that `scripts/migrate-graph-db-entity-types.js` exploits during type consolidation operations.

`GraphDatabaseAdapter` (in `integrations/mcp-server-semantic-analysis/src/storage/graph-database-adapter.ts`) is the bridge that the MCP semantic analysis server uses to interact with the graph. Its `initialize()` method performs the runtime decision: it attempts to dynamically import `VkbApiClient`, calls `isServerAvailable()`, and either retains the API client for HTTP routing or instantiates `GraphDatabaseService` directly. In the direct mode it additionally attaches a `GraphKnowledgeExporter` instance that subscribes to the service's `entity:stored` events, ensuring JSON exports stay synchronized with the underlying graph.

`PersistenceAgent` (in `integrations/mcp-server-semantic-analysis/src/agents/persistence-agent.ts`) defines an `EntityMetadata` shape that goes well beyond simple key-value metadata. It includes **bi-temporal staleness fields** inspired by Graphiti: `invalidating_commits[]` (an array of git commit hashes that have invalidated the entity), `staleness_score` (a 0–100 numeric measure), `staleness_check_at` (the timestamp of the last staleness evaluation), and `staleness_method` (the algorithm used). This enables git-commit-based knowledge decay tracking, where stored insights can be automatically aged out as the underlying code evolves. The metadata also supports `mixed_topics` detection via Jaccard similarity, identifying entities that have drifted into covering multiple distinct concerns.

The migration story is enforced by two complementary mechanisms. `scripts/migrate-graph-db-entity-types.js` consolidates legacy types — `TransferablePattern`, `WorkflowPattern`, `TechnicalIssue`, `AnalysisInsight`, and others — by mapping all of them onto `Pattern`, leaving only the three canonical members (System, Project, Pattern) at the Graphology layer. The script sets `isDirty=true` and explicitly calls `dbService._persistGraphToLevel()` to commit the in-memory mutation. Separately, the LevelDB→km-core migration test (`src/migration/migrate-leveldb-to-kmcore.test.ts`) seeds fixture databases using the same `'graph'` key blob format and asserts a canonical output shape requiring UUIDv7 ids and `layer='evidence'`, with a 10% error-budget exit-code threshold as the acceptance criterion.

## Integration Points

The component's primary integration is with the `SemanticAnalysis` sibling component through the MCP semantic analysis server. `PersistenceAgent` and `GraphDatabaseAdapter` are physically located inside that integration (`integrations/mcp-server-semantic-analysis/`), making them the principal seam between semantic analysis pipelines and the knowledge graph. The `OnlineLearning` child component formalizes this flow: the batch analysis pipeline routes extracted entities through `PersistenceAgent`, which calls `mapEntityToSharedMemory()` to pre-populate ontology metadata fields before persisting via the VKB HTTP API or the direct LevelDB fallback.

The `VkbApiClient` in `lib/ukb-unified/core/VkbApiClient.js` is the contract for HTTP-based interaction with the running server. Because it is referenced only via dynamic import, consumers can be TypeScript-compiled without requiring a typed declaration of the client — a deliberate trade-off favoring decoupling over compile-time safety at this boundary. The path-derivation logic (stripping `/.data/knowledge-graph` from `dbPath`) tightly couples the adapter to the conventional on-disk layout of the project.

The `ManualLearning` child handles the typed-node authoring side, ensuring that entities of all six types (System, Project, Component, SubComponent, Pattern, Detail) carry the rich metadata expected by downstream consumers. The `EntityMigrationScripts` child encapsulates the type-consolidation tooling for evolving the schema without full data reconstruction. The hierarchical parent/child relationships stored in graph metadata mirror the very component hierarchy in which KnowledgeManagement itself sits — the graph is self-describing about the codebase it documents.

Filesystem integration includes `.data/knowledge-export/` (the JSON mirror maintained by `GraphKnowledgeExporter`) and the LevelDB store at `.data/knowledge-graph/`. The export directory provides a human-readable, version-control-friendly view of the otherwise opaque LevelDB binary state.

## Usage Guidelines

When adding code that needs to persist entities to the knowledge graph, always go through `PersistenceAgent` or `GraphDatabaseAdapter` rather than instantiating `GraphDatabaseService` directly. Doing so ensures the dual-mode routing properly avoids LevelDB lock conflicts when the MCP server is running. Direct instantiation should be reserved for migration scripts and test fixtures that have explicit control over the server lifecycle.

Always classify entities into one of the canonical types (System, Project, Pattern at the graph layer, or the broader six-type set including Component, SubComponent, and Detail) before invoking `storeEntity()`. The hard no-fallback rule will throw rather than accept `'Unclassified'` or missing `entityType` values. If you are introducing a new conceptual entity category, prefer mapping it to `Pattern` and distinguishing via metadata rather than expanding the canonical type set — the migration script's consolidation philosophy makes this the path of least resistance.

When authoring entity metadata, populate the staleness fields (`invalidating_commits`, `staleness_score`, `staleness_check_at`, `staleness_method`) where applicable so that the bi-temporal decay tracking can function. Even an initial `staleness_score: 0` with a current `staleness_check_at` is more useful than absent fields, because it establishes the baseline for future decay calculations.

For migration work, follow the pattern established by `scripts/migrate-graph-db-entity-types.js`: mutate the in-memory graph, set `isDirty=true`, then explicitly call `dbService._persistGraphToLevel()`. When writing migration tests, replicate the `'graph'` key blob format with nodes/edges/metadata as `src/migration/migrate-leveldb-to-kmcore.test.ts` does, and respect the 10% error-budget threshold for exit-code-based acceptance.

Finally, never assume the JSON files under `.data/knowledge-export/` are authoritative — they are a debounced eventual-consistency mirror. The LevelDB store is the source of truth, and direct edits to the JSON exports will be overwritten on the next `entity:stored` event flush.

### Summary of Architectural Insights

1. **Architectural patterns identified**: dual-mode adapter routing (HTTP vs direct), dynamic-import boundary crossing (TS↔JS), event-driven eventual-consistency mirroring (debounced export sync), closed-ontology enforcement (hard no-fallback on `entityType`), and bi-temporal metadata modeling (Graphiti-inspired staleness).

2. **Design decisions and trade-offs**: choosing LevelDB lock-conflict avoidance over operational simplicity (forcing dual-mode complexity); choosing dynamic import over typed coupling (sacrificing compile-time safety for build-graph minimalism); choosing hard ontology enforcement over permissive defaults (forcing upstream discipline); choosing type consolidation over type proliferation (favoring a small canonical set + metadata).

3. **System structure insights**: the component cleanly separates authoring (`ManualLearning`), automated population (`OnlineLearning`), and schema evolution (`EntityMigrationScripts`) into distinct children, while integrating into `SemanticAnalysis` through the MCP server boundary.

4. **Scalability considerations**: the in-memory Graphology graph is the primary scalability ceiling — all operations work against the full graph in RAM, with LevelDB serving as a single-blob persistence layer rather than a queryable backend. The 2-second debounce on export sync and Jaccard-based `mixed_topics` detection introduce CPU costs that scale with write volume.

5. **Maintainability assessment**: the codebase is well-disciplined about classification and migration (clear canonical types, dedicated migration tooling, fixture-based migration tests with explicit error budgets), but the cross-boundary dynamic imports and path-derivation logic create implicit contracts that future developers must respect carefully. The hard no-fallback policy is a strong maintainability win because it surfaces classification gaps loudly rather than letting them accumulate silently.
