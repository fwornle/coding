# KnowledgeManagement — Technical Reference

## What It Is

KnowledgeManagement is the core knowledge graph infrastructure component of the Coding project, implemented primarily under `src/knowledge-management/` with migration and integration tooling in `integrations/mcp-server-semantic-analysis/scripts/`. It provides entity storage, querying, and lifecycle management for structured knowledge about systems, projects, and patterns — the persistent memory layer that siblings like SemanticAnalysis write into and components like ConstraintSystem and LiveLoggingSystem implicitly depend on for contextual awareness.

The component owns two distinct storage backends: a local LevelDB-backed Graphology in-memory graph (`GraphDatabaseService`) and a canonical shape store (`GraphKMStore`), plus the tooling to migrate between them. Its child components — ManualLearning, OnlineLearning, GraphKMStore, and KnowledgeDecayTracker — each address a distinct lifecycle concern: knowledge origin, knowledge ingestion, canonical persistence, and knowledge expiry respectively.

## Architecture and Design

The fundamental architectural decision is a **dual-store model** with an explicit migration path. `GraphDatabaseService` (`src/knowledge-management/GraphDatabaseService.js`) serves as the operational store: it holds a Graphology in-memory graph and flushes the entire graph as a single JSON blob (`{ nodes, edges, metadata }`) under the key `'graph'` in LevelDB via `_persistGraphToLevel()`. This single-key persistence is a deliberate simplicity trade-off — it avoids per-entity LevelDB operations at the cost of making partial reads impossible and requiring a full graph load on startup.

The canonical store, GraphKMStore, represents an architectural evolution. It abandons the monolithic blob in favor of per-domain JSON exports written to `<targetDataDir>/exports/<domain>.json`, each containing `{ nodes, edges }` for that domain. This enables domain-scoped inspection without reopening the database — a significant operational improvement for the SemanticAnalysis pipeline that needs to audit specific knowledge domains. GraphKMStore also adopts UUIDv7 entity IDs (time-ordered) rather than the `'{team}:{entityName}'` namespaced keys used by the LevelDB store, enabling chronological ordering and distributed ID generation without coordination.

The type system reflects a deliberate consolidation: `scripts/migrate-graph-db-entity-types.js` maps five legacy types (`TransferablePattern`, `WorkflowPattern`, `TechnicalIssue`, `SystemImplementation`, `AnalysisInsight`) onto three canonical types — `System`, `Project`, and `Pattern` — with unknown types defaulting to `Pattern`. This reduction trades expressiveness for query simplicity and prevents type proliferation as the SemanticAnalysis pipeline produces new entities.

## Implementation Details

`GraphDatabaseService` exposes persistence through an `isDirty` flag mechanism that supports both manual and auto-persist modes — callers can batch mutations and trigger `_persistGraphToLevel()` explicitly, or rely on auto-persist to flush on change. The entire graph state is serialized as one value, meaning persistence cost is proportional to total graph size regardless of how many nodes changed.

The LevelDB-to-km-core migration (`integrations/mcp-server-semantic-analysis/scripts/migrate-leveldb-to-kmcore.mjs`) implements an **error budget system**: if ≤5% of entities are malformed the migration exits 0 (success), but >5% malformed triggers a non-zero exit, blocking the pipeline. This is validated by `migrate-leveldb-to-kmcore.test.ts`, which uses a `makeLevelBlob` fixture helper that constructs test graphs with the `'{team}:{entityName}'` node key format (e.g., `'coding:MyEntity'`). The error budget prevents silent data loss from becoming an undetected migration failure while tolerating a small number of structurally odd entities that may have accumulated over time.

`UKBTraceReport` (`integrations/mcp-server-semantic-analysis/src/utils/ukb-trace-report.ts`) provides pipeline-stage auditability by tracking `inputCount`, `outputCount`, and `lossReasons` at each step — extraction, classification, and persistence. This directly addresses the risk of silent entity loss in a multi-stage pipeline where each stage filters or transforms entities. Without it, a classification step that silently drops 30% of entities would be invisible.

`CodeGraphAgent` (`integrations/mcp-server-semantic-analysis/src/agents/code-graph-agent.ts`) integrates with a Memgraph graph database via TCP, using `loadAgentTuningConfig().code_graph.memgraph_check_timeout_ms` for connection health checks before indexing — making the Memgraph timeout tunable without code changes, consistent with the configuration-driven patterns seen across the Coding project's LLMAbstraction and agent configuration.

## Integration Points

KnowledgeManagement is consumed most directly by SemanticAnalysis, which runs the extraction pipeline that populates it. The `CodeGraphAgent` within SemanticAnalysis writes to a Memgraph instance that sits alongside the LevelDB/GraphKMStore stores — suggesting KnowledgeManagement spans multiple physical backends depending on the query workload. The migration scripts represent the integration seam between the legacy LevelDB store and the canonical km-core shape.

The OnlineLearning child component draws from three automatic sources identified in `docs/architecture/memory-systems.md`: git history, LSL sessions (produced by LiveLoggingSystem's TranscriptAdapter pipeline), and code analysis (produced by SemanticAnalysis). This makes KnowledgeManagement a convergence point for outputs from at least two major sibling components. ManualLearning's provenance-protected entities ensure that human knowledge entered directly into the graph isn't silently overwritten when these automated pipelines run.

Within the parent Coding component, KnowledgeManagement functions as the shared memory substrate — the store that other components query to make context-aware decisions. ConstraintSystem's violation tracking and CodingPatterns' architectural conventions both represent knowledge that could flow into or out of this graph.

## Usage Guidelines

**Store selection**: Use `GraphDatabaseService`/LevelDB for operational, frequently-mutated graph state. Use GraphKMStore for canonical, domain-queryable exports. Do not treat them as interchangeable — they have different ID schemes (`'{team}:{entityName}'` vs UUIDv7) and different persistence granularities.

**Migration safety**: The error budget (≤5% malformed = acceptable) in `migrate-leveldb-to-kmcore.mjs` is a hard operational boundary. If a migration exits non-zero, investigate `UKBTraceReport` output to identify which pipeline stage is producing malformed entities before retrying.

**Type discipline**: New entity types should map to `System`, `Project`, or `Pattern`. The migration script's default-to-`Pattern` behavior means unrecognized types silently become Patterns — introducing a novel type without updating the canonical type map will result in quiet downcast, not an error.

**Temporal <USER_ID_REDACTED>**: Use KnowledgeDecayTracker's `validFrom`/`validUntil` fields for any query that should reflect current knowledge state. Querying the raw Graphology graph without these filters will return expired entities alongside active ones.

**Provenance on manual entries**: Any entity inserted through ManualLearning must carry provenance metadata. Entities without it are indistinguishable from pipeline-generated observations and may be overwritten during the next OnlineLearning ingestion cycle.

- [LiveLoggingSystem](./LiveLoggingSystem.md) -- [LLM] The TranscriptAdapter abstract class in `lib/agent-api/transcript-api.js` enforces a strict interface contract that all agent-specific adapters must satisfy. Subclasses must implement five methods: `getAgentType()` (returns a string identifier like 'claude' or 'copilot'), `getTranscriptDirectory()` (returns the filesystem path where native transcripts are stored), `readTranscripts()` (reads raw agent-native files), `convertToLSL()` (transforms them into the unified LSLSession/LSLEntry format), and `getCurrentSession()` (returns the active session for live ingestion). This adapter pattern means that adding a new agent source — say, a Cursor or Gemini CLI — requires only implementing this interface without touching any downstream pipeline code. The LSLMetadata type includes a `timeWindow` field (formatted as e.g. '0800-0900') that the adapter is responsible for populating, meaning the adapter layer also owns the hourly-bucketing logic that drives file routing downstream.
- [LLMAbstraction](./LLMAbstraction.md) -- LLMAbstraction is a multi-layered abstraction over LLM providers that enables provider-agnostic model calls through three distinct execution paths: mock mode (for testing), local inference via Docker Model Runner (DMR), and public cloud providers (Anthropic, OpenAI, Groq) routed through a rapid-llm-proxy. The system supports per-agent and global mode switching stored in `.data/workflow-progress.json`, allowing runtime toggling between modes without code changes. Provider selection follows a priority chain from per-agent overrides to global mode to legacy flags.

The architecture centers on a proxy-mediated request pattern where most LLM calls route through a local rapid-llm-proxy daemon (default port 12435) via `/api/complete`, enabling centralized token tracking, tier-based routing, and telemetry attribution. The `llm-with-process.ts` module exists specifically to inject a `process` tag into proxy requests — a gap in the SDK's `LLMService.complete()` that caused all wave-analysis calls to appear as `process='unknown'` in token-usage telemetry. DMR provider uses an OpenAI-compatible API at `localhost:${DMR_PORT}/engines/v1` for fully local inference.

Key patterns include: environment-variable-driven URL resolution with multiple fallback levels, singleton client instances with health-check caching, YAML-based provider configuration with env-var expansion, and SDK-shape response normalization ensuring downstream consumers work unchanged regardless of which provider path was taken.
- [DockerizedServices](./DockerizedServices.md) -- [LLM] The multi-service container architecture uses supervisord (docker/supervisord.conf) to manage multiple long-running processes within a single Docker container defined in docker/Dockerfile.coding-services. This is an intentional design choice that trades container isolation purity for operational simplicity — rather than running separate containers for the semantic analysis MCP server, constraint monitor API, and dashboard service, all are co-located under supervisord's process supervision. The entrypoint script handles startup orchestration sequencing before handing off to supervisord. A new developer should be aware that this means a crash in one supervised service does not terminate the container, supervisord will attempt restarts, but it also means a single container failure takes down all co-located services simultaneously. The bind mount pattern `CODING_ROOT=/coding` connects the host repository filesystem into the container, making the container dependent on the host directory layout at runtime rather than baking code into the image.
