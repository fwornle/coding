## Usage Guidelines  

1. **Choose the appropriate refresh mode** – For entities flagged with high‑severity validation failures, invoke the synchronous refresh method to guarantee immediate consistency.  For bulk or low‑priority updates, prefer the asynchronous API to keep the request thread responsive and to enable better throughput.

2. **Maintain cache coherence** – After a successful refresh, ensure the Redis entry for the entity is either updated or invalidated.  Do not rely on stale cache data; always follow the pattern of `update → cache.set` (or `cache.del`) within the same transactional context if possible.

3. **Handle errors centrally** – Wrap calls to EntityRefresher in try/catch blocks that surface `LoggingService` errors.  Do not swallow exceptions; instead, propagate them to the caller so that the parent ConstraintSystem can decide whether to retry, fall back, or surface a user‑facing error.

4. **Leverage the feedback API** – When users report issues with a refreshed entity, call `EntityRefresher.reportIssue(entityId, details)`.  This data should be persisted and correlated with validation reports to improve future validation rules.

5. **Do not bypass the GraphDatabaseAdapter** – All persistence must go through the adapter’s `update` method.  Directly accessing the underlying graph client would break the abstraction and hinder future database swaps.

### Design decisions and trade‑offs  

- **Abstraction via GraphDatabaseAdapter** trades a slight performance overhead for the ability to swap graph engines without touching refresh logic.  
- **Dual refresh modes** increase code complexity but give callers control over latency vs. throughput.  
- **Redis caching** reduces DB load and speeds reads, at the cost of cache invalidation complexity and potential consistency windows.  
- **Centralised logging** simplifies debugging but introduces a single point of failure if the logging infrastructure is unavailable; the component should degrade gracefully.  
- **Feedback mechanism** adds extra I/O (persisting reports) but improves user experience and future validation quality.
