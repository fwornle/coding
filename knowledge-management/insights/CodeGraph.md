# CodeGraph

**Type:** SubComponent

The CodeGraphManager provides a framework for managing the code graph, including updating and retrieving graph nodes and edges

## What It Is  

**CodeGraph** is the sub‑component responsible for turning raw source code into a navigable graph representation that downstream analysis modules can query. The core implementation lives in the **CodeGraphAgent** (`integrations/mcp-server-semantic-analysis/src/agents/code-graph-agent.ts`) and the **CodeGraphManager** (`integrations/mcp-server-semantic-analysis/src/agents/code-graph-agent.ts` – same file houses both classes, or they are co‑located in the *agents* package). The manager sits on top of a graph database, exposing an API that other parts of the **SemanticAnalysis** pipeline (e.g., the InsightGenerator, the OntologyClassificationAgent, and the Pipeline coordinator) use to retrieve nodes, edges, and derived metrics.

CodeGraph therefore acts as the “data‑model” layer that supplies the structural foundation for the insights and ontology‑driven reasoning performed elsewhere.

## Architecture and Design  

The observations reveal a **layered architecture** in which the *agent* handles the heavy‑weight parsing and graph construction, while the *manager* abstracts persistence and query concerns. This separation mirrors a classic **Builder / Director** pattern: the **CodeGraphAgent** (builder) incrementally creates graph nodes and edges from the abstract syntax tree generated by its recursive‑descent parser, and the **CodeGraphManager** (director) orchestrates storage in a graph database and serves queries through a clean API.

A **caching mechanism** is embedded directly inside the agent. By checking whether a source fragment has already been parsed, the agent can skip redundant work, effectively acting as a **Proxy** for the parsing routine. This design reduces CPU load and improves latency for repeated analyses of the same code base.

The **graph database** used by the manager (the exact technology is not named in the observations) serves as a **Repository** for graph entities. All CRUD (create‑read‑update‑delete) operations on nodes and edges flow through the manager, which shields the rest of the system from low‑level query language details. The manager also exposes an **API** that other modules invoke—this is a **Facade** that presents a simple, domain‑specific interface (e.g., “getAllFunctionsInFile”, “findCallersOf(node)”) while hiding the underlying persistence mechanics.

Interaction flow: the **SemanticAnalysisAgent** (located at `integrations/mcp-server-semantic-analysis/src/agents/semantic-analysis-agent.ts`) triggers the **CodeGraphAgent** with raw source code and ontology definitions. The agent parses, caches, and hands the resulting graph to the **CodeGraphManager**, which persists it and registers it with the query API. Downstream components such as the **InsightGenerator** and **OntologyClassificationAgent** then call the manager’s API to perform semantic enrichment, pattern detection, or visualization.

## Implementation Details  

### CodeGraphAgent  
* **Parsing** – The agent employs a **recursive descent parser** to walk the source text. This choice gives fine‑grained control over language constructs and allows the agent to emit graph nodes as soon as a syntactic element (e.g., class, function, variable) is recognized.  
* **Graph Construction** – As the parser discovers entities, it creates corresponding graph vertices and edges that capture relationships like “calls”, “inherits”, or “references”. The construction logic is encapsulated in methods that likely follow a *nodeFactory* style, ensuring each vertex adheres to a common schema expected by the downstream manager.  
* **Caching** – Before parsing a file, the agent checks an internal cache keyed by a hash of the file contents (or a version identifier). If a cache hit occurs, the previously built sub‑graph is reused, and only changed portions are re‑parsed. This reduces redundant computation, especially in large codebases where incremental analysis is common.  

### CodeGraphManager  
* **Persistence Layer** – The manager talks to a **graph database** (e.g., Neo4j, JanusGraph, or a custom store). All graph mutations—adding new nodes, updating edges, deleting stale elements—are routed through the manager, which translates domain objects into the database’s query language (Cypher, Gremlin, etc.).  
* **Query API** – The manager publishes a set of high‑level methods that other agents invoke. Typical signatures include `getNodeById(id)`, `findNeighbors(nodeId, relationshipType)`, and `searchByPattern(pattern)`. By providing these abstractions, the manager isolates the rest of the system from database‑specific quirks.  
* **Update Mechanics** – When the **CodeGraphAgent** produces a fresh sub‑graph (or an incremental diff), the manager merges it into the existing persisted graph. This may involve upserting nodes and reconciling edge changes, ensuring the stored representation stays consistent with the latest source snapshot.

Both classes are co‑located under the *agents* package, reinforcing the idea that they form a tightly coupled pair: the agent focuses on *how* to build the graph, while the manager focuses on *where* to store and *how* to expose it.

## Usage Guidelines  

* **Invoke through the Manager** – All callers should interact with the **CodeGraphManager** rather than the agent directly. This guarantees that any persisted state stays synchronized and that caching semantics are respected.  
* **Cache Awareness** – When updating source files, ensure that the file’s modification timestamp or content hash is refreshed so the **CodeGraphAgent** can invalidate stale cache entries. Failure to do so may lead to stale graph fragments persisting in the database.  
* **Batch Processing** – For large repositories, leverage the **Pipeline**’s DAG model to schedule the code‑graph step early and in parallel. The **Concurrency** wave controller can be tuned (e.g., `maxWorkers`) to match the hardware profile, but keep the cache shared across workers to maximize deduplication.  
* **Schema Consistency** – When extending the graph (e.g., adding custom node types for a new language feature), follow the existing node/edge schema defined by the manager. Adding new relationship types should be reflected in the manager’s API to keep downstream queries functional.  
* **Error Handling** – The agent’s parser may throw syntax‑related exceptions; wrap calls in try/catch blocks and surface meaningful error messages through the manager’s API. This helps the **InsightGenerator** distinguish between “no data” and “parsing failure”.

### Design decisions and trade‑offs  

* **Recursive descent parsing** gives precise control and easy extension at the cost of higher implementation effort compared to using an off‑the‑shelf parser generator.  
* **Local caching** reduces CPU load but introduces cache‑invalidation complexity; developers must ensure source changes are correctly detected.  
* **Graph database persistence** enables rich relationship queries and visualisation but adds operational overhead (deployment, backup, query tuning).  
* **Separate agent/manager layers** improve separation of concerns, making the parsing logic testable in isolation, yet they introduce an extra indirection that slightly increases latency for single‑step operations.  

### System structure insights  

* **CodeGraph** sits in the *agents* package, tightly coupled with the rest of the SemanticAnalysis module, reflecting a **modular pipeline** where each agent owns a distinct responsibility.  
* The **parent–child relationship** (SemanticAnalysis → CodeGraph) enforces a clear data‑flow: raw code → graph → ontology tagging → insight generation.  
* Sibling modules share the same execution DAG, meaning that any change to the graph construction step propagates automatically to downstream consumers.  

### Scalability considerations  

* **Parallel parsing** via the **Concurrency** wave controller allows the system to scale horizontally across CPU cores; the shared cache mitigates duplicated work.  
* The graph database can be scaled vertically (more memory/CPU) or horizontally (sharding) depending on the chosen technology, but the manager’s API must be designed to handle pagination or streaming for very large result sets.  
* Caching effectiveness grows with the size of the codebase; a well‑tuned hash‑based key reduces re‑parsing dramatically, especially in CI environments where only a subset of files change.  

### Maintainability assessment  

* The clear split between **CodeGraphAgent** (parsing & construction) and **CodeGraphManager** (persistence & API) yields high **cohesion** and low **coupling**, simplifying unit testing and future refactors.  
* Reliance on a **recursive descent parser** means that extending language support requires careful updates to the parser logic; however, the localized nature of the parser code makes such changes manageable.  
* The caching layer is a single responsibility and can be swapped out (e.g., for a distributed cache) without touching parsing logic.  
* Documentation should capture the cache key strategy and graph schema; otherwise, new contributors may inadvertently break downstream queries. Overall, the design promotes maintainability provided that schema evolution and cache invalidation are well‑documented.
